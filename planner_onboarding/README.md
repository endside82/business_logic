# 새 기획자 온보딩 문서 꾸러미

이 폴더는 개발 원문을 읽기 전에 서비스 구조를 먼저 이해하기 위한 비개발자용 Markdown 문서다. Notion에 옮겨도 문서 흐름이 깨지지 않도록 표, 짧은 도식, Mermaid 다이어그램을 섞어서 구성했다.

## 읽는 순서

1. `00_start_here.md`  
   서비스의 큰 목적, 사용자 유형, 14개 업무 영역을 먼저 파악한다.

2. `01_service_map.md`  
   전체 서비스가 어떤 영역으로 나뉘고 서로 어떻게 연결되는지 본다.

3. `02_core_journeys.md`  
   실제 사용자가 겪는 대표 흐름을 따라가며 기능 간 연결을 이해한다.

4. `03_unit_playbooks.md`  
   14개 업무 영역별로 목적, 핵심 행동, 연결 영역, 기획 체크포인트를 확인한다.

5. `04_state_policy_guide.md`  
   이벤트, 신청, 참석, 정산, 결제, 계정 삭제처럼 상태가 중요한 정책을 모아 본다.

6. `05_terms_glossary.md`  
   헷갈리기 쉬운 용어를 서비스 언어 기준으로 통일한다.

7. `06_feature_card_template.md`  
   이후 세부 기능을 Notion 카드로 쪼갤 때 쓰는 템플릿이다.

8. `07_full_feature_inventory.md`  
   실제 기능 단위 117개를 빠짐없이 훑는 전체 인벤토리다.

9. `08_scenario_coverage_matrix.md`  
   원문 시나리오 987개를 어떤 범주로 검산해야 하는지 정리한 커버리지 표다.

10. `09_actor_permission_matrix.md`  
    비로그인, 일반 사용자, 참가자, 호스트, 관리자, 크리에이터 등 역할별 허용 액션을 검토한다.

11. `10_planning_qa_checklist.md`  
    새 기능이나 화면을 기획할 때 빠뜨리기 쉬운 상태, 돈, 알림, 개인정보, 외부 시스템 체크리스트다.

12. `11_feature_coverage_index.md`  
    117개 기능 각각의 시나리오 수와 도식 수를 확인하는 누락 방지 인덱스다.

## 문서 작성 기준

- API 주소, 서버 클래스명, 파일 경로는 기본 노출하지 않는다.
- 개발 용어는 사용자 행동이나 운영 정책으로 번역한다.
- 상태값은 실제 정책 판단에 필요한 경우에만 영문 값을 병기한다.
- 한 문서는 하나의 질문에 답하도록 구성한다.
- 도식은 Mermaid를 우선 쓰되, 작은 흐름은 텍스트 도식으로 유지한다.
- "기능 누락 방지" 목적의 문서는 117개 기능 ID를 기준으로 검산한다.

## Notion 이전 시 주의

Notion에서 Mermaid 코드블록을 그대로 보관할 수 있지만, 팀 환경에 따라 렌더링 방식이 다를 수 있다. 최종 공유본에서는 중요한 Mermaid 도식만 이미지로 변환해 함께 첨부하는 방식을 권장한다.
