03. 이벤트

오프라인 모임의 전 사이클을 책임진다. 발견·생성·신청·승인·정원·대기열·체크인·리뷰까지, 승인 정책과 결제 흐름이 가장 복잡하게 얽히는 도메인.

이 도메인의 핵심 난점

이벤트는 “신청 심사”와 “참석 확정/대기”가 동시에 존재한다. 유료 + 승인제 이벤트의 경우, 호스트 승인은 결제 권한 부여이지 참석 확정이 아니다. 승인 → 결제 대기 → 결제 성공 → 참석 확정의 4단계를 모두 분리해서 보여줘야 한다. 이 흐름이 한 버튼에 섞이면 환불 분쟁이 시작된다.

핵심 흐름 — 유료 승인제 이벤트의 신청·결제·참석 확정

이 도메인에서 가장 자주 오해받는 흐름. 승인과 결제, 결제와 참석 확정이 모두 다른 단계라는 점이 핵심이다.

flowchart TD A["참가자 신청"] --> B["Application=PENDING"] B --> C{"호스트 심사"} C -->|"거절"| D["REJECTED
결제 없음"] C -->|"승인"| E["APPROVED_PENDING_PAYMENT
결제 권한 부여"] E --> F{"기한 내 결제?"} F -->|"성공"| G["EVENT_PAYMENT 거래 기록"] G --> H["Attendance=ATTENDING"] F -->|"실패/만료"| I["PAYMENT_EXPIRED
예약 정원 반환"] H --> J["체크인·위치·리뷰 자격 열림"]
서버 보강 필요

현재 서버 enum에는 APPROVED_PENDING_PAYMENT와 PAYMENT_EXPIRED가 없다. 이 상태가 생기기 전까지는 “승인 즉시 ATTENDING으로 만드는” 임시 처리는 돈 분쟁을 만든다. 자세한 정책은 결제·정산 정책에서 본다.

일반 신청·참석 상태 흐름

flowchart TD A["참석 신청 버튼"] --> B{"승인 필요?"} B -->|"예"| C["PENDING"] C -->|"승인"| D["ATTENDING"] C -->|"거절"| E["REJECTED"] C -->|"사용자 취소"| F["CANCELED"] B -->|"아니오"| G{"정원 여유?"} G -->|"예"| D G -->|"대기열 가능"| H["WAITING"] G -->|"대기열도 가득"| I["신청 실패"] H -->|"자리 발생"| D D -->|"사용자 취소"| J["CANCELLED"]
CANCELED와 CANCELLED를 구분한다

신청서 상태는 CANCELED(L 1개), 참석 상태는 CANCELLED(L 2개). 같은 글자로 보이지만 서버 enum이 다르다. 코드 리뷰·QA에서 자주 헷갈리는 부분이라 명시한다.

이 도메인의 부수효과

영향여부이유
알림강함승인·거절·대기 승격·취소·정원 변경마다 알림이 발생. 수신자가 호스트·참가자 양쪽이라 룰이 복잡.
결제·정산강함유료 이벤트의 결제·환불, 호스트 수익 정산, 클럽 이벤트의 기금 차감.
캘린더있음참석 확정 시 자동 등록, 일정 변경 시 자동 갱신.
위치조건부위치 공유 이벤트일 때만 위치 도메인이 활성화.
리뷰·신뢰있음참석 확정 + 종료 후 리뷰 자격이 생긴다. 노쇼는 신뢰점수에 반영.

레드 플래그

실제 분쟁 사례 모음
  • 정원·대기열·승인·결제가 한 버튼에 섞이는 경우 — 사용자가 자신이 어느 단계에 있는지 모르게 된다.
  • 호스트가 자기 이벤트에 신청 시도 — 권한상 차단해야 하고, 메시지가 명확해야 한다.
  • 정원 마지막 자리 동시 신청 — 둘 다 “정원 여유” 화면을 봤지만 한 명만 성공한다. 실패자에게 “방금 마감됐다”라는 정확한 안내가 필요.
  • 대기열 자동 승격 — 사용자 행동 없이 상태가 바뀐다. 알림·CTA 갱신이 필수.
  • 이벤트 취소 후 유료 참가자 환불 — 환불·알림·정산금이 동시에 움직이는 케이스.
  • 유료 승인제 미승인 사용자가 결제 시도 — 돈을 먼저 받으면 거절 후 환불해야 한다. 승인 후 결제만 허용.
