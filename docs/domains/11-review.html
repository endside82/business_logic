신뢰점수 산정식 — 활동·인증·패널티

F11-05의 핵심. 점수는 호출 시점에 즉시 계산해서 응답하고, 외부 도메인이 TrustScoreService.recalculate(userId)를 호출하면 변동치 ≥ 1.0인 경우에만 ScoreSnapshot이 추가된다. 서버 소스 기준 식이며, 절대값은 클라이언트에서 다시 그리지 않는다. 

activity   = 0.4·attended + 0.3·hosted + 0.15·50 + 0.15·50    (0~100)
verification = email30 + phone30 + profile20(image10+bio5+name5) + social10|20
penalty    = min(100, confirmedReports × 20)   ← RESOLVED + ESCALATED만 카운트

total      = clamp([0,100], 0.5·activity + 0.25·verification − 0.25·penalty)
grade      = 90+ DIAMOND  /  75+ PLATINUM  /  60+ GOLD  /  40+ SILVER  /  else BRONZE
컴포넌트가중치입력 신호책임
Activity +0.50 참석 수·주최 수 (클럽/연속 점수는 현재 50점 고정) ActivityScoreCalculator
Verification +0.25 이메일 인증, 전화번호, 프로필(사진·자기소개·실명), 소셜 연동 수 VerificationScoreCalculator
Penalty −0.25 RESOLVED·ESCALATED 상태 신고 1건당 20점, 최대 100점 PenaltyScoreCalculator

패널티는 어떻게 쌓이는가

패널티 점수는 신고 자체가 아니라 운영자가 신고를 어떻게 처리했는지로 정해진다. 신고가 들어와도 PENDING 상태에서는 점수가 깎이지 않고, 운영자가 RESOLVED 또는 ESCALATED로 결론을 내려야 그때 점수에 반영된다.

flowchart LR A["다른 사용자가 신고 접수
POST /api/v1/reports"] --> B["Report: PENDING"] B --> C{"운영자 처리"} C -->|"승인"| D["RESOLVED"] C -->|"심각·이관"| E["ESCALATED"] C -->|"기각"| F["DISMISSED"] C -->|"검토중"| G["IN_REVIEW"] D --> H["PenaltyScoreCalculator
+20점 누적"] E --> H H --> I["TrustScoreService.recalculate(userId)"] I --> J{"diff ≥ 1.0?"} J -->|"예"| K["ScoreSnapshot INSERT"] J -->|"아니오"| L["스킵 (노이즈 방지)"]
읽는 법
  • 신고 접수(POST /api/v1/reports) 자체로는 점수가 깎이지 않는다. 운영자가 RESOLVED 또는 ESCALATED로 처리해야 카운트.
  • 그래서 신뢰점수에는 “접수 즉시 반영”이 없다. 운영 SLA가 곧 패널티 반영 속도가 된다.
  • 변동치가 1.0 미만이면 스냅샷이 쌓이지 않는다. 잦은 미세 변동이 이력 그래프를 어지럽히지 않도록 한 장치.

레드 플래그

  • 미참석자의 리뷰 작성 — 평판 조작을 막아야 한다. 참석 확정자만 리뷰 자격.
  • 중복 리뷰·중복 신고 — 운영 큐와 점수 왜곡 방지.
  • 본인 신고 — 악용·오류 방지.
  • 리뷰 삭제 후 신뢰점수 재계산 누락 — “언제 다시 계산되는가”를 사용자가 모르면 점수가 잘못 보인다고 오해한다. 삭제 즉시 재계산하고 변동 이력에도 한 줄 남긴다.
  • 데이터 부족한 취향 프로필 — 빈 차트보다 “더 모일 때까지” 설명하는 빈 상태 필요.
