07. 모임 정산

모임이 끝난 뒤 비용을 참가자에게 나눠 받는 흐름. 핵심은 “돈을 어떻게 나누는가”가 아니라 “상태가 어떻게 흘러가는가”다. DRAFT 상태에서 알림이 새어 나가면 사용자가 “모르는 청구서”를 받는다.

절대 위반 금지

DRAFT 상태에서는 참가자에게 어떤 알림도 가서는 안 된다. 정산 생성 자체로 “돈 내라”라는 신호가 사용자에게 전달되면 분쟁이 시작된다. 알림은 DRAFT → ACTIVE 전이 시점에 한 번만 보낸다.

흐름 시퀀스

sequenceDiagram actor H as 호스트 actor P as 참가자 participant S as 정산 화면 participant W as 지갑·계좌 participant N as 알림 H->>S: 종료된 모임에서 정산 생성 (DRAFT) H->>S: 항목·참여자 선택 H->>S: 정산 활성화 (DRAFT → ACTIVE) S->>N: 참가자에게 납부 알림 (이때 최초 발송) P->>S: 내 분담금 확인 P->>W: 포인트 또는 계좌이체로 납부 W-->>S: 납부 상태 반영 H->>S: 계좌이체 건 수동 확인 S->>N: 완료 또는 미납 알림

호스트 정의 환불 규칙 (F07-09)

모임마다 호스트가 직접 설정한다. 시스템 기본값(F06-06의 STANDARD)과 별개로 작동한다.

  • 입력 단위: hoursBeforeStart(이벤트 시작 N시간 전) × refundPercentage(0~100, 5% 단위 권장)
  • 참가자 결제 방식: POINT 또는 BANK_TRANSFER
  • BANK_TRANSFER는 호스트가 수동 확인. 가상계좌 자동 입금은 PG webhook(현재 stub 401)으로 처리 예정
  • 화면 표시 예: “이벤트 3일 전 / 환불률: 50% / 환불금액: 5,000원” — 정책별 환불 금액을 항상 즉시 표시
  • 0% 룰일 때는 “환불 불가” 안내 명시
  • UI: RefundTimelineWidget으로 시간 흐름에 따른 환불률 그래프 시각화
F06-06과의 관계

F06-06의 시스템 STANDARD(24h=100%/12h=50%/0%)는 “이벤트 참가비 일반 결제”에 적용되고, F07-09의 호스트 정의 규칙은 “선입금 모임의 정산 환불”에 적용된다. 두 흐름이 같은 모임에서 동시에 일어나지 않도록 PRD가 분리되어 있다.

모임 정산은 PAID_ONLY — 무료 포인트 불가

실제 채무를 프로모션 포인트로 갚게 두지 않는다

모임 정산 송금·회수·선입금은 참가자→호스트로 흐르는 순수 P2P 채무 정산이라 사용처 분류상 PAID_ONLY다. 분담금·선입금 결제는 유료(paid) 잔액만 차감되고, 무료 포인트로는 납부할 수 없다. 정산금이 호스트에게 흘러간 뒤에도 유료/무료 split이 그대로 보존되며, 호스트의 외부 출금·정산금 인출 역시 유료분만 현금화 가능하다(무료분은 인출 불가). 전체 정책은 결제·정산 정책 참조.

레드 플래그

  • DRAFT인데 참가자에게 노출 — 정책상 가장 큰 사고. 노출 가드와 알림 가드를 이중으로 둔다.
  • 계좌이체 후 호스트 미확인 — 참가자는 “냈다”라고 생각하지만 시스템은 미완료. 자동 확인이 불가능하므로 호스트 액션이 필요함을 명시.
  • 포인트 vs 계좌이체 즉시성 차이 미안내 — 포인트는 즉시 반영, 계좌이체는 수동 확인. UI에서 분리.
  • 이의제기 수락/기각 시 금액·상태·알림·감사로그 동시 변경 누락.
  • 활성화 전 항목이 비어 있는데 활성화 시도 — 0원 청구는 차단해야 함.
