18. 분쟁 해결

기존 16개 도메인 곳곳에 흩어진 신고·이의·운영 이슈를 단일 통합 케이스 뷰로 묶는 경량 어댑터 도메인. 사용자가 직접 분쟁을 접수하거나, 다른 도메인에서 발생한 케이스에 이의를 제기하고, 증빙을 제출하며, 호스트는 운영 인박스 한 화면에서 자신과 관련된 모든 분쟁·문의를 처리한다. 각 원천 도메인의 해결 권위는 그대로 유지되며 — 이 도메인은 조회·접수·이의 창구만 제공한다.

이 도메인이 하는 것과 하지 않는 것

이 도메인은 원천 도메인의 status를 직접 변경하지 않는다. 통합 union 조회, 사용자 직접 분쟁 생성(USER_DISPUTE), 이의제기(DisputeAppeal) 생성·철회만이 공개 API 범위다. UPHELD/REJECTED 전이와 개별 원천 도메인의 실제 해결 처리(정산 재계산·환불 실행 등)는 각 도메인 또는 admin API가 소유한다. 단, 노쇼(EVENT_NO_SHOW) 이의제기 시에만 원본 도메인 상태와 best-effort 동기화가 이루어진다.

모든 케이스는 {prefix}:{sourceId} 형태의 통합 식별자로 표현된다. prefix는 원천 도메인을 나타내며 USER_DISPUTE:42, EVENT_NO_SHOW:7, WARNING_APPEAL:15 같은 형태다. 이 식별자는 URL 경로에 그대로 쓰이므로 URL-encoding이 필요하다.

운영자가 알아야 할 것

핵심 운영 원칙
  • SLA는 7일이다. 미해결 상태로 7일을 넘긴 케이스는 매일 06:00(KST)에 운영자 알림이 발행된다. 8개 scanOne 호출(OPERATIONAL_ISSUE·WARNING_REPORT·WARNING_APPEAL·SETTLEMENT_APPEAL·PUBLIC_REPORT·REFUND_DISPUTE·USER_DISPUTE·DISPUTE_APPEAL)이 독립적으로 실행되므로 한 source 처리 실패가 나머지를 차단하지 않는다. PUBLIC_REPORT는 PENDING+IN_REVIEW만 스캔(ESCALATED 제외).
  • legal hold는 계정 삭제를 차단한다. 활성 케이스(OPEN/IN_REVIEW/ESCALATED)에 연관된 사용자는 계정 삭제 및 데이터 익명화가 차단된다. 단, 연관 판정은 source별 finder 범위에 따라 다르다 — OperationalIssue·WarningReport는 reporter 측만, RefundDispute는 seller(판매자) 측만, USER_DISPUTE만 reporter/target 양측을 판정한다. SETTLEMENT_APPEAL appealer(비호스트)·PurchaseRefundDispute buyer 등 일부 시점은 현재 미검출(후속 개선 과제).
  • 증빙은 종결 후 1년이 지나야 정리된다. 활성 케이스의 증빙 파일은 삭제 불가(evidenceFrozen=true). 종결 + 1년 경과 후 매일 05:00에 자동 정리된다.
  • 이의제기 승인·기각은 admin API 전용이다. 공개 API로는 PENDING 이의를 본인이 철회하는 것만 가능하다. 호스트도 공개 API로 이의를 처리할 수 없다.
  • EVENT_NO_SHOW 이의만 원본과 동기화된다. 다른 source 이의제기는 audit-only로 dispute_appeal 테이블에만 기록되고 원천 케이스 상태는 변하지 않는다.

레드 플래그

현재 알려진 미구현·주의 사항
  • dispute 딥링크가 배선되지 않았다. FCM 알림을 눌러도 분쟁 화면으로 이동하지 않고 홈으로 떨어진다. USER_DISPUTE·CLUB_MEMBERSHIP_ACTION 관련 푸시 알림 클릭 경로가 누락 상태다.
  • 이의 승인·거절 공개 endpoint가 없다. 앱 화면에 호스트용 이의 승인 버튼이 존재하지만 서버 endpoint가 없어 동작하지 않는다. admin API 경로로만 처리 가능하다.
  • 분쟁 접수 시 증빙 첨부 UI가 없다 (v1). 사용자가 분쟁을 직접 접수할 때 파일 업로드 화면이 의도적으로 제외됐다. 서버 검증 로직은 배선되어 있어 v2에서 추가된다.
  • 플랜 마켓 환불 분쟁 증빙 보존 미구현. purchase_refund_dispute는 다른 source와 파일 구조가 달라 종결 1년 후 자동 정리 대상에서 제외되어 있다.
  • HostInboxService 성능 위험. stats 호출 시 size=500 파라미터로 collectAllItems를 호출하지만, 개별 finder는 각자 제한(호스트 이벤트 200건, 분쟁 케이스 100건, APPLICATION·REPORT·정산 이의·FailedRefund는 전량 로드)을 따른다. 수집 후 메모리 정렬·페이징이 적용되며 최종 페이지 크기는 최대 100건으로 캡된다. 활성 이벤트가 많은 호스트는 응답 지연 가능성이 있다.
