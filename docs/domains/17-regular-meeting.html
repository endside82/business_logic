17. 정기모임

호스트가 동일 멤버를 묶어 여러 회차를 운영하는 신규 경량 도메인. FIXED(고정형)과 VARIABLE(변동형)이 인원·정산의 바인딩 레벨을 전환한다. 클럽·EventRecurrence 어느 쪽도 부적합하다는 결론 후 별도 도메인으로 신설했다(regularmeeting). 하위 세션은 모두 일반 이벤트 머신 위에서 동작하되 정션 regular_meeting_event가 순번·교체·확정 상태를 잡는다.

FIXED는 코스(전체 세션) 단위로 인원·결제·정산이 묶인다. 멤버가 한 번 등록하면 모든 세션에 ATTENDING이 깔리고, 결제는 한 번에 일괄, 정산은 코스 종료 시 한 번 발생한다. VARIABLE은 세션마다 독립된 이벤트로 동작해 일반 이벤트 머신을 그대로 재사용한다. 모임 본체는 한 형태만 가지고 두 가지 운영 모드를 전환한다.

정산 후 환불 위험 0 — 다층 방어

close가 통과하려면 모든 세션이 FINALIZED여야 한다. 정산 row는 close 커밋 후 별도 트랜잭션(REQUIRES_NEW)에서 만들어지고, cancelEnrollment는 CLOSED/CANCELED를 거절한다. cancel도 findByIdForUpdate로 잠궈서 close 정산 스냅샷과 환불 경합을 차단한다. 환불은 항상 close 이전(finalize 시점)에 끝나야 정산 gross가 정확해진다.

하위 세션 공통점

두 타입 모두 세션은 event 테이블(eventType=REGULAR_MEETING(3)) 위에서 동작한다. 정션 regular_meeting_event가 모임-세션을 잇고 순번·취소대체본·머터리얼라이즈·확정 상태를 따로 관리한다. VARIABLE이라도 RM 세션은 일반 검색에서 EventScope 거름망(publiclyDiscoverable·신청 가드)이 적용된다.

레드 플래그

실제 분쟁 가능 사례
  • finalize 안 끝낸 채 close 시도 — close 가드가 모든 정션 FINALIZED 요구해서 막힌다. 안내 메시지 명확화.
  • BANK_TRANSFER 입금 확인 무대응 — 호스트가 확인 안 하면 paymentDueAt 만료까지 좌석 예약된 채. 멤버에게 안내 필수.
  • BANK_TRANSFER REFUND_REQUESTED 무대응 — 호스트가 계좌환불 후 refund-confirm 안 하면 환불 진행중 영구 잔존.
  • CLOSED 모임 자가 취소 시도 — 클로백 갭 + 중복 환불 위험이라 명시적으로 차단. UI에서 CTA 자체 비표시.
  • FORFEITED 재신청 — 한 번 박탈된 멤버는 영구 차단(ENROLLMENT_FORFEITED). 정책 안내 필요.
  • VARIABLE 본체에 가격·정원 입력 — 무시되지만 사용자는 입력 가능. 폼 단에서 비활성화 필요.
