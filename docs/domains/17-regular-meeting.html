17. 정기모임

호스트가 같은 사람들을 묶어 여러 회차를 운영하는 모임. 한 도메인 안에 두 가지 운영 방식이 함께 들어 있다 — 고정형은 정해 둔 회차 전체를 멤버가 한 번에 등록·결제·정산하고, 변동형은 회차마다 따로 모집·결제·정산한다. 두 방식이 같은 모임 카드 위에서 뱃지로 구분되고, 회차는 일반 이벤트와 같은 화면을 그대로 쓴다. 본 도메인은 코스 단위 돈 흐름이 핵심이라 환불·정산이 가장 민감하게 설계되어 있다.

이 도메인의 핵심 결정 — 한 모임 두 모드

고정형은 코스(전체 회차) 단위로 인원·결제·정산이 묶인다. 멤버가 한 번 등록하면 모든 회차의 참석 명단에 자동으로 올라가고, 결제는 한 번에 하고, 정산은 코스가 끝날 때 한 번 발생한다. 변동형은 회차마다 독립된 이벤트처럼 동작해 일반 이벤트의 신청·결제·정산을 그대로 재사용한다. 이 분기 하나가 인원 관리·돈 흐름·환불 산식의 기반을 바꾼다.

정산 후 환불 위험 0 — 다층 방어

코스 종료는 모든 회차의 출석이 확정된 다음에만 통과한다. 정산은 종료가 확정된 뒤 별개의 단계로 만들어지고, 그 사이 멤버 자가 취소는 차단된다. 자격박탈 환불은 회차 확정(즉 종료 이전) 시점에만 발생하므로 정산 금액에 자연스럽게 반영된다 — 정산이 만들어진 뒤에 환불이 새로 일어날 가능성이 0이다.

두 운영 방식의 공통점

두 방식 모두 회차는 일반 이벤트와 같은 화면을 그대로 쓴다. 정기모임 회차임을 알리는 뱃지·표기만 덧붙고, 일반 검색·홈·트렌딩에서는 자동으로 분류되어 노출 정책이 따로 적용된다. 호스트는 정기모임 전용 화면에서만 회차를 손볼 수 있고, 일반 이벤트 운영 화면을 통한 직접 수정·신청은 막혀 있다.

운영자가 알아야 할 것

현금 안전 원칙
  • 호스트는 한 번 정해지면 못 바꾼다. 정산 수익자와 환불 책임자가 흔들리지 않도록 v1 에서는 호스트 교체를 막아두었다.
  • 코스 종료는 출석이 다 확정돼야 통과한다. 미확정 회차가 남아 있으면 종료 버튼이 막힌다 — 호스트에게 안내가 필요하다.
  • 계좌이체 환불은 호스트 책임이다. 외부 송금이라 호스트가 직접 환불한 뒤 "환불 완료"를 눌러야 마감된다. 누락 시 환불 진행중 상태가 계속 남는다.
  • 유료 잔존액만 호스트 수익이다. 무료 보너스 포인트로 결제된 부분은 플랫폼 보조로 잡혀 호스트 지급 대상이 아니다.
  • 변동형 자동 종료는 1차 출시에 미적용. 마지막 회차 생성 시점에서 시간이 흘러도 자동으로 닫히지 않는다 — 후속에서 정책 결정.

레드 플래그

분쟁이 일어날 수 있는 케이스
  • 호스트가 출석 확정을 미루는 경우 — 종료 버튼이 막히고, 멤버는 다음 회차로 못 넘어간다. 자동 마감은 기본 비활성.
  • 계좌이체 입금 확인 무대응 — 호스트가 확인 안 하면 자리가 예약된 채로 시간이 흘러간다. 만료까지 다른 멤버가 못 들어옴.
  • 계좌이체 환불 마감 무대응 — 호스트가 외부 송금만 하고 "환불 완료"를 안 누르면 멤버 입장에서 영구 "환불 진행중".
  • 종료된 모임 자가 취소 시도 — 정산 후 환불 위험 때문에 명시적으로 차단했다. 모임이 끝난 뒤 환불은 운영자 중재 경로.
  • 자격박탈 후 재신청 시도 — 노쇼 한도로 박탈된 멤버는 같은 모임에 다시 들어올 수 없다(영구 차단). 정책으로 안내 필요.
  • 변동형 본체에 가격·정원 입력 — 의미 없는 값이 무시되지만 사용자는 입력할 수 있다. 폼에서 비활성 처리 필요.

06 결제·지갑강함지갑 결제는 기존 거래 이력을 재사용. 계좌이체는 외부 송금이라 회계에 잡히지 않는다.
07 모임 정산강함정산 모델이 확장되어 코스 단위 정산을 받아들인다. 유료/무료 잔존액 분리가 추가됐다.
11 리뷰·신고조건부고정형 회차는 실제 출석 확정자만 리뷰 자격. 매 회차마다 리뷰 가능.
12 알림조건부승인/거절/승격/만료/노쇼 알림은 1차 출시에 미배선. 후속에 푸시 라우팅 정의.
13 프로필·설정있음탈퇴·계정 비활성 시 활성 결제·운영중 모임·환불 책임이 있으면 차단된다.
