01. 인증 · 온보딩

가입·로그인·이메일 인증·온보딩·관심사 설정으로 사용자를 “추천 알고리즘이 무엇을 보여줄지 결정할 수 있는 상태”까지 끌어올린다. 모든 다른 도메인의 진입 조건이라, 여기서 깨지면 다른 모든 기능이 보이지 않는다.

왜 이 도메인이 중요한가

인증·온보딩은 사용자 입장에서 “1회만 거치는 화면”이라 가벼워 보이지만, 실제로는 가장 회귀 위험이 크다. 토큰 만료 처리 한 줄을 잘못 건드리면 전체 사용자가 갑자기 로그아웃되고, 마지막 로그인 수단 보호 로직이 빠지면 계정 잠김이 발생한다. “성공 흐름이 짧아도 실패 시나리오가 길다”는 게 이 도메인의 핵심 특성이다.

8개 기능과 각자의 역할

기능 ID는 다른 문서·이슈에서 그대로 참조한다.

F01-01

이메일 회원가입 · 로그인

이메일+비밀번호로 가입·로그인한다. 미성년자 차단, 중복 이메일, 비밀번호 정책, 이미 차단·탈퇴된 계정 처리가 핵심.

이메일 인증

가입 후 이메일 토큰 검증. 재발송 제한(스팸 방지), 만료·위조 토큰, 이미 인증된 사용자의 재진입 처리.

비밀번호 재설정

토큰 갱신 · 로그아웃

가장 자주 깨지는 기능. 동시 401, refresh 만료, 로그아웃 실패 시 로컬 토큰만이라도 비워야 함. 계정 상태 변화 시 강제 로그아웃 동기화 필요.

온보딩

첫 로그인 사용자가 거치는 4–5단계 흐름. 건너뛰기, 중간 재개, 사진·태그·위치 실패 시에도 “추천 가능 상태”까지 도달해야 한다.

관심사 태그 관리

온보딩 후에도 수정 가능. 태그 한도, 중복, 가중치, 추천 품질에 미치는 영향.

F01-08

소셜 계정 연결 해제

다중 소셜 연동 환경에서 “마지막 로그인 수단”은 절대 해제할 수 없다. 미연동 제공자 해제 시도, 멀티 디바이스 처리.

이 도메인의 부수효과

인증·온보딩은 다른 도메인 데이터의 진입 권한을 결정한다. 잘못 건드리면 다른 도메인이 같이 깨진다.

영향여부이유
알림간접가입·로그인 자체로 알림은 적지만, 푸시 디바이스 토큰 등록은 여기서 시작된다.
결제·정산간접로그인되지 않으면 결제 불가. 차단·탈퇴 계정의 결제 차단 정책이 여기로 수렴.
위치있음온보딩에서 위치 권한과 기본 주소를 받는다. 거부 시에도 추천이 동작해야 함.
리뷰·신뢰있음회원가입·소셜 로그인 케이스에서 차단·탈퇴 사용자의 재진입을 막아야 신뢰 시스템이 유지된다.

이메일 인증의 구체 정책 (F01-03)

항목값
인증 링크 유효 기간24시간 — 지나면 다시 발송받아야 한다
재발송 제한60초에 한 번 — 남은 시간 카운트다운 표시
인증 방법메일 속 링크를 누르면 앱이 열리며 자동으로 인증된다. 링크가 동작하지 않는 환경을 위한 수동 확인 경로도 함께 제공.
두 모드

모드 A (안내 + 재발송): 메일을 받지 못한 사용자가 다시 보내달라고 요청. 60초 쿨다운 적용.
모드 B (토큰 자동 인증): 메일 안의 Deep Link로 토큰이 자동 전달돼 즉시 검증.

자주 빠뜨리는 시나리오

레드 플래그
  • 마지막 로그인 수단 제거 — F01-08에서 마지막 남은 소셜 또는 이메일 인증을 해제하면 계정 잠김. 반드시 사전 차단.
  • 미인증 사용자 홈 진입 — 이메일 인증 전인데 다른 화면이 멀쩡히 열리는 경우. 같은 사용자가 어떤 화면은 막히고 어떤 화면은 통과하면 신뢰가 깨진다.
  • 토큰 갱신 무한 루프 — refresh 토큰까지 만료된 상태에서 401이 반복되며 사용자가 무한히 로딩만 보는 케이스.
  • 소셜 가입 후 추가정보 미입력 이탈 — 가입은 됐지만 닉네임·동의가 비어 있어 추천 불가능 상태. 진입 가드 분기 필요.
  • 이메일 인증 메일 재발송 제한 미안내 — 사용자가 “안 와요”라며 계속 누르는 케이스. 대기 안내·횟수 제한 표시 필요.
