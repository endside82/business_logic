상태 정책

이 문서는 화면 설명보다 “지금 이게 어떤 상태인지”가 중요한 기능만 모았다. 버튼·문구·알림·예외 처리를 결정할 때 첫 참고서다. 코드를 보지 않고도 “이 상태에서는 무엇이 가능한가”를 답할 수 있어야 한다.

이 문서를 보는 법

각 상태마다 사용자에게 보이는 의미, 허용 액션, 다른 도메인에 미치는 영향을 함께 정의한다. 상태 이름만 정해 두면 “이 상태에서 어떤 알림이 가는가”를 매번 다르게 해석하게 된다.

상태 PRD의 3원칙
  • 상태 이름만 정의하지 않는다. 사용자가 보는 문구와 가능한 액션까지 같이 정의한다.
  • 상태 변경은 알림·캘린더·결제·리뷰 자격에 영향을 줄 수 있다. 변경 시 그 영향을 함께 검토한다.
  • 취소·만료·삭제·차단은 성공 상태와 다른 복구 동선을 가져야 한다.

2. 이벤트 신청과 참석 상태

이벤트에는 “신청서 심사”와 “참석 확정/대기”가 같이 돌아간다. 두 흐름은 별개 — 한 상태로 묶지 않는 게 핵심.

자주 헷갈리는 부분
  • CANCELED(신청서)와 CANCELLED(참석)는 다른 enum이다. 코드 리뷰 시 가장 자주 어긋남.
  • 대기열 자동 승격은 사용자 행동 없이 시스템이 바꾼다. 따라서 알림이 필수.
  • “승인됨”과 “참석 확정”은 다르다. 유료 승인제 이벤트의 경우 결제 후에만 참석 확정이다.

F03-13 선입금 상태로 분리

APPROVED_PENDING_PAYMENT, PAYMENT_EXPIRED는 이벤트 참가 선입금의 핵심 상태다. 승인 즉시 ATTENDING으로 만들지 않고, 지갑 결제 또는 계좌이체 확인 이후에만 참석 확정으로 전이한다. 자세한 결제 row 상태는 F03-13 이벤트 참가 선입금에서 본다.

4. 모임 정산 상태

stateDiagram-v2 [*] --> DRAFT: 호스트가 정산 생성 DRAFT --> ACTIVE: 납부 요청 시작 ACTIVE --> COMPLETED: 모든 납부·확인 완료 ACTIVE --> CANCELLED: 호스트가 취소 COMPLETED --> [*] CANCELLED --> [*]
상태의미호스트 액션참가자 액션
DRAFT아직 참가자에게 요청하지 않은 정산항목 편집·활성화노출 없음
ACTIVE납부 진행중독촉·연장·이체 확인·취소분담금 확인·납부·이의제기
COMPLETED정산 완료내역 조회내역 조회
CANCELLED정산 취소내역 조회환불·취소 안내
절대 위반 금지
  • 정산 생성만으로 참가자에게 알림이 가면 안 된다. 알림은 DRAFT → ACTIVE 전이 시점에만.
  • 포인트 납부는 즉시 반영, 계좌이체는 호스트 확인 필요. 사용자가 “냈다”와 “시스템이 받았다”를 구분할 수 있어야 한다.
  • 이의제기와 감사로그는 분쟁을 줄이는 안전망이다. 임의로 끄지 않는다.

6. 플랜·마켓 상품 상태

flowchart LR A["플랜 DRAFT"] --> B["플랜 PUBLISHED"] B --> C["마켓 상품 등록"] C --> D["판매중"] D --> E["구매됨"] D --> F["판매중지"] E --> G["내 컬렉션"] G --> H["이 플랜으로 이벤트 생성"]
  • 플랜 초안은 크리에이터의 편집 대상, 마켓 상품은 구매자의 구매 대상, 컬렉션은 활용 대상. 셋이 같은 데이터의 다른 얼굴.
  • 이미 구매한 상품·일부 중복 번들·잔액 부족은 구매 바텀시트에서 명확히 분기.

7. 데이터 내보내기와 계정 삭제

flowchart TD A["데이터 내보내기 요청"] --> B["PROCESSING"] B --> C["COMPLETED"] B --> D["FAILED"] C --> E["다운로드 가능"] E --> F["EXPIRED"]
stateDiagram-v2 [*] --> NONE: 삭제 요청 없음 NONE --> PENDING: 삭제 예약 PENDING --> APPROVED: 처리 대기 PENDING --> CANCELLED: 사용자 취소 APPROVED --> CANCELLED: 사용자 취소 APPROVED --> COMPLETED: 유예 종료 후 삭제
  • 데이터 내보내기는 즉시 파일을 주는 기능이 아니라 비동기 요청이다. 처리 중 상태를 사용자에게 명확히 안내.
  • 계정 삭제 예약은 유예 기간과 취소 동선이 가장 중요. 유예 동안 사용자가 “취소”를 발견하지 못하면 의도치 않은 삭제 발생.
  • 즉시 비활성화는 삭제 예약과 사용자 영향이 다르므로 별도 문구.

8. 위치 공유 상태

flowchart LR A["위치 공유 가능 이벤트"] --> B{"사용자 동의"} B -->|"동의"| C["공유 활성"] B -->|"거부"| D["공유 비활성"] C --> E["위치 업데이트"] C --> F["공유 시간 연장"] C --> G["공유 중지"] G --> D
  • 위치 공유는 항상 opt-in. 동의 없이 위치가 노출되는 동작은 금지.
  • 중지 버튼과 프라이버시 대시보드가 함께 있어야 사용자가 통제권을 가진다.
  • 이벤트 종료 후 위치 데이터 보관·삭제 정책을 화면 문구와 맞춘다.

9. 알림 수신 정책

flowchart TD A["상태 변화 발생"] --> B{"알림 받을 사용자 존재?"} B -->|"없음"| C["종료"] B -->|"있음"| D{"카테고리 수신 허용?"} D -->|"아니오"| E["푸시 미발송"] D -->|"예"| F{"방해금지 시간?"} F -->|"예"| G["보류·무음 정책"] F -->|"아니오"| H["푸시 발송"] E --> I["알림함 반영 여부 판단"] G --> I H --> I
  • 알림은 “무엇을 보낼까”보다 “누가 받아야 하는가”가 먼저.
  • 결제·정산 알림은 마케팅 알림보다 중요도가 높다.
  • OS 권한 거부 상태에서는 인앱 알림함과 권한 회복 배너를 함께 고려.

수용 기준

  • 상태 이름만 정하지 말고, 그 상태에서 사용자에게 보이는 문구·할 수 있는 액션·알림 발송 여부까지 한 묶음으로 적어 둔다.
  • 상태 변경 후 이전 화면의 CTA가 오래된 상태를 보여주지 않아야 한다.
  • 취소·만료·삭제·차단 상태는 성공 상태와 별도의 복구 동선을 가진다.
