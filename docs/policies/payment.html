결제 · 정산 정책

포인트 충전, 결제, 환불, 호스트 정산금, 모임 정산, 클럽 기금을 서로 구분해 사용자·운영자가 돈의 상태를 오해하지 않게 한다. 한 줄로 요약하면 “돈의 흐름을 절대 섞지 말 것”.

돈의 6개 흐름

서로 섞이면 안 되는 거래 종류. 각자 별개 거래 타입과 별개 상태 모델을 가진다.

유료/무료 분리정산 (Point Split Flow-Through)

무료 포인트로 프로모션을 운영하되 무료가 절대 현금이 되지 않도록 유료/무료를 잔액·결제·정산·인출 전 구간에서 분리한다.

  • 핵심 원리: 지갑·클럽 기금 잔액을 유료(paid, 충전·현금환불 = 인출 가능) / 무료(free, 프로모션 지급 = 사용 가능·인출 불가)로 나눠 관리한다. 결제 시 유료/무료 split이 발생하지만, split이 수취자까지 전파되는지는 사용처 분류(아래 3분류)에 따라 다르다. flow-through(무료 수취자 존재)에서만 split이 수취자(창작자·호스트·기금)까지 그대로 전파(flow-through)되어 무료로 결제하면 수취자도 무료로 받아 현금화되지 않는다. free-burn(수취자 없음)은 무료분이 수취자 없이 지갑에서 차감(소각)된다. PAID_ONLY는 무료 결제 자체를 허용하지 않는다. 이벤트 참가비는 정책상 flow-through 목표이나 현재 legacy 경로 미이관(followup).
사용처 분류대상무료 처리
flow-through
(무료 수취자 존재)		마켓 구매, 플랜 직접 구매, 프라이빗 미팅비, 클럽 기부·가입비(→기금), 이벤트 참가비 (정책 목표, 현재 legacy 미이관 — followup)opt-in 시 무료 허용. split이 수취자/기금에 free로 적립, 유료분만 정산·인출
free-burn
(플랫폼 매출, 수취자 없음)		개인 구독, 클럽 구독, 프라이빗 호스팅비, 호스팅 티켓opt-in 시 무료 허용. 무료분은 지갑에서 차감(소각). spend 시점 프로모션 비용(PROMOTION_EXPENSE) 분개는 현재 미구현 — followup
PAID_ONLY
(순수 P2P 채무 정산)		모임 정산 송금·회수·선입금, 충전 취소무료 불가 — 실제 채무를 프로모션 포인트로 갚게 두지 않는다

차등 가격 (콘텐츠 생산자 opt-in)

  • 무료 허용 결정 필드는 도메인별로 다르다(PricingDecision) — 마켓 아이템/번들은 currencyType(PAID_POINT/FREE_POINT/ANY_POINT) + 구매자 fundingMode, 플랜 직접 구매는 allowFreePoints + 구매자 fundingMode로 결정한다. 무료 전용가는 양쪽 모두 freePointPrice(nullable).
  • 차등가 콘텐츠는 전액 무료(freePointPrice 또는 기본가) 또는 전액 유료(기본가) 중 택1. ANY_POINT + freePointPrice==null은 PAID_FIRST 혼합 허용.
  • 무료 미허용 콘텐츠(마켓 PAID_POINT / 플랜 allowFreePoints=false)에 무료 결제 요청 시 INVALID_REQUEST로 거부. 마켓 ANY_POINT + freePointPrice==null + fundingMode=FREE도 INVALID_REQUEST(기본가는 무료로 결제 불가).

인출(현금화) — paid-only

무료는 어떤 경로로도 현금이 되지 않는다

사용자·클럽 외부 출금은 유료(paid) 잔액만 대상이다. 탈퇴 시 유료는 환불·무료는 소멸(forfeit). 클럽 기금 인출 가용액 = paid − 예비금/미수금이며, 수수료(5%)·원천징수(3.3%)는 유료(현금)분에만 부과한다. 클럽 폐쇄 인출도 유료만 현금화하고 무료 기금은 소멸한다. 창작자 정산도 유료/무료를 분리 기록해 수수료·세금은 유료분에만 적용하고, 무료 매출은 무수수료 free_credit으로 창작자 free에 적립된다.

followup: refundToWallet(클럽 기부·가입비·구독 환불)의 split 보존은 refundByTransaction 전환 예정(결제 txId 스키마 보강 동반), EVENT 결제·환불의 spend() 이관, 마켓 번들 차등가 admin 배선(community_admin_api)은 미완.

결제 실패 처리 표준

사용자는 “돈이 빠졌는지 안 빠졌는지”를 가장 먼저 알고 싶어 한다. 실패 사유를 정확히 구분해서 보여준다.

상황사용자 안내
잔액 부족충전 또는 자동충전 안내
결제수단 없음결제수단 등록 화면으로 진입
PG 실패결제가 완료되지 않았고 재시도 가능함을 명시
콜백 지연“처리중” 상태와 재조회 동선 제공
중복 클릭한 번의 결제만 유효하도록 멱등 처리

유료 승인제 이벤트 결제 정책

한 이벤트가 “유료이면서 승인제”일 수 있다. 이 경우 호스트 승인과 결제, 결제와 참석 확정이 다 다른 단계라서 상태를 따로 둔다.

정책설명
승인 전 결제 금지승인되지 않은 사용자의 돈을 먼저 받지 않는다.
승인 후 결제 필요호스트 승인은 “결제 권한 부여”이지 참석 확정이 아니다.
결제 전 참석 권한 제한체크인·위치 공유·리뷰 자격은 결제 성공 후 열린다.
기한 만료 처리결제 대기 상태에는 기한이 있고, 만료 시 예약 정원을 반환한다.
서버 검증WalletService.payForApplication와 선입금 facade는 승인 대기 상태·기한·중복 결제를 검증해야 한다.
이벤트 참가 선입금으로 반영

APPROVED_PENDING_PAYMENT·PAYMENT_EXPIRED 상태와 “결제 후 attendance 생성” 계약은 F03-13 이벤트 참가 선입금에서 별도 결제 row로 다룬다. 승인 전 결제 금지와 승인 즉시 ATTENDING 금지 원칙은 그대로 유지된다.

모임 정산 vs 호스트 정산금 vs 클럽 기금

세 가지가 같은 “정산”이라는 단어로 묶여 헷갈리기 쉽다. 흐름·주체·상태가 모두 다르다.

모임 정산호스트 정산금클럽 기금
주체참가자 → 호스트플랫폼 → 호스트멤버 → 클럽
트리거호스트가 정산 생성·활성화유료 이벤트 종료·플랜 판매멤버 기부·이벤트 정산 반영
상태DRAFT → ACTIVE → COMPLETEDPENDING → APPROVED → PAYING → PAID잔액 + 거래 이력

수용 기준

  • 결제 성공과 기능 성공이 분리되는 경우 처리중·재조회 상태가 있어야 한다.
  • 환불은 원거래·환불 금액·환불 상태가 추적 가능해야 한다.
  • DRAFT 정산은 참가자에게 납부 요청으로 노출되지 않는다.
  • 계좌이체 납부는 호스트 확인 전까지 “완료”로 오해하지 않게 표시한다.
  • 유료 승인제 이벤트는 승인 전 결제·결제 전 참석 확정·기한 만료 후 결제가 모두 차단된다.
