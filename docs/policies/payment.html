결제 · 정산 정책

포인트 충전, 결제, 환불, 호스트 정산금, 모임 정산, 클럽 기금을 서로 구분해 사용자·운영자가 돈의 상태를 오해하지 않게 한다. 한 줄로 요약하면 “돈의 흐름을 절대 섞지 말 것”.

돈의 6개 흐름

서로 섞이면 안 되는 거래 종류. 각자 별개 거래 타입과 별개 상태 모델을 가진다.

흐름주체의미
포인트 충전사용자 → 지갑외부 결제 대행을 통한 결제로 잔액 증가
포인트 결제지갑 → 서비스이벤트·플랜·구독 등 비용 지불
환불서비스 → 지갑·원수단기존 결제 취소 또는 보상
모임 정산참가자 → 호스트모임 비용 분담
호스트 정산금플랫폼 → 호스트수익 지급
클럽 기금멤버·운영 → 클럽공동 자금

유료/무료 분리정산

무료 포인트로 프로모션을 운영하되 무료가 절대 현금이 되지 않도록 유료/무료를 잔액·결제·정산·인출 전 구간에서 분리한다.

  • 핵심 원리: 지갑·클럽 기금 잔액을 유료(충전·현금환불로 들어온 인출 가능 잔액) / 무료(프로모션으로 지급된, 사용은 되지만 현금 인출 불가한 잔액)로 나눠 관리한다. 결제 시 유료/무료가 나뉘어 빠지지만, 이 구분이 받는 쪽까지 전달되는지는 사용처 분류(아래 3분류)에 따라 다르다. 전파형(받는 사람이 있는 흐름)에서만 무료로 결제하면 받는 쪽(창작자·호스트·기금)도 무료로 받아 현금화되지 않는다. 소각형(받는 사람이 없는 흐름)은 무료분이 받는 쪽 없이 지갑에서 그대로 차감(소각)된다. 유료 전용은 무료 결제 자체를 허용하지 않는다. 이벤트 참가비는 정책상 전파형이 목표이나 현재 구형 경로라서 아직 이관되지 않았다(보강 예정).
사용처 분류대상무료 처리
전파형
(받는 사람이 있는 흐름)		마켓 구매, 플랜 직접 구매, 프라이빗 미팅비, 클럽 기부·가입비(→기금), 이벤트 참가비 (정책 목표, 현재 구형 경로 미이관 — 보강 예정)생산자가 허용하면 무료 결제 가능. 무료분은 받는 쪽·기금에 무료로 적립되고, 유료분만 정산·인출 대상이 된다.
소각형
(플랫폼 매출, 받는 사람 없음)		개인 구독, 클럽 구독, 프라이빗 호스팅비, 호스팅 티켓생산자가 허용하면 무료 결제 가능. 무료분은 지갑에서 그대로 차감(소각). 무료분을 마케팅 비용으로 회계에 잡는 처리는 현재 미구현 — 보강 예정
유료 전용
(개인 간 실제 채무 정산)		모임 정산 송금·회수·선입금, 충전 취소무료 불가 — 실제 채무를 프로모션 포인트로 갚게 두지 않는다

차등 가격 (콘텐츠 생산자가 직접 결정)

  • 무료로 결제할 수 있는지는 생산자가 정한다. 마켓 상품·번들은 결제 화폐(유료 전용 / 무료 전용 / 둘 다 허용)와 구매자의 결제 방식 선택으로, 플랜 직접 구매는 무료 허용 여부 설정과 구매자의 결제 방식 선택으로 결정된다. 무료로 결제할 때 적용할 별도 가격(무료 전용가)도 양쪽 모두 따로 정할 수 있다.
  • 차등가 콘텐츠는 전액 무료(무료 전용가 또는 기본가) 또는 전액 유료(기본가) 중 하나로만 결제된다. "둘 다 허용"이면서 무료 전용가를 따로 두지 않은 경우에만 유료 포인트를 먼저 쓰고 모자라면 무료로 채우는 혼합 결제가 허용된다.
  • 무료를 허용하지 않은 콘텐츠에 무료로 결제하려 하면 거부된다. "둘 다 허용"이면서 무료 전용가를 두지 않은 콘텐츠를 무료로 결제하려는 경우도 거부된다(기본가는 무료로 결제할 수 없다).

인출(현금화) — 유료분만

무료는 어떤 경로로도 현금이 되지 않는다

사용자·클럽 외부 출금은 유료 잔액만 대상이다. 탈퇴 시 유료는 환불, 무료는 소멸한다. 클럽 기금 인출 가용액 = 유료 잔액 − 예비금·미수금이며, 수수료(5%)·원천징수(3.3%)는 유료(현금)분에만 부과한다. 클럽 폐쇄 인출도 유료만 현금화하고 무료 기금은 소멸한다. 창작자 정산도 유료·무료를 분리 기록해 수수료·세금은 유료분에만 적용하고, 무료 매출은 수수료 없이 창작자의 무료 잔액으로 적립된다.

보강 예정: 클럽 기부·가입비·구독 환불 시 유료/무료 비율 보존, 이벤트 결제·환불의 분리정산 이관, 마켓 번들 차등가의 운영자 설정 연결은 아직 미완이다.

결제 실패 처리 표준

사용자는 “돈이 빠졌는지 안 빠졌는지”를 가장 먼저 알고 싶어 한다. 실패 사유를 정확히 구분해서 보여준다.

상황사용자 안내
잔액 부족충전 또는 자동충전 안내
결제수단 없음결제수단 등록 화면으로 진입
결제 대행 실패결제가 완료되지 않았고 재시도 가능함을 명시
콜백 지연“처리중” 상태와 재조회 동선 제공
중복 클릭같은 결제는 한 번만 유효하게 처리

유료 승인제 이벤트 결제 정책

한 이벤트가 “유료이면서 승인제”일 수 있다. 이 경우 호스트 승인과 결제, 결제와 참석 확정이 다 다른 단계라서 상태를 따로 둔다. 이 흐름은 사전결제를 사용하는 이벤트에 한해 적용되며, 사전결제를 쓰지 않는 유료 이벤트의 흐름은 후속 결정 대상이다.

정책설명
승인 전 결제 금지승인되지 않은 사용자의 돈을 먼저 받지 않는다.
승인 후 결제 필요호스트 승인은 “결제 권한 부여”이지 참석 확정이 아니다.
결제 전 참석 권한 제한체크인·위치 공유·리뷰 자격은 결제 성공 후 열린다.
기한 만료 처리결제 대기 상태에는 기한이 있고, 만료 시 예약 정원을 반환한다.
서버 검증결제 처리는 승인 후 결제 대기 상태인지, 기한이 지나지 않았는지, 이미 결제한 건은 아닌지를 모두 확인한 뒤에만 통과시킨다.
이벤트 참가 선입금으로 반영

"승인 후 결제 대기"·"결제 기한 만료" 상태와 "결제 후에만 참석 확정" 원칙은 F03-13 이벤트 참가 선입금에서 별도 결제 단계로 다룬다. 승인 전 결제 금지와 승인 즉시 참석 확정 금지 원칙은 그대로 유지된다.

모임 정산 vs 호스트 정산금 vs 클럽 기금

세 가지가 같은 “정산”이라는 단어로 묶여 헷갈리기 쉽다. 흐름·주체·상태가 모두 다르다.

모임 정산호스트 정산금클럽 기금
주체참가자 → 호스트플랫폼 → 호스트멤버 → 클럽
트리거호스트가 정산 생성·활성화유료 이벤트 종료·플랜 판매멤버 기부·이벤트 정산 반영
상태초안 → 납부 진행중 → 완료정산 대기 → 승인 → 지급 처리 → 지급 완료잔액 + 거래 이력
책임 도메인07 모임 정산06 결제·지갑04 클럽

수용 기준

  • 결제 성공과 기능 성공이 분리되는 경우 처리중·재조회 상태가 있어야 한다.
  • 환불은 원거래·환불 금액·환불 상태가 추적 가능해야 한다.
  • 초안 상태의 정산은 참가자에게 납부 요청으로 노출되지 않는다.
  • 계좌이체 납부는 호스트 확인 전까지 “완료”로 오해하지 않게 표시한다.
  • 유료 승인제 이벤트는 승인 전 결제·결제 전 참석 확정·기한 만료 후 결제가 모두 차단된다.
