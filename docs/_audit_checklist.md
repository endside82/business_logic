# 비즈니스 로직 문서 재정비 체크리스트

문서의 1차 사용자는 **비즈니스 의사결정자(PO/PM/운영/기획/디자이너)**다.
개발자가 API 명세를 찾으러 오는 곳이 아니다. 코드 디테일 노출이 의미 있을 때만
짧게 인용한다. 아래 기준으로 모든 페이지를 재정비한다.

## A. 즉시 제거 — 비즈니스 문서에 들어가면 안 되는 항목

각 페이지를 열어 다음 흔적을 모두 지운다.

- [ ] **"서버 계약 한눈에" 섹션 통째로 삭제**
  - HTTP method/Path/쿼리스트링이 노출된 표. API 문서가 아니므로 본문에서 제거.
  - 어떤 호출이 필요한지를 보여줘야 한다면 *비즈니스 동사*로만(예: "추천 이벤트 10개 조회").
- [ ] **DB 테이블명, 컬럼명, 상태 enum의 영문 원형 노출 제거**
  - `point_transactions`, `date_match`, `date_chat_room`, `refund_requests` 같은 원어 노출 금지.
  - "지갑 거래", "매치", "채팅방", "환불 요청" 같은 한국어 비즈니스 용어로 대체.
- [ ] **에러 코드 상수 노출 제거 또는 압축**
  - `DATE_VERIFICATION_REQUIRED`, `CLUB_INVALID_CATEGORY`, `INVALID_INPUT` 같은 코드를 본문에 노출하지 않는다.
  - 정책 위반의 *비즈니스 의미*만 적는다. (예: "본인인증 미통과 사용자는 후보 카드를 볼 수 없다.")
- [ ] **"이 단위가 만들거나 바꾸는 데이터" 표 삭제**
  - DB 테이블 단위로 INSERT/UPDATE를 그리는 표는 ERD 문서지 비즈니스 문서가 아니다.
  - 비즈니스 결과를 한 줄로 적는다. (예: "매칭이 성사되면 채팅방이 즉시 열린다.")
- [ ] **"수용 기준(AC) Given/When/Then" 표 삭제 또는 압축**
  - QA용 BDD 표는 비즈니스 결정자가 읽을 자료가 아니다.
  - 본문에 통합되거나 단위별 "확인 가능한 결과" 한 줄로 압축.
- [ ] **시퀀스/플로우 다이어그램의 코드 어휘 제거**
  - `POST /api/v1/...`, `INSERT`, `UPDATE`, `status=MATCHED` 같은 코드 어휘는 다이어그램에서 자연어로 교체.
- [ ] **pill(키워드 배지)에서 API 키워드 제거**
  - `GET /transactions`, `size=20`, `startTime ASC`, `memberCount DESC` 같은 칩 삭제.
  - 비즈니스 의미 칩만 남긴다. (예: "본인인증 필수", "비로그인 허용", "1인 1신청".)

## B. 어휘 — AI 번역체·개발 용어 정리

각 페이지를 본문 처음부터 끝까지 읽고 다음 어휘를 정리한다.

- [ ] **번역체로 어색한 단어 교체**
  - "그린다" → "보여준다" / "표시한다"
  - "합성한다" → "덧붙인다" / "함께 보여준다"
  - "정합성이 무너지면" → "데이터가 어긋나면"
  - "단일 트랜잭션" → "한 번에" / "분리될 수 없는 동작으로"
  - "rollbackFor" / "트랜잭션 분리" / "캐스케이드" 같은 코드 용어는 풀어 쓴다.
  - "퍼널" / "리프레시" / "런타임" 같은 영어 직역은 가능하면 한국어로.
- [ ] **불필요한 영어/약어 제거**
  - `OPEN` / `ACTIVE` / `PUBLISHED` 같은 상태 enum의 영문 원형은 본문에서 한국어 라벨로 교체.
  - 본문 첫 등장 시점에만 (원형) 형태로 한 번 병기 가능.
- [ ] **불필요하게 권위적인 어조 제거**
  - "절대 허용하지 않는다", "정답이다" 같은 표현은 정책 근거가 명확할 때만.
  - 그 외에는 "기본 정책은 ~다" / "현재 구현은 ~다"로 부드럽게.
- [ ] **시적·문학적 비유 금지** (이번 시범 1차 작성에서 다수 발생, 사용자 명시 지적 사항)
  - "손짓", "심장", "정점", "약속이 흔들리면", "신뢰의 다리", "납득의 비용", "구매 의사결정의 두 축",
    "결정의 무게를 크기로 표현", "깨끗한 면" 같은 명사화 비유는 모두 풀어서 직접 서술한다.
  - "X는 Y의 핵심 단서" 같은 추상 명사화도 직접 동사 문장으로 풀어 쓴다.
  - "0초", "95%" 같은 근거 없는 수치 비유 금지. 수치는 출처가 있을 때만.
  - 인용부호(\"...\")로 추상 개념을 강조하는 패턴(예: \"왜 추천인지\", \"이게 비즈니스에 무엇을 남기는가\")도 남용 금지 — 진짜 화면 카피/사용자 발화일 때만.

## C. 화면 구성 — "보이는 것"이 실제 화면이어야 한다

- [ ] **추상 명칭(AppBar / 배너 / 모달) 단독 나열 금지**
  - "AppBar — 좌측 로고, 우측 아이콘"만 적힌 칸은 정보가 0에 수렴한다.
  - 사용자가 화면을 안 봐도 *무엇이 왜 거기 있는지*를 알 수 있어야 한다.
- [ ] **화면 구성 칸에 다음 세 가지가 있어야 한다**
  1. 사용자가 *눈으로 보는 핵심 정보* (어떤 카드, 어떤 숫자, 어떤 라벨)
  2. 그 정보가 *왜 거기에 있는지* (의사결정·인지 단서)
  3. 사용자가 *바로 할 수 있는 행동*과 *그 행동의 결과*
- [ ] **화면이 단순한 hub(F02-05 같은 진입점)면 표를 강제로 만들지 않는다**
  - "검색 아이콘 / 알림 아이콘" 두 줄짜리 표는 의미 없음. 본문 1-2 문단으로 대체.

## D. 메트릭(헤더 4박스) — 강제 채움 금지

- [ ] **숫자가 비즈니스 결정에 영향을 줄 때만 표시**
  - "시나리오 9 / Happy 1 · 정책 5 · 횡단 3" 같은 문서 메타 통계는 삭제.
  - "하위 동작 4" 같은 자기참조 통계도 삭제. (목차에서 이미 보임)
- [ ] **남길 메트릭의 조건**
  - 운영·정책·SLA에 직접 영향 (예: "PENDING 유효 7일", "기간 90일 이내", "추천 10건 노출")
  - 사용자 경험의 핵심 약속 (예: "캐시 5분", "본인인증 1회")
- [ ] **메트릭이 부족하면 박스 수를 줄인다 (2~3개 허용)**

## E. 시나리오 — 형식 평탄화 금지

- [ ] **시나리오 개수 강제 폐지**
  - F09-03 같은 핵심 단위는 9개도 적을 수 있고, F02-05 같은 hub는 3개가 자연스러울 수 있다.
  - "Happy 1 · 정책 5 · 횡단 3" 같은 강제 균등 분배 폐지.
- [ ] **각 시나리오에 "비즈니스 의미" 한 줄 추가**
  - 시나리오 종료 후 "이게 비즈니스에 무엇을 남기는가" 한 줄. (예: "차단 한 번이 향후 노출과 매칭을 모두 차단한다 — 안전한 빠른 이탈 경로의 핵심.")
- [ ] **개발 흐름 단계는 압축**
  - "서버: 인증 가드 → 차단 검사 → 기존 매치 조회 → 액션 갱신" 같은 7단계는 1~2단계로 압축한 비즈니스 흐름으로.

## F. 깊이 — 무게에 맞춰 조절

페이지가 다 같은 분량이면 안 된다. 단위의 비즈니스 무게를 우선 분류한다.

- [ ] **HEAVY 단위** (도메인의 심장 — 매출/안전/핵심 가치 결정)
  - 예시: F03 이벤트 신청/결제, F04 클럽 가입/멤버십, F06 결제·환불, F07 정산, F09-03 스와이프, F09-07 차단
  - 추가 섹션: **설계 의도 / 비즈니스 가치 / 운영 책임 / 측정 지표 / 대안 거부 이유** 중 적용 가능한 것
  - 시나리오 6~9개, 다이어그램 2~3개
- [ ] **NORMAL 단위** (도메인의 일반 흐름)
  - 핵심 의도 + 시나리오 4~6개 + 다이어그램 1~2개
  - 운영 책임/측정 지표는 필요할 때만
- [ ] **HUB/THIN 단위** (라우팅·진입점·새로고침 등)
  - 본문 2~3 문단, 시나리오 2~4개, 다이어그램 0~1개
  - "메트릭 4박스" 강제 폐지, "화면 구성 표" 강제 폐지

## G. 빠진 정보 — 비즈니스 문서가 진짜 필요한 것

가능한 곳에 다음 정보를 추가한다(억지로 다 채우지는 않는다).

- [ ] **"왜 이 단위가 존재하는가"** — 핵심 의도 콜아웃 위에 1문장
- [ ] **"누가 영향을 받는가"** — 사용자/호스트/멤버/운영자 등 페르소나 매핑
- [ ] **"어디서 가치를 만드는가"** — 사용자 가치 + 사업 가치
- [ ] **"무엇이 어긋나면 위험한가"** — 비즈니스 리스크 (현재 "자주 빠뜨리는 케이스"가 개발 회귀에 치우쳐 있음 — 비즈니스 리스크 섹션과 분리)
- [ ] **"어떻게 측정하는가"** — 가능한 KPI/지표 한 줄
- [ ] **"다른 단위와의 경계"** — 본 단위에서 책임지지 않는 것 명시 (이미 일부 있음)

## H. "자주 빠뜨리는 케이스" 분리

지금은 개발 회귀 경고와 운영 리스크가 섞여 있다.

- [ ] **레드 플래그 = 개발 회귀**는 짧게 유지 (3~5개)
- [ ] **운영·정책 리스크**는 별도 콜아웃 분리 또는 본 섹션 폐지

## I. 페이지 구조 표준 — 재정비 후 권장 골격

```
헤더(브레드크럼 + h1 + lede 3~5문장)
↓
핵심 메트릭 (2~4개, 비즈니스 의미 있는 숫자만)
↓
설계 의도 콜아웃 (왜 이 단위가 이렇게 생겼는가)
↓
이 기능 안의 하위 항목 카드 (3~5개)
↓
다이어그램 1~3개 (필요한 만큼만, HUB 단위는 0~1개)
↓
사용자 시나리오 (무게에 맞춰 3~9개, 각 시나리오에 비즈니스 의미 1줄)
↓
화면 구성 — 사람이 보는 것 (실제 콘텐츠가 있을 때만)
↓
운영·비즈니스 리스크 (선택 — 운영자에 의미 있는 정보가 있을 때만)
↓
이어지는 단위
```

비즈니스 문서가 안 필요로 하는 섹션은 **만들지 않는다**. 모든 페이지가 같은 골격을 가질 필요 없다.

## J. 우선순위 — 어떤 페이지부터

1. **HEAVY 도메인 먼저** — 03 이벤트, 06 결제, 07 정산, 09 데이팅, 04 클럽 핵심 단위 (~30 페이지)
2. **NORMAL 단위** — 나머지 도메인 본체 (~50 페이지)
3. **HUB/THIN 단위** — 라우팅·진입점·새로고침 (~35 페이지) — 가장 강하게 압축

도메인 단위로 묶어 코드 농도/번역체/구조 동시 정리.

## K. 작업 단위

- 도메인 1개당 1 라운드.
- 각 라운드: HEAVY → NORMAL → HUB 순으로 깊이 차등.
- 라운드 마무리 시 1 페이지 샘플을 직접 확인 후 다음 라운드 진행.
