F01-03. 이메일 인증

가입자가 입력한 이메일이 실제 본인 소유인지 확인한다. 인증 메일에 담긴 링크를 누르면 앱이 다시 열리고 토큰 검증 → email_verified 갱신이 자동으로 이뤄진다. 짧은 흐름이지만 만료·재사용·재발송 쿨다운 등 토큰 생명주기 분기가 많고, 외부 메일 서버 장애가 트랜잭션 경계와 함께 얽혀 있어 실무에서 가장 자주 회귀가 발생하는 단위 중 하나다.

로그인된 사용자가 자기 이메일을 인증할 수 있게 한다. 외부 메일 서버로 인증 메일을 보내고, 메일 안의 딥 링크가 다시 앱을 열며, 앱은 받은 토큰을 서버에 보내 본인 사용자 레코드의 인증 플래그를 참으로 만든다. 비밀번호 변경이나 계정 복구 같은 다른 이메일 흐름은 본 단위가 아니다.

이 기능 안의 하위 항목

하위 기능 1

인증 메일 발송 (모드 A 화면)

“{이메일}로 인증 메일을 보냈습니다” 안내 + “메일 앱 열기” + 재발송 텍스트 링크. 사용자가 메일을 못 받았다고 느끼면 60초 쿨다운이 지난 뒤 다시 발송할 수 있고, 남은 시간은 텍스트에 카운트다운으로 노출된다.

60초 쿨다운스팸 방지
하위 기능 2

딥 링크 자동 인증 (모드 B 화면)

메일 안의 링크를 누르면 앱이 토큰과 함께 다시 열린다. 화면은 자동으로 검증을 시도하고 진행 / 성공 / 실패 세 가지 상태 중 하나를 보여준다. 실패 시에는 “재전송”과 “로그인 복귀” 두 갈래만 제공해 사용자가 절대 막다른 길에 갇히지 않게 한다.

진행·성공·실패
하위 기능 3

토큰 생명주기 관리

토큰은 한 번 사용되거나 재발송되면 무효화된다. 새 발송 시 사용자 단위로 이전 토큰을 일괄 삭제하고 새 토큰을 발급해 “동시에 살아있는 토큰은 최대 1개” 규칙을 보장한다. 24시간 지난 토큰은 만료 에러로 응답하여 클라이언트가 재발송을 안내한다.

사용자 시나리오 10개

S1 · Happy Path

가입 직후 메일 발송 + 링크 클릭으로 인증 완료

페르소나 — 신규 가입자, 토큰 보유, 이메일 미인증.

  1. 인증 안내 화면 진입 — “{이메일}로 인증 메일을 보냈습니다”
  2. 메일 앱에서 인증 메일 수신
  3. 메일 안 링크 탭 → 딥 링크로 앱 재진입(토큰 첨부)
  4. 화면이 자동으로 검증 시작
  5. 서버: 토큰 발견 → 미만료 → 미사용 → 인증 플래그 갱신, verifiedAt 기록
  6. “이메일 인증이 완료되었습니다” + “로그인으로 이동” 버튼
  7. 사용자가 “로그인으로 이동” → 로그인 화면

종료 — 사용자의 인증 플래그 = 참, 메인 진입 가능.

S2 · 재발송

메일 미수신, 60초 후 재발송 요청

페르소나 — 스팸 처리되어 메일을 못 받은 사용자.

  1. “인증 이메일 재발송” 탭
  2. 서버: 기존 토큰 일괄 삭제 → 새 토큰 INSERT → 메일 재발송
  3. “메일을 다시 보냈습니다” 토스트 + 60초 카운트다운 시작
  4. 카운트다운 중에는 텍스트가 “인증 이메일 재발송 (45초)” 식으로 변하고 클릭 무효

종료 — 새 토큰 발급, 메일 재발송, UI에 남은 시간 표시.

S3 · 토큰 만료

만료된 토큰 링크 클릭

페르소나 — 24시간 이상 뒤에 메일을 본 사용자.

  1. 딥 링크로 자동 검증 시작
  2. 서버: 만료 확인 → 400 EMAIL_VERIFICATION_TOKEN_EXPIRED
  3. 실패 화면 + 에러 메시지 + “인증 메일 재전송” 버튼
  4. 사용자가 “재전송” → 새 토큰 발급 + 메일 재발송

종료 — 만료 안내 후 재발송 가능. 사용자가 막다른 길에 갇히지 않음.

S4 · 토큰 재사용

이미 인증된 사용자가 토큰 재사용 시도

  1. 인증 완료된 토큰 링크를 두 번째 클릭
  2. 서버: 토큰 발견 → verifiedAt이 이미 있음 → 409 EMAIL_ALREADY_VERIFIED
  3. 실패 화면 + 메시지

종료 — 변화 없음. 같은 토큰으로 두 번 인증 시도해도 안전.

S5 · 위조 토큰

잘못된·위조된 토큰

  1. URL 직접 변조 또는 임의의 UUID로 진입
  2. 서버: 토큰 조회 실패 → 400 EMAIL_VERIFICATION_TOKEN_INVALID
  3. 실패 화면 표시

종료 — 인증 실패. 주의: 만료와 위조의 에러 메시지를 사용자에게 같은 톤으로 보여 “토큰을 모아둔 사람도 차이를 못 알게” 한다.

S6 · 로그인 후 자동 발송

로그인 직후 미인증 상태에서 발송

페르소나 — 가입 후 로그아웃했다가 다시 로그인한 사용자.

  1. 화면 진입 시 자동 또는 “재발송” 탭으로 발송 요청
  2. 서버: 사용자 조회 → 미인증 확인 → 토큰 INSERT + 메일 발송

종료 — 새 토큰 발급, 메일 발송.

S7 · 이미 인증된 사용자

이미 인증된 사용자가 발송 요청

  1. “재발송” 탭
  2. 서버: 사용자 조회 → 이미 인증됨 → 409 EMAIL_ALREADY_VERIFIED
  3. 충돌 토스트

종료 — 메일 미발송. 불필요한 메일 전송과 토큰 생성을 사전에 차단.

S8 · 외부 장애

메일 서버 장애로 발송 실패

  1. 발송 요청
  2. 서버: 토큰 INSERT 성공 → 메일 전송 단계에서 예외 발생 → 트랜잭션 롤백으로 토큰도 함께 삭제
  3. 500 응답 → 클라이언트는 일반 서버 오류 토스트

종료 — DB에 “발송 실패한 토큰”이 남지 않음. 설계 의도: “메일은 안 갔는데 토큰은 활성”이라는 어긋난 상태를 만들지 않는다.

E2E-2 · 실패 surface

위조·만료 토큰 딥 링크 → 실패 화면 노출

서버 계약 한눈에

실제로 쓰는 엔드포인트는 두 개. 같은 기능이 계정 설정 경로(/users/me/email/verify/*)에도 미러링되어 있으나, 본 단위 클라이언트는 /auth/* 경로를 사용한다.

Method Path 인증 핵심 동작
POST /api/v1/auth/send-verification-email 로그인 본인 이메일로 인증 메일 발송. 기존 활성 토큰은 일괄 삭제.
GET /api/v1/auth/verify-email?token={token} 불필요 토큰으로 인증 완료. 딥 링크가 자동으로 호출.
에러 코드 매트릭스
  • 400 EMAIL_VERIFICATION_TOKEN_INVALID — 토큰이 DB에 없음(위조 또는 폐기)
  • 400 EMAIL_VERIFICATION_TOKEN_EXPIRED — 24시간 초과
  • 409 EMAIL_ALREADY_VERIFIED — 이미 인증된 사용자/토큰
  • 400 INVALID_INPUT — 사용자에게 이메일이 없음(일부 소셜 케이스)
  • 404 USER_NOT_FOUND — 사용자 미존재
  • 500 — 메일 서버 장애로 트랜잭션 롤백

자주 빠뜨리는 케이스

레드 플래그
  • 재발송 쿨다운 미표시 — “안 와요”라며 사용자가 계속 누르는 케이스. 남은 시간을 텍스트로 노출.
  • “메일은 보냈는데 토큰은 없음” 일관성 깨짐 — 메일 발송 실패 시 토큰 INSERT도 함께 롤백해야 한다.
  • 이미 사용된 토큰을 만료와 같은 에러로 응답 — 사용자는 “왜 두 번째는 안 되지?”라고 헷갈린다. 별도 코드로 구분.
  • 인증 완료 후 막다른 화면 — 성공 화면이 “로그인으로 이동” 버튼 없이 끝나면 사용자가 갇힌다. 항상 다음 행동을 제공.
  • 이미 인증된 사용자에게 메일 보내기 — 발송 단계에서 사전 차단(서버에서 409).
  • 이메일 없는 소셜 사용자에게 발송 시도 — 발송 단계에서 400으로 거부해야 한다.
  • 토큰을 평문으로 노출하는 로그 — 인증 토큰은 일시적이라도 비밀. 로그에서 마스킹.
