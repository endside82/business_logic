F01-03. 이메일 인증

가입 시 입력한 이메일이 실제로 사용자 본인의 것인지 확인한다. 인증 메일이 사용자 메일함으로 발송되고, 메일 안의 링크를 한 번 누르면 인증이 완료된다. 가입과 분리되어 있어서 사용자는 가입을 끝낸 직후가 아니라 본인이 편한 시점에 인증을 마칠 수 있다.

24시간

재발송 쿨다운
60초
스팸 처리·메일 지연 대비
링크 재사용
1회
한 번 인증된 링크는 다시 안 됨
설계 의도 — 가입을 막지 않되, 인증이 끝나기 전엔 중요한 동작을 미룬다

이메일 인증을 가입 직후로 강제하면 가입 완료율은 떨어진다. 그래서 본 서비스는 가입은 그대로 통과시키고 인증이 안 된 상태에서는 결제·신청 같은 결정 동작이 시작될 때만 안내를 띄운다. 사용자에게는 "가입은 됐지만 메일 한 번만 확인해 주세요" 정도의 가벼운 부담을 남긴다.

메일 링크 한 번 클릭이 일으키는 일

메일 안의 링크를 누르면 앱이 자동으로 열리고, 사용자가 별도 입력 없이 인증이 끝난다.

사용자 시나리오

기본 흐름

가입 직후 메일을 확인해 인증 완료

가입을 마친 사용자가 인증 화면을 본다. "이메일을 보냈습니다" 안내와 함께 "이메일 앱 열기" 버튼이 있다. 메일 앱에서 인증 메일을 찾아 링크를 누르면 앱이 자동으로 다시 열리고 짧은 진행 표시 후 "인증이 완료되었습니다" 화면이 뜬다.

이 흐름이 남기는 것 — 사용자가 직접 입력하지 않고도 인증이 끝난다. 4자리 인증 코드 입력 방식보다 사용자 실수가 적다.

메일 지연

메일이 안 와서 재발송

스팸함으로 분류되거나 메일 서버 지연으로 메일이 늦게 도착한다. 사용자는 화면의 "재발송" 링크를 누른다 — 60초 쿨다운이 시작되고 카운트다운이 보인다. 새 메일은 직전 토큰을 무효로 만들고 새 토큰으로 발송된다.

이 흐름이 남기는 것 — 사용자가 답답함을 토로하기 전에 스스로 해결할 동선을 제공. 운영 측 CS 부담을 줄인다.

만료 분기

하루 뒤에 메일을 열어봄

24시간이 지난 메일의 링크를 누르면 앱이 열리고 "인증에 실패했습니다" 화면이 나타난다. 화면에 곧장 "인증 메일 재전송" 버튼이 보여서 사용자는 한 번 누르면 새 메일을 받을 수 있다.

재사용 차단

이미 인증한 링크를 다시 누름

인증을 끝낸 후 같은 메일의 링크를 다시 눌러도 다시 인증되지 않는다. "이미 인증된 사용자입니다" 안내가 뜬다 — 한 번 사용된 토큰은 즉시 무효가 되어 노출되어도 재사용할 수 없다.

이 흐름이 남기는 것 — 인증 링크가 어디선가 노출되어도 한 번 이상은 쓰일 수 없다는 보안 약속.

미인증 진입

인증 안 한 사용자가 결제 시도

가입은 했지만 메일 링크를 안 누른 사용자가 결제 동작을 시작한다. 본 단위는 직접 차단하지 않지만, 결제 도메인이 인증 여부를 확인해서 인증 화면으로 안내한다 — 이 안내는 처음 마주치는 결정 동작에서만 뜬다.

화면에서 사용자가 인지하는 것

화면 상태사용자가 보는 것그 자리에 둔 이유
발송 안내 "{사용자 이메일}로 인증 메일을 보냈습니다" 한 줄 + 이메일 앱 열기 버튼 + 재발송 링크(쿨다운 표시) + 로그인 복귀 링크 사용자가 자기 이메일을 한 번 더 확인하고, 메일 앱으로 한 번에 갈 수 있게.
인증 진행 짧은 진행 표시와 함께 "이메일 인증 중..." 한 줄 토큰 검증이 1~2초 걸리기 때문에 시각 피드백 없이는 "앱이 멈췄나" 오해가 생긴다.
인증 성공 큰 체크 아이콘 + "이메일 인증이 완료되었습니다" + "이제 모든 기능을 사용할 수 있습니다" + 로그인으로 이동 버튼 인증의 의미를 사용자가 분명히 인지하도록 — "메일 한 번 누른 게 무슨 의미였는지" 그 자리에서 보여준다.
인증 실패 큰 경고 아이콘 + 실패 안내 + "인증 메일 재전송" 버튼 + "로그인으로 돌아가기" 링크 실패에서 곧장 다음 행동(재전송)으로 이동할 수 있는 동선이 같은 화면에 있어야 사용자가 막히지 않는다.

운영자가 알아야 할 것

메일 발송 정책과 결정점
  • 메일 도착이 늦거나 안 오는 사례는 운영 측에서 모니터링한다. 사용자 측에는 안 보이는 인프라 문제(SMTP 장애, 스팸 필터)이므로 발송 성공률과 인증 완료율을 별도로 추적해야 한다.
  • 재발송 60초 쿨다운은 메일 시스템에 부담을 주지 않기 위한 최소 간격이다. 악의적 반복 호출 차단 효과도 있다. 이 수치를 줄이면 외부 메일 서비스 차원의 발송 제한이 걸릴 위험.
  • 인증 메일의 보낸 사람 주소와 본문은 도메인 신뢰도에 직접 영향한다. 운영 측이 메일 본문·발신 도메인의 SPF/DKIM 설정을 일관되게 유지해야 사용자의 스팸 처리율이 낮아진다.
  • 인증이 가입의 강제 조건은 아니다. 결제·신청 같은 결정 동작에 한해 인증을 요구하는 정책이므로, 운영 측이 "인증해야 들어올 수 있는 화면" 목록을 별도로 관리해야 한다 — 정책이 흩어지면 사용자가 같은 화면에서 다른 경험을 한다.
