F13-02. 프로필 편집 (닉네임·자기소개·사진)

마이페이지에서 "프로필 편집"으로 들어왔을 때 사용자가 직접 닉네임·자기소개·사진을 바꾸는 화면. 사용자가 다른 사람에게 어떻게 보이는지를 본인 의지로 통제하는 곳이며, 잘못 저장되거나 변경 도중 빠져나가 작업이 사라지는 일이 없도록 두 단계의 보호 장치를 둔다 — 저장 전 입력 길이 가드, 변경된 채로 화면을 빠져나가려 할 때 띄우는 폐기 확인.

닉네임 길이
2~30자
중복 불가
자기소개
200자
선택 입력
변경 폐기 가드
있음
뒤로가기 시 확인
설계 의도 — 가벼운 편집이지만 두 가지는 양보하지 않는다

닉네임 중복과 변경 폐기 가드. 두 가지가 본 화면의 정책 핵심이다. 닉네임은 서비스 안에서 "다른 사람이 나를 부르는 이름"이라 중복을 허용하면 모집·매칭·신고 흐름이 모두 흔들리고, 한 글자 닉네임도 동일한 이유로 거른다. 변경 폐기 가드는 "사진까지 업로드해 두고 뒤로가기를 잘못 눌렀다"가 가장 큰 사용자 좌절이라 다이얼로그로 한 번 막는다.

사용자 시나리오

기본 흐름

닉네임과 자기소개를 동시에 바꾼다

편집 화면에 들어오면 현재 값이 미리 채워져 있다. 닉네임과 자기소개를 고친다. 둘 중 하나라도 바뀌면 우상단 "저장"이 켜진다. 저장을 누르면 닉네임 중복이 검사되고, 통과하면 새 값이 저장된 뒤 "프로필이 수정되었습니다" 토스트와 함께 마이페이지로 돌아간다.

이 흐름이 남기는 것 — 사용자가 다른 사람에게 보여주는 정체성을 본인 의지로 갱신할 수 있다는 통제감.

사진 교체

모임 가기 전에 최근 사진으로 바꾼다

원형 사진을 누르면 갤러리가 열린다. 사진을 고르면 자동으로 업로드되고, 저장을 눌러야 비로소 반영된다. 업로드는 됐는데 저장을 안 누른 채 뒤로가기를 누르면 변경 폐기 다이얼로그가 떠 사용자가 한 번 더 결정한다.

운영 관찰 포인트 — 카메라 직접 촬영은 현재 흐름에 없다. 사용자 요구가 많아지면 추가 검토 항목.

짧은 입력

한 글자 닉네임은 저장 자체가 안 된다

"a" 한 글자로 저장을 누르면 서버 호출 전에 클라이언트가 막고 "닉네임은 2자 이상이어야 합니다" 안내가 뜬다. 이 가드가 없으면 사용자는 서버 응답을 기다린 뒤 같은 안내를 받게 되어 대기 시간이 늘어난다.

중복 닉네임

이미 누가 쓰고 있는 닉네임

중복은 본인을 제외하고 검사한다. 다른 사용자가 이미 쓰고 있으면 저장이 실패하고 "프로필 수정에 실패했습니다" 안내가 뜬다. 사용자는 다른 닉네임을 시도해 본다.

이 흐름이 남기는 것 — 동일 닉네임이 허용되면 신고·매칭·검색에서 사람을 특정할 수 없다. 검사 단계가 무너지면 도메인 전체가 흔들리는 가드.

변경 폐기 보호

입력하던 중 뒤로가기를 잘못 누름

닉네임·자기소개·사진 어느 하나라도 바뀐 상태로 뒤로가기를 시도하면 "변경 사항을 저장하지 않고 나가시겠습니까?" 다이얼로그가 뜬다. "취소"를 누르면 편집 화면에 머물고, "나가기"를 누르면 변경분이 폐기된다.

이 흐름이 남기는 것 — 사용자가 가장 자주 불평하는 회귀("입력한 게 다 사라졌어요")를 한 단계로 막는다.

화면 구성 — 사람이 보는 것

영역보이는 것그 자리에 둔 이유
상단 앱바 "프로필 수정" 제목, 우측 "저장" 텍스트 액션. 변경 사항이 없으면 저장이 흐릿하게 표시된다. 저장 버튼이 항상 켜져 있으면 "지금 저장될 게 있는가"가 사용자에게 안 보인다. 비활성 색상으로 의도가 표현된다.
원형 사진과 카메라 아이콘 현재 프로필 사진 또는 기본 아이콘 위에 카메라 모양 오버레이. "사진 변경" 텍스트 버튼이 같이 보인다. 사진 영역 자체가 누를 수 있는 자리임을 카메라 오버레이로 분명히 한다 — 텍스트 없이 두면 사용자가 누르지 않는다.
닉네임 입력 "2~30자, 한글/영문/숫자" 안내 문구가 입력 아래에 한 줄. 저장을 누른 뒤 거절당하는 경험을 줄이기 위한 사전 안내.
자기소개 입력 여러 줄 입력 영역, 최대 200자. 마이페이지 카드의 2줄 말줄임 정책과 짝을 이룬다 — 길이 자체가 통제되지 않으면 카드가 깨진다.

운영자가 알아야 할 것

편집 화면에서 무너지면 다음 모든 표시가 어긋난다
  • 닉네임 중복 검사는 본인을 제외해야 한다. 본인을 포함하면 "내 닉네임을 그대로 저장해도 중복"이라는 회귀가 발생.
  • 변경 사항 폐기 다이얼로그를 빼면 사용자 좌절이 가장 크다. 사진 업로드까지 끝낸 상태에서 뒤로가기를 누르고 다시 들어왔을 때 입력이 사라져 있는 게 가장 흔한 불만 패턴.
  • 저장 성공 직후 인증 상태를 다시 한 번 확인해 동기화한다. 그래야 마이페이지로 돌아왔을 때 새 값이 바로 보인다 — 이 단계가 빠지면 사용자가 "왜 안 바뀌었지" 하고 새로고침을 또 시도한다.
  • 닉네임 변경 이력은 별도로 기록된다. 운영팀이 사용자를 식별할 때 닉네임만으로 추적하면 변경 이력 직전 닉네임을 놓친다.
