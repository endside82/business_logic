F03-06. 신청서 승인·거절 (호스트)

승인제 이벤트에서 호스트가 신청자를 받거나 거절하는 면. 한 번의 승인 안에서 정원 가드, 참석 확정, 신청자 알림, 그리고 유료 승인제의 결제 권한 부여까지가 결정된다. 호스트의 결정이 사용자의 비용과 직접 연결되기 때문에, 승인·거절 가드가 풀리면 환불 분쟁이 즉시 시작되는 단위.

두 호스트 동시 승인 차단
설계 의도 — 승인 한 번 안에서 결정되는 세 가지

호스트가 "승인"을 누르는 한 번의 동작 안에 (1) 정원 여유 확인, (2) 신청자의 참석 확정 처리, (3) 신청자에게 승인 알림 도달까지가 일어난다. 정원이 가득 차면 승인 자체가 차단되고, 정원이 풀리지 않은 채 승인이 일어나면 호스트가 "정원 초과 모임"을 운영하는 사고가 발생한다. 유료 승인제는 여기에 한 단계 더 — 승인은 결제 권한 부여이지 참석 확정이 아니다. 이 분리가 흐려지면 환불 분쟁이 시작된다.

호스트가 보는 면

상세 화면의 "신청서 관리" 진입점에서 들어오는 호스트 전용 화면.

탭 분류

4개 탭으로 단계별 신청을 분리

전체 / 심사 중 / 승인 / 거절. 심사 중 탭에는 새 신청 건수가 빨간 점으로 표시되어 호스트가 놓치지 않게 한다.

신청자 카드

한 카드에 필요한 결정 단서가 모임

아바타·닉네임·신청 시각·신청 메시지(2줄 + 더보기)·상태 배지. 심사 중 카드에는 "승인"·"거절" 두 버튼이, 처리된 카드에는 상태 배지만 남는다.

일괄 처리

여러 신청을 한 번에

우상단 "선택" 토글로 체크박스 노출. 여러 신청을 한 번에 승인하거나 거절. 정원 한계가 있으면 부분 처리되고 결과 요약이 토스트로 표시된다 — "3건 승인, 2건 실패 (정원 초과)" 같은 형태.

신청자 프로필

호스트가 결정하기 전에 신청자를 본다

신청자 닉네임을 누르면 간단한 프로필 미리보기 시트가 열린다. 신뢰점수와 자기소개를 보고 결정할 수 있다.

사용자 시나리오

기본 흐름

새 신청을 단건 승인

호스트가 새 신청 푸시 알림을 받고 진입. 심사 중 카드의 "승인" 버튼을 누르면 같은 동작 안에서 참석 확정 처리 + 정원 +1 + 신청자에게 푸시 알림이 모두 일어난다. 카드는 "승인됨" 배지로 바뀐다.

이 흐름이 남기는 것 — 승인 한 번이 매출과 신뢰의 출발점. 알림이 도달하지 않으면 신청자는 자기가 받아진 줄 모르고 다른 모임으로 옮긴다.

정원 가드

정원이 가득 찬 상태에서 승인 시도

같은 시점에 이미 다른 신청자가 승인되어 정원이 가득 찼다. 본 신청의 "승인"을 누르면 "정원이 가득 찼습니다. 정원을 늘리거나 다른 참석자를 제거한 후 다시 시도해주세요" 안내 + "정원 관리" 버튼이 떠오른다. 호스트가 정원을 확장하거나 다른 참석자를 제거한 뒤 다시 승인.

이 흐름이 남기는 것 — 정원 가드가 풀리면 호스트의 운영 문제가 즉시 발생한다. 본 단위의 가장 중요한 안전장치.

동시성

호스트와 공동호스트가 동시 승인

마지막 한 자리가 남은 상태에서 호스트와 공동호스트가 동시에 다른 신청자를 승인. 서버 측 잠금으로 한 명만 처리되고, 다른 한 명에게는 정원 초과 안내가 뜬다. 호스트가 정원을 확장하거나 거절을 선택.

이 흐름이 남기는 것 — 운영자가 여러 명일 때 동시성 가드가 풀리면 정원 초과가 발생. 본 단위의 두 번째 안전장치.

거절

신청자 거절

"거절" 버튼 → 확인 → 거절 처리. 신청자에게 푸시 알림이 도달하고, 신청자의 상세 화면 액션바는 "다시 신청" 버튼으로 바뀐다. (현재 거절 사유 입력은 지원하지 않음.)

이 흐름이 남기는 것 — 거절도 사용자에게 알림이 도달해야 다음 모임으로 사용자가 빠르게 옮겨 갈 수 있다. 거절 알림 누락이 사용자 이탈로 이어짐.

유료 승인제

유료 승인제 — 승인은 결제 권한 부여

유료 + 승인 필요 이벤트는 본 단위에서 가장 분쟁이 자주 일어나는 흐름. 승인 즉시 참석 확정이 아니다. 승인 후 신청자에게 "결제 필요" 상태로 전환되고, 결제 기한 안에 결제가 일어나면 그제서야 참석 확정. 결제가 일어나지 않으면 예약 정원이 반환된다.

이 흐름이 남기는 것 — "승인 = 참석 확정"으로 처리하면 거절 사용자에게 환불 처리가 발생하고 호스트의 운영 비용이 폭증. 현재 구현은 승인 즉시 참석 확정을 만드는 회귀가 일부 존재해서, 유료 승인제 정책을 안전하게 운영하려면 향후 보강이 필요한 영역 (자세한 사용자 측 흐름은 F03-05로 위임).

일괄 처리

5건을 한 번에 승인 → 부분 성공

심사 중 5건을 일괄 선택 후 "일괄 승인". 정원 여유가 3자리뿐이면 처음 3건만 승인되고 나머지 2건은 "정원 초과"로 멈춤. 결과 토스트 "3건 승인, 2건 실패" + 정원 관리 안내.

이 흐름이 남기는 것 — 일괄 처리 결과를 사용자(호스트)가 명확히 인지하지 않으면 "왜 다 안 됐지" 혼란이 누적. 부분 성공의 결과 요약이 본 동선의 핵심.

신청자 자가 취소

심사 중 사용자가 본인 취소

신청자가 직접 본인 신청을 취소한 경우, 호스트 화면에 카드가 "취소됨" 배지로 바뀐다. 호스트는 추가 액션 없이 단순 표시만.

권한

일반 사용자가 신청서 관리 화면 진입

호스트와 공동호스트가 아닌 사용자가 화면 URL로 진입해도 "권한이 없습니다" 안내만 보인다. 우회 호출도 서버에서 차단.

이미 처리됨

이미 처리된 신청을 다시 처리 시도

승인됨/거절됨 카드의 버튼은 화면에서 사라진다. 우회해도 "이미 처리된 신청입니다" 안내로 차단. 호스트가 잘못 누르는 사고를 막는다.

운영자가 알아야 할 것

신청서 처리의 비즈니스 리스크
  • 유료 승인제는 승인 = 결제 권한 부여여야 한다. 승인 즉시 참석 확정으로 처리하면 거절 사용자에게 환불 처리가 필요해진다. 현재 구현은 이 분리를 완전히 보장하지 않아 환불 분쟁의 잠재 영역. 운영 측에서 유료 승인제 이벤트는 별도 모니터링 필요.
  • 정원 가드는 어떤 경우에도 풀려서는 안 된다. 정원 초과 승인이 한 번이라도 일어나면 호스트는 운영 현장에서 "신청한 사용자를 받지 못하는" 사고를 겪는다.
  • 동시성 가드 — 두 호스트의 동시 승인. 호스트 + 공동호스트 운영 모델에서 매번 발생할 수 있는 시나리오. 한 명만 처리되고 다른 한 명에게는 정원 초과 안내가 떠야 한다.
  • 승인·거절 알림 도달. 알림이 누락되면 신청자는 자기 상태를 모른 채 다음 모임으로 옮긴다. 알림 도달률은 본 단위의 핵심 신호.
  • 대기열 활성 이벤트에서도 호스트의 "승인"은 참석 확정만 만든다. "정원 초과면 자동 대기열"은 사용자의 신청 경로(F03-05)에서만 일어나고, 호스트의 승인 경로에서는 자동 대기열로 전환되지 않는다. 정원 초과면 무조건 거절 또는 정원 확장 동선으로 이동해야 한다.
  • 거절 사유 입력은 현재 미지원. 추후 호스트가 거절 이유를 사용자에게 안내할 수 있는 경로를 열어 두는 것이 분쟁 감소에 도움. 운영 분석에서 거절 후 사용자 이탈이 크면 우선 도입.

측정해야 하는 것

신청서 처리가 정상이라고 부르려면
  • 신청 도착 → 호스트 처리까지의 시간 — 호스트의 응답 시간. 길수록 신청자 이탈이 늘어난다. 시간이 길면 알림·진입점 노출 위치 점검 필요.
  • 승인 → 신청자 인지(상세 재진입)까지의 시간 — 푸시 알림 도달의 직접 신호.
  • 승인 시도 → 정원 초과 차단 발생률 — 호스트가 정원 가드를 자주 마주치면 정원 설정 UX가 정원 패턴과 어긋난다는 신호.
  • 유료 승인제의 승인 → 결제 완료 비율 — 승인 후 결제 누수의 직접 측정. 결제 기한·결제 안내의 적합성 신호.
  • 일괄 승인의 부분 성공률 — 부분 성공이 자주 발생하면 호스트가 정원 한계를 모른 채 일괄 처리를 누른다는 신호.
