F03-06. 신청서 승인·거절 (호스트)

승인제 이벤트의 호스트가 PENDING 신청을 단건/일괄로 처리한다. 승인 시점에 자동으로 정원·대기열·결제 흐름과 맞물리고, 정원 가득 시 안전하게 차단된다. 공동호스트와 동시에 작업할 때도 정원 초과가 일어나지 않게 비관적 잠금으로 보호된다.

이 기능 안의 하위 항목

동작 1

탭별 신청 목록

전체/심사중/승인/거절 4탭.

동작 2

단건 승인/거절

승인 시 ATTENDING INSERT + 알림.

동작 3

일괄 승인

다중 선택 → 순차 처리, 부분 실패 집계.

동작 4

유료 승인 → 결제 대기

승인은 결제 권한 부여, 결제 후 ATTENDING 확정(정책 목표).

승인 처리 흐름

flowchart TD A["승인 탭"] --> B["비관적 잠금 + event 조회"] B --> C{"PENDING?"} C -->|"아니오"| E1["400 APPLICATION_INVALID_STATUS"] C -->|"예"| D{"host 또는 cohost?"} D -->|"아니오"| E2["403 UNAUTHORIZED"] D -->|"예"| E{"정원 여유?"} E -->|"가득"| E3["409 CAPACITY_FULL"] E -->|"여유"| F["APPROVED + ATTENDING INSERT"] F --> G["capacity++ · 알림 발송"]

사용자 시나리오 11개

S1 · Happy Path

단건 승인

  1. 알림 푸시 → 신청서 화면 진입
  2. “승인” 탭 → ATTENDING + capacity++
  3. 신청자에게 APPLICATION_APPROVED 푸시
S2 · 정원 가득

승인 시도 → 409

  1. 정원 가득 + 대기열 OFF 상태에서 승인
  2. 409 + “정원 관리” 안내 다이얼로그
  3. 호스트가 정원 확장 후 재승인
S3 · 정책

승인 경로는 ATTENDING만

  1. 호스트의 승인은 대기열로 자동 등록하지 않음
  2. 대기열 자동 등록은 사용자 신청 경로(apply)에서만
S4 · 거절

단건 거절 + 알림

  1. “거절” 확인 → REJECTED 저장
  2. APPLICATION_REJECTED 푸시 → 사용자 액션바 “다시 신청”
S5 · 일괄

5건 일괄 승인

  1. 다중 선택 → “일괄 승인”
  2. 순차 호출 중 정원 초과 시 멈춤
  3. “3건 승인, 2건 실패” 집계 토스트
S6 · 재처리 차단

이미 처리된 신청

  1. 버튼은 UI 숨김
  2. 우회 시 400 APPLICATION_INVALID_STATUS
S7 · 권한

비호스트 진입 차단

  1. 403 UNAUTHORIZED
  2. “권한이 없습니다” + 돌아가기
S8 · 자가 취소

신청자가 PENDING 취소한 카드

  1. 호스트 목록에 “취소됨” 그레이아웃
  2. 호스트는 추가 액션 없음
S9 · path 변조

이벤트 ID 불일치

  1. 다른 이벤트의 application ID로 호출
  2. 400 INVALID_REQUEST
S10 · 동시성

호스트 + 공동호스트 동시 승인

  1. 비관적 잠금으로 직렬화
  2. 한 쪽 성공 → 다른 쪽은 정원 초과 시 409
  3. 안전한 정원 초과 방지
S11 · 유료 정책

승인 → 결제 대기 → ATTENDING

  1. 유료 승인제에서 승인은 결제 권한 부여
  2. APPROVED_PENDING_PAYMENT + paymentDueAt 기록(정책 목표)
  3. 결제 성공 시 ATTENDING 확정, 만료 시 예약 정원 반환

현재 갭: 일부 경로는 승인 즉시 ATTENDING. 결제 검증 강화 예정.

자주 빠뜨리는 케이스

레드 플래그
  • 비관적 잠금 없는 승인 — 동시 승인 시 정원 초과 발생.
  • 승인 시 알림 누락 — 사용자가 결과를 모름.
  • 일괄 승인 중 실패를 침묵 — 결과 집계 토스트 필수.
  • 유료 승인이 즉시 ATTENDING 만드는 경로 — 결제 권한 부여로 분리되어야 한다.
