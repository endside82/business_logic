F01-04. 비밀번호 재설정

비밀번호를 잊은 사용자가 본인 이메일로 받은 30분 짜리 토큰을 써서 비밀번호를 다시 설정한다. 비로그인 상태에서 처리되며, 보안상 “이메일이 가입돼 있는지”는 절대 외부에 노출하지 않는다. 미가입 이메일에도 가입자와 똑같은 “이메일을 확인해주세요” 화면을 보여주는 것이 의도된 정책.

핵심 보안 정책 세 가지
  1. 이메일 존재 여부 비노출 — 미가입 이메일에도 가입자와 동일한 화면을 보여준다. 응답도 200 동일.
  2. 토큰과 이메일 교차 검증 — 토큰을 가로채도 본인 이메일이 아니면 변경 불가.
  3. 일회용 토큰 — 사용 즉시 폐기 + Redis TTL 30분.

이 기능 안의 하위 항목

하위 기능 1

재설정 요청 화면

이메일 한 칸과 “재설정 링크 발송” 버튼. 성공이든 미가입이든 결과 화면은 같은 “이메일을 확인해주세요” 안내로 통일된다. 사용자는 메일이 안 와도 화면상으로는 차이를 알 수 없다.

하위 기능 2

재설정 확인 화면

딥 링크로 진입한다. 새 비밀번호와 확인 입력을 받고 강도·일치를 실시간 검증. “비밀번호 변경” 성공 시 별도 자동 로그인을 시키지 않고 로그인 화면으로 명시 이동하여 새 비밀번호로 다시 로그인하도록 유도한다.

사용자 시나리오 10개

S1 · Happy Path

비밀번호 분실 후 정상 재설정

  1. 로그인 화면 → “비밀번호 찾기” 탭
  2. 이메일 입력 → “재설정 링크 발송” 탭
  3. “이메일을 확인해주세요” 안내로 화면 전환
  4. 메일 앱에서 링크 탭 → 딥 링크로 확인 화면 진입
  5. 새 비밀번호 입력 + 확인 입력 → “비밀번호 변경”
  6. 서버: 토큰 검증 → 이메일 교차 검증 → 비밀번호 해싱 후 UPDATE → 토큰 폐기
  7. “변경 완료” 화면 → “로그인으로 이동”

종료 — 비밀번호 갱신, 토큰 폐기, 사용자 재로그인 유도.

S2 · 보안 정책

미가입 이메일로 재설정 요청

  1. 가입한 적 없는 이메일 입력 → “재설정 링크 발송”
  2. 서버: 사용자 미발견 → 조용히 종료, 200 OK 반환
  3. 화면은 가입자와 동일한 “이메일을 확인해주세요” 안내

종료 — 메일은 실제로 발송되지 않음. 이메일 존재 여부 노출 차단이 목적.

S3 · 만료

30분 지난 토큰으로 변경 시도

  1. 30분 후 메일 링크 클릭 → 확인 화면 → 새 비밀번호 입력 → 변경
  2. 서버: Redis TTL 만료로 자동 삭제됨 → 400 TOKEN_INVALID
  3. 실패 토스트, 사용자가 다시 “비밀번호 찾기” 진입

종료 — 변경 실패, 재요청으로 새 토큰 발급.

S4 · 토큰 탈취 차단

공격자가 가로챈 토큰으로 다른 이메일 변경 시도

  1. 공격자가 다른 사용자의 토큰을 입수
  2. 본인 이메일과 함께 변경 요청
  3. 서버: 토큰의 userId로 사용자 조회 → 저장된 이메일과 요청 이메일 비교 → 불일치 → 400 INVALID_INPUT

종료 — 계정 탈취 차단. 토큰 단독 보유는 충분하지 않다는 정책이 동작.

S5 · 재사용 차단

같은 토큰 두 번 사용 시도

  1. 첫 번째 변경 성공 → Redis에서 토큰 즉시 DELETE
  2. 같은 토큰으로 두 번째 변경 시도 → Redis에 없음 → 400 TOKEN_INVALID

종료 — 재사용 차단. 일회용 토큰 정책.

S6 · 정책 위반

약한 비밀번호로 변경 시도

  1. 클라이언트는 “8자 + 영문/숫자”까지만 검증해 통과시킴
  2. 서버가 추가로 대소문자·특수문자 패턴까지 검증 → 422 VALIDATION_ERROR
  3. 실패 토스트

종료 — 변경 실패. 이중 방어: 클라이언트 우회 시도에도 서버가 막아준다.

S7 · 외부 장애

메일 서버 장애로 발송 실패

  1. 요청 → Redis SET 성공 → 메일 발송 단계에서 예외
  2. 본 단위 요청 메서드에는 별도 쓰기 트랜잭션이 없어 Redis SET은 자동 롤백되지 않음
  3. 사용자는 메일이 오지 않아 다시 요청 → 새 토큰 발급
  4. 이전 토큰은 30분 후 자연 만료

종료 — 사용자가 재시도로 복구 가능. 운영 노트: 메일 발송 실패율이 높아지면 “미사용 토큰이 30분 동안 살아있는 상태”가 누적될 수 있어 모니터링 후보.

S8 · 보안 강화 후보

비밀번호 변경 후에도 다른 기기 세션 유지

  1. 비밀번호 변경 성공
  2. 그러나 본 단위는 다른 기기의 리프레시 토큰을 폐기하지 않음 → 기존 세션 유지됨

종료 — 다른 기기는 새 비밀번호 없이도 기존 토큰으로 계속 사용 가능. 운영 결정 필요: 비밀번호 변경 시 모든 세션 강제 종료를 추가할지 여부는 향후 보안 강화 항목.

E2E-1 · 요청 surface

재설정 요청 → 발송 안내 surface 노출

  1. 로그인 화면 도달 후 재설정 화면으로 이동
  2. “비밀번호 찾기” 헤더 + 이메일 입력 필드 가시
  3. 시드 이메일 입력 후 “재설정 링크 발송”
  4. 15초 이내 “이메일을 확인해주세요” 헤더 등장
  5. 입력한 이메일 텍스트 노출 + “로그인으로 돌아가기” CTA 가시

종료 — 발송 안내 surface 안정 노출.

E2E-2 · 확인 검증 surface

확인 화면의 인라인 검증 (8자 미만 + 불일치)

  1. 딥 링크로 확인 화면 진입
  2. 새 비밀번호에 “short” 입력 → “최소 8자 이상 입력해주세요” 표시
  3. 확인 비밀번호에 다른 값 입력 → “비밀번호가 일치하지 않습니다” 표시
  4. “비밀번호 변경” CTA는 존재하나 실패 입력으로는 전송되지 않음

종료 — 두 인라인 에러 동시 노출, 사용자가 수정할 때까지 변경 차단.

자주 빠뜨리는 케이스

레드 플래그
  • 미가입 이메일에 다른 응답을 주는 회귀 — 에러 코드·메시지·화면 어느 것이라도 달라지면 이메일 enumeration 공격에 노출.
  • 토큰·이메일 교차 검증 누락 — 토큰만 가지면 누구나 어느 계정 비밀번호든 바꿀 수 있게 된다.
  • 토큰을 GET만 하고 DELETE 안 함 — 같은 토큰으로 두 번 변경되는 회귀.
  • 변경 후 자동 로그인 — 본 단위는 자동 로그인을 시키지 않는다. 명시적 재로그인을 통해 새 비밀번호가 동작하는지 사용자가 확인하게 한다.
  • 비밀번호 정책을 클라이언트만 검증 — 서버에서도 동일/더 엄격한 검증을 적용해야 한다.
  • 30분 카운트다운을 화면에 보여줌 — 의도적으로 표시하지 않는다. 시간 정보는 메일 본문에만.
