F01-04. 비밀번호 재설정

비밀번호를 잊은 사용자가 이메일 한 번으로 계정 통제권을 회복할 수 있게 한다. 가입자에게는 메일이 발송되지만, 누가 가입자인지 화면상으로는 절대 드러내지 않는다. 메일로 받은 링크는 30분 동안만 유효하고 한 번 사용된 토큰은 즉시 폐기된다.

설계 의도 — 본인 비밀번호 회복을 막지 않되, 그 과정이 공격 통로가 되지 않게 한다

비밀번호 재설정은 본인이 계정 통제권을 회복하는 가장 빠른 길이지만, 동시에 공격자가 다른 사람의 계정에 들어갈 수 있는 통로이기도 하다. 그래서 본 단위는 두 가지를 동시에 지킨다 — (1) 미가입 이메일을 입력해도 화면은 가입자와 똑같이 "메일을 보냈습니다" 안내를 보여서 누가 가입자인지 알 수 없게 하고, (2) 메일 링크의 토큰만으로는 비밀번호를 못 바꾸게 하고 사용자가 본인 이메일을 한 번 더 입력해야 한다.

두 단계로 분리된 흐름

메일 요청 단계와 비밀번호 변경 단계는 다른 시점에 일어난다. 사이 시간은 사용자가 메일함을 확인하는 시간.

사용자 시나리오

기본 흐름

비밀번호를 잊어 새로 설정

로그인 화면의 "비밀번호 찾기"를 누른 사용자가 자기 이메일을 입력하고 "재설정 링크 발송"을 누른다. "이메일을 확인해주세요" 안내가 뜬다. 메일에서 링크를 누르면 새 비밀번호 입력 화면이 열리고, 두 번 입력해 일치를 확인한 뒤 변경하면 로그인 화면으로 돌아간다 — 새 비밀번호로 다시 로그인한다.

이 흐름이 남기는 것 — 사용자가 본인 계정 통제권을 회복하는 가장 짧은 경로. CS 문의의 가장 큰 비중을 차지하는 비밀번호 관련 부담을 셀프 서비스로 흡수한다.

보안 정책

가입 안 한 이메일을 입력

오타로 다른 이메일을 입력하거나 가입한 적 없는 이메일을 시도해도 화면에는 똑같이 "이메일을 확인해주세요" 안내가 뜬다. 메일은 발송되지 않지만 사용자 측에서는 가입자 입력과 구분할 수 없다.

이 흐름이 남기는 것 — 공격자가 "이 이메일이 가입자냐"를 알아내는 통로가 차단된다. 본 단위의 가장 중요한 보안 약속.

만료

30분 지난 링크를 사용

한 시간 뒤에 메일을 본 사용자가 링크를 누른다. 비밀번호 입력 화면까지는 진입하지만 변경을 시도하면 거절된다 — "다시 요청해 주세요" 안내가 떠오르고 사용자는 비밀번호 찾기를 다시 한다.

토큰 탈취 차단

공격자가 메일 링크를 가로채 본인 이메일로 변경 시도

공격자가 사용자 A의 재설정 링크를 어떤 경로로 가져왔다고 해도, 변경 화면에서 본인의 이메일을 입력해서 통과할 수는 없다 — 토큰에 묶인 사용자의 이메일과 입력 이메일이 같아야만 변경이 진행되기 때문.

이 흐름이 남기는 것 — 메일 링크 자체만으로는 계정을 탈취할 수 없다는 보안 약속. 메일 노출 사고의 영향을 차단.

재사용 차단

같은 링크로 두 번 변경 시도

한 번 비밀번호를 바꾸고 나면 같은 메일의 같은 링크는 즉시 무효가 된다 — 다시 누르면 거절된다. 필요하면 사용자가 비밀번호 찾기를 다시 진행한다.

검증

새 비밀번호가 정책에 안 맞음

화면에서 즉시 검증이 일어난다 — 8자 미만이거나 확인 칸과 다르면 인라인 안내가 뜨고 변경 버튼이 비활성화된다. 사용자가 검증을 통과해도 서버에서 한 번 더 정책 검사를 하므로, 가입과 동일한 강도의 비밀번호가 강제된다.

운영자가 알아야 할 것

보안 약속과 정책 결정점
  • "이메일을 확인해주세요" 안내는 가입 여부와 무관하게 동일해야 한다. 만약 미가입 이메일에 다른 안내가 뜨는 회귀가 생기면 공격자가 가입자 목록을 추출할 수 있다 — 본 단위의 가장 중요한 보안 약속.
  • 토큰 30분 만료는 이메일 인증(24시간)보다 짧다. 비밀번호 변경이 더 민감한 동작이라는 정책의 발현. 운영 측이 이 수치를 늘릴 때는 보안 영향 평가가 필요하다.
  • 변경 화면에서 본인 이메일을 다시 입력받는 정책은 토큰 탈취 방어선이다. UX 개선 요청으로 이 단계를 없애자는 제안이 들어오면 보안 약속과 충돌함을 사전 안내.
  • 비밀번호 변경 후 다른 기기 세션은 자동으로 끊지 않는다. "다른 기기에서도 자동 로그아웃되길 기대했다"는 사용자 문의가 있을 수 있다 — 보안 강화 방안의 후보로 운영 측이 인지하고 있어야 한다.
  • 메일 시스템 장애 시에도 화면 응답은 200을 유지한다. 가입 여부 노출 방지 정책 때문에 메일이 안 가도 사용자에게는 "메일을 보냈습니다"로 보인다 — 발송 실패율은 운영 측에서 별도로 모니터링해야 한다.
