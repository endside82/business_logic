F09-01. 본인 인증

데이팅을 사용하려는 모든 사용자가 가장 먼저 통과해야 하는 관문. 외부 본인인증 사업자(토스)의 절차를 거쳐 실명·생년월일·성별·고유식별값을 검증받고, 이 단계를 통과해야 비로소 프로필 작성, 후보 카드 조회, 호감 표시, 채팅까지 모든 후속 기능에 접근할 수 있다. 데이팅에서 발생할 수 있는 안전 문제는 이 단계의 강도에 의해 1차로 걸러진다.

설계 의도 — 데이팅 도메인에서 한 번 무너지면 다른 모든 안전 장치가 의미를 잃는다

데이팅은 다른 도메인과 다르다. 사용자끼리 실제로 만나서 오프라인으로 이어지는 흐름이기 때문에, 허위 신원·미성년 가입·중복 계정이 한 건이라도 통과하면 그 한 건이 곧 사고로 이어진다. 그래서 본 단위는 "데이팅을 시작하기 전에 가장 강한 관문을 한 번 통과시키되, 한 번 통과한 사용자는 1년 동안 다시 묻지 않는다"는 균형으로 설계되어 있다. 너무 자주 묻으면 사용자가 지치고, 너무 길게 두면 신원의 신뢰가 헐거워진다 — 1년은 그 사이의 결정점.

누가 영향을 받는가

본 단위는 데이팅을 처음 만나는 모든 사용자에게 가장 먼저 작동한다
  • 신규 진입자 — 데이팅 탭을 처음 누른 사람이 마주하는 첫 화면이 본 인증이다. 이 자리에서 이탈하면 데이팅 도메인 전체 사용자 수가 그만큼 줄어든다.
  • 1년 후 재사용자 — 한 번 인증했지만 1년이 지난 사용자. 다시 데이팅을 켰을 때 같은 흐름을 한 번 더 거쳐야 한다 — 이 재인증의 불편이 너무 크면 휴면 사용자가 돌아오지 않는다.
  • 운영팀 — 인증 통과율, 실패 사유 분포, 재시도 패턴을 본 단위 메트릭으로 관찰한다. 외부 사업자 측 장애가 데이팅 신규 가입에 직접 영향을 주기 때문에 이상 신호를 빠르게 잡아야 한다.

인증 한 번이 거치는 단계

사용자에게는 한 동작처럼 보이지만, 그 안에서 외부 사업자 호출과 개인정보 암호화가 순서대로 일어난다.

한 사용자의 인증은 시간에 따라 다음 네 가지 상태를 거친다.

사용자 시나리오

기본 흐름

데이팅을 처음 켜는 사용자

데이팅 탭에 처음 진입한다. 후보 카드 대신 본인 인증 안내가 보인다. "본인 인증하기"를 누르면 토스 인증 페이지가 화면 안에 열린다. 통신사·이름·생년월일·휴대폰을 입력하고 문자 인증까지 마치면, 화면이 자동으로 닫히고 "본인인증이 완료되었습니다" 토스트가 뜬다. 뒤로 가서 다시 데이팅 탭으로 들어오면, 이번에는 후보 카드 또는 프로필 작성 안내로 바로 넘어간다.

이 흐름이 남기는 것 — 데이팅의 모든 후속 기능이 이 한 번의 통과를 전제로 한다. 첫 인증까지의 시간이 짧을수록 가입 후 첫 매칭까지의 시간도 짧아진다.

중도 이탈

인증 페이지를 띄웠다가 닫음

토스 인증 화면이 열렸지만 사용자가 X를 눌러 닫는다. 화면은 다시 안내 화면으로 돌아오고, 사용자에게 별다른 강제 안내는 뜨지 않는다. 필요하면 사용자가 같은 자리에서 다시 시도할 수 있다.

이 흐름이 남기는 것 — 사용자가 마음을 바꿀 권리를 막지 않는다. 한 번 닫은 사용자가 다른 시간에 돌아와 다시 시도하는 게 자연스러운 사용자 흐름.

반복 실패 차단

하루에 다섯 번 넘게 시도

같은 사용자가 하루에 5번까지 인증 페이지를 띄울 수 있다. 여섯 번째 시도부터는 인증 요청이 차단되고, 클라이언트는 "본인인증 요청에 실패했습니다" 토스트만 보여 준다. 다음 날이 되면 자동으로 풀린다.

이 흐름이 남기는 것 — 외부 인증 사업자에게 무한 호출이 가지 않도록 막는다. 동시에 사용자에게는 "내일 다시 시도"라는 인지로 자연스러운 정리.

외부 장애

외부 사업자가 일시적으로 응답하지 않는다

사용자는 인증 페이지에서 정상 입력을 마쳤지만, 서버가 결과를 받아오는 단계에서 외부 사업자가 응답하지 않는다. 서버는 짧게 세 번 다시 시도한 뒤 "인증 확인에 실패했습니다. 다시 시도해주세요." 라는 안내로 사용자에게 결과를 돌려준다. 사용자의 인증 상태는 "진행중"으로 남아 있어, 잠시 후 같은 동작을 다시 시도할 수 있다.

이 흐름이 남기는 것 — 외부 장애가 사용자 한 명의 신원에 영원한 흔적을 남기지 않는다. 잠시 후 다시 시도하면 그대로 통과한다.

1년 후 재인증

작년에 인증한 사용자가 다시 데이팅을 켰을 때

작년에 데이팅을 잠깐 써 보고 나갔다가 1년 뒤에 다시 들어온 사용자. 앱은 만료 시점을 인지하고 "본인 인증이 만료되었습니다" 화면과 만료일을 함께 보여 준다. "다시 인증하기"를 누르면 첫 인증과 동일한 흐름이 진행되고, 통과하면 또 다시 1년 동안 데이팅을 사용할 수 있다.

이 흐름이 남기는 것 — 휴면 사용자가 돌아왔을 때 안전 검증을 한 번 더 거치되, 새 가입처럼 모든 정보를 다시 받지는 않는다 — 데이팅 프로필 본문은 그대로 살아 있다.

이미 인증된 사용자

호기심에 다시 인증을 시도

이미 인증 완료 상태에서 사용자가 "본인 인증하기" 버튼을 다시 누르는 경우. 서버는 재인증이 필요 없다는 의미로 요청을 막는다. 사용자에게는 "본인인증 요청에 실패했습니다" 안내가 뜬다.

운영 관찰 포인트 — 현 안내는 "이미 인증되어 있어 추가 인증이 필요 없습니다" 같은 적극적 메시지가 아니다. 사용자가 혼란스러워하는 신호가 들어오면 안내 카피의 분기를 따로 둘 여지 있음.

화면에서 사용자가 인지하는 것

화면 요소사용자가 보는 것그 자리에 둔 이유
안내 헤더 "안전한 데이팅을 위해 본인 인증이 필요합니다" 라는 한 줄과 보조 설명. 인증을 단순한 행정 절차가 아닌 "내 안전을 위한 절차"로 사용자에게 다시 한 번 자리매김한다. 본인 인증을 "왜 하라는 거지?"의 의문 없이 자연스럽게 받아들이게.
안전 체크리스트 세 줄 "실명 인증으로 안전한 매칭 / 나이·성별 자동 확인 / 개인정보는 인증 목적으로만 사용". 사용자가 가장 자주 갖는 세 가지 의심에 대한 답을 미리 둔다. 동의 화면이 아니라 안내 화면이지만, 사용자가 본인 정보를 넘기는 것에 대한 심리적 저항을 줄이는 자리.
본인 인증하기 버튼 화면 하단에 고정된 큰 버튼. 누르면 "인증 요청 중..."으로 잠깐 라벨이 바뀐 뒤 토스 인증 페이지가 자리에 들어온다. 의사결정 동작이 단 하나여야 한다. 인증을 하느냐, 안 하느냐 — 다른 선택지를 한 줄도 두지 않는다.
인증 페이지 안착 풀스크린 모달이 아니라 같은 화면의 본문 자리에 인증 페이지가 자연스럽게 들어온다. "외부 페이지로 떠난다"는 인상을 주지 않는다. 사용자에게는 "같은 앱 안에서 한 단계 진행 중"으로 보이는 게 자연스럽다.
만료 안내 화면 1년이 지난 사용자에게만 보이는 ⏰ 아이콘 + 만료일 + "다시 인증하기" 버튼. "왜 갑자기 또?" 라는 의문에 미리 답한다. 만료일을 직접 보여 줘 사용자가 시간 흐름을 자기 눈으로 확인하게 한다.
실패 사유 카드 실패 시 노출 — "입력한 정보가 일치하지 않을 수 있습니다 / 일시적 장애일 수 있습니다 / 일일 시도 횟수를 초과했을 수 있습니다" 세 가지 후보. 실패 사유를 단정하지 않는다. 정확한 원인은 외부 사업자에게 있어 알기 어렵기 때문 — 사용자가 자신의 상황과 대조해 다음 행동을 정할 수 있게 후보를 나열한다.

측정해야 하는 것

본 단위가 건강하다고 말하려면 이 네 가지 신호가 추적되어야 한다
  • 데이팅 첫 진입 → 인증 완료 전환율 — 신규 사용자가 인증을 거쳐 데이팅 사용자가 되는 비율. 이 숫자가 떨어지면 본 단위의 안내·인증 흐름 자체가 진입 장벽이 되고 있다는 신호.
  • 인증 요청 → 인증 완료 시간 — 사용자가 "본인 인증하기"를 누른 시점부터 완료 토스트까지의 시간. 너무 길면 외부 환경 또는 사용자의 입력 단계 어딘가에서 막힌다.
  • 실패 사유 분포 — "일치 안 함 / 외부 장애 / 일일 한도 초과" 어느 사유가 늘어나는지. 외부 장애 비중이 커지면 외부 사업자 측 상태를 의심한다.
  • 1년 만료 → 재인증 회수율 — 만료된 휴면 사용자가 돌아와 다시 인증하는 비율. 너무 낮으면 만료 → 재인증 흐름 자체가 사용자 이탈 지점이 되고 있을 가능성.

운영자가 알아야 할 것

본 단위에서 신뢰가 무너지면 데이팅 도메인 전체가 신뢰를 잃는다
  • 외부 본인인증 사업자에 강하게 의존한다. 사업자 측 장애가 그대로 데이팅 신규 가입 정지로 나타난다 — 외부 상태 모니터링이 본 도메인 모니터링과 함께 가야 한다.
  • 일일 5회 한도는 사용자 1명 단위다. 같은 사람이 빠르게 다섯 번 시도하면 그날 안에는 더 이상 못 한다. 이 한도가 너무 낮다는 신호가 들어오면 조정 여지 있음 — 다만 외부 사업자 호출 비용·악용 방지 측면에서 균형이 필요.
  • 저장된 개인정보는 모두 암호화되어 있다. 실명·생년·휴대폰·고유식별값 어느 것도 평문으로 노출되지 않는다. 운영팀이 사용자 식별이 필요할 때도 별도 권한 절차를 거친다.
  • 1년 만료 처리가 정확히 동작하지 않으면 두 가지 위험이 동시에 발생. 너무 빨리 만료시키면 멀쩡한 사용자가 갑자기 데이팅에서 튕겨 나가고, 너무 늦게 만료시키면 신원이 바뀐 사용자가 통과 상태로 머문다.
  • 본 인증 통과는 데이팅 모든 후속 기능의 전제다. 프로필 작성, 후보 카드 조회, 호감 표시, 채팅 어느 한 곳이라도 인증 가드가 풀리는 사고가 발생하면 안전 보장이 전 도메인에서 의미를 잃는다.
