F07-04. 정산 현황 · 요약 · 영수증

정산이 어떻게 굴러가고 있는지를 호스트와 참가자가 함께 확인하는 화면. 총액·1인당·결제 진척률 같은 요약과, 참가자별 결제 상태, 영수증 사진이 한 자리에 모인다. 호스트는 "지금 누가 보냈지"를 확인하고, 참가자는 "내가 얼마를 내야 하는지"와 "영수증이 진짜인지"를 확인한다. 모든 상태(작성 중 · 진행 중 · 완료 · 취소됨)에서 같은 화면이 열리며, 상태에 따라 보이는 행동 버튼만 달라진다.

설계 의도 — 같은 진실을 호스트와 참가자가 같은 화면에서 본다

"내가 보낸 돈이 도착했는지"와 "누가 아직 안 보냈는지"는 호스트와 참가자의 관심사가 다르지만, 그 기반이 되는 데이터는 같다. 둘이 서로 다른 화면을 보면 "나는 보냈는데 호스트는 안 받았다 한다"는 분쟁이 생긴다. 본 단위는 같은 데이터를 두 페르소나에게 공유하고, 각자에게 의미 있는 행동 버튼만 노출하는 방식으로 신뢰의 기반을 만든다.

화면에 늘 함께 있는 세 가지 정보

요약

이 정산의 한 줄

총액, 1인당 금액, 결제 진척률(예: 2/5명, 40%). 호스트가 화면을 열자마자 "지금 어디까지 왔는지" 한눈에 보이는 자리. 참가자에게도 본인이 속한 정산의 규모가 즉시 인지된다.

참가자별 상태

누가 얼마를 냈고 누가 아직인지

참가자 한 명당 카드 한 장. 분담 금액, 결제 완료 여부, 완료 시각이 함께 보여 호스트가 누구의 응답을 기다리는지 알 수 있다. 본인은 카드에서 자기 분담만 강조되어 한눈에 찾힌다.

영수증 갤러리

이 정산이 근거하는 영수증들

호스트가 항목 입력 시 첨부한 영수증 사진의 썸네일. 누르면 핀치 줌이 가능한 뷰어로 확대. "이 금액이 진짜였느냐"의 분쟁이 생기면 이 자리가 답이 된다.

사용자 시나리오

호스트 · 진척 확인

활성화 직후 "누가 보냈는지" 1분마다 새로고침

호스트가 정산을 활성화하자 알림이 모두에게 나갔다. 잠시 후 본인 화면을 아래로 당겨 새로고침. 요약 카드의 진척률이 "0/5명"에서 "2/5명 (40%)"으로 바뀌고, 참가자 카드 중 두 장에 "완료" 배지와 완료 시각이 떠 있다.

이 흐름이 남기는 것 — 호스트가 "지금 누구를 기다리고 있는지" 안다는 사실 자체가 운영의 통제감. 여기서 누가 안 보냈는지가 보여야 다음 단계(독촉)로 자연스럽게 넘어간다.

참가자 · 분담 확인 + 영수증 열람

알림을 받고 들어와서 본인 분담과 영수증을 본다

참가자가 정산 알림을 누르고 화면에 들어온다. 본인이 얼마를 내야 하는지가 자기 카드에 보이고, 영수증 썸네일을 누르면 전체 화면 뷰어로 사진이 떠오른다. 핀치 줌으로 가게 이름과 금액을 확인한 뒤 닫고 결제로 넘어간다.

이 흐름이 남기는 것 — 참가자가 영수증을 직접 볼 수 있어야 호스트의 청구가 "진짜"라는 믿음이 형성된다. 영수증을 못 보면 청구는 늘 한 단계 의심을 받는다.

차단 · 비참가자

이 모임에 없는 사람이 정산 정보를 보려 한다

모임에 참석하지 않은 다른 사용자가 정산 화면 주소로 들어온다. 호스트도 참가자도 아니라는 이유로 거절되고 "정산 정보를 불러올 수 없습니다" 안내가 뜬다.

이 흐름이 남기는 것 — 정산은 그 모임의 참가자만 보는 사적 정보다. 누구의 분담이 얼마인지가 외부에 노출되면 사생활 침해로 이어진다.

차단 · 임의 영수증 ID

다른 모임의 영수증 ID로 영수증을 보려 한다

사용자가 주소 끝의 영수증 식별자를 임의로 다른 값으로 바꿔서 요청한다. 그 영수증이 본인이 참여한 정산에 속한 것이 아니라는 이유로 거절된다.

이 흐름이 남기는 것 — 영수증은 정산이라는 범위 안에서만 공개되어야 한다. 파일 단위로 권한을 푸는 회귀가 들어가면 누군가의 영수증이 다른 사람에게 노출되는 사고가 난다.

작성 중 단계

참가자가 활성화 전에 들어왔다

호스트가 아직 활성화하지 않은 상태에서 참가자가 화면을 열면 "호스트가 정산을 준비 중입니다" 안내만 보이고 결제 버튼은 없다. 본인 분담 금액은 미리 노출되지만, 송금 라인이 만들어지지 않은 상태라 결제는 시작되지 않는다.

완료 단계

모든 송금이 끝난 정산의 회고

모든 송금과 분담이 완료되면 정산 상태가 자동으로 "완료"로 바뀐다. 화면 상단 라벨에 "정산이 완료되었습니다"가 뜨고, 참가자 카드는 모두 완료 배지로 표시된다. 호스트는 영수증·감사 로그를 통해 사후 점검만 한다.

취소 단계

취소된 정산의 사후 모습

취소된 정산을 열면 송금 카드들이 "취소됨", "수동 환불 대기", "환원 실패" 같은 여러 상태가 혼재한다. "수동 환불 대기" 카드에는 "계좌이체 환불 필요"라는 짧은 사유가 함께 표시된다.

이 흐름이 남기는 것 — 취소 이후에도 화면이 "지금 누가 무엇을 기다리고 있는지"를 그대로 보여 줘야 남은 환불 책임을 누구도 잊지 않는다.

실시간 갱신

호스트가 아래로 당겨 진척률을 확인한다

호스트는 결제 알림을 기다리는 동안 화면을 아래로 당겨 새로고침할 수 있다. 참가자 카드의 완료 배지가 늘어나면서 진척률이 갱신된다. 마지막 송금이 들어오는 순간 정산이 자동으로 완료 상태로 바뀐다.

화면에서 사용자가 인지하는 것

화면 요소사용자가 보는 것그 자리에 둔 이유
요약 카드 "총 80,000원, 1인당 16,000원, 2/5명 납부 (40%)" 같은 한 줄 요약. 호스트가 화면을 열자마자 "지금 어디까지 왔는지"를 먼저 보게 — 진척률이 본 화면에 들어오는 첫 인지.
참가자 카드 리스트 이름·사진·분담 금액·결제 상태(완료/대기)·완료 시각. 본인 카드는 색상이나 위치로 강조. "누가 어디까지"를 한 카드씩 보여 줘야 호스트가 다음에 누구에게 말을 걸지 결정할 수 있다.
영수증 썸네일 갤러리 항목별 영수증 사진 썸네일. 누르면 전체 화면 뷰어로 확대. 분쟁이 발생하기 전 영수증이 미리 노출되어 있어야 청구의 정당성이 확보된다.
액션 바 호스트일 때: "독촉 / 취소" (진행 중 상태). 완료 후엔 라벨만. 참가자에게는 액션 노출 없음. 같은 화면이라도 페르소나·상태에 따라 다음에 할 일이 다르다 — 보이는 행동이 페르소나에 맞아야 한다.
아래로 당겨 새로고침 표시 없이 화면을 위에서 아래로 당기면 데이터가 다시 불러와짐. 호스트가 짧은 간격으로 진척을 확인하고 싶을 때 폴링 없이도 통제감을 줄 수 있는 표준 동작.

운영자가 알아야 할 것

조회 화면이지만 가장 자주 분쟁의 기준이 되는 면
  • 호스트와 참가자가 같은 데이터를 본다. 두 화면이 다른 데이터를 보여 주는 회귀가 들어가면 "나는 보냈는데 호스트가 안 받았다 한다" 같은 분쟁이 즉시 발생한다.
  • 영수증은 정산이라는 범위 안에서만 공개된다. 파일 ID 단위로 권한을 푸는 회귀가 들어가면 다른 모임의 영수증이 노출되는 사고가 난다 — 정산에 속한 영수증인지 항상 확인.
  • 진척률은 표시 시점에 다시 계산된다. 완료 트리거는 결제 시점에 일어나지만, 화면에서 새로고침을 누르면 항상 최신 상태가 반영된다.
  • 작성 중 단계에 참가자가 들어와도 본인 분담은 보인다. 호스트가 작성을 끝내기 전 "어차피 받을 청구의 윤곽"을 참가자가 미리 보는 셈인데, 이는 의도된 노출이다 — 호스트가 작성 시점에 빠르게 활성화하도록 압박하는 효과도 있다.
  • 완료된 정산을 다시 작성 중으로 되돌리지 않는다. 완료 라벨이 한 번 떠오른 뒤에는 호스트가 손댈 동작이 사라진다 — 분쟁 처리는 이의제기·감사 로그 경로로.
