F03-11. 위시리스트 (찜)

마음에 드는 이벤트를 가벼운 한 번의 탭으로 모아 두는 면. 카드와 상세의 하트 토글은 즉시 반응해 사용자가 망설임 없이 찜할 수 있고, 프로필 탭의 "찜한 이벤트"에서 한 곳에서 다시 둘러볼 수 있다. 신청 결정 직전의 1차 후보 모음 자리.

목록 페이지 사이즈
20장
하트 토글 반응
즉시
서버 응답을 기다리지 않음
설계 의도 — 망설이지 않게, 그러나 안전하게

찜은 사용자가 가장 가벼운 마음으로 누르는 동작이다. 하트를 누른 순간 화면이 즉시 채워져야 사용자가 망설임을 느끼지 않는다. 동시에 서버에서 거절(종료된 이벤트, 비공개 클럽 이벤트의 비멤버 등)이 일어나면 자연스럽게 원래 상태로 돌아간다 — 사용자에게는 시각적으로 자연스러운 롤백.

사용자 시나리오

기본 흐름

탐색 결과에서 하트 토글

카드의 회색 하트를 누른다. 같은 순간에 하트가 빨갛게 채워지고 "찜했어요" 토스트. 서버 응답이 실패하면 자연스럽게 회색으로 돌아가고 "잠시 후 다시 시도해주세요" 안내.

이 흐름이 남기는 것 — 화면 반응이 즉시 일어나야 사용자가 망설임 없이 다음 카드로 이어 갈 수 있다. 본 단위의 응답 속도가 탐색 행동 자체의 흐름을 결정.

목록

"찜한 이벤트"에서 정리

프로필 탭 → "찜한 이벤트" → 한 번에 20장씩 카드 표시. 스크롤하면 다음 묶음. 마음에 안 드는 카드는 하트를 다시 눌러 목록에서 즉시 사라진다.

종료된 이벤트

이미 끝난 이벤트에 찜 시도

다른 화면에서 봤던 카드가 그 사이 종료된 경우, 하트를 누르면 서버 거절. 화면은 자연스럽게 회색으로 돌아가고 "잠시 후 다시 시도해주세요" 안내. 종료된 이벤트가 찜 목록에 쌓이는 사고는 발생 안 함.

비공개 클럽

비멤버가 클럽 비공개 이벤트 찜 시도

비공개 클럽 이벤트는 클럽 멤버만 찜 가능. 비멤버가 누르면 서버 거절. 화면이 회색으로 돌아간다.

중복 동작

두 화면에서 같은 이벤트 찜

검색 결과와 상세 화면에서 같은 이벤트를 빠르게 두 번 찜. 한 번만 찜 처리되고 두 번째 호출은 자연스럽게 같은 결과를 반환. 중복으로 저장되지 않음.

비로그인

비로그인 사용자가 하트 탭

탐색 카드에서 하트를 누른 비로그인 사용자는 로그인 화면으로 이동. 로그인 후 같은 이벤트로 자동 복귀해 다시 찜 가능.

네트워크 단절

토글 도중 네트워크 끊김

화면은 잠시 빨갛게 채워지지만, 서버 응답이 안 오면 회색으로 자연스럽게 돌아가고 안내가 뜬다. 사용자가 "내가 뭘 잘못했나" 혼란을 겪지 않게.

화면에서 사용자가 인지하는 것

영역사용자가 보는 것그 자리에 둔 이유
하트 아이콘 찜 안 됨 = 회색 외곽선, 찜 = 빨간 채움. 토글 즉시 색 변화. 한 번의 시각적 변화로 결과를 알게. 결제·결제 후 같은 무게의 결정이 아니므로 망설임 없이 누르도록.
찜 목록 프로필 탭의 "찜한 이벤트". 일반 이벤트 카드와 같은 형태로 정렬. 빈 상태에 안내 카피. 찜은 사용자의 신청 결정 직전의 1차 후보. 한 곳에서 다시 비교할 수 있어야 결정으로 이어진다.
토스트 안내 찜 추가 시 "찜했어요", 해제 시 "찜 해제했어요", 실패 시 "잠시 후 다시 시도해주세요". 실패를 분명히 알려야 사용자가 "왜 안 됐지" 혼란을 겪지 않는다. 단, 종료/비공개 같은 구체 이유는 현재 일반 메시지로 통합 — 추후 분기 가능.

운영자가 알아야 할 것

위시리스트의 결정점
  • 화면 즉시 반응이 본 단위의 핵심 UX. 서버 응답을 기다린 뒤 화면을 바꾸면 사용자가 누른 결과를 인지하지 못한 채 다음 카드로 이어 가는 사고가 발생.
  • 실패 시 자연스러운 롤백. 종료된 이벤트나 비공개 클럽 비멤버 등의 거절은 시각적으로 자연스럽게 원상복귀되어야 한다.
  • 실패 메시지가 현재 일반화되어 있다. "종료된 이벤트는 찜할 수 없습니다" 같은 구체 메시지가 필요한 경우는 운영 분석에서 거절 빈도가 큰 거절 사유부터 우선 분리.
  • 찜은 사용자의 1차 후보 신호. 찜 → 신청 전환율은 추천 품질의 직접 신호. 찜이 많이 일어나는데 신청이 적으면 카드의 결정 단서가 부족하다는 의미.
