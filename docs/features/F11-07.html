F11-07. 호스트 리뷰 모더레이션 (답변 · 임시 숨김)

이벤트 호스트가 본인 이벤트에 달린 리뷰에 직접 대응할 수 있는 두 가지 도구다. 답변은 리뷰 1건에 공개 댓글을 붙이고(1:1, 24시간 이내만 수정 가능), 임시 숨김은 6가지 사유코드 중 하나를 선택해 리뷰를 일시적으로 가린다. 허위 리뷰로 숨기면 운영자 검토 신고가 자동으로 접수된다(F11-04 연동). 모더레이션 권한은 해당 이벤트의 호스트에게만 있다.

설계 의도 — 호스트에게 발언권을 주되, 삭제 권한은 주지 않는다

모임 후기는 다른 사용자의 판단 근거다. 사실과 다른 후기에 호스트가 침묵해야 한다면 커뮤니티 신뢰는 유지되지 않는다 — 답변 기능이 그 발언권이다. 동시에 호스트가 부정적인 후기를 마음대로 지워버릴 수 있다면 리뷰 자체가 무력화된다. 그래서 삭제는 없고 임시 숨김만 있다. 숨김도 사유 코드를 남기고, 운영자가 검토할 수 있는 신고 경로와 연동된다. "호스트가 불쾌하다고 숨겼느냐, 진짜 문제 있는 후기를 정리했느냐"를 운영 기록이 추적한다.

답변 — 발언권의 구조

호스트의 답변은 리뷰 카드 아래에 녹색 카드로 붙는다. 다른 사용자가 리뷰를 볼 때 호스트의 시각도 같이 보인다.

현재 앱에서 이미 답변이 달린 리뷰의 더보기를 누르면 수정 경로 대신 작성 경로(createReply)가 호출된다. 서버는 이미 답변이 있으면 409 REVIEW_REPLY_ALREADY_EXISTS를 반환한다. 호스트가 24시간 이내 답변을 수정하려 할 때 에러가 발생한다. 수정(PUT /reply)과 작성(POST /reply)을 분기하는 코드가 화면에 추가되어야 한다.

임시 숨김 — 사유 코드가 있는 가림

숨김은 리뷰를 지우는 게 아니다. 사유 코드를 남기고 가리는 것이다. 운영자가 나중에 그 흔적을 볼 수 있다.

알려진 배선 갭 — autoEscalate 기본값 불일치

앱의 _autoEscalate 초기값은 true이며, 사용자가 체크박스로 해제할 수 있다. FAKE_REVIEW 선택 시 true로 재초기화되지만 비활성화는 되지 않아 여전히 해제 가능하다. 서버의 기본값은 false다. 사용자가 체크박스를 해제하지 않으면 모든 임시 숨김 요청이 autoEscalate=true로 전송되어 신고를 자동 생성한다. 이것이 의도된 운영 정책이라면 문서화하고 서버 기본값을 true로 통일해야 한다. 의도하지 않았다면 앱 기본값을 false로 수정해야 한다.

숨겨진 리뷰를 사용자가 보면

숨긴다는 것은 없애는 게 아니다. 내용이 가려지고 왜 가려졌는지가 대신 보인다.

숨김 상태 리뷰 카드의 모습
  • 리뷰 작성자와 별점은 그대로 보인다. 누가 쓴 리뷰인지, 몇 점을 줬는지는 가려지지 않는다 — 신뢰점수에도 계속 반영된다.
  • 본문 자리에 "임시 숨김" 안내가 들어온다. reasonText가 있으면 그 내용이, 없으면 사유 코드의 의미가 안내문으로 보인다.
  • 숨김 배지가 붙는다. FAKE_REVIEW이면 "허위 리뷰 검토 중", 그 외이면 "임시 숨김" 라벨이 카드 상단에 붙는다.
  • 호스트 답변이 있으면 숨김 상태에서도 함께 보인다.

사용자 시나리오

기본 흐름 — 답변

억울한 후기에 입장을 남긴다

호스트가 본인 이벤트 리뷰 목록을 보다 사실과 다른 내용이 적힌 후기를 발견한다. 카드 더보기를 눌러 "답변하기"를 선택하고, 공식 입장을 적는다. 확인하면 녹색 카드가 리뷰 아래에 붙는다. 다른 사용자가 그 리뷰를 볼 때 호스트의 입장도 같이 보인다.

이 흐름이 남기는 것 — 호스트가 침묵하지 않아도 된다는 장치. 리뷰 한 줄이 전부인 상황에서 호스트 시각이 추가되어 조회자가 더 균형 잡힌 판단을 할 수 있다.

수정 윈도우

잘못 입력한 답변을 24시간 안에 수정

답변을 달았는데 오탈자가 심각하다. 24시간 이내라면 더보기 → "답변 수정"으로 고칠 수 있다. 24시간이 지나면 수정 메뉴가 비활성화된다 — 공개 답변은 함부로 고치지 못한다는 정책이다.

이 흐름이 남기는 것 — 답변을 언제든 수정할 수 있으면 호스트가 후기에 따라 입장을 바꾸는 식으로 악용될 수 있다. 24시간은 교정 시간이지 번복 시간이 아니다.

임시 숨김

광고성 스팸 후기를 잠시 가린다

이벤트와 전혀 무관한 광고 문구가 적힌 후기가 달렸다. 더보기 → "임시 숨김" → "광고·스팸(SPAM)" 선택. 확인하면 후기 본문 자리에 "임시 숨김" 안내가 들어온다. 다른 사람은 그 후기의 내용 대신 "임시 숨김 처리된 리뷰"라는 안내를 본다.

이 흐름이 남기는 것 — 리뷰가 사라지지 않는다는 것을 사용자가 인지한다. 호스트가 부정적인 후기를 모두 없앨 수 있다는 오해를 막는다.

허위 리뷰 신고 연동

"이건 허위다" — 숨김과 신고가 한 번에

참석하지 않은 사용자가 리뷰를 남긴 것이 의심된다. 더보기 → "임시 숨김" → "허위 리뷰(FAKE_REVIEW)" 선택. 확인하면 리뷰가 가려지는 동시에 운영자 신고 큐에 우선 검토 플래그가 붙은 신고가 자동으로 접수된다.

이 흐름이 남기는 것 — 호스트가 "이건 신고감이다"고 판단할 때 두 번 행동(숨김 + 신고)하지 않아도 된다. 허위 리뷰를 발견한 순간 한 번의 선택으로 운영 검토까지 이어진다.

숨김 해제

오판이었다 — 되돌린다

숨겼는데 돌아보니 과한 판단이었다. 더보기 → "숨김 해제"를 누르면 리뷰가 다시 보이기 시작한다. 별점과 본문이 원래대로 복구된다.

이 흐름이 남기는 것 — 호스트의 실수를 되돌릴 수 있다. 단, 숨김 중에 자동 생성된 신고는 이미 접수된 상태라 함께 취소되지 않는다 — 운영자 검토는 별도로 진행된다.

권한 경계

다른 이벤트의 호스트가 내 리뷰에 접근 시도

B 이벤트 호스트가 A 이벤트 리뷰에 답변을 달려 한다. 서버에서 "해당 이벤트의 호스트가 아닙니다" 거절이 반환된다. 앱에서는 본인이 아닌 이벤트의 리뷰에는 더보기 모더레이션 메뉴 자체가 보이지 않는다.

이 흐름이 남기는 것 — 답변과 숨김 권한이 이벤트 단위로 격리된다. 호스트 A가 B의 이벤트에 개입할 수 없다.

화면에서 사용자가 인지하는 것

호스트와 일반 사용자가 같은 리뷰를 볼 때 각각 다른 것이 보인다.

화면 요소일반 사용자호스트(revieweeId 본인)
리뷰 카드 더보기 신고 아이콘만 신고 + "답변하기" + "임시 숨김" / "숨김 해제" (상태에 따라)
숨김 상태 카드 본문 자리에 "임시 숨김" 안내 + 사유 배지 동일. 단, 더보기에 "숨김 해제" 메뉴가 붙는다.
호스트 답변 카드 리뷰 아래 녹색 카드 — 호스트 닉네임 + 답변 내용 동일. 24h 이내이면 더보기에 "답변 수정" 메뉴 노출. (현재 미배선 — Gap)
답변 수정 여부 보이지 않음 allowEdit=true이면 수정 메뉴 노출. false이면 비활성.

운영자가 알아야 할 것

모더레이션 흔적은 남는다
  • 임시 숨김 해제의 별도 audit 로그는 없다. 숨김 해제(unhide)는 hidden_reason_code·hidden_by_user_id 등 컬럼을 null로 초기화할 뿐이다(ReviewService.java:454-459). 별도 audit 로그 테이블 기록은 없다. 다만 autoEscalate=true로 숨김 시 생성된 Report row는 삭제되지 않고 그대로 남아, 반복 숨김·해제 패턴은 Report 큐를 통해 운영에서 감지할 수 있다.
  • FAKE_REVIEW 숨김은 자동으로 신고가 접수된다(우선 검토 플래그 포함). 운영자 큐에서 priority=true로 상단 노출. 숨김 해제를 해도 신고 레코드는 남는다. 운영 처리(승인/기각)는 별도로 진행되어야 한다.
  • 별점은 숨겨도 신뢰점수에서 제외되지 않는다. 현재 정책은 "임시 숨김 중에도 별점을 신뢰점수 계산에서 빼지 않는다"다(ReviewService.java 주석). 이 정책이 바뀌면 신뢰점수 재계산이 필요하다.
  • 답변 삭제는 운영자도 사용자 경로로는 못 한다. DELETE 엔드포인트가 없다. soft delete 경로는 서비스 레이어에 있지만 컨트롤러에 미노출. 필요 시 admin API 경로를 통해야 한다.

알려진 갭 요약

지금 바로 처리해야 하는 항목
  • 답변 수정 UI 미배선 — 이미 답변이 있는 리뷰에서 더보기를 누르면 서버에서 409 에러가 발생한다. review_list_screen에서 reply 존재 여부에 따라 createReply / updateReply를 분기해야 한다.
  • autoEscalate 기본값 운영 정책 결정 — 앱 기본값은 ON이나 사용자가 시트의 체크박스에서 해제 가능(해제 시 false 전송). 서버 기본값은 false. 체크박스 가시성과 기본값 방향이 의도된 정책인지 확인하고, 필요하면 UI 강조 또는 기본값 통일이 필요하다.
  • 답변 알림 미구현 — 리뷰 작성자가 호스트 답변을 알 방법이 없다. NotificationType 추가 + 서비스 연동 검토 필요.
