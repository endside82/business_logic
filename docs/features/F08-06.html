F08-06. 마켓 아이템 관리

마켓에 노출되는 상품(아이템) 단위를 크리에이터가 직접 관리하는 화면. 플랜과 별개로 상태가 흐른다 — 작성 중 → 심사 제출 → 심사 승인 → 판매 중 → 판매 중지 → 삭제됨. 운영자 심사를 통과해야 판매를 시작할 수 있고, 가격·설명·재고·노출 순서를 손볼 수 있다. 반려 시 사유를 안내받고 수정 후 재제출한다. 플랜은 콘텐츠 자체의 라이프사이클, 아이템은 마켓 진열의 라이프사이클 — 같은 콘텐츠를 두 단계로 통제하는 구조다.

플랜(F08-02)이 콘텐츠의 라이프사이클(작성·발행·숨김)을 다룬다면, 아이템은 마켓 진열의 라이프사이클을 다룬다. 플랜은 그대로 두고 진열만 잠시 내릴 수 있고, 같은 플랜을 다른 가격·설명으로 다시 진열할 수도 있다. 크리에이터가 자기 콘텐츠의 운영을 더 세밀하게 다룰 수 있는 한 단계 위의 통제권이다.

아이템 상태의 흐름

새 아이템은 항상 작성 중 상태로 만들어진다. 크리에이터가 심사를 제출하면 운영자가 검토한 뒤 승인하거나 반려한다. 승인된 아이템만 판매를 시작할 수 있다. 삭제도 데이터는 보존하는 표시 변경이다.

이 화면이 다루는 여섯 가지

동작 1

메타 수정

제목·설명·가격·재고·노출 순서를 손본 뒤 "저장"을 누른다. 아이템 종류와 통화는 생성 시 고정이라 본 화면에서는 바뀌지 않는다.

동작 2

심사 제출 · 재제출

작성 중 상태에서 "심사 제출"을 누르면 운영자 검토 대기 상태로 넘어간다. 반려 시 사유가 화면 상단에 표시되고, 수정 후 다시 제출할 수 있다.

동작 3

판매 시작 / 재개 (승인 후)

운영자 승인이 난 아이템에서 "판매 시작"을 누르면 마켓 검색·인기·최신 목록에 노출된다. 판매를 중지했다가 다시 시작하는 재개도 같은 버튼.

동작 4

판매 중지 (판매 중에서)

판매 중인 아이템을 잠시 내려놓는다. 마켓에서 사라지지만 데이터는 그대로 — 다시 판매 재개 버튼을 누르면 같은 아이템이 돌아온다.

동작 5

예약 판매시작

승인됐거나 판매 중지 상태의 아이템에서 날짜·시각을 지정하면, 그 시각이 되면 자동으로 판매 중으로 전환된다. 예약을 걸어 두고 다른 일을 해도 된다. 예약을 취소하는 것도 가능하다.

동작 6

삭제

아이템 자체를 정리한다. 데이터는 보존되지만 마켓에서 영구히 사라진다. 위험 색상 버튼 + 확인 다이얼로그로 한 번 더 잡는다.

사용자 시나리오

첫 심사 제출

작성을 마친 아이템을 심사에 올린다

발행해 둔 플랜으로 만든 첫 마켓 아이템. 상태 배지가 "작성 중"으로 떠 있다. "심사 제출"을 누르고 확인하면 상태가 "심사 중"으로 바뀌고, 운영자 검토 대기열에 들어간다. 이 상태에서는 크리에이터가 할 수 있는 액션이 없고, 결과를 기다린다.

반려 후 재제출

사유를 확인하고 수정 후 다시 올린다

운영자가 반려하면 화면 상단에 반려 사유 배너가 나타난다. 크리에이터는 사유를 확인하고 제목·설명·가격 등을 수정한 뒤 "심사 제출"을 다시 누른다. 서버는 수정 저장과 재제출을 순서대로 처리하고 상태가 다시 "심사 중"으로 바뀐다.

이 흐름이 남기는 것 — 거절이 막힌 벽이 아니라 개선 요청임을 사유 배너가 전달한다.

판매 시작

승인받은 아이템을 마켓에 푼다

운영자 승인 알림이 온다. 아이템 화면에서 상태가 "승인됨"으로 바뀌고 "판매 시작" 버튼이 나타난다. 버튼을 누르고 확인하면 상태가 "판매 중"으로 바뀌고 마켓 메인의 최신 목록에 들어간다.

예약 판매시작

특정 시각에 자동으로 판매가 시작되게 한다

승인은 났지만 이번 주말 이후에 열고 싶다. "예약 판매시작" 버튼을 눌러 날짜·시각을 지정한다. 지정한 시각이 되면 자동으로 판매 중으로 전환된다 — 크리에이터가 직접 그 시각에 손댈 필요가 없다. 예약을 취소하고 싶으면 같은 화면에서 예약을 없앤다.

이 흐름이 남기는 것 — 발매 타이밍이 마켓 성과에 영향을 주는 크리에이터라면 정확한 시각 통제가 필요하다.

가격 변경

5,000P를 4,500P로 내린다

잘 안 팔리는 아이템의 가격을 낮춘다. 가격 칸을 수정하고 "저장"을 누르면 곧장 반영되고, 마켓 카드에 새 가격이 표시된다. 상태는 그대로 — 가격 변경 자체는 진열 상태에 영향을 주지 않는다.

이 흐름이 남기는 것 — 가격은 자주 손보는 운영 변수라, 별도 상태 전이 없이 한 번에 바꿀 수 있게 둔다.

중지

일정상 판매를 잠시 멈춘다

운영이 어려운 시즌에 판매 중 아이템을 내려놓는다. "판매 중지" 버튼 + 확인 다이얼로그. 마켓에서 사라진다. 시즌이 풀린 뒤 같은 화면에서 "판매 재개"를 누르면 같은 아이템이 돌아온다.

정책 비공개

운영자가 아이템을 내렸다

화면 상단에 "정책 비공개" 배너가 뜨고, 크리에이터가 직접 할 수 있는 판매 관련 액션이 사라진다. 재공개 경로는 운영자와 별도로 소통해야 하며, 본인이 이 상태를 단독으로 해제할 수는 없다.

동시 조작 안전

크리에이터와 운영자가 같은 순간 누른다

크리에이터가 "판매 시작"을 누른 것과 운영자가 "비공개 처리"를 한 것이 거의 동시에 일어난다. 두 요청이 겹쳐도 서버는 한 쪽만 처리하고 나머지는 안전하게 거절한다 — 어느 쪽도 중간 상태에 빠지지 않는다.

이 흐름이 남기는 것 — 두 명이 같은 버튼을 동시에 눌러도 마지막 결과가 하나로 확정된다.

삭제

운영을 정리한다

더 이상 운영하지 않을 아이템을 정리한다. "아이템 삭제" + "되돌릴 수 없습니다" 확인 다이얼로그. 데이터는 보존되지만 마켓에는 다시 노출되지 않는다.

권한 가드

다른 사람의 아이템을 손보려 한다

임의 경로로 다른 크리에이터의 아이템 편집에 도달한다. 화면은 열리지만 저장을 누르면 서버가 본인 소유가 아니라는 이유로 거절한다. 사용자에게는 일반 실패 안내가 뜬다.

무료 아이템

0원으로 등록한다

가격을 0으로 두고 등록한다. 마켓에서는 "무료" 라벨이 가격 자리에 표시된다. 구매 흐름은 동일하지만 결제는 일어나지 않는다.

운영자가 알아야 할 것

진열 통제와 정산이 같은 화면에서 갈린다
  • 플랜 상태와 아이템 상태는 독립적이다. 플랜이 발행 중이어도 아이템은 심사 대기 상태일 수 있다. 마켓 노출 여부는 아이템 상태가 결정한다.
  • 판매 시작은 운영자 승인이 선행되어야 한다. 크리에이터가 바로 판매 중으로 진입할 수 없다. 심사 제출 → 운영자 승인 → 판매 시작이 순서다. 반려 시 사유가 안내되고 수정 후 재제출할 수 있다.
  • 운영자는 승인된 아이템을 언제든 비공개 처리할 수 있다. 판매 중이거나 중지 상태여도 비공개로 전환된다. 비공개 상태의 아이템은 크리에이터가 직접 재공개할 수 없고, 운영자와의 별도 소통이 필요하다. 크리에이터와 운영자가 같은 아이템에 동시에 다른 조작을 해도 한 쪽만 반영되고 충돌이 일어나지 않는다.
  • 아이템 종류와 통화는 생성 후 바꿀 수 없다. 잘못 만든 경우 삭제 후 새로 등록해야 한다. 이 규칙이 화면에서 분명하지 않으면 사용자가 같은 폼에서 종류를 바꿔도 저장되지 않는 회색 지대를 만난다. "어떤 포인트로 살 수 있는가"(유료만 / 무료만 / 둘 다)도 이 통화 설정에 묶여 있어 생성 시점에만 정해진다 — 수정 단계에서는 무료 결제 가격(차등가)만 손댈 수 있다.
  • 무료 포인트로 팔린 분은 크리에이터에게도 무료 형태로만 들어온다. 무료 결제로 받은 값은 플랫폼 안에서 쓸 수 있을 뿐 현금으로 인출되지 않는다(유료/무료 분리정산). 현금 정산·인출 대상은 유료 결제분뿐이다 — 무료를 허용한 아이템의 수익을 운영·정산할 때 이 구분을 전제로 본다(상세는 결제·정산 정책 문서).
  • 판매 시작 시점에 수수료·원천세 계산 기준이 정해진다. 10% 수수료 + 3.3% 원천세 — 크리에이터 수익은 판매 단가에서 두 비율을 뺀 값. 가격을 자주 바꾸는 운영은 정산 페이지에서 사용자가 직접 누적 수익을 추적하기 어려워질 수 있다.
  • 월 50건의 구매 한도는 구매자 측 정책이다. 한 아이템이 인기를 끌어도 한 사용자가 무한 구매할 수 없다 — 이는 마켓 운영 안정과 결제 부정 방지 동시 목적.
  • 재고 칸이 비면 무제한. 일반 플랜은 보통 무제한이지만, 인원 한정 이벤트 패키지 같은 종류에는 재고를 설정한다. 사용자가 의도하지 않게 한도가 걸리는 사고를 피하려면 입력 비어 있을 때의 의미가 화면 카피에 분명해야 한다.
