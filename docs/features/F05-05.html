F05-05. 저장된 검색

매주 같은 조건으로 모임을 찾는 사용자가 키워드와 필터를 묶어 이름과 함께 저장하고, 한 번에 다시 실행할 수 있다. 저장 시 "새 결과 알림"을 켜 두면, 같은 조건에 새로 올라온 이벤트가 생길 때 푸시 알림이 도착한다. 최근 검색어(F05-04)가 자동으로 만들어지는 가벼운 기록이라면, 저장된 검색은 사용자가 의식적으로 "이 조건을 다시 쓰겠다"고 표시한 묶음이다.

저장 시점 조건으로 즉시 검색
설계 의도 — "자주 쓰는 조건"을 한 번 만들어 두면 다음부터는 한 탭이다

키워드만으로 결과를 좁히기 어려운 사용자가 있다. "강남 + 무료 + 음식"처럼 세 가지 조건을 매번 다시 만드는 것은 부담스럽다. 저장된 검색은 이 조건 묶음을 이름과 함께 저장해 두고, 카드 한 번 탭으로 결과 화면에 도착하게 한다. "새 결과 알림"을 켜 두면 사용자가 앱을 열지 않아도 시스템이 30분 주기로 새 결과를 확인해 푸시를 보낸다 — 검색을 "사용자가 찾는 것"에서 "조건에 맞는 것이 생기면 알림"으로 확장하는 동력.

이 기능이 다루는 일

저장

현재 조건을 이름과 함께 묶는다

검색 결과 화면의 북마크 아이콘으로 다이얼로그를 띄워, 이름과 알림 여부를 입력한다. 이름 최대 50자, 사용자당 최대 10개까지 저장 가능.

실행

저장된 조건으로 한 번에 검색

저장 목록의 검색 아이콘을 누르면 저장 시점의 키워드와 필터가 복원되어 결과 화면으로 도착한다. 사용 횟수와 마지막 실행 시각이 함께 갱신되어, 자주 쓰는 검색이 위로 올라온다.

새 결과 알림

조건에 맞는 새 이벤트가 생기면 푸시

알림 토글이 켜진 저장 검색에 대해 시스템이 30분마다 새 결과를 확인한다. 새 카드가 있으면 푸시 알림을 보내고, 저장 카드에 빨간 배지로 새 결과 개수가 표시된다.

관리

이름·알림·필터 수정과 삭제

각 카드에서 이름·알림·필터를 수정하거나 삭제 가능. 마지막 실행 시각 순으로 자동 정렬되어, 자주 쓰는 묶음이 항상 맨 위에 보인다.

저장한 검색이 다시 실행되는 흐름

사용자가 저장 목록에서 카드를 누른 순간부터 결과 화면에 도착하기까지.

사용자 시나리오

기본 흐름

매주 같은 조건으로 검색하던 사용자가 묶어 둔다

"강남 + 무료 + 음식" 조건으로 자주 검색하는 사용자가 결과 화면 북마크 아이콘을 누른다. 이름을 "강남 무료 음식"으로 입력하고 알림 토글을 켠 채 저장한다. 다음 주에는 저장 목록에서 카드 한 번 탭으로 같은 결과에 도착한다.

이 흐름이 남기는 것 — 매번 세 가지 필터를 다시 만들지 않아도 되는 경험. 검색의 "재방문 비용"을 한 탭으로 줄여 주는 핵심 기능.

새 결과 알림

앱을 열지 않아도 새 모임이 도착했음을 안다

저장 검색 "강남 무료 음식"의 알림이 켜져 있다. 시스템이 30분 주기로 새 결과를 확인하다가 새 이벤트가 등록된 것을 발견하면 푸시 알림을 보낸다. 사용자가 알림을 누르면 저장 목록 화면에 도착하고, 카드에 빨간 배지로 새 결과 개수가 보인다.

이 흐름이 남기는 것 — 검색을 "사용자가 찾는 것"에서 "조건에 맞는 것이 생기면 알림"으로 확장. 앱을 열지 않은 시간대에도 사용자에게 가치를 만들어 주는 동력.

갯수 초과

이미 10개를 저장한 사용자가 새 저장을 시도

저장 다이얼로그에서 "저장"을 누르면 "저장에 실패했습니다" 토스트만 뜬다. 사용자에게는 갯수 초과 사실이 명시되지 않아, 왜 저장이 안 됐는지 추측해야 한다.

이 흐름이 남기는 것 — "최대 10개까지 저장할 수 있어요. 기존 항목을 정리해 보세요" 같은 구체적 안내가 필요하다. 현재 메시지는 운영 결함.

알림 토글

저장 후 알림만 따로 끄거나 켠다

이미 저장된 카드의 "새 결과 알림" 토글을 누르면 즉시 알림이 켜지거나 꺼진다. 알림이 켜진 카드는 우상단에 알림 아이콘이 표시되어 한눈에 구분된다.

잠재 결함

저장된 검색의 필터를 수정했는데 그대로다

사용자가 저장된 카드의 편집을 눌러 필터 시트로 가격 슬라이더만 바꾼 뒤 적용한다. 화면에서는 변경 사항이 사라진 듯하지만, 다시 실행하면 옛 조건으로 결과가 받아 와진다.

이 흐름이 남기는 것 — 현재 서버 수정 처리가 이름과 알림 여부만 반영하고 필터 본체는 갱신하지 않는다. 사용자의 의식적 수정이 무시되는 결함이라 빠른 정비가 필요하다.

삭제

더 이상 쓰지 않는 묶음을 정리

카드의 휴지통 아이콘을 누르면 "OO 저장된 검색을 삭제하시겠습니까?" 확인 다이얼로그가 뜨고, 확인하면 즉시 목록에서 사라진다.

빈 상태

저장된 검색이 한 건도 없다

저장 목록에 처음 들어온 사용자에게는 북마크 아이콘과 "저장된 검색이 없습니다 — 자주 사용하는 검색을 저장해 보세요" 안내만 보인다. 사용자는 어디서 어떻게 저장하는지를 검색 결과 화면에서 자연스럽게 배운다.

화면에서 사용자가 인지하는 것

영역사용자가 보는 것그 자리에 둔 이유
저장 카드 이름, 적용된 키워드와 주요 필터가 칩으로, "마지막 실행: M/d HH:mm" 그리고 "N회 사용". 새 결과가 있으면 빨간 배지로 개수. "내가 마지막으로 본 것이 언제이고 얼마나 자주 쓰는지" 사용자가 카드 단계에서 인지하게 한다. 사용 횟수가 자주 쓰는 묶음의 손쉬운 정렬 기준이 된다.
알림 토글 카드 내 "새 결과 알림" 행과 스위치. 알림 활성 시 카드 우상단에 작은 알림 아이콘. 저장과 알림은 분리된 의사 결정이다. 사용자가 "이 조건을 다시 보고 싶다"와 "결과가 새로 생기면 알림 받고 싶다"를 따로 끄고 켤 수 있어야 한다.
액션 세 버튼 검색(실행) · 편집 · 삭제 세 아이콘이 카드 우측에 한 줄. 저장된 묶음의 라이프사이클은 "실행 / 수정 / 정리" 세 가지가 전부. 그 외 동작을 카드에 끼우면 의사결정이 흐려진다.
저장 다이얼로그 이름 입력(50자) + 알림 토글 + 취소/저장 두 버튼. 저장은 사용자가 의식적으로 하는 동작이므로 모달로 명확하게 분리. 알림 여부는 저장 시점에 함께 결정한다.
빈 상태 안내 북마크 아이콘 + "저장된 검색이 없습니다" + "자주 사용하는 검색을 저장해 보세요". 처음 들어온 사용자에게 "여기가 무엇을 위한 곳인지" 한 줄로 설명한다.

운영자가 알아야 할 것

저장된 검색의 정책 결정점 + 알면서 둔 결함
  • UI 스펙은 사용자당 20개, 서버 한도는 10개로 어긋나 있다. 현재 10개에서 막히며, 11번째 저장 시도는 "저장에 실패했습니다"만 떠올라 사용자는 이유를 모른다. 한도를 맞추고 초과 시 안내 문구를 구체화하는 정비가 필요하다.
  • 편집에서 필터를 바꿔도 서버에 반영되지 않는다. 현재 서버 수정 로직이 이름과 알림 여부만 갱신하므로, 필터를 다시 짠 사용자의 의식적 수정이 무시된다. 빠른 정비가 필요한 결함.
  • 새 결과 알림은 30분 주기 일괄 처리다. 새 이벤트가 등록되자마자 사용자가 알림을 받지는 않는다 — 사용자에게 "최대 30분 안에 알려 드린다"는 약속이 명시되지 않은 점은 후속 검토 대상.
  • 정렬은 마지막 실행 시각 기준만 지원한다. UI 스펙의 "이름순" 정렬은 현재 미구현. 저장 갯수가 10개 안쪽이라 큰 문제는 아니지만 스펙 정정 또는 구현 중 하나가 필요.
  • 실행은 카운트만 올리고, 검색은 별도로 호출된다. "실행" 동작이 사용 횟수를 늘리고 조건 묶음을 돌려주면, 화면이 그 묶음으로 다시 검색을 실행한다 — 두 단계가 분리된 구조라 한쪽 실패가 전체를 무너뜨리지 않는다.
  • 다른 사용자의 저장 검색 ID로는 어떤 동작도 할 수 없다. 실행·수정·삭제 모두 본인 소유 확인이 선행되며, 그 외에는 차단된다.
