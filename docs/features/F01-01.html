F01-01. 이메일 가입·로그인

외부 소셜 계정 없이도 이메일과 비밀번호만으로 서비스에 들어올 수 있게 한다. 가입 한 동작 안에서 본인 정보·약관 동의·연령 검증이 모두 처리되고, 로그인 한 동작 안에서 잠금·정지·탈퇴 상태가 한꺼번에 점검된다. 사용자가 서비스에 처음 자기 신원을 맡기는 면이라, 여기서 만든 신뢰가 이후 모든 도메인의 출발점이 된다.

설계 의도 — 가입 한 동작에 네 가지 검증이 같이 끝나야 한다

"가입하기"를 누르는 순간 서비스는 (1) 같은 이메일·닉네임으로 이미 가입한 사람이 없는지, (2) 생년월일이 정상이고 만 19세 이상인지, (3) 필수 약관 두 가지에 동의했는지, (4) 비밀번호가 충분히 강한지를 한 번에 점검하고, 통과하면 곧장 로그인까지 끝낸 상태로 사용자를 로그인 화면에 돌려보낸다. 이 네 가지를 단계마다 끊어 묻거나 일부만 검사하면 가입 도중 이탈이 급격히 늘어난다 — 첫 진입이 답답하면 사용자는 그 앱이 자기 시간을 존중하지 않는다고 인지한다.

이 단위가 도메인 잔류·이탈을 결정한다

가입을 끝낸 사람만 다음 도메인을 볼 수 있다 — 그래서 본 단위는 가장 짧고 가장 안전해야 한다
  • 가입 화면의 길이가 곧 첫 이탈률 — 입력 항목이 한 줄 늘어날 때마다 가입 완료율이 떨어진다. 그래서 본 단위는 필수 정보(이메일·비밀번호·닉네임·생년월일·필수 약관 2종)만 받고, 휴대폰·성별은 선택으로 둔다.
  • 잠금·정지·탈퇴 상태는 로그인 시점에 한 번에 처리 — "비밀번호가 맞는데도 들어가지지 않는다"는 답답함을 줄이기 위해 자격 증명 확인과 계정 상태 확인을 같은 응답으로 끝낸다. 단계마다 다른 화면을 띄우지 않는다.
  • 가입 후 곧장 자동 로그인 시키지 않고 로그인 화면으로 돌려보낸다 — 사용자가 자기 비밀번호를 한 번 더 손으로 확인하는 동작을 통해 "내가 이 계정의 주인이다"라는 통제감을 만든다. 자동 로그인은 편의지만 가입 직후의 비밀번호 망각 비율을 가시화하지 못한다.

가입 한 동작이 통과하는 네 검증

중복·연령·약관·비밀번호 네 가지가 같은 응답 안에서 결정된다. 어느 하나라도 어긋나면 전체가 실패하고 같은 화면에서 사용자가 바로 고쳐 입력할 수 있다.

로그인이 한 응답 안에서 확인하는 것

자격 증명 확인과 계정 상태 확인이 분리되어 있지 않다 — 한 번의 로그인 시도가 정상 사용자/정지·차단 사용자/탈퇴 사용자/잘못된 비밀번호를 한꺼번에 가른다.

사용자 시나리오

기본 흐름

가입을 마치고 곧장 로그인

이메일·비밀번호·닉네임·생년월일을 채우고 필수 약관 둘에 동의한 뒤 가입하기를 누른다. 가입 완료 안내가 토스트로 뜨고 로그인 화면으로 돌아온다. 방금 입력한 자격 증명으로 로그인을 누르면 첫 로그인 사용자로 식별되어 이메일 인증 또는 온보딩 화면으로 진입한다.

이 흐름이 남기는 것 — 사용자는 가입 직후 자기 비밀번호를 한 번 더 손으로 확인했다. 비밀번호 망각으로 인한 재설정 요청이 가입 첫 주에 몰리는 것을 줄인다.

이미 가입함

같은 이메일로 다시 가입 시도

이전에 가입했는지 잊은 사용자가 같은 이메일로 다시 가입한다. "이미 사용 중인 이메일입니다" 안내가 뜨고 로그인 화면으로 안내된다. 비밀번호 찾기 동선이 같은 화면에 보이기 때문에 다음 행동까지의 거리가 짧다.

이 흐름이 남기는 것 — 같은 사용자가 두 계정을 만드는 사고를 막아 데이터가 두 갈래로 흩어지지 않게 한다.

연령 정책

만 19세 미만이 가입을 시도

2010년생이 모든 필드를 채우고 가입하기를 누른다. 서비스는 가입을 차단하고 "만 19세 이상만 이용 가능합니다" 안내를 띄운다. 계정 자체가 만들어지지 않는다 — 나중에 만 19세가 되어 같은 이메일로 다시 가입할 수 있다.

이 흐름이 남기는 것 — 본 서비스가 다루는 데이팅·결제·모임 분야의 법적·사회적 책임 경계가 가입 시점에서 강제된다.

자격 증명 실패

비밀번호를 잘못 입력했다

"이메일 또는 비밀번호가 올바르지 않습니다" 한 줄이 뜬다. "이메일은 맞는데 비밀번호만 틀렸다"는 식의 세부 안내는 하지 않는다 — 공격자가 가입 여부를 알아내는 단서가 되기 때문이다. 같은 화면에 비밀번호 찾기 링크가 함께 보여서 사용자가 다음 행동을 바로 잡을 수 있다.

계정 상태

운영자가 차단한 계정으로 로그인

정상 자격 증명을 입력했지만 운영자가 정책 위반으로 계정을 정지·차단했다. 로그인은 거부되고 계정 상태에 맞는 안내가 표시된다. 사용자는 운영팀에 별도 문의로 흐른다.

이 흐름이 남기는 것 — 차단된 사용자가 계속 시도하지 않게 분명한 거절을 한 번에 전달한다.

기기 정책

다른 기기에서 새로 로그인

폰과 태블릿을 함께 쓰는 사용자가 태블릿에서 새로 로그인한다. 폰의 기존 세션은 즉시 끊어지고, 폰에서 다음 동작을 시도하면 자동으로 로그인 화면으로 돌아간다. 두 기기를 동시에 유지하지 않는다는 정책의 발현 지점.

이 흐름이 남기는 것 — 데이팅·결제 같은 민감 도메인에서 "내 계정이 다른 기기에서 살아 있다"는 불안을 없앤다.

미인증 분기

이메일 인증을 마치지 않은 채 로그인

가입은 됐지만 메일 링크를 아직 누르지 않았다. 로그인 자체는 성공하지만 이메일 인증 화면으로 안내된다. 일부 화면을 둘러볼 수는 있어도 결제·신청 같은 결정 동작은 인증 후로 미뤄진다.

이 흐름이 남기는 것 — 이메일 인증을 가입 직후로 강제하지 않고 "필요해질 때 마주치는" 흐름으로 둠으로써 가입 완료율과 인증 완료율 사이의 균형을 잡는다.

비밀번호 분실

기억나지 않아 비밀번호 찾기로 빠짐

같은 화면 우측의 "비밀번호 찾기" 링크로 빠진다. 메일로 재설정 링크가 가고 새 비밀번호로 다시 로그인한다. 비밀번호 재설정의 상세 흐름은 F01-04에서 정의된다.

화면에서 사용자가 인지하는 것

로그인 화면과 가입 화면, 그리고 두 화면을 잇는 진입 동선이 같이 결정된다.

화면 요소사용자가 보는 것그 자리에 둔 이유
로그인 화면 상단 서비스 로고 + "Community" + 한 줄 카피 "함께하는 즐거움을 발견하세요" 비로그인 상태에서 처음 보는 화면이라 서비스 정체성이 가장 먼저 노출되어야 한다.
이메일/비밀번호 입력 두 칸 + "로그인" 큰 버튼 + 우측 "비밀번호 찾기" 작은 링크 로그인은 두 칸 입력만 받는 가장 짧은 형태로 유지. 비밀번호 분실 동선이 같은 시야 안에 있어 다음 행동 거리가 짧다.
소셜 로그인 영역 "또는" 디바이더 아래 카카오·네이버·구글·애플 4종 버튼 세로 나열 이메일 가입을 선호하지 않는 사용자에게 즉시 분기 가능한 대안을 같은 화면에서 제시.
회원가입 진입 "계정이 없으신가요? 회원가입" 화면 하단 링크 가입은 부차적 동선으로 둔다 — 대부분의 진입은 재방문 로그인이라는 사실 반영.
가입 화면 입력 묶음 이메일·비밀번호·확인·닉네임·생년월일·휴대폰(선택)·성별(선택) 필수 항목은 위쪽, 선택 항목은 아래쪽. 가입 완료의 인지 거리를 짧게 유지한다.
비밀번호 강도 표시 입력하는 동안 3단계 막대(약함/보통/강함)가 색상으로 변화 가입 시점에 약한 비밀번호를 쓰지 않도록 즉시 시각 피드백. 거부가 아닌 권장으로 작동.
약관 동의 카드 "전체 동의"·이용약관(필수)·개인정보(필수)·마케팅(선택). 각 항목 옆 화살표로 본문 확인 필수와 선택을 명확히 구분. "전체 동의" 한 번으로도 진행 가능한 동선이 있어 입력 마찰이 줄어든다.

측정해야 하는 것

본 단위가 정상이라고 부르려면 이 다섯 지표가 추적되어야 한다
  • 가입 시작 → 완료 전환율 — 가입 화면 진입 후 가입하기 탭까지의 전환. 입력 항목 변경이 가입 의향에 미치는 영향을 가장 빨리 잡는 신호.
  • 가입 직후 로그인 성공률 — 가입 완료 후 5분 내 같은 자격 증명으로 로그인 성공 비율. 낮으면 비밀번호 정책이 사용자 인지와 어긋난다는 신호.
  • 중복 이메일/닉네임 시도 비율 — 너무 높으면 사용자가 자기 가입 여부를 잊었다는 신호. 비밀번호 찾기 동선의 가시성을 강화해야 한다.
  • 비밀번호 잘못 입력 횟수의 분포 — 한 사용자가 평균 몇 번 만에 로그인에 성공하는지. 3회 이상이 정상 사용자의 30% 이상이면 비밀번호 정책이 너무 빡빡하다.
  • 로그인 → 첫 화면 도달까지의 시간 — 로그인부터 홈/온보딩/인증 화면 진입까지의 체감 시간. 너무 길면 첫 인상이 손상된다.

운영자가 알아야 할 것 — 비즈니스 리스크

본 단위에서 어긋나면 도메인 전체의 신뢰가 무너진다
  • 로그인 실패 메시지를 세분화하면 가입 여부 노출 위험. "이메일 또는 비밀번호가 올바르지 않습니다" 한 줄을 유지해야 한다 — "이메일은 맞는데 비밀번호가 틀렸다"는 표현이 공격자에게는 가입 여부의 단서가 된다.
  • 가입과 로그인의 검증 정책이 어긋나면 안 된다. 가입 시 허용된 비밀번호가 로그인 시 거부되면 사용자는 본인이 누군지 모르게 된다 — 두 화면의 검증 기준은 같이 움직여야 한다.
  • 동시 로그인 1대 정책은 운영팀에 분명히 공유되어야 한다. "왜 갑자기 로그아웃됐냐"는 문의의 상당수가 다른 기기에서 새로 로그인한 본인 행동의 결과 — CS 운영 매뉴얼에 우선 포함.
  • 차단·정지 사용자에게 분명한 거절을 한 번에 전달해야 한다. 여러 화면에 흩어진 안내는 사용자가 "어떻게든 다시 시도하면 통과될 것"으로 오해하게 한다 — 그 결과 동일 사용자의 로그인 재시도가 누적된다.
  • 만 19세 미만 차단은 가입 시점에서만 막힌다. 만약 다른 채널(소셜 가입)에서 미성년 차단이 누락된 회귀가 발생하면 데이팅·결제 도메인 전체가 노출된다 — 본 정책과 F01-02 소셜 가입의 정책이 같이 움직여야 한다.
  • 비밀번호는 단방향 해싱되어 저장된다. 운영 측에서 사용자의 비밀번호 자체를 조회·복호화할 수 없다 — 어떤 경우에도 사용자에게 비밀번호를 알려주지 않는다는 정책의 기술적 근거.
