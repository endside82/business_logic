F01-01. 이메일 회원가입 · 로그인

이메일과 비밀번호만으로 신규 사용자를 등록하고, 기존 사용자를 다시 들여보낸다. 모든 사용자의 첫 접점이라 “여기서 멈추면 다른 모든 기능이 보이지 않는다”는 점이 본 기능의 무게다. 가입은 사용자·약관·생년월일·로그인 수단까지 한 번에 만들어내는 단일 트랜잭션이고, 로그인은 상태 검증과 토큰 재발급이 합쳐진 짧지만 분기가 많은 흐름이다.

이 기능이 다루는 범위

여기서는 이메일 + 비밀번호만 다룬다. 소셜 로그인(F01-02), 이메일 인증 메일 발송(F01-03), 비밀번호 재설정(F01-04), 토큰 갱신(F01-05), 첫 로그인 뒤의 온보딩(F01-06)은 각자 별도 단위로 분리되어 있다. 본 단위는 로그인 화면에 진입한 사람이 어떻게 사용자 레코드를 갖게 되고, 어떤 조건에서 다음 화면으로 이동하게 되는지까지를 책임진다.

이 기능 안의 하위 항목

한 카드 = 사용자가 한 번에 의식하는 흐름 단위. 시나리오는 이 단위 안에서 분기한다.

하위 기능 1

신규 회원가입

이메일·비밀번호·닉네임·생년월일·약관 동의를 받아 한 번의 가입 요청으로 사용자·회원정보·약관·로그인 수단 4가지 레코드를 동시에 만든다. 만 19세 미만, 중복 이메일/닉네임, 약한 비밀번호는 가입 자체를 막는다.

5 시나리오 미성년자 차단 409 중복
하위 기능 2

이메일 로그인

등록된 이메일과 비밀번호로 다시 진입한다. 잘못된 자격 증명·잠금/탈퇴 상태는 거부. 처음 로그인하는 사용자는 별도 플래그로 구분되어 이어지는 화면이 온보딩으로 분기한다.

3 시나리오 잠금/탈퇴 거부 첫 로그인 분기
하위 기능 3

스플래시 진입 분기

앱을 열 때 저장된 토큰을 확인해 로그인 화면 / 온보딩 / 홈 중 어디로 보낼지 결정한다. 토큰이 없으면 무조건 로그인 화면. 사용자는 이 화면에서 직접 액션을 하지 않지만, 잘못 분기하면 “앱이 매번 로그인 화면을 띄운다”는 가장 흔한 회귀가 발생한다.

자동 분기 F01-05 의존
하위 기능 4

개발 환경 바이패스

개발자가 시드 계정으로 한 번에 들어갈 수 있는 임시 진입점. 프로덕션 빌드에서는 노출되지 않는다. E2E 테스트는 이 버튼을 쓰지 않고 실제 가입/로그인 흐름을 검증한다.

DEV 전용

화면 전이

스플래시가 토큰을 보고 갈래를 정한다. 회원가입은 곧장 홈으로 가지 않고 한 번 로그인 화면으로 돌려보내 명시적 로그인을 유도한다.

회원가입 — 사용자 시퀀스

사용자가 한 번의 “가입하기” 탭을 누른 뒤 서버 안에서 일어나는 일. 중복 검사 → 나이 검증 → 4테이블 INSERT → 토큰 발급이 한 트랜잭션 안에서 일어난다.

로그인 — 사용자 시퀀스

짧지만 분기가 많다. 사용자 존재 → 상태 정상 → 비밀번호 일치 세 조건을 모두 통과해야 토큰이 발급된다. 어느 단계에서든 실패하면 같은 “자격 증명 오류” 메시지로 응답해서 공격자가 “이메일이 존재하는지”를 추측하지 못하게 한다.

가입 검증 분기

가입 요청 하나가 서버에서 통과해야 할 4단계 검증. 어디서 막혀도 사용자 레코드는 만들어지지 않는다.

사용자 시나리오 10개

아래 시나리오는 모두 동일한 형식 — 페르소나 / 시작 상태 / 단계 / 종료 상태. 시퀀스의 각 단계는 사용자가 인지하는 행동 또는 화면 응답 기준으로 정리한다.

S1 · Happy Path · 가입

신규 회원이 이메일로 가입한다

페르소나 — 30대 직장인, 친구 추천으로 앱 설치. 이메일 가입 선호.

시작 — 앱 첫 실행, 토큰 없음. 스플래시가 로그인 화면으로 보내준 상태.

  1. 로그인 화면 하단 “회원가입” 링크 탭 → 회원가입 화면 진입
  2. 이메일 입력 → 정규식 검증 통과
  3. 비밀번호 입력 → 강도 인디케이터가 “강함”으로 표시
  4. 비밀번호 확인 입력 → 일치 표시
  5. 닉네임 입력 (2~16자) → 통과
  6. 생년월일 picker로 선택
  7. 약관 동의: 이용약관·개인정보 필수 체크, 마케팅 선택
  8. “가입하기” 탭 → 서버에 가입 요청
  9. 서버: 4개 레코드 동시 INSERT 후 토큰 발급
  10. “회원가입이 완료되었습니다. 로그인해주세요.” 토스트 + 로그인 화면 복귀

종료 — 사용자 레코드 생성 (상태=정상, 이메일 미인증), 토큰 저장, 로그인 화면 표시.

S2 · Happy Path · 첫 로그인

가입 직후 로그인하여 이메일 인증 화면으로 진입한다

페르소나 — S1을 막 마친 신규 회원.

시작 — 로그인 화면, 토큰은 있으나 이메일 미인증.

  1. 이메일 + 비밀번호 입력 후 “로그인” 탭
  2. 서버: 비밀번호 검증 후, 이전 로그인 시각이 없는 것을 보고 첫 로그인 사용자로 판정
  3. 토큰 발급 + 사용자 정보 반환
  4. 클라이언트: 첫 로그인 신호를 받고 온보딩 저장소 초기화
  5. 본인 정보 API로 온보딩 완료 여부 확인
  6. 이메일 미인증 사용자라 우선 이메일 인증 화면(F01-03)으로 보냄

종료 — 사용자 상태=정상, loginAt 기록, 이메일 인증 또는 온보딩 화면 진입.

S3 · 실패 · 자격증명

잘못된 비밀번호로 로그인 실패

페르소나 — 등록 사용자, 비밀번호 오타.

시작 — 로그인 화면, 이메일은 맞지만 비밀번호 오타.

  1. “로그인” 탭 → 서버에 로그인 요청
  2. 서버: 사용자 발견, 비밀번호 해시 비교 실패 → 403 LOGIN_FAILURE_BAD_CREDENTIAL
  3. 클라이언트: “이메일 또는 비밀번호가 올바르지 않습니다” 토스트
  4. 사용자가 비밀번호 재입력 또는 “비밀번호 찾기” 탭

종료 — 로그인 화면 유지, 토큰 미발급. 보안 노트: 사용자 미존재일 때도 동일한 에러 코드/메시지로 응답.

S4 · 실패 · 중복

이메일 중복 가입 시도

페르소나 — 이미 가입한 이메일로 다시 가입을 시도하는 사용자.

시작 — 회원가입 화면.

  1. 같은 이메일 + 새 비밀번호/닉네임 입력
  2. “가입하기” 탭 → 서버 응답 409 USER_ALREADY_EXISTS
  3. 클라이언트: “이미 사용 중인 이메일입니다” 토스트

종료 — 회원가입 화면 유지.

S5 · 정책 차단 · 미성년자

만 19세 미만 가입 차단

페르소나 — 만 18세, 모든 필드 입력 완료.

시작 — 회원가입 화면.

  1. 생년월일 picker에서 만 19세 미만 날짜 선택
  2. “가입하기” 탭 → 서버에서 만 19세 미만 거부 → 403 UNDERAGE_NOT_ALLOWED
  3. 클라이언트: 차단 토스트

종료 — 사용자 레코드 미생성, 화면 유지. 주의: 클라이언트 picker는 기본값이 2000-01-01이라 “별다른 입력 없이 통과”하는 케이스가 없도록 picker UI를 항상 명시적 선택으로 둔다.

S6 · 정책 차단 · 상태

잠금·탈퇴 상태 사용자의 로그인 시도

페르소나 — STOP / BAN / TRYEXIT / EXIT 상태 사용자.

시작 — 로그인 화면.

  1. 정상 이메일·비밀번호 입력 후 “로그인” 탭
  2. 서버: 비밀번호는 맞지만 상태가 비정상 → 403 LOGIN_FAILURE_USER_STATUS_*
  3. 클라이언트: 상태에 맞는 차단 메시지 토스트

종료 — 로그인 거부, 화면 유지. 토큰 미발급.

S7 · 실패 · 중복

닉네임 중복으로 가입 실패

페르소나 — 이미 사용 중인 닉네임을 시도.

  1. “가입하기” 탭 → 서버 응답 409 NICKNAME_ALREADY_EXISTS
  2. 클라이언트: 토스트 표시
  3. 닉네임만 바꿔 재시도

종료 — 가입 실패. 사용자가 닉네임 수정 후 재시도하면 통과한다.

S8 · 검증 · 비밀번호

비밀번호 검증 정책 위반

페르소나 — 단순 비밀번호(“abc123”)를 시도한 사용자.

  1. 비밀번호 입력 즉시 클라이언트 측 검증 실패 → 인라인 에러 “최소 8자 이상 입력해주세요”
  2. 가입 버튼은 비활성
  3. (클라이언트 검증을 우회한 경우) 서버에서 동일 정책으로 한 번 더 검증 → 422 VALIDATION_ERROR

종료 — 가입 차단. 이중 방어: 화면과 서버 둘 다 같은 정책을 적용.

E2E-1 · 단일 세션 라운드트립

가입 → 로그인 → 메인 셸 진입

출처 — integration_test/auth_flow_test.dart

페르소나 — 첫 실행 사용자, 가입 직후 즉시 로그인까지 확인하려는 신규 회원.

  1. 로그인 화면(슬로건 “함께하는 즐거움을 발견하세요”) 도달
  2. “회원가입” 탭 → 가입 폼 진입
  3. 이메일/비밀번호/비밀번호 확인/닉네임 입력
  4. 생년월일 picker → 확인
  5. “전체 동의” 탭 → “가입하기” 탭 → 로그인 화면 복귀
  6. 동일 자격증명으로 로그인
  7. 최대 30초 폴링: 메인 탭(홈/검색/클럽/알림/프로필) 또는 온보딩 첫 페이지가 보이면 통과

종료 — 메인 셸 또는 온보딩 첫 페이지 진입, 둘 중 하나는 반드시 도달.

E2E-2 · 시드 사용자 단발 로그인

기존 시드 사용자 로그인 → 홈/온보딩 분기 허용

출처 — integration_test/seed_login_flow_test.dart

페르소나 — 시드 데이터에 미리 존재하는 사용자.

  1. 콜드 스타트 → 로그인 화면 도달 확인(슬로건 가시)
  2. 시드 자격증명 입력 후 “로그인” 탭
  3. 30초 폴링: 메인 탭 또는 온보딩 첫 페이지가 보이면 통과

종료 — 온보딩 완료 여부에 따라 둘 중 하나로 분기. 정확한 분기 검증은 별도 회귀 테스트가 담당한다.

화면 구성 — 사람이 보는 것

코드 레벨 디테일은 생략. “사용자가 화면을 열었을 때 무엇이 보이고 무엇을 할 수 있는가”만 정리한다.

서버 계약 한눈에

기능을 책임지는 두 엔드포인트. 응답 형식이 같아서 클라이언트는 가입과 로그인 이후의 후속 처리를 동일하게 한다.

Method Path 인증 핵심 동작
POST /api/v1/auth/signup 불필요 신규 회원 생성 + 토큰 발급 (201 Created). 사용자·회원·약관·이메일 로그인 수단 4개 레코드 동시 INSERT.
POST /api/v1/auth/login 불필요 자격 증명 검증 + 상태 검증 + 토큰 재발급 (200 OK). 이전 리프레시 토큰은 단일 기기 정책으로 일괄 폐기.
응답 본문 (가입·로그인 공통)

액세스 토큰(JWT, 1시간) · 리프레시 토큰(UUID, 30일) · 사용자 ID · 이메일 · 닉네임 · 역할 · 첫 로그인 여부. “첫 로그인 여부”는 가입 직후이거나 이전 로그인 시각이 비어 있을 때 참이 되고, 클라이언트가 온보딩/이메일 인증으로 분기하는 핵심 신호다.

에러 코드 매트릭스
  • 409 USER_ALREADY_EXISTS — 이미 가입된 이메일
  • 409 NICKNAME_ALREADY_EXISTS — 중복 닉네임
  • 403 UNDERAGE_NOT_ALLOWED — 만 19세 미만 가입 시도
  • 400 INVALID_INPUT — 생년월일 누락
  • 403 LOGIN_FAILURE_BAD_CREDENTIAL — 이메일 미존재 또는 비밀번호 불일치 (동일 코드로 응답해 사용자 존재 여부 노출 방지)
  • 403 LOGIN_FAILURE_USER_STATUS_BAN / STOP / TRYEXIT / EXIT — 비정상 상태 사용자
  • 422 VALIDATION_ERROR — 비밀번호 정책 위반(서버 측 재검증)

자주 빠뜨리는 케이스

레드 플래그
  • 가입 성공 후 “자동 로그인된 채로” 화면이 그대로 머무는 회귀 — 현재 정책은 토스트 후 로그인 화면 복귀. 이걸 “자동 진입”으로 바꾸면 약관·이메일 인증 분기와 어긋난다.
  • 이메일 미존재 vs 비밀번호 불일치를 다른 메시지로 응답 — 사용자 열거 공격에 노출. 둘은 같은 코드/메시지를 유지한다.
  • 생년월일 picker 기본값을 그대로 두고 가입 통과 — picker는 명시 선택을 강제해야 한다.
  • 가입 트랜잭션 부분 실패 — 사용자·회원·약관·로그인 수단 INSERT가 모두 같은 트랜잭션이어야 한다. 중간 실패 시 전부 롤백.
  • 로그인 시 이전 리프레시 토큰을 남겨두는 경우 — 단일 기기 정책 위반. 다른 기기에서 동시 세션이 살아남아 토큰 갱신 충돌(F01-05) 회귀로 이어진다.
  • 잠금/탈퇴 상태에서 비밀번호 검증을 먼저 하는 경우 — 상태 검증을 먼저 통과해야 비밀번호를 비교한다(타이밍 정보 노출 방지).
