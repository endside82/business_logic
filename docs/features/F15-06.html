F15-06. 경고 부여 & 원장 조정

운영진이 제보를 거치지 않고 직접 점수를 부여하거나(한 명/여러 명), 이미 부여된 경고를 감경·정정·만료로 조정하는 면. 경고 원장은 “추가만 되는” 장부라 한 번 부여된 기록은 지워지지 않고, 조정도 새 기록으로 쌓여 점수에 반영된다.

설계 의도 — 같은 작업을 두 번 눌러도 두 번 부여되지 않는다

운영 화면은 네트워크 재시도나 더블클릭이 흔하다. 그래서 부여·일괄 만료는 같은 요청이 반복돼도 한 번만 처리된다. 원장 자체도 “같은 출처의 같은 건”은 한 번만 들어가도록 고유키로 막혀 있어, 이중으로 중복 부여를 방지한다. 감경은 남은 유효 점수를 넘을 수 없고, 정정·만료는 한 경고당 한 번만 가능해 “과하게 깎거나 두 번 무효화”되는 일을 막는다.

사용자 시나리오

직접 부여

제보 없이 한 멤버에게 경고를 준다

대상·유형·사유로 부여. 원장에 부여 기록이 생기고 유효 점수가 올라간다. 여러 명을 한 번에 부여할 수도 있다.

중복 방지

같은 부여 요청이 두 번 도착한다

같은 요청이 다시 들어오면 처음 결과를 그대로 돌려주고 새 경고를 만들지 않는다. 더블클릭·재시도에 안전.

감경

경고 점수 일부를 깎아준다

남은 유효 점수 범위 안에서만 감경 가능. 초과하면 “유효 점수 초과”로 막힌다.

정정/만료

잘못 부여된 경고를 정정하거나 만료한다

정정·만료는 한 경고당 한 번만. 이미 정정·만료된 경고에 다시 시도하면 막힌다.

일괄 만료(사면)

여러 경고를 한 번에 만료한다

사면 같은 정책으로 여러 경고를 일괄 만료. 이미 만료된 건은 건너뛰고 “성공 N건/실패 M건”으로 결과가 온다.

운영자가 알아야 할 것 — 비즈니스 리스크

일괄 부여의 부분 성공과 응답 형태
  • 여러 명 일괄 부여는 “부분 성공”이 가능하다. 일부 멤버만 부여되고 나머지는 실패 목록으로 돌아올 수 있어, 운영진이 실패한 대상을 다시 처리해야 한다. 실패 목록 처리 UX가 명확해야 한다.
  • 부여 결과의 응답이 타입이 느슨하다(dynamic Map). 필드 오타가 런타임에야 드러날 수 있어 전용 결과 모델로 정리하는 게 좋다.
  • 조정 시 사유를 빼먹으면 “유효 점수 초과”라는 엉뚱한 에러가 뜬다. 실제 원인과 메시지가 다르다.
