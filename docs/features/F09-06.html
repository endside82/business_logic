F09-06. 만남 제안

매칭된 두 사용자가 오프라인으로 만나기 위한 일시·장소·메모를 주고받는 단위. 채팅이 무르익은 시점에 한쪽이 제안하고, 상대가 받아들이면 약속이 확정된다. 데이팅의 가치 사슬에서 가장 마지막 전환 — 온라인 매칭이 실제 만남으로 이어지는 자리다.

제안 가능 기간
90일
오늘부터 3개월 이내
제안 상태
4단계
제안 → 확정 → 취소 / 완료
위치 공유
안내만
실제 공유 기능은 미운영
설계 의도 — 데이팅 가치 사슬의 마지막 전환을 가볍게 만든다

채팅까지 잘 흘러간 두 사람이 만남으로 넘어가지 못하는 자리에서 데이팅의 가치가 사라진다. 그래서 본 단위는 만남 제안을 한 화면 안에서 끝낼 수 있게 — 일시·장소·메모만 받고 곧장 제안 발송 — 짧게 설계되어 있다. 제안은 수정하지 않고 새로 보낸다. 같은 매칭에 새 제안이 누적되면 클라이언트가 가장 최신 활성 제안만 화면에 보여 줘 사용자가 혼란스러워하지 않는다. SOS·체크인 같은 강한 안전 기능은 의도적으로 미운영 상태로 두고, 위치 공유 안내 메시지로 사용자가 안전 의식을 갖게 하는 데 그친다.

사용자 시나리오

기본 흐름

채팅에서 만남 제안

매칭 후 일주일 정도 대화한 사용자가 채팅 헤더의 📅 아이콘을 누른다. 만남 제안 화면이 열리고, 일시(달력 + 시간)·장소·메모(선택)를 입력한 뒤 "만남 제안하기"를 누른다. "만남을 제안했습니다" 토스트가 떠오르고 자동으로 채팅 화면으로 돌아간다. 상대가 같은 매칭에서 진입하면 노란 "응답 대기 중" 상태 카드를 본다.

이 흐름이 남기는 것 — 데이팅에서 가장 가치 있는 전환을 한 화면, 한 동작으로 끝낸다. 제안의 무게를 가볍게 둬서 사용자가 망설이지 않게 한다.

필수 입력

일시 또는 장소를 빠뜨림

장소만 입력하고 일시를 비운 채로 제안하기를 누른다. "일시를 선택해주세요" 토스트가 뜨고 송신은 막힌다. 반대로 일시만 채우고 장소를 빠뜨리면 "장소를 입력해주세요" 토스트.

이 흐름이 남기는 것 — 만남 제안에서 핵심 두 정보(언제·어디)를 빠뜨리지 않게 한다. 메모는 비워도 된다.

수락

상대가 제안을 받아들임

상대가 같은 화면으로 진입해 "약속 확정" 의사를 표시한다. 두 사용자 모두 화면 다시 진입 시 노란 "응답 대기 중" 대신 초록 "약속 확정" 카드를 본다. 카드 안에 "위치 공유 안내" 버튼이 있어 누르면 "미팅 시작 1시간 전부터 종료까지 자동으로 위치가 공유됩니다" 라는 안내 다이얼로그가 떠오른다.

운영 관찰 포인트 — 위치 공유 실제 기능은 현재 미운영 상태. 안내 텍스트만 노출되는 한계가 있어 사용자가 "공유되고 있다고 믿는다"는 오해 가능. 안내 카피 분리 검토 여지.

새 제안

응답이 없어 다른 일정으로 다시 제안

1차 제안에 상대가 며칠 응답하지 않는다. 사용자가 같은 화면에서 다른 일시·장소로 새 제안을 입력해 보낸다. 화면은 항상 가장 최신 제안만 카드로 보여주므로 사용자에게는 자연스럽게 "새 제안이 활성"으로 보인다.

운영 관찰 포인트 — 서버에는 두 제안이 모두 살아 있다(누적 저장). 1차 제안이 PROPOSED 상태로 그대로 남기 때문에, 운영팀이 매칭의 제안 이력을 봤을 때 여러 row가 보일 수 있다. 향후 1차 자동 취소 정책 고려 여지.

차단 발동

제안 후 한쪽이 상대를 차단

제안을 보낸 후 한쪽이 상대를 차단한다. 매칭 자체가 차단 상태로 바뀌어, 그 매칭에 새 제안을 보내려고 하면 "만남 제안에 실패했습니다" 토스트가 뜬다. 기존 제안 데이터는 그대로 보존된다 — 분쟁 시 운영팀이 참조할 수 있도록.

이 흐름이 남기는 것 — 차단이 매칭과 만남 제안 양쪽을 동시에 끊는다. 안전 가드의 일관성.

화면에서 사용자가 인지하는 것

화면 요소사용자가 보는 것그 자리에 둔 이유
진행 중 상태 카드 현재 활성 제안이 있으면 상단에 색이 다른 카드가 자리. 제안 = 노란색, 확정 = 초록색, 취소·완료 = 회색. "지금 우리 약속이 어디까지 와 있는가"를 색 하나로 인지하게. 본문을 읽기 전에 상태가 먼저 들어온다.
상대 정보 카드 분홍색 fallback 아바타 + "{상대 이름}님과의 만남" + "매칭한 지 N일" 같은 보조문. 제안 화면에서도 누구와의 약속인지 사용자에게 다시 한 번 보여 준다. 매칭 후 시간 감각도 함께 제공.
일시 입력 영역 📅 + "날짜와 시간을 선택하세요" 자리. 누르면 달력 → 시간 선택기 두 단계로 이어진다. 두 단계로 나눠 사용자가 "어느 날, 몇 시"를 차례로 결정. 한 화면에 둘 다 띄우면 결정 무게가 커진다.
장소 입력 📍 + "만남 장소를 입력하세요" 입력창. 자유 텍스트. 장소 검색·지도 연동이 없는 단순 텍스트 입력. 의사결정의 빠른 흐름을 우선.
메모 영역 여러 줄 텍스트 + "{현재}/200" 카운터. 선택 입력. "옷차림, 모임 인사, 늦을 수도 있어요" 같은 추가 안내를 사용자가 자유롭게 보낼 수 있는 자리.
제안 버튼 "💕 만남 제안하기" 분홍색 큰 버튼. 의사결정 동작 하나만 둔다. 제안의 무게를 가볍게 만드는 핵심 디자인.
위치 공유 안내 확정 상태 카드 안의 "위치 공유 안내" 버튼. 누르면 정적 안내 다이얼로그가 떠오른다. 실제 공유 기능은 없지만 사용자가 안전 의식을 갖게 하는 자리. 이 안내가 자리하는 것 자체가 "데이팅이 안전을 신경 쓴다"는 신호.

운영자가 알아야 할 것

현 단위의 한계와 정책 포인트
  • 제안 상태 변경 화면이 아직 없다. 상대가 받은 제안을 수락·거절하는 별도 화면이 현 구현에는 없다. 서버에는 상태를 바꾸는 경로가 열려 있지만 사용자 화면에서 호출하는 자리가 비어 있어, 향후 매칭 상세 또는 알림 진입에서 보완 필요.
  • SOS·체크인·긴급연락처는 의도적으로 운영하지 않는다. 이전 결정으로 서버·클라이언트 양측에서 모두 삭제된 상태. 스펙 문서에 흔적이 남아 있어도 본 단위에는 포함되지 않는다 — 향후 재도입은 별도 작업으로 다룬다.
  • 위치 공유는 안내 메시지일 뿐 실제 공유는 동작하지 않는다. 사용자가 안내 카피를 보고 "공유되고 있다"고 오해할 가능성이 있다 — 안내 카피를 미운영 상태에 맞게 조정하거나, 실제 공유 기능 도입 시점까지 안내 자체를 빼는 선택지를 고려할 여지.
  • 제안 데이터는 누적 저장된다. 같은 매칭에 새 제안이 들어와도 1차 제안이 자동 취소되지 않는다. 운영상 한 매칭에 여러 PROPOSED row가 살아 있을 수 있으므로, 이력 조회 시 가장 최신만 유효 제안으로 본다는 정책을 운영팀이 인지해야 한다.
  • 차단된 매칭에 새 제안은 들어가지 않는다. 안전 가드가 일관되게 작동한다 — 매칭이 차단되면 만남 제안도 함께 막힌다.
  • 제안 일시는 디바이스 로컬 시간 그대로 송신된다. 사용자가 한국에서 입력하든 외국에서 입력하든 동일한 값으로 저장 — 운영상 한국 시간 기준으로 가정한다.
