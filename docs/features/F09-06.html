F09-06. 만남 제안 · 안전 흐름

채팅 중에 일시·장소를 합의해 “만남 제안”을 만들고, 상대가 수락하면 CONFIRMED. 확정 후에는 미팅 1시간 전부터 종료까지 위치 공유가 안내된다. 매치가 차단·만료되면 새 제안 자체가 막히고, 기존 row는 분쟁 기록용으로 보존된다.

SOS·체크인·긴급연락처는 제품 결정으로 서버·클라이언트 양쪽에서 삭제되었다. 스펙 문서(SCR-PD-006)의 관련 항목은 deprecated. 재도입 시 별도 작업.

이 기능 안의 하위 항목

동작 1

만남 제안

일시·장소·메모 입력 → PROPOSED row INSERT. 매치 MATCHED만 허용.

동작 2

수락 / 거절 / 취소

상대 또는 본인이 status 전환. 참여자 검증 필수.

동작 3

위치 공유 안내(CONFIRMED 이후)

“미팅 1시간 전 ~ 종료” 위치 공유 정책 안내. 실제 공유는 F14-01.

동작 4

제안 누적

같은 매치에 새 제안 가능. 서버 가드 없음 → 클라이언트가 “최신 활성” 1건만 노출.

동작 5

차단·만료 시 차단

매치가 BLOCKED/EXPIRED면 새 제안 차단. 기존 row는 보존(분쟁용).

제안 상태 머신

stateDiagram-v2 [*] --> PROPOSED: 사용자가 일시·장소 제안 PROPOSED --> CONFIRMED: 상대 수락 PROPOSED --> CANCELED: 상대 거절 또는 본인 취소 CONFIRMED --> CANCELED: 본인 또는 상대가 취소 CONFIRMED --> COMPLETED: 만남 시각 이후 CANCELED --> [*] COMPLETED --> [*]

제안 → 수락 시퀀스

sequenceDiagram actor A as 사용자 A actor B as 사용자 B participant API as 미팅 API participant SVC as 미팅 서비스 participant DB as DB A->>API: GET /meetings/match/{matchId} API-->>A: [] (없음) A->>API: POST /meetings {date, time, locationName, note} API->>SVC: 제안 등록 SVC->>SVC: 참여자 + 매치 MATCHED 검증 SVC->>DB: meeting INSERT (status=PROPOSED, proposedBy=A) API-->>A: 201 + 카드 갱신 Note over B: B가 같은 라우트 진입 시 PROPOSED 카드 노출 B->>API: PATCH /meetings/{id} {status=CONFIRMED} API->>SVC: 상태 전이 SVC->>SVC: 참여자 검증 SVC->>DB: status=CONFIRMED 갱신 API-->>B: 200 Note over A: A가 다시 진입 시 초록 "약속 확정" 카드 + 위치 공유 안내 버튼

사용자 시나리오 10개

S1 · Happy Path

제안 → 수락

  1. A가 채팅 헤더 📅 → 미팅 화면 진입
  2. 일시·장소·메모 입력 → “만남 제안하기”
  3. POST → PROPOSED + proposedBy=A INSERT
  4. B가 같은 라우트 진입 → PROPOSED 카드 노출
  5. B가 수락 → PATCH status=CONFIRMED
  6. A에 “약속 확정” + 위치 공유 안내 버튼

현재 갭: B에게 자동 푸시는 없음. 채팅 메시지 또는 라우트 재진입으로 인지.

S2 · 검증

일시·장소 미입력

  1. 장소만 입력 → “일시를 선택해주세요”
  2. 일시만 선택 → “장소를 입력해주세요”
  3. 송신 차단
S3 · 권한

다른 매치의 미팅 조작

  1. 자기 매치 아닌 meetingId로 PATCH 시도
  2. 참여자 검증 실패 → 403 FORBIDDEN
S4 · 매치 게이트

BLOCKED/EXPIRED 매치에 제안

  1. POST → match.status≠MATCHED → DATE_MATCH_NOT_FOUND
  2. 토스트 “만남 제안에 실패했습니다”
S5 · 누적

같은 매치에 새 제안

  1. 1차 PROPOSED 미응답 상태에서 새 제안
  2. 서버 단순 INSERT → row 누적
  3. 클라이언트가 id 최댓값으로 “최신 활성”만 노출

운영 노트: 서버 가드가 없어 1차가 PROPOSED로 남는다. 분쟁 시 전체 row가 기록.

S6 · 위치 공유

CONFIRMED 후 안내

  1. 카드 → “위치 공유 안내” 버튼
  2. AlertDialog: “미팅 시작 1시간 전부터 종료까지 자동으로 위치가 공유됩니다”
  3. 실제 공유 구현은 F14-01에 의존
S7 · 미운영

SOS / 체크인

의도적으로 운영하지 않음. 서버·클라이언트 양쪽 삭제. 별도 작업으로만 재도입.

S8 · 차단

차단 발동 → 미팅 중단

  1. PROPOSED 상태에서 차단(F09-07)
  2. 매치 BLOCKED → 메시지 송신 차단 + 새 제안 차단
  3. 기존 미팅 row는 보존(분쟁 기록)
S9 · 신고

본 화면에 신고 액션 없음

차단으로 단절. 신고 자체는 F11-04(외부 도메인).

S10 · 시간대

KST 가정

서버는 LocalDateTime/LocalTime, 클라이언트는 디바이스 로컬 시간 그대로 송신. 운영상 KST 가정.

자주 빠뜨리는 케이스

레드 플래그
  • 매치 상태 게이트 누락 — 차단/만료된 매치에 미팅이 만들어지면 사용자가 갇히는 흐름.
  • 제안 누적 시 1차 PROPOSED를 자동 CANCELED 처리하지 않음 — 운영상 분쟁 추적은 가능하지만 사용자 인지가 혼란.
  • CONFIRMED 후 알림 누락 — 상대가 약속을 인지 못 함. 자동 푸시 도입 후보.
  • 참여자 검증 누락된 PATCH — 외부자가 임의로 status 전이 가능.
  • 위치 공유 안내를 실제 활성화로 오해 — 안내는 F14-01에 의존. F09-06 단독으로 활성화 안 됨.
  • SOS 흐름을 다시 추가 — 의도적 미운영. 재도입은 별도 작업으로.
