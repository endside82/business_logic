F08-11. 아이템 · 번들 · 플랜 구매

마켓의 모든 구매가 통과하는 단일 경로. 단건 아이템, 번들(여러 아이템 + 보너스 포인트), 플랜 직접 구매를 분기 처리한다. 잔액 · 재고 · 월 한도 · 본인 차단 네 가드를 통과해야 결제·컬렉션 INSERT·크리에이터 수익 PENDING이 한 트랜잭션 안에서 일어난다.

한 구매가 만드는 데이터 6가지
  1. 지갑 차감 + PointTransaction(PAY)
  2. MarketPurchase INSERT (단건/번들 구분)
  3. Collection INSERT × N (번들은 아이템 수만큼)
  4. CreatorEarning INSERT (PENDING, gross/fee/tax/net 계산)
  5. 아이템 purchaseCount++ + 재고 차감
  6. 번들 보너스 포인트 적립 (BundleCurrency가 있을 때)

이 기능 안의 하위 항목

동작 1

단건 아이템 구매

가장 흔한 경로. 잔액 → 재고 → 월 한도 가드 통과 후 결제 + 컬렉션 + 수익 PENDING 동시 INSERT.

동작 2

번들 구매

여러 아이템 묶음 가격. 컬렉션 N건 INSERT + 번들 보너스 포인트 자동 적립. 번들별 “인당 구매 횟수” 한도 별도.

보너스 포인트
동작 3

플랜 직접 구매

마켓 아이템을 거치지 않고 플랜 자체 구매. 본인 플랜 차단·1인 1구매는 플랜 경로에서만 적용(마켓 아이템은 중복 가능).

1인 1구매
동작 4

무료 플랜 구매

가격 0인 플랜은 결제 스킵. PlanPurchase만 생성(transactionId=null).

동작 5

크리에이터 수익 PENDING

구매 시점에 gross/fee/tax/net을 계산해 CreatorEarning을 PENDING으로 적립. 정산 사이클에서 PAID로 전이(F06-10).

크리에이터에게 가는 금액은 PENDING으로 적립되고, 정산 사이클에 PAID로 전이된다.

사용자 시나리오 8개

S1 · Happy Path

단건 구매 — 잔액 충분

잔액 50,000P, 30,000P 아이템.

  1. “구매하기” 탭 → spinner 표시
  2. 가드 통과 → 지갑 −30,000 + PAY 거래
  3. MarketPurchase + Collection(isActivated=false, qty=1) INSERT
  4. CreatorEarning(gross=30,000 / fee=3,000 / tax=891 / net=26,109) PENDING INSERT
  5. “구매 완료” 다이얼로그 → 컬렉션 화면(F08-12)으로 이동

종료: 비활성 상태 카드 표시, 사용자가 “활성화” 시점에 사용 가능.

S2 · 잔액 부족

충전 분기로 전환

  1. POST /purchase → 400 INSUFFICIENT_BALANCE + PaymentShortfall
  2. “포인트 부족” 다이얼로그 → “충전하기”
  3. 충전 화면을 부족액 prefill로 push
  4. 충전 후 다시 구매 시도
S3 · 번들

3개 아이템 + 1,000P 보너스

  1. “이 번들로 구매하기” 탭
  2. 번들 가드(countPerPerson=1) + 월 한도 통과
  3. 지갑 차감 + 번들 재고 차감
  4. MarketPurchase(BUNDLE) + Collection INSERT × 3
  5. BundleCurrency 1,000 보너스 → 지갑 +1,000(FREE_POINT)

참고: 보너스는 무료 포인트로 적립되어 우선 소비된다.

S4 · 중복

이미 구매한 아이템 재시도

  1. 마켓 아이템: 정책상 중복 허용 (수량 누적). 컨트롤러 가드 없음.
  2. 플랜 직접 구매: existsByPlanIdAndBuyerId → 409 PLAN_ALREADY_PURCHASED
  3. 클라이언트: 409 시 “이미 구매한 플랜입니다” 토스트

UX 부정합: 마켓 아이템도 “보유 중” 상태에서는 UI가 중복 구매를 막아야 하지만 서버는 허용. 추후 정렬 필요.

S5 · 동시성

마지막 한 개에 두 명 동시 구매

  1. stockQuantity=1 상태
  2. 첫 번째 사용자: 트랜잭션 내 stockQuantity=0 + 구매 완료
  3. 두 번째 사용자: stockQuantity≤0 감지 → 400 ITEM_STOCK_EXHAUSTED

UI 보강: 현재 ITEM_STOCK_EXHAUSTED 전용 다이얼로그 미구현 → 일반 400 핸들러로 폴백.

S6 · 한도

월 50건 도달

  1. 한 달 안에 50건 누적 후 51번째 시도
  2. checkMonthlyLimit → 400 PURCHASE_LIMIT_EXCEEDED
  3. 익월까지 마켓 구매 차단
S7 · 본인 차단

크리에이터가 자기 플랜 구매 시도

  1. plan.creatorId == buyerId → 400 PLAN_CANNOT_PURCHASE_OWN
  2. 토스트로 안내

주의: 마켓 아이템 경로에는 가드 없음. 본인 플랜이라도 구매 가능 → 정책 정렬 필요.

S8 · 무료

가격 0 플랜 구매

  1. plan.price=0 → 결제 스킵, transactionId=null
  2. PlanPurchase INSERT(pricePaid=0)
  3. purchaseCount 증가

종료: 무료 플랜 보유함 추가, 지갑 무변동, CreatorEarning도 생성 안 됨.

자주 빠뜨리는 케이스

레드 플래그
  • 가드 순서 뒤집힘 — 잔액 가드를 재고/한도보다 먼저 실행하면, 잔액 없는 사용자에게 재고가 “예약 차감”되는 회귀.
  • 마켓 아이템 중복 구매를 UI는 막는데 서버는 허용 — 빠른 더블 탭에서 두 번 결제되는 사용자 불만 발생.
  • CreatorEarning 누락 — 정산 사이클에 수익이 안 잡혀 크리에이터 미정산 상태.
  • 번들 보너스가 유료 포인트로 적립 — 환불 정책과 충돌. 보너스는 반드시 무료 포인트.
  • 본인 플랜 차단을 클라이언트만 — UI 우회 시 본인이 본인 수익 만들기 가능.
  • 무료 플랜에 CreatorEarning 생성 — net=0인 PENDING이 쌓여 정산 처리에 혼란.
  • 재고 차감을 트랜잭션 밖에서 — 결제 실패 시 재고가 영구 차감.
