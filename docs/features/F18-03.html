F18-03. 분쟁 이의제기 (DisputeAppeal)

통합 케이스 조회에서 확인한 케이스 결정에 동의하지 않을 때 공식 이의를 제기하는 기능. 이의는 케이스당 본인 1건으로 제한되며, PENDING 상태에서만 본인이 철회할 수 있다. 승인·기각은 admin API가 처리한다. 노쇼(EVENT_NO_SHOW) 케이스에 한해서만 이의제기 시 원본 도메인 상태가 동기화된다 — EVENT_NO_SHOW 외 prefix는 이의 기록(audit)만 남는다. 강퇴/차단(CLUB_MEMBERSHIP_ACTION) prefix는 제출 전 본인 가시성 검증이 이루어진다.

이의제기는 케이스 결과에 대한 재검토 요청이다. 사용자가 이의를 제기해도 원천 케이스의 상태는 즉시 바뀌지 않는다. admin이 이의를 인용(UPHELD)하거나 기각(REJECTED)하고 필요한 경우 원천 케이스 상태를 별도로 처리한다. 유일한 예외가 노쇼 이의인데, 이 경우에 한해서만 이의 제출 시 EventNoShow.status가 APPEALED로 자동 전환된다.

이의 상태가 거치는 길

stateDiagram-v2 [*] --> PENDING: 이의 제출\n(케이스당 본인 1건) PENDING --> CLOSED: 본인 철회\n(공개 API) PENDING --> UPHELD: admin 인용\n(admin API 전용) PENDING --> REJECTED: admin 기각\n(admin API 전용) UPHELD --> [*]: 종결 (재전이 불가) REJECTED --> [*]: 종결 (재전이 불가) CLOSED --> [*]: 종결 (재전이 불가) note right of PENDING canWithdraw = true 철회 버튼 표시 end note note right of UPHELD EVENT_NO_SHOW라면 원본 노쇼 상태도 변경됨 end note
UPHELD/REJECTED/CLOSED는 모두 종결 — 재전이 불가

이의가 인용(UPHELD)됐거나 기각(REJECTED)됐거나 본인이 철회(CLOSED)했다면 더 이상 상태가 바뀌지 않는다. 동일 케이스에 이의를 다시 제기하는 것도 불가능하다. 재심이 필요하면 직접 분쟁 접수(F18-02)로 새 케이스를 만들어야 한다.

사용자 시나리오

기본 흐름

노쇼로 기록됐는데 실제로는 참석했다

케이스 상세 화면에서 "이의제기" 버튼을 탭한다. 사유 코드(FACT_INCORRECT)를 선택하고 "현장에 있었지만 체크인이 안 됐습니다"를 20자 이상 입력한 뒤 제출한다. 이의가 접수되면 케이스 상세에 PENDING 배지가 표시된다. EVENT_NO_SHOW 케이스이므로 노쇼 원본 기록도 APPEALED 상태로 전환된다.

이 흐름이 남기는 것 — 참가자 입장에서 일방적 노쇼 기록을 그냥 받아들이지 않아도 된다는 표시가 서비스 내에서 가능해진다.

기타 원천

클럽 경고 처분이 부당하다고 이의

WARNING_APPEAL 케이스에 대해 이의를 제기할 수 있다. 이의 기록이 dispute_appeal에 남지만 원본 WarningAppeal의 상태는 변하지 않는다. 이 케이스의 최종 처리는 admin이 담당한다.

이 흐름이 남기는 것 — 모든 케이스에 이의 경로가 열려 있지만, 노쇼를 제외한 원천 케이스는 즉각 변화가 없다는 것을 사용자에게 알려야 한다.

이의 철회

제출 후 마음이 바뀌었다

이의가 PENDING 상태인 동안은 철회할 수 있다. 케이스 상세에서 "철회" 버튼을 누르면 CLOSED로 전환된다. UPHELD 또는 REJECTED 이후에는 철회 버튼이 사라진다.

중복 차단

이미 이의를 낸 케이스에 다시 시도

동일 케이스에 이미 이의를 제출한 상태에서 다시 제출하려 하면 서버가 409를 반환한다. PENDING이든 UPHELD/REJECTED든 기존 이의가 있으면 재접수가 차단된다.

결과 확인

admin이 이의를 인용하거나 기각했다

admin 처리 후 케이스 상세에서 UPHELD 또는 REJECTED 배지로 확인할 수 있다. 인용(UPHELD)됐다면 별도 알림 또는 케이스 목록 재조회로 결과를 인지한다. 결과에 따른 원천 케이스 처리(환불·제재 조정 등)는 admin이 별도로 수행한다.

이의 사유 코드

클라이언트 드롭다운 5값 / 서버는 임의 문자열 허용

앱 이의제기 화면은 아래 5가지 코드를 드롭다운으로 제공한다. 서버 AppealCreateParam.reasonCode는 @NotBlank @Size(max=40)으로만 검증하며 enum whitelist가 없으므로, 40자 이내 임의 문자열을 허용한다.

사유 코드 (클라 드롭다운)언제 쓰나
FACT_INCORRECT결정의 기반이 된 사실이 틀렸다
EVIDENCE_MISSING판단 시 중요한 증빙이 누락됐다

WarningAppeal(F15-03)과의 경계

이름이 비슷하지만 완전히 다른 두 이의 시스템

클럽 경고·징계에 대한 이의는 F15-03 WarningAppeal이 별도 경로(POST /api/v1/clubs/{clubId}/warnings/appeals)로 처리한다. 이 기능(F18-03)의 DisputeAppeal과 F15-03의 WarningAppeal은 서버 enum, endpoint, 테이블, 상태값이 모두 다르다. F18-01 통합 조회에서는 WarningAppeal도 WARNING_APPEAL:{id}로 보이지만, F18-03 이의제기를 통해 이 케이스에 이의를 낼 경우 원본 WarningAppeal 상태는 변하지 않는다.

운영자가 알아야 할 것

운영 주의사항
  • EVENT_NO_SHOW 외 prefix 케이스는 이의 제출 후 원본이 바뀌지 않는다. audit 기록만 생기며, 실제 원천 케이스를 처리하려면 admin이 별도로 각 원천 도메인에서 조치해야 한다. CLUB_MEMBERSHIP_ACTION prefix는 제출 전 본인(영향받은 멤버) 가시성을 검증한다.
  • 이의 기한 정책이 v1에 없다. 케이스 종결 후 수개월이 지나도 이의 제출이 이론적으로 가능하다. 원천 케이스가 종결됐더라도 이의를 막는 기한 제약이 없다.
  • 이의 승인·기각 공개 endpoint가 없다. 호스트용 appeal 처리가 앱에서 가능해 보이도록 버튼이 있지만 실제 서버 endpoint가 미구현이다. admin 화면을 통해서만 처리 가능하다.
  • 노쇼 sync는 best-effort다. EventNoShowService 호출이 실패해도 이의 기록 자체는 남는다. 노쇼 상태가 APPEALED로 바뀌지 않았더라도 이의는 접수된 것이다.

레드 플래그

현재 알려진 미구현·주의 사항
  • 이의 승인·기각 공개 API 없음. actorPermissions.canApproveAppeal=true인 케이스에서 호스트 앱에 버튼이 있으나 동작하지 않는다. admin API 경로만 존재한다.
  • audit-only 이의의 의미 미안내. EVENT_NO_SHOW 외 케이스에 이의를 제출해도 원본 케이스 상태가 바뀌지 않는다는 안내가 앱에 없다. 사용자가 이의 제출 후 원본이 바뀌길 기대할 수 있다.
  • 이의제기 기한 미구현. 서버 소스에 v1 미구현으로 명시되어 있다. 노쇼 확정 수개월 후에도 이의가 가능하다.
