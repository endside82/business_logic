F20-02. 운영 이슈

특정 트랜잭션·정산·신고 객체에 직접 연결해 접수하는 구조화된 지원 채널. 결제·환불·거래·정산·신고·정기모임 결제·정기모임 정산 7가지 소스 타입이 있고, 각 타입마다 소유권 검증 방식이 다르다. 접수 즉시 서버가 호스트 매핑 가능한 source(PAYMENT/REFUND/SETTLEMENT/REGULAR_MEETING_*)에 한해 관련 호스트를 자동으로 찾아 라우팅 키를 심어두고, WALLET_TRANSACTION·REPORT는 null(CS 직접 처리)이다. 증빙 파일을 최대 5개 첨부할 수 있다.

왜 "특정 객체에 연결"하는가

결제나 환불 문제는 "언제 무엇이 얼마"라는 맥락이 없으면 처리가 느리다. 소스 객체 ID를 필수로 받아 서버가 자동으로 거래 이력을 추적하고 관련 호스트를 찾아두면, 담당자가 배경 조사 없이 바로 처리할 수 있다. 증빙 첨부와 가시성 분류(양측 공개·호스트 전용·CS 전용)로 민감 정보도 안전하게 전달된다.

사용자 시나리오

결제 / 환불 문의

"환불 요청한 지 3일이 됐는데 처리가 안 돼요"

거래 내역에서 해당 거래 항목 옆 "문의하기"를 누른다. sourceType=REFUND, sourceId=거래 ID가 자동으로 채워진다. 내용을 쓰고 접수하면 서버가 그 거래가 어느 이벤트의 환불인지 찾아 이벤트 호스트 ID를 라우팅 키에 저장한다.

이 흐름이 남기는 것 — HostInbox에서 호스트도 상황을 볼 수 있고, CS도 거래 번호를 따로 요청하지 않고 바로 처리할 수 있다.

정산 문의

"이번 모임 정산 금액이 제가 계산한 것과 달라요 (호스트)"

정산 상세 화면에서 "정산 문의하기"를 누른다. sourceType=SETTLEMENT, sourceId=정산 ID가 채워진다. 정산 참가자(creator 또는 항목 holder/share user)만 접수 가능하다. 증빙으로 계산 근거 파일을 첨부하면 가시성이 PARTIES로 설정돼 운영자와 호스트 모두 볼 수 있다.

신고 결과 문의

"제가 낸 신고 결과가 어떻게 됐는지 모르겠어요"

신고 내역에서 해당 신고를 누르고 "문의하기"를 탭한다. sourceType=REPORT, sourceId=신고 ID로 접수된다. 신고를 낸 사람(reporter)만 접수 가능하다. 증빙 가시성이 자동으로 CS_ONLY로 설정돼 피신고자에게 내용이 노출되지 않는다.

자동 호스트 라우팅

호스트 매핑 가능한 source만 자동 라우팅, WALLET_TRANSACTION·REPORT는 CS 직접 처리

운영 이슈가 생성되는 순간 서버가 sourceType에 따라 관련 호스트를 자동으로 추적한다. PAYMENT/REFUND/SETTLEMENT/REGULAR_MEETING_PAYMENT/REGULAR_MEETING_SETTLEMENT는 PointTransaction 또는 MeetingSettlement를 통해 hostUserId·creatorUserId를 채운다. WALLET_TRANSACTION은 일반 거래라 단일 호스트 매핑이 모호해 null, REPORT는 target에 따라 단일 호스트로 환원되지 않아 null — 두 타입 모두 CS가 직접 처리한다. HostInbox는 이 값으로 이슈를 필터해 호스트에게 라우팅한다.

flowchart LR A["PAYMENT/REFUND"] --> B["PointTransaction"] --> C["Event"] --> D["hostUserId"] E["SETTLEMENT"] --> F["MeetingSettlement"] --> G["creatorUserId"] H["REGULAR_MEETING_PAYMENT"] --> I["PointTransaction"] --> J["RegularMeetingPayment"] --> K["RegularMeeting"] --> L["hostUserId"] M["WALLET_TRANSACTION\nREPORT"] --> N["null — CS 직접 처리"]

이슈 상태 흐름

stateDiagram-v2 [*] --> OPEN : 사용자 접수 OPEN --> IN_PROGRESS : 운영자 처리 시작 IN_PROGRESS --> RESOLVED : 완료 IN_PROGRESS --> REJECTED : 반려 OPEN --> RESOLVED : 즉시 완료 OPEN --> REJECTED : 즉시 반려 RESOLVED --> [*] REJECTED --> [*]

RESOLVED 또는 REJECTED 상태가 되면 사용자는 추가 메시지를 보낼 수 없다. 상세 화면에서 상태가 닫힌 경우 메시지 입력창이 숨겨진다.

Flutter 앱에서 증빙 첨부 불가

서버는 파일 첨부를 받을 준비가 돼 있지만, Flutter 접수 화면에 파일 선택 UI가 없고 OperationalIssueCreateParam 모델에 evidenceFileIds 필드가 없다. 현재는 내용만 전달할 수 있다.

알려진 격차

Flutter 불완전 구현
  • 정기모임 소스 타입(REGULAR_MEETING_PAYMENT/REGULAR_MEETING_SETTLEMENT)이 Flutter enum에 없다. 앱에서 정기모임 관련 이슈를 접수하면 잘못된 소스 타입이 전달될 수 있다.
  • 증빙 첨부 UI가 없다. 서버는 최대 5개 파일을 받도록 설계됐다.
  • 답변 알림이 앱으로 오는지 확인이 필요하다. SUPPORT_ISSUE_UPDATED(66) 알림 코드는 정의됐지만 emit 경로 확인 필요.
