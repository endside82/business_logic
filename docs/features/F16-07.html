F16-07. 호스트 제안

이벤트 호스트가 자기 모임에서 활약한 멤버에게 줄 점수를 제안하고, 운영진이 검토 후 승인하면 적립이 실행되는 흐름. 호스트는 직접 점수를 줄 수 없고 제안만 한다 — 점수를 주는 권한과 제안하는 권한을 일부러 분리했다.

설계 의도 — 현장을 아는 사람이 제안하고, 책임지는 사람이 승인한다

누가 모임에서 활약했는지는 호스트가 가장 잘 안다. 하지만 점수를 직접 주게 하면 남용 위험이 있다. 그래서 호스트는 "이 멤버에게 이만큼"이라고 제안만 하고, 운영진이 검토 큐에서 승인할 때 비로소 적립이 일어난다. 운영진은 승인 시 점수를 조정할 수도 있다.

제안 한 번이 거치는 흐름

flowchart LR A["호스트: 멤버·점수·사유 제안"] --> B["제출 (SUBMITTED) + 검토 큐 적재"] B --> C{"운영진 결정"} C -->|"승인"| D["적립 실행 + APPROVED"] C -->|"반려 (사유 필수)"| E["REJECTED + 기록"]

사용자 시나리오

제안

호스트가 활약 멤버를 제안

모임 후 호스트가 "이 멤버에게 50점"을 사유와 함께 제안한다. 제안은 검토 큐에 자동으로 올라간다.

권한 가드

호스트가 아닌 사람이 제안 시도

해당 이벤트의 호스트가 아니면 제안 자체가 거절된다.

승인

운영진이 점수를 조정해 승인

운영진이 50점 제안을 30점으로 조정해 승인하면 30점이 적립되고 제안은 승인 상태가 된다.

반려

운영진이 사유와 함께 반려

반려는 사유가 필수다. 사유 없이는 반려할 수 없다.

운영자가 알아야 할 것

제안 워크플로의 한계
  • "추가 정보 요청"·"호스트 철회" 상태가 정의만 있고 동작하지 않는다. 현재는 제출 → 승인/반려만 가능하다.
  • 호스트가 결과를 확인할 경로가 제한적이다. 승인/반려는 운영 화면에서 일어나고, 호스트는 자기 이벤트의 제안 목록만 본다. 승인/반려 결과 알림 흐름의 보강이 필요하다.
  • 제안 적립도 등급에 반영된다. 호스트 제안으로 받은 점수가 등급 산정에 포함되는 것이 의도인지 확인이 필요하다.
