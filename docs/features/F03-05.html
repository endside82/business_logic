F03-05. 이벤트 신청·참석 (참가자)

사용자가 모임 상세에서 "참석 신청"을 누르는 순간 일어나는 모든 분기 — 무료/유료, 승인 불필요/필요, 정원 여유/초과, 대기열 활성/비활성, 본인 취소, 자동 승급, 클럽 멤버십 가드 — 가 본 단위에서 결정된다. 매출이 직접 발생하고, 호스트와 참가자의 신뢰가 처음 맺어지는 자리.

신청 경로
2가지
즉시 참석 / 승인 신청서
참석 상태
3종
참석·대기·취소
동시성 안전
필수
마지막 자리 동시 신청 처리
설계 의도 — 한 버튼 안에 다섯 분기가 동시에 결정된다

사용자가 "참석 신청"을 누르는 한 번의 동작 안에 (1) 호스트인가 아닌가, (2) 이미 신청했는가, (3) 승인 필요한가 즉시 가능한가, (4) 정원 여유 있는가 초과인가, (5) 대기열로 들어가는가 거부되는가가 모두 결정된다. 그 결과로 사용자가 받는 결과는 "참석 확정", "대기 N번째", "심사 중", "거절", "정원 가득"의 다섯 가지로 갈린다. 한 버튼 안에서의 분기가 흐려지면 "내가 신청한 건지 안 한 건지", "참석된 건지 대기인지"를 사용자가 헷갈리고, 그 혼란이 환불 분쟁의 출발점이 된다.

이 단위가 매출의 직접 발생점이다

신청 한 번이 결제·정산·체크인·리뷰의 출발점
  • 유료 이벤트의 결제는 본 단위에서 일어난다. 신청과 결제가 같은 동선 위에 있어야 사용자가 "신청은 했는데 결제가 따로 있다"는 혼란을 겪지 않는다.
  • 본인 상태의 시각적 정확성이 본 단위의 안전 기준. 사용자가 "참석된 거 같은데 아닌가" 혼란을 느끼면 즉시 운영 문의가 들어온다.
  • 대기열 자동 승급이 사용자에게 도달하는 첫 면. 푸시와 화면 강조가 함께 동작하지 않으면 "내가 언제 참석으로 바뀌었지"의 혼란이 누적된다.
  • 본인 취소와 자동 승급의 연쇄. 한 사용자가 취소하면 다음 대기자가 자동으로 승급되고, 그 사용자에게 알림이 도달해야 한다. 이 연쇄가 풀리면 "정원이 비어 있는데 아무도 못 들어오는" 상태가 발생.

사용자 시나리오

기본 흐름

무료 + 정원 여유 + 승인 불필요

사용자가 "참석 신청"을 누른다. 같은 동작 안에서 참석 확정이 일어나고, 액션바가 "참석 취소"로 바뀐다. 현재 인원이 +1로 즉시 갱신된다. 호스트에게 별도 알림은 가지 않는다 (자동 신청 경로이므로).

이 흐름이 남기는 것 — 신청에서 결정까지의 시간이 가장 짧은 경로. 본 경로의 응답이 느리면 사용자가 "버튼이 안 눌리는 줄" 알고 두 번 누르는 사고가 발생.

정원 초과

늦게 발견해서 대기열 등록

정원은 이미 가득. 액션바가 "대기열 등록"으로 바뀌어 있고, 누르면 대기 N번째로 들어간다. 화면에 "대기 3번째로 등록되었습니다" 토스트.

이 흐름이 남기는 것 — "정원 가득"을 사용자가 카드 단계에서부터 인지하게 하는 것이 핵심. 상세에서 처음 알면 답답함이 누적.

자동 승급

대기열에서 참석으로 자동 전환

다른 참석자가 취소하면 1번째 대기자가 자동으로 참석 확정. 푸시 알림 "참석으로 확정되었습니다"가 사용자에게 도달하고, 상세에 다시 들어가면 화면에 1회성 "참석 확정!" 강조가 보인다. 사용자가 한 번 확인한 뒤에는 강조가 사라진다.

이 흐름이 남기는 것 — 자동 상태 전환을 사용자가 명확히 인지해야 노쇼를 막을 수 있다. 알림과 화면 강조가 함께 작동해야 한다.

승인 필요

승인제 이벤트에 신청서 제출

"참석 신청" 대신 "신청서 작성"이 액션바에 보인다. 시트에서 신청 메시지("꼭 참여하고 싶어요") 입력 후 제출. 액션바가 "심사 중"으로 바뀌고, 호스트와 공동호스트에게 새 신청 알림이 도달한다. 승인 후 푸시 → 사용자가 상세 다시 진입 → 액션바 "참석 확정" 표시.

이 흐름이 남기는 것 — 사용자에게 "지금 어느 단계인지"가 액션바에 항상 보여야 한다. 심사 중/승인됨/거절됨이 흐려지면 사용자가 다시 신청하는 사고가 발생.

매출 핵심 분기

유료 + 승인제 — 결제는 승인 후에만

유료 + 승인제 이벤트는 본 도메인에서 가장 분쟁이 자주 일어나는 흐름. 신청 시점에는 결제하지 않는다. 호스트 승인 전까지 돈이 차감되지 않아야 하고, 승인 후 사용자에게 "결제하고 참석 확정" 액션이 노출된다. 결제 성공 후에만 참석 확정 상태로 전환된다.

이 흐름이 남기는 것 — "신청 단계에서 결제"가 일어나면 거절 사용자에게 환불 처리가 필요해진다. 승인을 결제 권한 부여 행위로 분리하는 것이 본 단위의 가장 중요한 정책. 현재 일부 구현이 승인 즉시 참석 확정으로 처리되고 있어, 유료 승인제에서 환불 분쟁 위험이 있음 (자세한 호스트 측 정책은 F03-06으로 위임).

본인 취소

참석 확정 사용자가 취소

액션바 "참석 취소" 탭 → 환불 정책 안내 시트 → 확인 → 취소 처리 + 다음 대기자 자동 승급. 사용자에게는 "참석이 취소되었습니다" 토스트, 자동 승급된 다음 사용자에게는 별도 푸시.

이 흐름이 남기는 것 — 본인 취소 한 번이 다음 사용자의 참석으로 이어지는 연쇄. 두 사용자에게 일관되게 알림이 도달해야 정원·대기열이 정상 동작.

잔액 부족

유료 이벤트 결제 시 잔액 부족

결제 확인 시트에서 "잔액이 부족합니다 + 충전" CTA가 떠오른다. 사용자가 충전 화면(06 결제 도메인)으로 이동해 충전 후 돌아오면 같은 자리에서 결제 다시 시도.

클럽 멤버십 가드

클럽 비멤버가 클럽 이벤트 신청

"클럽 멤버만 참석할 수 있습니다 + 클럽 가입" 안내. 사용자가 클럽 가입 동선(04 클럽 도메인)으로 이동. 본 단위는 가드 트리거만 책임지고 실제 가입은 위임.

중복 신청

이미 신청한 모임에 다시 신청 시도

액션바가 이미 "참석 취소" 또는 "심사 중"으로 바뀌어 있어 정상 흐름에서는 중복 신청이 일어나지 않는다. 우회 호출이 일어나도 "이미 신청한 모임입니다" 안내로 차단.

동시성

마지막 자리 한 자리에 두 사용자가 동시 신청

두 사용자가 같은 순간에 "참석 신청"을 누른다. 한 명만 참석으로 처리되고, 다른 한 명은 "정원이 가득 찼습니다" 안내를 받는다. 서버 측에서 잠금으로 처리되어 두 사용자 모두 참석되는 사고는 없다.

이 흐름이 남기는 것 — 인기 모임의 핵심 안전장치. 동시성 가드가 풀리면 정원 초과가 발생하고, 호스트에게 운영 문제가 즉시 전가된다.

화면에서 사용자가 인지하는 것

영역사용자가 보는 것그 자리에 둔 이유
액션바 라벨 사용자 상태에 따라 다른 글자. "참석 신청"/"대기열 등록"/"신청서 작성"/"심사 중"/"참석 취소"/"대기 취소 (N번째)"/"결제하고 참석 확정". 지금 사용자가 다음에 할 수 있는 한 가지 동작만 같은 자리에 띄워, 결정을 미루지 않게 한다.
신청서 시트 승인 필요 이벤트에서만 등장. 신청 메시지(선택), 본인 정보 자동 표시, "제출" 버튼. 호스트가 누구를 받을지 결정하기 위한 단서. 메시지가 비어 있어도 제출은 가능하지만, 호스트의 결정에 도움이 되는 정보를 사용자가 더할 수 있는 자리.
결제 확인 시트 유료 이벤트에서만 등장. 가격·환불 정책 미리보기·현재 잔액·결제 후 잔액. "결제 후 신청" 버튼. 결제와 신청을 같은 시점에 시각화. "내가 얼마를 내고 얼마를 돌려받을 수 있는지"를 신청 전에 본다.
취소 확인 시트 환불 정책 안내(유료) + "취소 확인" 버튼. 취소 전에 환불 조건을 한 번 더 보여 줘, 사용자가 "취소했더니 환불이 적다"는 분쟁을 미리 차단.
본인 상태 배지 상세 화면 또는 카드에 "참석 중"/"대기 N번째"/"심사 중"/"결제 필요" 라벨. 본인이 어느 단계인지 한눈에. 이 라벨이 흐려지면 사용자의 다음 행동이 결정되지 않는다.
자동 승급 강조 대기 → 참석 자동 전환된 사용자에게 1회성 "참석 확정!" 강조. 자동 상태 전환은 사용자가 인지하기 어렵다. 한 번이라도 명확히 보여 줘야 노쇼를 막을 수 있다.

운영자가 알아야 할 것

신청·참석의 비즈니스 리스크
  • 유료 승인제는 신청 시점에 결제하지 않는다. 결제는 승인 후에만 일어나야 한다. 승인 즉시 결제가 일어나면 거절 사용자에게 환불 처리가 발생하고, 호스트의 운영 비용이 폭증. 현재 일부 구현이 이 분리를 완전히 보장하지 않아 향후 보강이 필요한 영역.
  • 마지막 자리 동시 신청의 동시성 가드가 풀리면 정원 초과가 발생한다. 인기 모임에서는 매번 발생하는 시나리오이므로, 본 단위 회귀 테스트의 최우선 시나리오.
  • 대기열 자동 승급 후 알림 도달. 푸시가 실패하면 사용자가 "참석으로 바뀐 것을 모른 채" 해당 시간에 나타나지 않아 노쇼가 된다. 푸시 도달률 모니터링이 본 단위의 핵심 신호.
  • 본인 취소 → 다음 대기자 자동 승급 → 알림 도달까지가 한 동선. 어느 한 단계가 깨지면 "정원은 비어 있는데 아무도 못 들어오는" 상태가 발생.
  • 액션바 라벨이 본인 상태와 어긋나는 회귀가 가장 자주 발생하는 사고 유형. "심사 중인데 참석 신청 버튼이 다시 보인다", "대기 중인데 마감 버튼이 보인다" 같은 상태 표시 회귀를 본 단위 출시 전마다 확인 필요.
  • 유료 이벤트의 결제 확인 시트는 신청 전에 떠야 한다. 결제 후에 환불 정책을 보여 주면 "알았다면 안 했을 텐데"의 분쟁이 시작된다. 환불 조건 사전 공지가 본 단위의 안전 정책.

측정해야 하는 것

신청·참석이 정상이라고 부르려면
  • 상세 진입 → 신청 완료 전환율 — 신청 결정의 마지막 한 걸음이 본 단위에서 어떻게 작동하는지의 직접 지표.
  • 유료 이벤트의 결제 확인 시트 진입 → 결제 완료 전환율 — 결제 직전 누수의 지점. 잔액 부족 비율이 크면 충전 동선의 부하가 큰 신호.
  • 대기열 자동 승급 → 사용자 인지(상세 재진입)까지의 시간 — 푸시 알림의 도달과 사용자 인지의 차. 시간이 길수록 노쇼 위험.
  • 유료 승인제의 승인 후 결제 완료 비율 — 승인 후 결제 누수의 지점. 너무 낮으면 결제 안내·기한 정책 점검 필요.
  • 본인 취소 → 다음 대기자 자동 승급 도달률 — 100%가 정상. 1건이라도 누수가 있으면 정원 회복 동선이 깨진다.
