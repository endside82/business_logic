F13-07. 계정 즉시 비활성화 (탈퇴)

유예 없는 즉시 탈퇴. 단, 차단 항목(blocking) × 자동 처리 항목(resolvable) 두 분류를 먼저 확인한다. 미정산·진행 중 결제 같은 차단 항목이 있으면 탈퇴 자체가 막히고, 무료 이벤트·클럽 멤버십 같은 자동 처리 항목은 탈퇴 진행 중에 일괄 정리된다. 사전 체크와 DELETE 시점 재검증으로 이중 안전망을 둔다.

두 단계 검증의 이유

사전 체크 통과와 DELETE 호출 사이에 다른 트랜잭션이 새 차단 항목(예: 새 정산 대상)을 만들 수 있다. DELETE 시점에 validateAndResolve가 한 번 더 체크한다.

이 기능 안의 하위 항목

동작 1

사전 체크 (deactivation-check)

차단 / 자동 처리 / 정보 안내 항목을 분류해서 반환. UI는 차단이 있으면 “탈퇴 불가” 다이얼로그.

동작 2

최종 확인 다이얼로그

자동 처리 항목 목록을 그대로 보여줘 사용자가 인지하고 탈퇴를 결정.

동작 3

DELETE + 재검증 + 자동 정리

DELETE 시점에 다시 체크 → 통과 시 무료 이벤트 취소·클럽 탈퇴·매치 만료 일괄 → 익명화 + status=TRYEXIT.

동작 4

세션 종료

리프레시 토큰 일괄 삭제 → 다른 기기는 다음 호출에서 401 → 자연 로그아웃.

사용자 시나리오 5개

S1 · Happy Path

차단 없는 즉시 탈퇴

  1. “계정 탈퇴” → GET /deactivation-check → deactivationAllowed=true
  2. 최종 확인 다이얼로그 → 확정
  3. DELETE /users/me → 자동 처리 실행 + 익명화 + TRYEXIT
  4. 토스트 “계정이 탈퇴되었습니다” + 로그인 화면
S2 · 차단

미정산으로 탈퇴 불가

  1. 사전 체크에서 deactivationAllowed=false + blockingItems
  2. 각 차단 항목의 제목·설명·상세 3건·초과 카운트 표시
  3. DELETE 호출하지 않음, 사용자가 해결 후 재시도
S3 · 자동 정리

무료 이벤트·클럽만 있는 사용자

  1. blockingItems 없음, resolvableItems만 다수
  2. 최종 확인 메시지에 자동 처리 목록 노출
  3. DELETE → 무료 이벤트 취소·클럽 탈퇴·매치 만료 일괄
  4. 익명화 + 세션 종료
S4 · API 실패

사전 체크 실패

  1. 네트워크 등으로 deactivation-check Result.failure
  2. “탈퇴 가능 여부를 확인할 수 없습니다” 토스트
  3. 최종 확인 다이얼로그 미노출
S5 · 재검증

DELETE 시점에 새 차단 발생

  1. 사전 체크는 통과
  2. 그 사이 새 정산 대상 생성
  3. DELETE → validateAndResolve가 다시 차단 감지 → DEACTIVATION_BLOCKED
  4. “계정 탈퇴에 실패했습니다” 토스트

UI 보강 후보: DELETE 실패에서 받은 차단 항목을 다이얼로그에 매핑하는 부분은 현재 미확인. 사용자에게 “무엇이 차단인지” 그 자리에서 보여주는 게 이상적.

자주 빠뜨리는 케이스

레드 플래그
  • 사전 체크 결과를 신뢰 — DELETE 시점에 재검증 안 하면 새 차단을 놓친다.
  • 소유 클럽을 자동 처리에 포함 — 소유권 이전 없이 사라지면 클럽 좀비화. 반드시 차단.
  • 리프레시 토큰 삭제 누락 — 다른 기기가 탈퇴 후에도 살아있는 회귀.
  • 익명화 누락(레코드 그대로) — 개인정보 노출 위험.
  • 유료 이벤트도 자동 취소로 분류 — 환불·정산 일관성 깨짐. 유료는 미정산이면 차단.
  • 차단 항목 사유를 사용자에게 안 보여줌 — 무엇을 해결해야 하는지 모름. 항목별 제목·상세 노출 필수.
