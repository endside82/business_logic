F13-07. 계정 즉시 비활성화 (탈퇴)

30일 유예 없이 "지금 당장 서비스 이용을 멈추고 비공개로 만들고 싶다"는 사용자가 누르는 흐름. 하지만 사용자에게 의무(미정산, 진행 중인 결제, 소유한 클럽, 호스팅 중인 유료 이벤트 등)가 남아 있으면 즉시 비활성화가 차단된다. 그래서 본 단위는 두 단계로 동작한다 — 먼저 "탈퇴해도 되는지" 점검하고, 차단 항목이 없을 때만 실제 비활성화를 수행한다. 진행 중 자동 처리 가능한 활동(무료 이벤트 신청, 클럽 멤버십, 데이팅 매칭 등)은 사용자에게 명시한 뒤 한 번에 정리한다.

설계 의도 — "당장 떠나고 싶다"와 "다른 사용자의 권리"를 동시에 지킨다

한 사용자의 즉시 탈퇴가 다른 사용자의 환불·납부·정산을 막아서는 안 된다. 그래서 호스트가 받아야 할 정산, 참가자가 받아야 할 환불, 클럽 운영 중인 호스트의 책임 같은 항목이 남아 있으면 본 단위는 탈퇴를 의도적으로 차단한다. 반대로 사용자 본인만의 활동(무료 이벤트 신청, 클럽 멤버십, 데이팅 매칭)은 자동 처리 항목으로 묶어 사용자에게 "이걸 한 번에 정리하고 떠납니다"라고 명시한 뒤 같은 동작 안에서 해결한다. 30일 유예 삭제(F13-06)와의 차이는 명확하다 — F13-06은 "충분한 회복 가능성", 본 단위는 "지금 비공개".

한 번의 탈퇴 탭이 통과하는 두 관문

사전 점검 → 차단 검토 → 자동 처리 안내 → 최종 확인 → 실제 비활성화. 한 동작 같지만 두 번의 검증과 세 번의 사용자 의사 확인이 들어 있다.

flowchart LR A["사용자: 계정 탈퇴 탭"] --> B["사전 점검 호출"] B --> C{"점검 결과"} C -->|"실패"| D["확인 불가 안내 → 진행 차단"] C -->|"차단 항목 있음"| E["탈퇴 불가 다이얼로그(상세 최대 3건 + 외 N건)"] E --> F["사용자가 항목 해결 후 재시도"] C -->|"차단 없음"| G{"자동 처리 항목 있음?"} G -->|"예"| H["최종 확인에 자동 처리 목록 포함"] G -->|"아니오"| I["최종 확인(짧은 형태)"] H --> J{"사용자가 탈퇴 확정?"} I --> J J -->|"취소"| Z["원상태로"] J -->|"확정"| K["실제 비활성화 호출"] K --> L{"서버 재검증 통과?"} L -->|"아니오"| M["탈퇴 실패 안내(다른 사용자가 사이에 결제 등 발생)"] L -->|"예"| N["자동 처리 실행 + 포인트 처리 + 익명화 + 토큰 폐기"] N --> O["강제 로그아웃 + 로그인 화면 복귀"]

탈퇴 점검에서 막히는 것과 자동 처리되는 것

사용자에게 "이 활동을 먼저 정리해야 합니다"라고 안내할 항목과 "이건 우리가 한 번에 처리합니다"라고 안내할 항목을 명확히 가른다.

차단

금전·운영 의무가 남은 경우

호스트 정산 대기, 크리에이터 수익, 모임 정산 분배·송금, 클럽 자금 인출, 진행 중인 결제, 정산 정정, 클럽 구독 중, 클럽 소유, 유료 이벤트 호스팅 중. 이 중 하나라도 남아 있으면 탈퇴 자체가 막힌다 — 사용자는 해당 항목을 먼저 해결해야 한다.

자동 처리

본인만의 활동

이벤트 신청 자동 취소, 무료 이벤트 호스팅 자동 취소, 클럽 멤버십 자동 탈퇴, 데이팅 매칭 자동 만료, 데이팅 프로필 자동 비활성화. 탈퇴 확정 시 같은 동작 안에서 한 번에 정리.

정리 결과

익명화와 토큰 폐기

이메일·닉네임 익명화, 멤버 정보 비우기, 푸시 토큰·소셜 로그인·결제 자격증명 삭제, 데이팅 프로필·사진·매칭·차단 기록 삭제. 강제 로그아웃 후 로그인 화면으로 빠진다.

사용자 시나리오

기본 흐름

차단 없이 즉시 탈퇴

데이터 프라이버시 화면에서 "계정 탈퇴"를 누른다. 사전 점검이 통과되고 자동 처리 항목도 없다. "정말 탈퇴하시겠습니까?" 최종 확인이 뜨고 사용자가 확인을 누른다. 계정이 즉시 익명화되고 토큰이 폐기되며 사용자는 로그인 화면으로 빠진다.

이 흐름이 남기는 것 — "당장 떠나고 싶다"는 사용자의 의사가 그 자리에서 해결된다. 30일 유예와의 본질적 차이가 사용자에게 분명히 전달되어야 한다.

금전 차단

호스트인데 정산이 안 끝났다

"계정 탈퇴"를 누르면 사전 점검이 차단을 반환한다. "탈퇴 불가" 다이얼로그에 차단 항목 제목과 설명, 개수, 상세 항목 최대 3건과 "외 N건"이 표시된다. 사용자는 정산을 먼저 처리해야 한다.

이 흐름이 남기는 것 — 다른 사용자(참가자)가 받아야 할 환불·정산이 한 호스트의 탈퇴로 막히면 안 된다 — 본 단위의 가장 큰 정책 책임.

자동 처리 동반

무료 이벤트 신청과 클럽 멤버십이 있는 채로 탈퇴

차단 항목은 없지만 자동 처리 항목이 있다. 최종 확인 다이얼로그에 "탈퇴 시 다음 항목이 자동 처리됩니다" 라는 안내와 함께 항목 목록이 포함된다. 사용자가 확인하면 같은 동작 안에서 신청 취소·클럽 탈퇴·데이팅 매칭 만료가 모두 일어나고 계정이 비활성화된다.

이 흐름이 남기는 것 — 사용자가 "내가 떠나면 어떤 일이 자동으로 일어나는지" 미리 보고 결정한다 — 그 안내가 없으면 "예고 없이 사라진" 모임·매칭에 다른 사용자가 당황한다.

사전 점검 실패

네트워크가 끊긴 채 탈퇴 시도

사전 점검 자체가 실패하면 "탈퇴 가능 여부를 확인할 수 없습니다" 안내가 뜨고 최종 확인 다이얼로그는 열리지 않는다. 사용자는 신호 회복 후 다시 시도한다.

이 흐름이 남기는 것 — 점검 없이 탈퇴를 진행하면 정산·결제 의무를 놓친 채 계정이 사라질 수 있다. 점검이 실패하면 무조건 멈추는 게 정책.

최종 단계 재검증

점검 통과 후 확정하기 직전 다른 결제가 들어옴

사전 점검 시점과 확정 시점 사이에 다른 도메인에서 결제·정산이 발생하면 실제 비활성화 직전 서버가 한 번 더 검증해 새로운 차단 항목을 찾는다. 찾으면 비활성화는 일어나지 않고 "계정 탈퇴에 실패했습니다" 안내가 뜬다.

이 흐름이 남기는 것 — 점검 결과를 그대로 믿고 실행하면 사이에 들어온 거래가 사라질 위험이 있다. 두 번 검증하는 게 본 흐름의 안전 장치.

포인트 처리

지갑 잔액이 남은 채로 탈퇴

비활성화 동작 안에서 지갑이 함께 정리된다 — 유료 포인트는 환불 처리, 무료 포인트는 소멸. 사용자에게 "탈퇴하면 무료 포인트는 사라진다"는 안내가 사전에 보이는지가 운영 관찰 포인트.

운영 관찰 포인트 — 사용자가 잔액 인지 없이 탈퇴해 환불 문의로 들어오는 비율이 신호.

화면 구성 — 사람이 보는 것

영역보이는 것그 자리에 둔 이유
탈퇴 섹션 "계정 탈퇴" 제목, "계정을 탈퇴하면 즉시 로그아웃되며 더 이상 서비스를 이용할 수 없습니다" 설명, 위험 색상의 "계정 탈퇴" 버튼. 30일 유예 삭제와 같은 화면에 있어 사용자가 두 가지를 의식적으로 구분할 수 있어야 한다. 버튼 색·설명 문구로 "즉시·비공개"임을 명확히.
탈퇴 불가 다이얼로그 차단 항목별 제목·설명·개수, 상세 항목(최대 3건)과 "외 N건". 확인 버튼으로 닫는다. 사용자가 "왜 탈퇴가 안 되는지"를 항목 단위로 알아야 다음 행동을 할 수 있다. 상세 라인은 사용자 본인의 의무를 식별할 수 있는 최소 단서.
최종 확인 다이얼로그 본문 + 자동 처리 항목 목록(있을 때) + 취소·탈퇴 두 버튼. 자동 처리 항목을 별도 화면이 아닌 확인 문구에 포함해 사용자가 한 곳에서 결정하게 한다.

측정해야 하는 것

본 단위가 정상이라고 부르려면 이 다섯 가지가 추적되어야 한다
  • 탈퇴 시도 대비 차단 비율 — 호스트·결제·정산 의무가 있는 사용자가 탈퇴를 시도하는 빈도. 너무 높으면 사용자가 의무를 인지하지 못한 채 탈퇴를 시도한다는 신호.
  • 차단 카테고리별 분포 — 어떤 의무가 사용자를 가장 자주 막는지. 가장 흔한 차단 카테고리는 사용자 안내 강화 대상.
  • 점검 통과 → 실제 탈퇴 전환율 — 사용자가 차단 없이 최종 확인까지 가서 정말 떠나는 비율. 너무 낮으면 자동 처리 항목 안내가 사용자를 망설이게 만든다는 신호일 수도.
  • 탈퇴 후 재가입(같은 이메일·전화번호)률 — 본 단위가 "정리 도구"로 잘못 쓰이는지의 신호.
  • 점검 통과 후 최종 단계에서 재검증으로 차단된 비율 — 사용자에게는 가장 답답한 케이스. 너무 높으면 점검 시점과 실행 시점 사이의 지연이 길다는 신호.

운영자가 알아야 할 것 — 비즈니스 리스크

본 단위에서 무너지면 다른 사용자의 권리가 막힌다
  • 호스트가 정산 의무가 있는 채로 탈퇴되면 참가자가 받아야 할 환불·정산이 동결될 수 있다. 차단 검사 누락은 사용자 한 명의 문제가 아니라 다른 여러 사용자의 권리가 묶이는 사고.
  • 차단 점검과 실제 비활성화 사이의 재검증이 없으면 점검 직후 들어온 결제·정산이 그대로 사라질 위험이 있다 — 두 번 검증이 본 흐름의 안전 장치.
  • 자동 처리 항목 안내가 빠지면 사용자가 "내가 갑자기 나간 모임" 때문에 다른 사용자에게 미안한 상황이 된다. 자동 처리 항목 목록은 사용자에게 미리 보여 주는 것이 정책.
  • 30일 유예 삭제(F13-06)와 즉시 비활성화의 결과 계정 상태가 다르다. 운영 데이터에서 두 상태를 한 묶음으로 취급하면 사용자 흐름이 잘못 분석된다.
  • 지갑 잔액 처리가 비활성화 동작 안에 들어 있다. 유료 포인트 환불은 외부 결제사 연동이라 실패할 수 있다 — 실패한 케이스의 후속 처리가 운영 책임.
  • 강제 로그아웃이 동작하지 않으면 사용자가 비활성화된 계정으로 계속 로그인 상태가 유지된다. 본인이 자기 계정 상태를 잘못 인지하는 가장 위험한 회귀.
