F01-02. 소셜 로그인 (카카오·네이버·구글·애플)

비밀번호를 새로 만들지 않고도 이미 쓰던 외부 계정으로 들어올 수 있게 한다. 신규 사용자에게는 가입+로그인 한 동작으로 끝낼 수 있는 가장 짧은 경로이고, 기존 사용자에게는 비밀번호를 외울 필요 없는 재방문 경로다. 4종 제공자 모두 같은 흐름을 따르되, 토큰 검증 방식은 제공자별로 다르다.

지원 제공자
4종
카카오·네이버·구글·애플
최소 가입 연령
만 19세
이메일 가입과 동일하게 강제
사전 입력
2가지
닉네임·생년월일을 외부 인증 전에 받음
설계 의도 — 외부 인증 한 번으로 가입과 로그인이 같이 끝난다

사용자가 카카오 버튼을 누르고 마지막 외부 인증을 마치면, 서비스는 그 사람을 이미 알고 있는지 아닌지를 같은 응답으로 결정한다. 이미 아는 사용자면 곧장 홈으로, 처음 보는 사용자면 곧장 온보딩으로. 가입과 로그인의 분기를 사용자가 직접 결정하지 않는다 — "처음이신가요/아니신가요" 같은 질문은 받지 않는다. 이 분기를 사용자에게 떠넘기면 매번 같은 외부 계정으로 두 번째, 세 번째 가입이 만들어지는 사고가 발생한다.

한 번의 소셜 버튼이 통과하는 세 단계

사용자가 보기에는 한 번의 동작이지만, 서비스는 세 단계를 거친다 — 사전 정보 수집, 외부 인증, 가입/로그인 결정.

사용자 시나리오

신규

카카오로 처음 시작

사용자가 카카오 버튼을 누른다. 닉네임·생년월일을 입력하고 필수 약관에 동의하면 카카오 로그인 화면이 뜨고, 카카오 인증을 마치면 곧장 온보딩 화면으로 들어간다. 별도 가입 화면 없이 한 번에 끝난다.

이 흐름이 남기는 것 — 사용자가 처음 본 화면에서 가장 짧은 가입 경로. 이메일 가입 흐름보다 입력 항목이 적기 때문에 가입 완료율이 일반적으로 더 높다.

재방문

이전에 카카오로 가입한 사용자가 다시 로그인

같은 카카오 버튼을 누른다. 닉네임·생년월일을 다시 입력하라는 시트가 떠도 사용자가 입력한 정보는 사용되지 않고 외부 인증을 마치면 곧장 홈으로 들어간다 — 서비스는 이미 그 사람을 알고 있다.

이 흐름이 남기는 것 — 비밀번호를 외울 필요 없는 재방문 경로. 비밀번호 분실 문의를 가장 강하게 줄이는 동선.

중단

외부 인증 화면에서 취소

시트까지 입력했지만 카카오 인증 화면에서 마음이 바뀌어 취소한다. 토스트 없이 로그인 화면 그대로 돌아온다 — 사용자의 명시적 취소이므로 실패로 인지시키지 않는다.

연령 정책

만 19세 미만 신규 가입 시도

시트에서 2010년생을 입력하고 외부 인증을 마쳤다. 서비스는 신규 사용자임을 확인한 직후 연령을 검사해서 가입을 차단한다. 계정 자체가 만들어지지 않는다. 이메일 가입 화면과 정확히 같은 연령 기준이 발현된다.

이 흐름이 남기는 것 — 가입 채널이 달라도 연령 정책이 흩어지지 않게 한다.

계정 상태

차단된 기존 사용자가 다시 소셜 로그인 시도

운영자가 정책 위반으로 차단한 사용자가 같은 카카오 계정으로 다시 로그인한다. 외부 인증은 통과해도 서비스 내부의 계정 상태가 차단 상태라서 로그인이 거부된다.

외부 인증 실패

외부 토큰 검증 실패

외부 제공자의 일시적 장애 또는 키 설정 오류로 인증 토큰이 검증되지 않는다. "소셜 로그인에 실패했습니다" 안내가 떠오르고 사용자는 다른 제공자 또는 이메일 로그인으로 시도할 수 있다.

화면에서 사용자가 인지하는 것

화면 요소사용자가 보는 것그 자리에 둔 이유
4종 소셜 버튼 카카오(노란색)·네이버(녹색)·구글(흰색)·애플(검정) 4개 버튼이 같은 크기로 세로 나열 각 브랜드의 표준 디자인을 그대로 따라야 사용자가 1초 안에 "이 버튼이 뭔지" 인지한다.
추가 정보 시트 외부 인증 전에 닉네임·생년월일·필수 약관 두 가지를 한 화면에서 묻는 바텀시트 외부 인증이 끝난 뒤 추가 화면을 끼우면 사용자가 "끝났는데 또 뭘 묻나" 답답해한다 — 외부 인증 전에 한 번에 묻는다.
로딩 오버레이 외부 인증 응답을 기다리는 동안 화면 위에 반투명 진행 표시 외부 인증은 1~3초 걸리기 때문에 시각적 피드백 없이는 "버튼이 안 눌렸나" 오해가 생긴다.
실패 안내 토스트 "소셜 로그인에 실패했습니다" / "이미 가입된 계정입니다" 같은 짧은 한 줄 사용자가 본인 잘못이 아니라는 사실을 분명히 인지하고 다른 경로로 시도할 수 있게.

운영자가 알아야 할 것

제공자별 검증 방식과 정책 결정점
  • 4종 제공자는 검증 방식이 모두 다르다. 카카오·네이버는 외부 API에 사용자 정보를 묻고, 구글·애플은 토큰 자체를 검증한다. 한 제공자의 장애가 다른 제공자에 영향을 주지 않는다 — 한 곳이 막혀도 나머지 셋은 정상 동작.
  • 외부 식별자 형식이 바뀌면 같은 사용자가 신규로 보일 수 있다. 카카오는 과거 식별자 방식에서 한 번 변경된 적이 있다. 이런 변경이 다시 일어나면 같은 사용자에게 새 계정이 만들어져 데이터가 갈라진다 — 운영 측에 마이그레이션 절차가 사전에 준비되어야 한다.
  • 연결 해제 후 재로그인 시 새 계정이 만들어진다. 사용자가 카카오 연결을 끊은 뒤 다시 카카오로 로그인하면 서비스는 그 사람을 처음 보는 사용자로 인지한다 — 이전 활동 데이터가 새 계정에 자동으로 합쳐지지 않는다는 사실을 CS가 알고 있어야 한다.
  • 소셜만 쓰는 사용자에게는 비밀번호가 없다. 비밀번호 재설정·이메일 인증·소셜 연결 해제 등의 동선이 이메일 가입자와 다르게 흐른다 — F01-08과 함께 정책을 점검.
  • 외부 인증 사용자에 대한 닉네임·생년월일은 시트에서 받는다. 만약 시트를 건너뛰는 동선이 만들어지면 닉네임이 임의 문자열로 자동 생성되고 추천·매칭 품질이 떨어진다 — 사전 정보 시트는 항상 외부 인증 직전에 노출되어야 한다.
