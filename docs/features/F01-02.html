F01-02. 소셜 로그인 (Apple · Google · Kakao · Naver)

외부 제공자에서 받은 토큰 하나로 가입과 로그인을 동시에 처리한다. 이메일 가입과 달리 비밀번호 단계가 없고, 사용자에게 받는 추가 정보(닉네임·생년월일·약관)는 서비스가 직접 모은다. 단 한 번의 서버 호출로 “신규 가입 + 토큰 발급” 또는 “기존 로그인”이 결정되며, 응답의 첫 로그인 여부가 다음 화면을 가른다.

이 기능이 다루는 범위

버튼 4개 + 추가 정보 시트 + 4가지 외부 토큰 검증 경로를 단일 엔드포인트로 모은다. 기존 사용자라면 시트에서 입력한 닉네임/생년월일은 무시되고 토큰만 새로 발급된다. 신규 사용자에게는 시트 입력값이 그대로 사용자 레코드에 저장된다. 이메일 인증 메일은 보내지 않는다(소셜은 외부 제공자가 이미 검증한 것으로 본다).

이 기능 안의 하위 항목

한 카드 = 사용자가 한 번에 의식하는 흐름 단위.

하위 기능 1

4종 소셜 버튼

로그인 화면 하단에 항상 4개가 동일 순서(카카오 → 네이버 → 구글 → 애플)로 노출된다. 각 버튼은 자기 브랜드 색을 따른다(카카오 노랑·네이버 그린·구글 화이트·애플 블랙). 한 번에 한 버튼만 활성화돼 동시 다중 SDK 호출을 막는다.

시각 통일
하위 기능 2

추가 정보 시트

SDK 인증을 시작하기 전에 닉네임·생년월일·약관 동의를 받는 바텀시트. 사용자가 시트를 닫으면 SDK 호출 자체가 일어나지 않는다. 외부 제공자가 보내주는 정보를 믿지 않고 본 서비스가 직접 수집하는 핵심 설계 결정.

필수 약관 2종만 19세 검증
하위 기능 3

제공자별 SDK 인증

시트 완료 후 해당 제공자의 SDK 화면으로 이동. 사용자가 SDK 화면에서 취소하면 토스트 없이 조용히 로그인 화면으로 돌아온다(취소는 에러가 아님). 인증 성공 시 토큰을 받아 서버 호출 단계로.

취소 = 무토스트
하위 기능 4

서버 토큰 검증 + 가입/로그인 결정

서버가 제공자의 토큰을 다시 검증해 소셜 식별자를 얻는다. DB에 같은 식별자가 있으면 기존 사용자로 로그인, 없으면 즉시 신규 가입. 결과 응답의 “첫 로그인 여부”가 클라이언트의 다음 분기를 결정한다.

기존 사용자 재로그인 시퀀스

시트 입력은 그대로 받지만 사용하지 않는다. 잠금/탈퇴 상태 검사를 먼저 통과한 뒤에만 토큰을 재발급.

사용자 시나리오 8개

정상 두 가지, 정책/검증 실패 다섯 가지, 그리고 마이그레이션 회귀 한 가지.

S1 · Happy Path · 신규 카카오

신규 회원이 카카오로 가입+로그인 완료

페르소나 — 카카오톡 사용자, 비밀번호 입력을 싫어함.

시작 — 로그인 화면.

  1. “카카오로 시작하기” 탭
  2. 추가 정보 시트 표시 — 닉네임·생년월일·필수 약관 2개 체크
  3. “시작하기” 탭 → 시트 닫히고 SDK 인증 시작
  4. 카카오 로그인 화면에서 인증·동의 → 토큰 획득
  5. 서버: 카카오 사용자 API 호출 → 식별자가 DB에 없음 → 신규 가입 분기
  6. 만 19세 검증 통과 → 사용자·회원·소셜 로그인 수단 INSERT → 토큰 발급
  7. 응답: 첫 로그인 = 참 → 클라이언트는 온보딩 저장소 초기화

종료 — 새 사용자 생성, 온보딩 첫 화면 진입.

S2 · Happy Path · 기존 카카오

기존 카카오 사용자 재로그인

페르소나 — 이전에 카카오로 가입한 사용자, 새 기기 또는 토큰 만료 상태.

  1. “카카오로 시작하기” → 시트 입력 → SDK 인증
  2. 서버: 식별자로 기존 소셜 로그인 발견 → 사용자 조회 → 상태 정상 확인
  3. 이전 로그인 시각이 있어 첫 로그인 = 거짓
  4. 토큰 재발급 (이전 리프레시 토큰은 폐기)
  5. 클라이언트: 온보딩 완료로 표시 → 홈으로 이동

종료 — 메인 피드 진입. 시트의 닉네임·생년월일·약관 입력값은 사용되지 않는다(기존 사용자라 변경 안 함).

S3 · 사용자 취소

SDK 인증 화면에서 사용자가 취소

페르소나 — 시트는 채웠지만 SDK 화면에서 마음이 바뀐 사용자.

  1. SDK 화면에서 “취소” 또는 시스템 뒤로
  2. SDK가 인증 결과 없음을 반환
  3. 로딩 해제, 토스트 미표시
  4. 로그인 화면 그대로 유지

종료 — 변화 없음. 취소를 에러로 다루지 않는 것이 핵심.

S4 · 외부 검증 실패

Apple 토큰 검증 실패 (개발자 설정 오류)

페르소나 — iOS 사용자, 백엔드의 Apple JWKS 검증 설정 불일치.

  1. Apple Sign-In 성공, 토큰을 서버로 전송
  2. 서버: Apple JWKS 키로 JWT 검증 → 발급자/만료 검증 실패 → 401 INVALID_AUTH_TOKEN
  3. 클라이언트: “소셜 로그인에 실패했습니다” 토스트

종료 — 로그인 화면 유지. 운영 관점: 사용자 실수보다 서버 설정 오류일 가능성이 높음 → 모니터링 알림 후보.

S5 · 정책 차단

미성년자 신규 소셜 가입 차단

페르소나 — 시트에 만 19세 미만 생년월일을 입력한 신규 사용자.

  1. SDK 인증 성공 → 서버 호출
  2. 서버: 식별자로 사용자 조회 → 없음(신규) → 생년월일 파싱
  3. 만 19세 미만 → 403 UNDERAGE_NOT_ALLOWED
  4. 클라이언트: 차단 토스트

종료 — 가입 차단, 사용자 레코드 미생성.

S6 · 마이그레이션 회귀

과거 해시 방식 사용자의 신규 인식 문제

페르소나 — 과거에 카카오 식별자를 SHA-256 해시로 저장하던 시기에 가입한 사용자.

  1. 카카오 토큰 검증 → 실제 카카오 사용자 ID 형식의 식별자
  2. DB에는 SHA-256 해시 식별자만 존재 → 매칭 실패
  3. 서버는 신규 사용자로 처리 → 같은 카카오 계정인데 별도의 새 사용자 레코드 생성

종료 — 동일 카카오 계정에 두 개의 별도 사용자 레코드가 존재하게 됨(기존 데이터와 단절). 대응: 운영 마이그레이션으로 옛 해시 식별자를 새 형식으로 일괄 갱신 필요. 이 항목은 버그 분류상 데이터 마이그레이션 미시행으로 추적된다.

S7 · 정책 차단 · 상태

차단(BAN) 사용자의 소셜 로그인 시도

페르소나 — 운영자가 차단한 기존 소셜 사용자.

  1. SDK 인증 성공 → 서버 호출
  2. 서버: 식별자로 기존 사용자 발견 → 상태 검증에서 BAN 거부 → 403 LOGIN_FAILURE_USER_STATUS_BAN
  3. 클라이언트: 차단 토스트

종료 — 로그인 거부, 토큰 미발급.

S8 · 입력 오류

잘못된 제공자 문자열 전송 (개발 실수)

  1. 제공자 문자열로 “facebook” 같은 미지원 값이 전송됨
  2. 서버: 제공자 파싱 실패 → 400 INVALID_REQUEST
  3. 클라이언트: 일반 에러 토스트

종료 — 화면 유지. 참고: 정상 경로에서는 발생 불가(버튼이 4종으로 고정). 개발자 디버깅 시나리오.

서버 계약 한눈에

Method Path 인증 핵심 동작
POST /api/v1/auth/social 불필요 외부 토큰 검증 + 자동 가입 또는 로그인 (200 OK). 응답의 “첫 로그인 여부”로 신규/기존 구분.
제공자별 외부 호출
  • Google — Google ID Token Verifier로 JWT 검증 (audience = 서비스 클라이언트 ID). 식별자 = google_.
  • Kakao — https://kapi.kakao.com/v2/user/me 호출로 사용자 ID 획득. 식별자 = kakao_.
  • Naver — https://openapi.naver.com/v1/nid/me 호출. 식별자 = naver_.
  • Apple — Apple JWKS 키 캐시로 JWT 검증 (발급자 = appleid.apple.com, 시계 오차 허용 300초). 식별자 = apple_.
에러 코드 매트릭스
  • 400 INVALID_REQUEST — 미지원 제공자 문자열
  • 401 INVALID_AUTH_TOKEN — 외부 토큰 검증 실패(어떤 제공자든)
  • 400 INVALID_INPUT — 신규 가입 시 생년월일 누락
  • 403 UNDERAGE_NOT_ALLOWED — 만 19세 미만
  • 403 LOGIN_FAILURE_USER_STATUS_BAN / STOP / TRYEXIT / EXIT — 비정상 상태

자주 빠뜨리는 케이스

레드 플래그
  • SDK 취소를 에러로 표시 — 사용자가 자발적으로 닫은 화면에 “실패” 토스트가 뜨면 신뢰가 깨진다. 인증 결과 없음과 검증 실패는 분리해서 다룬다.
  • 외부 토큰을 그대로 신뢰 — 클라이언트가 보낸 토큰만 보고 가입을 시키면 위조에 노출. 서버가 제공자 API/JWKS로 다시 검증해야 한다.
  • 제공자 변경 시 식별자 형식 마이그레이션 누락 — S6 시나리오 처럼 같은 사용자가 새 사용자로 인식되는 회귀. 형식 변경 전후 데이터를 모두 처리할 마이그레이션 필수.
  • 시트 입력값을 기존 사용자에게 덮어쓰기 — 닉네임·약관 동의는 신규일 때만 사용한다. 기존 사용자는 변경 안 함.
  • 제공자별 응답 차이로 동일 흐름 가정 깨짐 — Google·Apple은 JWT, Kakao·Naver는 OAuth 토큰. 서버에서 통일된 인터페이스로 추출해야 클라이언트가 단일 엔드포인트로 다룬다.
  • 이메일을 자동 생성하지 않거나 빈 값으로 둠 — “식별자@제공자” 규칙을 깨면 UNIQUE 제약 위반·로그인 충돌 가능.
