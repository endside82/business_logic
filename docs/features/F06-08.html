F06-08. 개인 구독 관리

베이직·프리미엄 두 멤버십 중 하나를 30일 단위로 구독하는 화면. 첫 결제와 자동 갱신 결제 모두 지갑 포인트에서 차감된다 — 외부 카드 결제는 충전(F06-02)에서만 일어난다. 한 번 구독하면 자동 갱신이 켜진 상태로 시작하고, 사용자는 자동 갱신만 따로 끌 수 있다 — 자동 갱신을 꺼도 이미 결제한 기간까지는 이용이 유지된다. 자동 갱신 결제가 여러 번 실패하면 구독이 잠시 정지되고, 사용자가 재활성을 누르면 그 자리에서 다시 결제하고 30일이 다시 시작된다.

설계 의도 — 사용자가 가벼운 마음으로 구독을 시작하고 끊을 수 있게 한다

구독은 사용자가 "끊을 자유"를 신뢰할 때만 시작한다. 본 화면은 자동 갱신만 끄는 동작과 구독을 완전히 끝내는 동작을 분리해 두지 않는다 — 자동 갱신을 끄면 이미 결제한 기간까지는 그대로 이용하고, 그 이후 결제가 자동으로 일어나지 않는다. 결제는 지갑 포인트에서만 차감된다 — 외부 카드 결제의 사고 위험을 충전(F06-02)에 격리한 정책이 본 단위에도 그대로 적용된다. 자동 갱신 결제가 반복 실패하면 구독이 정지로 잠시 멈춘다 — 사용자가 재활성을 명시적으로 결정하지 않는 한 잔액이 다시 차감되지 않는다.

구독의 라이프사이클

한 사용자의 구독은 네 상태 사이를 오간다. 사용자의 명시적 동작과 자동 갱신 결제 결과가 상태를 움직인다.

사용자 시나리오

첫 구독

프리미엄 멤버십에 가입

지갑 메인의 멤버십 진입 카드를 누른 사용자. 두 플랜이 혜택 체크리스트와 함께 비교되어 보인다. 프리미엄 카드의 "지갑으로 구독"을 누르면 확인 안내가 떠오르고 "9,900원이 지갑에서 차감됩니다 · 30일 유지 · 자동 갱신"이 안내된다. 동의 시 잔액이 차감되고 카드가 "이용 중"으로 바뀐다.

이 흐름이 남기는 것 — 사용자가 30일짜리 결정을 부담 없이 시작할 수 있다 — 외부 카드 결제가 끼지 않아 결제 마찰이 거의 없다.

자동 갱신 해제

다음 결제 전에 자동 갱신을 끈다

이용 중인 사용자가 "자동갱신 해제"를 누른다. 안내가 "해제해도 {만료일}까지는 멤버십을 이용할 수 있습니다"를 명시한다. 사용자가 확정하면 상태가 "해지 예약"으로 바뀐다 — 만료일까지는 이용이 그대로 유지되고, 그 이후 결제가 자동으로 일어나지 않는다.

이 흐름이 남기는 것 — "구독을 끊는다"는 행위가 즉시 이용을 끊지 않는다는 정책. 사용자가 가벼운 마음으로 자동 갱신을 끌 수 있게 한다.

잔액 부족

구독 시도 직전에 잔액이 모자라다

잔액 5,000원인 사용자가 프리미엄(9,900원) 구독을 시도한다. 화면이 사용자에게 "잔액이 부족합니다 — 충전한 뒤 다시 시도해 주세요" 안내를 띄우고 서버 호출을 보내지 않는다. 사용자가 충전 화면(F06-02)으로 이동해 잔액을 채운 뒤 돌아와 구독을 다시 시도한다.

이 흐름이 남기는 것 — 잔액 부족이 결제 호출 전에 차단되어 사용자에게 "결제 실패"가 보이지 않게 한다.

정지 → 재활성

자동 갱신이 반복 실패해 정지된 구독을 살린다

이용 중이었지만 만료일에 잔액이 부족해 자동 갱신 결제가 여러 번 실패한 사용자. 화면에 "정지" 배지와 "재활성화" 버튼이 보인다. 누르면 안내가 "9,900원이 지갑에서 차감되고 구독 기간이 다시 시작됩니다"를 명시한다. 동의 시 결제가 일어나고 상태가 "이용 중"으로 돌아간다.

이 흐름이 남기는 것 — 사용자가 잠시 빠진 구독을 명시적으로 다시 시작할 수 있다. 시스템이 사용자 몰래 결제하지 않는다는 신뢰.

만료 후 재가입

만료된 구독을 다시 시작한다

만료된 구독이 있는 사용자에게는 안내 카드가 "멤버십이 만료되었습니다. 아래 플랜에서 다시 시작할 수 있습니다"를 보여 준다. 플랜 카드에서 한 번 더 구독을 시작하면 첫 구독과 동일한 흐름이 진행된다.

중복 방지

이미 이용 중인 사용자가 다른 플랜으로 바꾸려 함

이용 중인 사용자에게는 다른 플랜 카드의 구독 버튼이 비활성으로 보인다 — 사용자가 잘못된 동작을 시도하지 않게 미리 막는 가드. 플랜 변경이 필요한 사용자는 현재 구독의 자동 갱신을 끄고 만료를 기다린 뒤 새 플랜으로 시작한다.

이 흐름이 남기는 것 — 플랜 즉시 변경 기능이 없다는 사실. 향후 결정에 따라 보강할 수 있는 영역.

실패 알림

자동 갱신 결제가 실패했음을 푸시로 안내받는다

본 화면이 아니라 별도 푸시·배너로 사용자에게 결제 실패 사실이 전달된다. 사용자가 본 화면에 진입하면 상태가 "정지"로 바뀌어 있고 재활성 버튼이 떠 있다.

화면에서 사용자가 인지하는 것

영역사용자가 보는 것그 자리에 둔 이유
멤버십 헤로 카드 화면 상단의 큰 카드. "Community Plus / 더 적은 마찰로 모임을 운영하세요" + 현재 잔액 표시. 구독이 가져다주는 가치를 한 문장으로 명시. 잔액을 함께 보여 줘 사용자가 "지금 결제 가능한가"를 즉시 판단.
현재 구독 카드 이용 중인 구독의 상태 배지(이용 중 / 해지 예약 / 정지 / 만료) + 단가 + 만료일 + 액션 버튼. "내 구독이 지금 어느 상태인지"를 한눈에. 상태별로 다음 액션(자동갱신 해제 / 재활성)이 달라진다.
플랜 카드 두 개 베이직과 프리미엄. 단가, 기간, 혜택 체크리스트, "지갑으로 구독" 버튼. 두 플랜을 같은 자리에서 비교. 이용 중이면 그 플랜은 "이용 중"으로 비활성, 다른 플랜도 변경 불가로 비활성된다 — 잘못된 시도를 막는 시각적 가드.
확인 안내 구독·해제·재활성 각각 다른 문구의 확인 안내가 한 번씩. 결제(구독·재활성)에는 차감 금액과 결과 안내, 해제에는 만료까지 이용 유지 안내 — 사용자가 결정의 결과를 정확히 인지하게.
정책 안내 "구독은 지갑 잔액에서 결제되며, 자동갱신 해제 후에도 만료일까지 이용할 수 있습니다" 같은 짧은 안내. 본 화면의 두 가지 핵심 정책(잔액 결제 · 해제 후 유지)을 사용자에게 명문화해 알린다.

측정해야 하는 것

본 단위가 정상이라고 부르려면 이 다섯 가지 지표가 추적되어야 한다
  • 구독 → 첫 갱신 통과율 — 첫 30일이 지나고 자동 갱신 결제가 통과되는 비율. 너무 낮으면 사용자가 첫 30일 안에 가치를 충분히 발견하지 못한다는 신호.
  • 자동 갱신 해제율 — 한 달 안에 자동 갱신을 끈 사용자 비율. 너무 높으면 구독 가치가 약하거나 안내가 부정확하다는 신호.
  • 자동 갱신 결제 실패율 — 잔액 부족으로 갱신이 실패하는 비율. 너무 높으면 자동 충전(F06-05) 가입 유도가 필요하다는 신호.
  • 정지 → 재활성 복귀율 — 결제 실패로 정지된 사용자가 재활성을 누르고 돌아오는 비율. 너무 낮으면 정지 안내가 사용자에게 잘 전달되지 않는다는 신호.
  • 구독 유지 기간 분포 — 사용자별 구독 누적 일수. 한두 달 후 끊는 비율이 높으면 구독 가치 자체를 재설계할 시점.

운영자가 알아야 할 것 — 비즈니스 리스크

구독은 사용자의 "끊을 자유"가 신뢰의 핵심
  • 자동 갱신 해제가 이용 자체를 끊는다고 잘못 인지되면 안 된다. 안내 문구가 "만료일까지는 이용이 유지된다"를 명확히 알려야 한다. 이 인지가 흐려지면 사용자가 자동 갱신을 못 끄고 누적된 결제를 받게 되거나, 반대로 끊고 나서 "이용이 안 된다"고 신고가 들어온다.
  • 이미 결제한 기간의 중도 환불은 없다. 플랜 변경 기능이 없는 현재, 사용자가 잘못된 플랜을 선택한 경우 환불 요청이 운영팀으로 들어올 수 있다 — 정책으로 명확히 두는 것과 환불 사례별 결정 사이의 균형이 필요한 영역.
  • 정지된 구독이 재활성 없이 만료되도록 두면 그 사용자는 사실상 떠난 상태다. 정지 안내가 사용자에게 명확히 전달되지 않으면 "어느 날 갑자기 구독이 없어졌다"는 신고가 늘어난다 — 푸시·배너의 명확성이 본 단위의 사용자 유지에 핵심.
  • 플랜 즉시 변경(베이직 → 프리미엄)은 현재 지원되지 않는다. 알면서 둔 현재 정책 — 사용자가 자동 갱신을 끄고 만료를 기다린 뒤 새 플랜으로 시작해야 한다. 사용자 신고가 늘어나면 정책 보강 검토.
  • 자동 갱신 결제가 외부 카드를 직접 청구하지 않는다. 결제는 지갑 포인트에서만 차감되므로, 잔액이 모자라면 갱신이 실패한다 — 자동 충전(F06-05) 가입이 본 단위의 유지율과 직접 연관된다.
  • 같은 갱신이 두 번 청구되지 않는다 (2026-06-06). 자동 갱신은 정해진 시각에 일괄로 도는데, 그 전까지는 처리 서버가 여럿이면 같은 구독을 동시에 갱신해 한 번의 갱신에 두 번 결제될 수 있었다. 이제는 갱신 작업이 한 번에 하나만 실행되도록 잠금이 걸려, 한 구독은 한 주기에 한 번만 청구된다 — 사용자 잔액과 구독 매출이 동시에 어긋나던 사고가 막혔다.
  • 구독 환불은 결제했던 구성 그대로 복원되고, 두 번 환불되지 않는다 (2026-06-06). 운영자가 구독 환불을 처리할 때도 본 결제 시스템과 같은 규칙을 따른다 — 유료/무료 구성 보존, 이중 환불 차단이 운영 경로에도 동일하게 적용된다.
