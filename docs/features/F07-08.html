F07-08. 정산 이의제기 · 처리 · 감사 로그

정산 도메인의 안전망. 참가자가 share/item에 이의를 제기하면 해당 subject의 결제 자체가 차단되고, 호스트가 인정·거절을 결정해야 풀린다. 모든 단계는 감사 로그에 actorRole(USER/HOST/SYSTEM)과 함께 기록되어 “누가 언제 무엇을 했는지” 한 화면에서 추적 가능하다.

이의 대상
SHARE·ITEM·TRANSFER
시나리오
8
상태
PENDING·APPROVED·REJECTED
감사 actor
USER·HOST·SYSTEM
핵심 게이트

PENDING 이의가 있는 subject는 결제 자체가 차단된다(MEETING_SETTLEMENT_SUBJECT_UNDER_APPEAL). 이의 해결 후에야 잔액이 움직일 수 있다. 잔액 이동 후 환원이 매우 어렵기 때문에 사전 차단이 안전망 역할을 한다.

이의 처리 상태 머신

stateDiagram-v2 [*] --> PENDING: 참가자 이의 제기 PENDING --> APPROVED: 호스트 인정 + 응답 PENDING --> REJECTED: 호스트 기각 + 응답 APPROVED --> [*]: subject 수정 또는 정산 재구성 REJECTED --> [*]: subject 결제 차단 해제

이 기능 안의 하위 항목

동작 1

이의 제기 (SHARE/ITEM)

참가자가 자기 share 또는 항목에 사유와 함께 제기. 동일 subject + PENDING 중복 차단.

사유 최대 1500자
동작 2

호스트 인정 / 기각

hostResponse(공개 메시지) 입력 + status 결정. RESOLVED 후 결제 차단 해제. APPROVED 시 호스트가 항목/share를 직접 수정하거나(DRAFT만 가능) 정산 cancel 후 재생성.

동작 3

결제 차단 게이트

모든 결제 호출이 validateNoPendingAppeal을 통과해야 한다. transfer·share·confirm 어디서든 동일.

전역 가드
동작 4

감사 로그

모든 정산 변경(SETTLEMENT_CREATED, ITEM_CREATED, TRANSFERS_REGENERATED, APPEAL_*, TRANSFER_PAID_*…)을 actor 정보(USER/HOST/SYSTEM)와 payload로 기록. 시간순 타임라인 + 페이징.

이의 제기와 결제 차단 효과

sequenceDiagram actor U as 참가자 actor H as 호스트 actor P as 송금자 participant SVC as 정산 서비스 participant AUD as 감사 로그 U->>SVC: 이의 제기 (subject=SHARE/42, reason) SVC->>SVC: 중복 PENDING 검사 SVC->>SVC: appeal INSERT (PENDING) SVC->>AUD: APPEAL_CREATED (USER role) SVC-->>U: 이의가 제기되었습니다 Note over P: subject 42 결제 시도 P->>SVC: 포인트 결제 SVC->>SVC: validateNoPendingAppeal(SHARE,42) SVC-->>P: 400 SUBJECT_UNDER_APPEAL H->>SVC: resolve (APPROVED 또는 REJECTED + hostResponse) SVC->>SVC: appeal status·resolvedAt 갱신 SVC->>AUD: APPEAL_RESOLVED (HOST role + hostResponse) SVC-->>H: 이의가 처리되었습니다 Note over P: 결제 차단 해제 P->>SVC: 포인트 결제 재시도 SVC->>SVC: 정상 처리

사용자 시나리오 8개

S1 · 이의 제기

참가자가 share 16k에 이의

  1. my-shares 카드 “이의제기” → 사유 입력 → 제기
  2. 중복 PENDING 검사 통과 → appeal INSERT(PENDING)
  3. audit APPEAL_CREATED(USER, payload: appealId·reason)
  4. “이의가 제기되었습니다” 토스트

결과: 본인 share의 결제가 차단된다(payShareByPoint/confirmBankTransfer 모두 SUBJECT_UNDER_APPEAL).

S2 · 인정

호스트 APPROVED + 응답

  1. 이의제기 관리 화면 → PENDING 카드
  2. status=APPROVED + hostResponse 입력 → resolve PATCH
  3. appeal 갱신 + audit APPEAL_RESOLVED(HOST role)
  4. 호스트가 항목/share를 수정하거나 정산 cancel 후 재구성
S3 · 기각

REJECTED + 응답

  1. “추가 영수증 확인 결과 적정합니다” 등 응답 + REJECTED
  2. 차단 해제 → 사용자가 다시 결제 가능
S4 · 중복

동일 subject 재제기

  1. PENDING 1건이 이미 있는 상태에서 같은 사용자가 다시 제기
  2. 409 MEETING_APPEAL_ALREADY_PENDING
S5 · 호스트 본인 제기

UI 가드 vs 서버 가드

  1. UI: 호스트의 share 카드는 이의 버튼 미노출
  2. 우회 시 백엔드는 현재 별도 차단 없음
  3. → 호스트가 자기 이의를 자기 처리하는 self-loop 가능

참고: 향후 서버 차단(appealerUserId == settlement.creatorUserId) 추가 후보.

S6 · 결제 차단

PENDING 이의로 transfer 결제 차단

  1. transfer 5에 이의 PENDING
  2. 송금자 “포인트 결제” → validateNoPendingAppeal → 400 SUBJECT_UNDER_APPEAL
  3. 호스트 resolve 후 재시도 가능
S7 · 감사 로그

변경 이력 타임라인

  1. “감사 로그” 진입 → 페이지 30건씩
  2. SETTLEMENT_CREATED → ITEM_CREATED × N → SETTLEMENT_ACTIVATED → TRANSFERS_REGENERATED(SYSTEM) → TRANSFER_PAID_POINT(USER) → APPEAL_CREATED(USER) → APPEAL_RESOLVED(HOST) → SETTLEMENT_COMPLETED(SYSTEM)
  3. 각 행에 actorUserName · actorRole · payload · createdAt
S8 · 잘못된 상태 전이

이미 RESOLVED된 appeal에 resolve 호출

  1. 호스트가 동시에 두 번 처리 시도
  2. 두 번째 호출: PENDING 아님 → 400 MEETING_APPEAL_INVALID_STATUS
  3. 토스트로 “이미 처리됨” 안내

자주 빠뜨리는 케이스

레드 플래그
  • 결제 가드 누락 — PENDING 이의 동안 잔액이 움직이면 환원이 매우 복잡. 모든 결제 경로에 validateNoPendingAppeal 적용.
  • 이의 사유를 알림에 포함 — 호스트만 보는 정보. 알림은 “이의가 제기되었습니다” 같은 일반 안내로.
  • APPROVED 후 정산 수정 가능 여부 미안내 — DRAFT 아닌 정산은 항목 수정 불가. 사용자는 정산 cancel 후 재생성을 안내받아야 한다.
  • actorRole 누락 — “시스템이 한 일”과 “사용자가 한 일”이 섞이면 책임 추적 불가.
  • 감사 로그 페이징 누락 — 큰 정산에서 메모리·렌더링 부담.
  • 호스트 self-loop 허용 — 호스트가 자기 share에 이의 제기 후 본인이 처리하는 경로가 열려 있음. 정책 강화 시 차단.
