F18-02. 분쟁 직접 접수 (USER_DISPUTE)

사용자가 어떤 도메인의 케이스와도 무관하게 직접 분쟁을 신규 접수하는 기능. 9가지 분쟁 유형 중 하나를 선택하고 사유를 10~2000자로 작성한 뒤 제출하면 즉시 케이스가 생성된다. 이렇게 만들어진 케이스는 USER_DISPUTE source로 통합 분쟁 목록에 포함되며, 이후 상태 전이는 admin이 처리한다. 24시간 10건 한도와 동일 상대 활성 케이스 중복 방지가 남용을 막는다.

설계 의도 — 어디에도 소속되지 않은 분쟁을 접수할 창구

이벤트·클럽·정산 등 기존 도메인에는 각자의 이의 경로가 있다. 그러나 어느 화면에서도 처리되지 않는 복합적 상황이나 새로운 유형의 분쟁은 갈 곳이 없었다. F18-02는 그 공백을 채우는 직접 접수 창구다. 기존 케이스와 연결 맥락(sourceType/sourceId, 이벤트/클럽 id)을 함께 전달할 수 있어 "어떤 일에서 비롯됐는지"를 운영팀이 추적하기 쉬운 구조다.

분쟁 접수 흐름

생성 후 상태 변경 방법 없음

케이스가 OPEN으로 생성된 뒤에는 사용자가 앱에서 상태를 바꾸는 방법이 없다. 운영팀이 처리하기 전까지 OPEN이 유지된다. 접수 완료 화면에서 처리에 시간이 걸린다는 안내가 필요하나 v1에서 이 안내 문구가 빠져 있다.

사용자 시나리오

기본 흐름

현장에서 불쾌한 일이 있었다 — 안전 신고

사용자가 "안전 관련(SAFETY)" 유형을 선택하고 사유를 15자 이상 작성한다. 책임 동의 체크박스는 없으며, 유형 선택 + 사유 10자 이상 입력 시 제출 버튼이 활성화된다. 제출하면 즉시 케이스가 생성되고 케이스 상세 화면으로 이동한다. 케이스 id는 USER_DISPUTE:42 형태로 이후 통합 분쟁 목록에 나타난다.

이 흐름이 남기는 것 — 사용자가 고객센터 채널을 별도로 찾을 필요 없이 앱 내에서 즉시 접수가 가능하다.

중복 방지

같은 상대에게 두 번 접수 시도

이미 처리 중인 활성 케이스가 동일 상대와 존재하면 서버가 409를 반환한다. "이미 동일한 상대로 진행 중인 분쟁이 있어요" 안내가 표시된다. 이전 케이스가 RESOLVED 또는 CLOSED로 종결되면 다시 접수할 수 있다.

이 흐름이 남기는 것 — 같은 상대에 대한 병행 케이스가 운영팀에 쌓이는 것을 방지한다.

연결 맥락

이벤트에서 비롯된 분쟁을 접수

다른 화면에서 "이 이벤트에 대해 분쟁 접수"로 진입하면 이벤트 id와 source 맥락이 자동으로 채워진다. 사용자는 유형과 사유만 추가로 입력한다. 이 맥락은 운영팀이 어떤 이벤트에서 비롯된 분쟁인지 추적하는 데 사용된다.

한도 초과

하루에 너무 많이 접수 시도

24시간 안에 10건을 초과해 접수하면 429가 반환된다. "요청이 너무 많습니다. 잠시 후 다시 시도해주세요" 안내 후 재시도할 수 있다.

v1 제약

증빙 파일을 첨부하고 싶은데 방법이 없다

v1에서는 분쟁 접수 시 파일 첨부 UI가 의도적으로 제외됐다. 접수는 사유 텍스트만으로 이루어진다. 서버는 파일 첨부를 지원하지만 앱 화면에 업로드 기능이 없다. v2 wave에서 추가된다.

운영자가 알아야 할 것

운영 주의사항
  • targetUserId가 없는 케이스는 중복 방지가 작동하지 않는다. 특정 상대를 지정하지 않은 일반 분쟁은 동일 유형으로 여러 건을 접수할 수 있다. 24시간 10건 한도만 적용된다.
  • 이벤트·클럽 맥락 정보 전달 경로 확인 필요. Dart 모델에는 eventId, clubId, reasonCode 필드가 선언되어 있으나, 실제 분쟁 접수 화면 진입 시 어떤 경로에서 이 값들이 채워지는지 확인이 필요하다. 채워지지 않는 경로에서는 null로 전송된다.
  • 접수 후 처리 기간 안내 없음. v1에서 "운영팀이 N일 내 처리 예정" 같은 안내 문구가 없다. 사용자가 상태 변화가 없다고 중복 접수를 시도할 수 있다.
  • rate-limit 구현 검증 필요. 컨트롤러 문서에 429 조건이 명시되어 있으나 서비스 레이어 내 실제 rate-limit 로직 구현 여부가 직접 확인되지 않았다.

레드 플래그

현재 알려진 미구현·주의 사항
  • 증빙 첨부 v1 미제공. 서버는 최대 5개 파일을 받을 준비가 되어 있지만 앱에 업로드 UI가 없다. 접수 후 증빙을 추가할 방법이 없다.
  • 이벤트·클럽 맥락 주입 경로 미확인. eventId, clubId, reasonCode 필드는 모델에 있지만, 어떤 화면 진입 경로에서 값이 채워지는지 확인이 필요하다. 채워지지 않는 진입 경로에서는 null로 전송된다.
  • 접수 완료 후 처리 기간 안내 없음. 케이스가 OPEN으로 만들어진 뒤 상태가 바뀌지 않으면 사용자가 "처리됐는지 아직인지" 모른다.
