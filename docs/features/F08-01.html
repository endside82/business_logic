F08-01. 내 플랜 목록 관리

크리에이터가 자기 손으로 만든 플랜과 마켓에서 구매해 보유 중인 플랜을 두 탭으로 나눠 관리하는 화면이다. 새 플랜이 시작되는 자리이자 — 하단 "+ 새 플랜 만들기" 버튼이 누구나 첫 플랜을 만드는 진입점 — , 작성한 플랜을 마켓에 풀 수 있는 단축 진입점("판매 시작")까지 한 면에 두었다. 한 사람이 가질 수 있는 플랜의 수에는 한도가 있어서(일반 5개, 호스트 권한 10개), 새 플랜이 만들어지지 않을 때 가장 흔한 이유가 여기서 정해진다.

설계 의도 — 만든 플랜과 산 플랜은 사용자의 머릿속에서 다른 흐름이다

크리에이터가 직접 만든 플랜은 "내가 편집하고 발행하는 대상"이고, 마켓에서 산 플랜은 "내가 활용하는 대상"이다. 두 흐름은 카드를 탭했을 때 도착하는 화면도 다르고(편집 가능한 상세 vs 읽기 전용 미리보기), 카드 안에서 가능한 행동도 다르다(판매 시작 vs 이벤트 만들기). 그래서 한 화면 안에 두지만 두 탭으로 명확히 갈라 둔다.

이 화면이 하는 세 가지

동작 1

내가 만든 플랜 탭

내가 작성한 모든 플랜을 상태와 무관하게 보여 준다(작성 중·발행 중·숨김·삭제 모두 포함). 작성 중이나 숨김 상태의 카드에는 "판매 시작" 버튼이 함께 노출되어, 상세 화면을 거치지 않고 바로 마켓에 풀 수 있다.

동작 2

구매한 플랜 탭

다른 크리에이터의 플랜을 산 기록이 별도 탭에 모인다. 두 탭은 서로 다른 호출을 쓰며, 한쪽 탭을 보고 있어도 다른 쪽 데이터는 비치지 않는다. 한 사람이 만들기도 하고 사기도 하는 사용자에게 두 행동을 섞지 않게 두기 위해서다.

동작 3

새 플랜 생성

하단 "+ 새 플랜 만들기"로 작성 중 상태의 플랜 한 건을 만들고 곧장 블록 편집 화면으로 보낸다. 한도를 초과한 사용자는 이 자리에서 막힌다.

새 플랜이 만들어지는 흐름

한 번의 "만들기" 탭이 실제로는 한도 검사 → 새 플랜 생성 → 편집 화면으로 이동까지를 한 번에 끝낸다.

사용자 시나리오

첫 작성

신규 크리에이터가 첫 플랜을 만든다

호스팅 경험을 살려 데이트 코스를 상품화하려는 사용자. 마이페이지에서 "내 플랜"으로 들어가면 만든 플랜 탭에 "아직 만든 플랜이 없습니다" 빈 안내가 떠 있다. 하단의 "+ 새 플랜 만들기"로 제목 한 줄을 입력하고 만들기를 누르면 곧장 블록 편집 화면이 열린다.

이 흐름이 남기는 것 — 새 플랜의 입구가 화면 하단에 항상 있어, 글을 안 쓴 사람도 빈 화면 앞에서 다음 행동을 찾을 수 있다.

탭 전환

만든 플랜과 산 플랜 사이를 오간다

작성한 플랜 세 개(작성 중과 발행 중 섞임)와 구매한 플랜 두 개를 가진 사용자. 만든 탭에서는 작성 중 카드에 "판매 시작" 버튼이 함께 보인다. 구매한 탭으로 가면 같은 카드 모양이지만 탭하면 편집 화면이 아니라 읽기 전용 미리보기로 들어간다.

이 흐름이 남기는 것 — 같은 사람이 만드는 동시에 사는 양면 사용자에게도 행동이 섞이지 않는다.

인라인 발행

작성 중 카드에서 곧장 마켓에 푼다

제목·블록·설명을 다 채운 작성 중 카드에서 "판매 시작" 버튼을 누른다. 확인 다이얼로그가 뜨고, 확인하면 그 카드의 상태 배지가 "판매 중"으로 바뀌면서 "판매 시작" 버튼은 사라진다. 상세나 발행 화면을 거치지 않는 단축 경로다.

이 흐름이 남기는 것 — 발행 화면(F08-05)의 체크리스트 가드를 건너뛰는 길이라, 정식 발행 경로와 별도로 운영 모니터링이 필요하다.

한도 차단

다섯 개를 채운 일반 사용자가 새 플랜을 시도

이미 다섯 개를 가진 일반 사용자가 새 플랜을 만들려 한다. 제목을 입력하고 "만들기"를 누르면 다이얼로그는 닫히지만 다음 화면으로 넘어가지 않는다. 호스트 권한이 있다면 같은 행동이 열 번째까지는 통과된다.

이 흐름이 남기는 것 — 작성 중 플랜을 삭제하면 한도가 회복된다. 다만 현재는 한도 차단을 알리는 별도 안내가 없어 사용자가 "버튼이 안 먹었나"로 오인할 수 있다 — 후속 개선 대상.

빈 입력

빈 제목으로 만들 수 없다

공백만 입력하고 만들기를 눌러도 다이얼로그가 닫히지 않는다. 서버 호출 자체가 일어나지 않는 클라이언트 가드.

동기화

다른 디바이스에서 추가한 플랜을 끌어온다

한 사용자가 두 디바이스를 동시에 쓰는 경우. 한쪽에서 새 플랜을 만들고 다른 쪽에서 당겨서 새로고침를 하면 그 플랜이 카드 리스트에 들어온다. 당겨서 새로고침는 현재 보고 있는 탭만 다시 부른다.

화면 구성 — 사용자가 보는 것

영역보이는 것그 자리에 둔 이유
상단 탭 두 개 "내가 만든 플랜" / "구매한 플랜". 우측 상단에는 작은 차트 아이콘(판매 현황으로 이동). 두 흐름을 같은 화면에 두되 시각적으로 분리한다. 차트 아이콘은 크리에이터가 자기 활동의 결과를 보고 싶을 때 곧장 빠져나갈 단축 경로.
플랜 카드 썸네일 + 제목 + 요약 + 상태 배지. 작성 중·숨김 카드에는 "판매 시작" 인라인 버튼이 함께. 상태 배지는 사용자가 "이 플랜이 지금 마켓에 풀려 있는가"를 카드 단계에서 즉시 알게 한다. 인라인 발행 버튼은 잘 만든 작성 중 플랜이 풀리지 않은 채 묻혀 있는 상황을 줄인다.
구매한 플랜 카드 썸네일 + 제목 + 가격(0이면 "무료"). 상태 배지 자리 없음. 이 탭의 카드는 상태가 의미 없는 자산이라, 가격을 그 자리에 둔다.
하단 "+ 새 플랜 만들기" 화면 하단에 고정된 큰 버튼. 누르면 제목 한 줄만 받는 작은 다이얼로그. 새 플랜 생성을 어디서든 한 번에 시작할 수 있는 자리. 제목 하나만 받는 가벼운 입구라 작성 부담이 없다.

운영자가 알아야 할 것

이 화면의 정책 결정점
  • 한도 5/10은 양산형 플랜 도배를 막는 정책이다. 한도를 키우면 호스트 한 명이 작성 중 플랜을 누적해 두고 마켓 카탈로그가 빠르게 늘어나지만, 진짜 완성된 플랜이 묻히는 위험이 커진다.
  • 인라인 "판매 시작"은 발행 화면의 체크리스트 가드를 우회한다. 제목·설명·블록·커버 이미지가 충분히 채워졌는지 확인하지 않고 마켓에 푸는 단축 경로이므로, 여기로 풀린 플랜의 품질 분포는 별도 추적이 필요하다.
  • 한도 초과 시 안내가 비어 있다. 현재는 다이얼로그만 닫히고 사용자에게 이유가 전달되지 않는다. "내 플랜이 5개라 더 만들 수 없습니다" 같은 안내가 추가되어야 한도 정책이 사용자에게 도달한다.
  • 삭제는 실제 삭제가 아니다. 데이터는 남고 상태만 "삭제됨"으로 바뀐다. 한도 카운트에서만 빠진다. 이전에 발행했던 플랜의 통계는 운영 측에서 그대로 추적 가능하다.
  • 플랜은 무료 포인트로 살 수 있게 허용할지와 무료 결제 가격을 가질 수 있다. 생성 단계에서는 제목만 받으므로 기본값으로 시작하고, 무료 허용·무료 결제 가격을 실제로 정하는 자리는 발행/수정(F08-05)이다. 무료로 팔린 분은 작성자에게도 무료 형태로만 들어와 현금으로 인출되지 않는다(유료/무료 분리정산, 상세는 결제·정산 정책 문서).
