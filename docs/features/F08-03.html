F08-03. 블록 에디터

크리에이터가 자기 플랜의 본문을 만드는 화면. 문단·제목·이미지·장소·시간표·체크리스트 같은 여러 종류의 블록을 추가·편집·정렬·삭제하면서 트리 형태로 쌓아 올린다. 텍스트는 입력하는 동안 0.3초마다 자동으로 저장되고, 마크다운 단축 입력으로 블록 종류를 바꿀 수 있다. 이 화면은 작성 중 상태의 플랜에서만, 그것도 작성자 본인에게만 열린다 — 다른 사람의 손에서는 본문이 흔들리지 않게 하는 것이 이 단위의 가장 중요한 약속이다.

플랜 본문은 코스 안내라 글·이미지·장소·시간표가 섞이고 위·아래 구조가 깊어진다. 그래서 한 종류의 입력창이 아니라 종류별로 다른 카드 형태의 블록을 트리로 쌓는다. 텍스트는 사용자가 인지하지 못하는 동안 자동으로 저장되고, 삭제는 한 번에 일어나지만 곧장 "실행 취소"로 되돌릴 수 있다. 잘 작성된 본문이 실수로 한 번에 사라지는 일을 막는 것이 이 화면의 가장 큰 책임이다.

이 화면이 다루는 다섯 가지

동작 1

블록 추가

하단 "+ 블록 추가" 버튼으로 종류 선택 시트를 열고 추가한다. 새 블록은 가장 마지막 자리(끝)에 들어간다. 종류는 문단·제목 세 종류·목록 두 종류·할 일·토글·인용·강조·구분선·이미지·동영상·시간표·장소·파일·코드·표·임베드·북마크 — 모두 20종.

동작 2

인라인 입력 + 자동 저장

문단·제목·인용 등 텍스트 블록은 화면에 입력창이 그대로 노출돼 바로 글을 쓸 수 있다. 마지막 입력 후 0.3초가 지나면 저장 요청이 무음으로 나가고, 우측에 작은 인디케이터가 "편집 중 → 저장 중 → 저장됨" 순으로 깜빡인다. 입력 중에 다른 데를 터치하면 즉시 한 번 더 저장한다.

동작 3

마크다운 단축 입력

문단에 "## "를 입력하면 그 블록이 큰 제목으로 바뀐다. "- "는 글머리표, "[] "는 할 일, "> "는 인용 등. 같은 자리에서 블록 종류 자체가 바뀌어 — 글을 쓰면서 흐름이 끊기지 않는다.

동작 4

전용 다이얼로그 편집

이미지·할 일·시간표·장소·목록·강조·임베드·북마크는 인라인이 아니라 풀스크린 다이얼로그에서 편집한다. 이미지는 갤러리에서 고른 뒤 업로드까지 한 번에 끝나고, 시간표는 행을 추가하는 표 형태로 다룬다.

동작 5

삭제와 실행 취소

블록을 길게 누르면 컨텍스트 시트에서 "삭제"를 고를 수 있다. 삭제는 즉시 화면에서 사라지지만 하단 토스트에 "실행 취소"가 함께 떠 있다. 다시 누르면 같은 자리에 그대로 복원된다(자손 블록까지 함께).

한 블록을 추가·편집할 때 일어나는 일

사용자가 보는 것은 한 번의 입력이지만, 서버 입장에서는 권한 가드 → 한도 검사 → 저장이라는 세 단계를 거친다.

사용자 시나리오

첫 본문 작성

새 플랜의 첫 블록을 채운다

플랜을 막 만든 직후 자동으로 들어온 빈 에디터. 화면 가운데에 "블록을 추가해보세요" 안내가 떠 있다. "+ 블록 추가"를 눌러 "텍스트"를 고르면 빈 문단 카드가 생기고 입력창에 포커스가 잡힌다. 한 줄 입력하고 손을 떼면 우측 인디케이터가 잠시 회전하다 "저장됨"으로 바뀐다.

이 흐름이 남기는 것 — 처음 들어와도 "무엇을 해야 하는지" 한 번에 보이고, 첫 입력이 사라질까 의심하지 않게 한다.

단축 입력

"## "로 제목 블록을 만든다

문단 블록에 "## "를 입력한다. 그 자리에서 입력창이 비워지고 카드가 큰 제목 글꼴로 바뀐다. 사용자가 손을 떼지 않고도 같은 자리에서 제목을 이어 쓸 수 있다.

이 흐름이 남기는 것 — 본문을 쓰는 동안 종류 선택 시트를 다시 열지 않아도 되어, 작성 흐름이 끊기지 않는다.

이미지 블록

커버 옆에 본문 이미지를 끼워 넣는다

"+ 블록 추가"에서 "이미지"를 고르면 빈 이미지 카드가 만들어진다. 그 카드를 누르면 풀스크린 다이얼로그가 열리고, 갤러리에서 사진을 고르면 업로드가 진행되어 카드에 썸네일이 채워진다. 업로드 실패 시에는 안내만 뜨고 빈 카드는 그대로 남는다.

삭제와 복원

잘못 만든 블록을 되돌린다

잘못 추가한 블록을 길게 누른 뒤 "삭제"를 선택한다. 화면에서 즉시 사라지고 하단에 "블록이 삭제되었습니다 / 실행 취소" 토스트가 뜬다. "실행 취소"를 누르면 그 자리에 그대로 복원된다. 자식 블록을 가지고 있던 경우 자식도 함께 살아난다.

이 흐름이 남기는 것 — 한 번의 실수가 본문 한 묶음을 통째로 날리지 않게 한다. 작성 자체에 대한 공포를 낮추는 안전망.

한도 차단

200개를 채운 플랜에 블록을 더 추가하려 한다

이미 200개의 블록을 가진 플랜에 새 블록을 더 추가하려 한다. 추가 호출이 한도 검사에서 거절되고, 하단에 "블록 추가에 실패했습니다 / 다시 시도" 토스트가 뜬다. "다시 시도"를 눌러도 같은 결과이므로, 사용자는 불필요한 블록 일부를 삭제한 뒤 다시 작성해야 한다.

상태 차단

발행 중인 플랜은 편집되지 않는다

이전에 발행한 플랜의 편집 화면에 어떤 경로로 들어왔다. 화면 자체는 열리지만 블록을 추가하려 하면 거절된다. 서버는 "작성 중 상태인 본인 플랜에서만 편집 가능"이라는 규칙을 항상 검사한다.

이 흐름이 남기는 것 — 이미 산 사람들이 보는 본문이 작성자에 의해 무심코 바뀌는 사고를 막는다. 발행 후에 다시 손대고 싶다면 판매 중지 → 편집 → 재발행 순서로 가야 한다.

권한 가드

다른 사람의 편집 화면에 들어왔다

딥링크나 잘못된 경로로 다른 크리에이터의 플랜 편집에 도달했다. 화면이 즉시 "접근 권한이 없습니다"로 바뀌고, 서버 호출은 아예 일어나지 않는다.

저장 실패

인라인 입력 중에 네트워크가 끊긴다

지하철 진입 직전에 한 블록의 텍스트를 편집한다. 저장 요청이 실패해 우측 인디케이터가 "재시도 필요"로 바뀐다. 사용자가 같은 입력을 한 번 더 손대면 다음 저장 시도가 다시 트리거된다(현재는 자동 재시도 없음 — 사용자 입력이 트리거).

운영 관찰 포인트 — 사용자가 인지하지 못한 채 저장 실패가 누적될 위험이 있다. 실패 인디케이터의 가시성과 자동 재시도 도입 필요성은 관찰해서 판단할 운영 결정점.

계층 만들기

토글 블록 아래 자식을 끼운다

토글 블록 아래 자식 블록을 두고 싶다. 본 화면에서는 자식 추가가 직접 노출되지 않고, 일단 "+ 블록 추가"로 같은 자리에 새 블록을 만든 뒤 별도 재정렬 화면(F08-04)으로 가서 들여쓰기로 자식 관계를 만든다.

화면 구성 — 사용자가 보는 것

영역보이는 것그 자리에 둔 이유
상단 액션 "블록 편집" 제목 + 우측에 "저장됨" 인디케이터(작은 클라우드 아이콘) + "재정렬" 텍스트 액션. 자동 저장이 잘 돌고 있다는 신호가 항상 보인다. 재정렬은 별도 화면이라 진입 동선만 둔다.
본문 — 블록 카드 트리 블록 종류마다 다른 카드 모양. 텍스트 계열은 인라인 입력창, 그 외는 짧은 요약. 자식 블록은 들여쓰기 + 약간 밝은 배경. "지금 어떤 종류의 블록인가"가 색·들여쓰기로 즉시 인지된다. 글을 쓰는 동안 종류 단서가 시야에 있어야 다음 블록 결정이 빨라진다.
하단 + 블록 추가 가운데가 살짝 떠 보이는 큰 둥근 버튼. 누르면 종류 선택 시트가 위로 올라온다. 새 블록 추가는 가장 자주 일어나는 동작이라 한 번에 손이 닿는 자리에 둔다.
길게 누르기 메뉴 블록을 길게 누르면 떠오르는 시트 — "편집(인라인 블록은 생략) · 복제 · 삭제". 모든 블록에 작은 메뉴 버튼을 붙이면 화면이 빽빽해진다. 흔한 동작만 길게 누르기 뒤로 숨긴다.
삭제 후 토스트 "블록이 삭제되었습니다" + 우측에 "실행 취소" 텍스트. 다이얼로그로 확인을 받지 않는 대신 실행 취소를 항상 띄워 둔다. 작성 흐름이 끊기지 않으면서도 실수에 안전하다.

운영자가 알아야 할 것

본문 데이터는 한 번 깨지면 사용자가 직접 복원할 길이 없다
  • 편집은 작성자 본인 + 작성 중 상태에서만 가능하다. 발행 중인 플랜은 편집이 차단된다 — 이미 산 사람들이 보는 본문이 작성자의 단순 클릭으로 바뀌는 사고를 막는 보호선.
  • 자동 저장의 실패 인지가 약하다. 우측 인디케이터가 "재시도 필요"로 바뀌는데, 사용자가 그 자리를 보지 않으면 변경분이 누적된 채 사라질 수 있다. 자동 재시도 또는 더 강한 시각 신호 도입은 관찰 후 결정.
  • 삭제는 자손 블록까지 한 번에 사라진다. 실행 취소로 복원할 때도 자손까지 함께 살아난다. 부모만 복원하고 자식이 사라지는 회귀가 일어나면 사용자가 본문 일부를 영구히 잃게 된다.
  • 200블록 한도는 마켓 노출 비용과 본문 가독성의 균형점이다. 한도를 키우면 사용자는 더 길게 쓸 수 있지만, 마켓 미리보기·검색 성능·구매 결정 속도가 모두 떨어진다.
  • 자식 블록은 본 화면에서 직접 만들지 않는다. 재정렬 화면(F08-04)에서 들여쓰기로 만든다 — 한 화면에 추가 동작이 너무 많아지지 않게 분리한 정책. 사용자에게는 "재정렬 화면이 들여쓰기도 한다"는 안내가 필요할 수 있다.
  • 이미지·파일 블록 안의 파일 키는 본 화면이 검증하지 않는다. 파일 업로드 자체는 별도 인프라가 책임지고, 본문에는 결과 키만 저장된다. 잘못된 키가 들어오면 마켓 노출 시점에서야 깨진 썸네일로 드러난다.
