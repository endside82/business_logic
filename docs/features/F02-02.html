F02-02. 홈 피드 새로고침

사용자가 명시적으로 “최신 상태”를 원할 때 캐시를 우회해 4섹션을 모두 다시 가져온다. 5분 캐시가 살아있어도 사용자가 풀투리프레시를 하면 무조건 새로 가져오는 게 본 단위의 단호한 정책. 새 데이터로 갱신 후에도 스크롤 위치는 유지되도록 설계된다.

새로고침은 사용자가 “정말 최신을 보고 싶다”는 신호다. 클라이언트 캐시는 우회하지만, 중복 트리거는 RefreshIndicator 자체가 막아준다(진행 중에는 추가 트리거 무시). 실패 시 사용자에게 명확한 재시도 경로를 제공.

이 기능 안의 하위 항목

동작 1

풀투리프레시

화면 상단에서 아래로 당기면 spinner 후 4섹션 강제 재호출.

동작 2

에러 화면 재시도

전체 실패 또는 섹션 실패에서 “다시 시도” 버튼을 통한 재호출. 풀투리프레시와 동일한 경로.

새로고침 흐름

flowchart TD A["풀투리프레시 트리거"] --> B{"진행 중?"} B -->|"예"| C["추가 트리거 무시"] B -->|"아니오"| D["4섹션 강제 호출 (캐시 우회)"] D --> E["성공한 섹션만 캐시 갱신"] E --> F{"4섹션 모두 실패?"} F -->|"예"| G["전체 화면 에러"] F -->|"아니오"| H["섹션 단위 렌더 + 스크롤 위치 유지"]

사용자 시나리오 6개

S1 · Happy Path

풀투리프레시로 새 콘텐츠 확인

  1. 홈 진입 후 7분 경과(캐시 만료)
  2. 아래로 당김 → spinner 등장
  3. 4섹션 병렬 재호출 → 200
  4. 새 카드 1개가 추천 섹션 맨 앞에 추가
  5. spinner 종료, 캐시 시각 갱신
S2 · 캐시 hit 우회

1분 만에 강제 새로고침

  1. 1분 전 진입(캐시 유효)
  2. 풀투리프레시 → 캐시 검사 우회
  3. 4섹션 호출 → DB 변화 없으면 동일 데이터 다시 렌더
  4. “최신 상태” 시각적 확인
S3 · 전체 실패

네트워크 끊김 중 새로고침

  1. 풀투리프레시 → 4섹션 모두 네트워크 오류
  2. 전체 화면 “피드를 불러올 수 없습니다” + “다시 시도”
  3. 네트워크 복구 후 사용자가 재시도
S4 · 부분 실패

인기 클럽만 실패

  1. 4섹션 호출 중 1개만 500
  2. 3섹션은 새 데이터로 갱신, 1섹션만 재시도 UI
  3. 실패 섹션의 캐시는 비워둠(이전 데이터 표시 안 함)
S5 · 위치 유지

스크롤한 상태에서 새로고침

  1. 4번째 섹션까지 스크롤한 상태에서 위로 올라가 풀투리프레시
  2. 새 데이터로 갱신 → 항목 수 같으면 스크롤 위치 유지

참고: 별도 reset 코드 없이 sliver rebuild 특성으로 자연스럽게 유지됨.

S6 · 반복 트리거

짧은 시간 반복 풀투리프레시

  1. 1차 풀투리프레시 진행 중에 다시 당김 → 무시
  2. 1차 완료 후 즉시 다시 당기면 2차 실행
  3. throttle 없음 → 사용자 의지대로 호출 가능

향후 검토: 클라이언트 최소 간격(throttle) 도입 여부.

자주 빠뜨리는 케이스

레드 플래그
  • 새로고침이 캐시 hit으로 떨어지는 회귀 — “새로고침해도 같은 데이터”라는 사용자 불신 발생.
  • 실패 섹션에 이전 캐시를 그대로 두는 회귀 — “재시도했는데 옛 데이터”라는 혼란.
  • 스크롤 위치 0으로 reset되는 회귀 — 사용자가 봤던 위치를 잃는다.
  • 중복 트리거가 동시에 4섹션을 두 번 호출 — RefreshIndicator의 자체 보호가 필요.
