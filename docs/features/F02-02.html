F02-02. 홈 새로고침

홈 화면을 아래로 당기면 일어나는 동작을 정의한다. 사용자가 "지금 이 화면이 진짜 최신인가"를 스스로 한 번 더 확인하고 싶을 때 누르는 경로다. 본 단위는 메인 조회(F02-01)와 같은 네 영역을 다시 불러오되, 캐시는 무시하고 항상 최신을 받아 온다.

설계 의도 — 사용자의 의심을 한 동작으로 해소

홈은 5분 캐시로 동작하기 때문에 "다른 사용자에게는 새 카드가 있을 수도 있다"는 의심이 자연스럽게 생긴다. 새로고침은 사용자가 "지금 진짜로 비교해 보고 싶다"고 표시하는 동작이며, 캐시 만료 여부와 상관없이 네 영역을 모두 다시 불러와 최신 상태를 한 번에 보여준다. 다시 시도(부분 실패 복구)와는 다르다 — 새로고침은 "내가 보고 싶다"는 의지의 표시고, 다시 시도는 "방금 실패한 걸 한 번 더"의 의미다. 결과는 같지만 사용자 맥락이 다르다.

사용자 시나리오

기본 흐름

점심시간에 새 모임이 올라왔나 확인

홈을 띄운 지 7분 정도 지났다. 사용자가 화면 위쪽을 당기면 짧은 진행 표시가 돌고, 새로 올라온 카드가 추천 자리 맨 앞에 들어와 있다.

이 흐름이 남기는 것 — "내가 직접 갱신했다"는 통제감. 자동 갱신과 본질적으로 다른 사용자 만족.

아주 짧은 간격

방금 본 화면을 또 새로고침

1분 전에 들어왔지만 한 번 더 당긴다. 화면은 같은 카드를 다시 보여 줄 가능성이 높지만, "방금 확인했다"는 감각을 사용자에게 남긴다.

이 흐름이 남기는 것 — 빈번한 새로고침을 막지 않는 것이 정책. 의사 표시를 차단하면 답답함이 누적된다.

실패 분기

지하철 진입 직전에 새로고침

네트워크가 끊긴 채 당기면 네 영역이 모두 비고, "피드를 불러올 수 없어요" 풀스크린 안내가 뜬다. 자동 재시도는 하지 않는다. 사용자가 신호를 회복한 뒤 직접 다시 시도한다.

부분 분기

새로고침 도중 한 영역만 실패

인기 클럽만 응답이 안 오면, 다른 세 영역은 새 데이터로 갱신되고 인기 클럽 자리만 "다시 시도" 카드로 바뀐다. 직전에 보던 클럽 카드를 그대로 두지는 않는다 — 새로고침은 "이전 상태를 의식적으로 버리는 행위"이기 때문에.

운영자가 알아야 할 것

새로고침 정책 결정점
  • 최소 간격 제한을 두지 않는다. 사용자가 의도적으로 당기는 동작을 막으면 "데이터가 진짜인가" 불신만 커진다. 필요해지면 서버 부하 기반으로 클라이언트 측에 한도를 두는 것을 검토할 여지가 있다.
  • 새로고침 후 스크롤 위치는 그대로 둔다. 새 데이터를 받았다고 사용자를 맨 위로 끌어올리지 않는다 — 사용자가 보던 자리를 잃는다.
  • 당겨서 새로고침 동작 영역은 화면 최상단으로 제한된다. 모바일 운영체제의 표준 동작과 충돌하지 않게 두는 것이 원칙.
