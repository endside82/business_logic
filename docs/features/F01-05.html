F01-05. 토큰 갱신 · 로그아웃

모든 인증 API의 뒤에 매번 동작하는, 가장 자주 깨지는 인프라성 기능. 액세스 토큰이 만료(1시간)되면 클라이언트가 백그라운드에서 리프레시 토큰으로 새 토큰을 받아 같은 요청을 다시 보낸다. 동시 다발적 401, 다른 기기와의 단일 기기 정책, 명시적 로그아웃, 계정 라이프사이클(탈퇴·삭제 요청)까지 — “사용자가 토큰 흐름을 의식해서는 안 된다”는 게 본 단위의 유일한 성공 기준이다.

이 기능이 다루는 범위

한 사용자의 세션 시작과 끝 사이를 모두 책임진다. 자동 갱신은 사용자가 알지 못해야 성공이고, 명시 로그아웃은 토큰 폐기 + 다른 기기 자동 로그아웃 + 토스트 안내까지 묶어야 완성이다. 계정 탈퇴·삭제 요청은 세션 종료와 같은 흐름을 공유하기 때문에 본 단위에서 함께 다룬다.

이 기능 안의 하위 항목

하위 기능 1

백그라운드 자동 갱신

모든 401 응답을 가로채 리프레시 토큰으로 새 토큰을 받아 원 요청을 다시 보낸다. 동시에 발생한 다수의 401은 큐로 모아 단 한 번만 갱신한다. 사용자는 짧은 지연만 경험하고 화면 전환이나 토스트는 보이지 않는다.

큐잉분산 락
하위 기능 2

명시적 로그아웃

프로필 화면의 “로그아웃”에서 시작. 서버가 액세스 토큰을 즉시 블랙리스트 등록하고 그 사용자의 모든 리프레시 토큰을 폐기한다. 다른 기기는 다음 요청 시 자연 로그아웃. 네트워크 오류로 서버 호출이 실패해도 로컬 토큰은 반드시 삭제한다.

블랙리스트로컬 보장
하위 기능 3

세션 만료 처리

리프레시 토큰도 만료되었거나 차단·탈퇴 상태로 갱신이 거부되면 모든 토큰을 폐기하고 로그인 화면으로 강제 이동시킨다. 사용자에게는 별도 토스트 없이 자연스럽게 “세션이 만료되었네”라고 알 수 있게 한다.

단일 기기 정책
하위 기능 4

계정 라이프사이클

데이터 내보내기·계정 삭제 요청·탈퇴는 본 단위가 노출하는 “세션 종료” 표면 위에 동작한다. 탈퇴/삭제 후에는 같은 자격증명으로 재로그인이 차단되어야 한다.

사용자 시나리오 15개

S1 · Happy Path

백그라운드 자동 갱신 후 정상 응답

  1. 1시간 이상 앱 사용 → 일반 API 호출이 401
  2. 인터셉터가 갱신 API 자동 호출
  3. 서버: 분산 락 확보 → 토큰 갱신
  4. 새 토큰으로 원 요청 재시도 → 정상 응답

종료 — 사용자는 짧은 지연만 인식. 토큰은 갱신됨.

S2 · 세션 만료

리프레시 토큰까지 만료 (30일 미사용)

  1. 401 → 갱신 시도
  2. 서버: 토큰은 있으나 만료됨 → 만료 토큰 삭제 → REFRESH_TOKEN_EXPIRED
  3. 클라이언트: 모든 토큰 폐기 → 로그인 화면으로 강제 이동

종료 — 별도 토스트 없이 자연스러운 “세션 종료”.

S3 · 동시성

동시 다발적 401 → 갱신은 단 한 번

  1. 홈 진입 시 5개 위젯이 병렬 API 호출 → 모두 401
  2. 인터셉터 큐가 첫 401만 갱신 진행, 나머지는 대기
  3. 서버 분산 락이 멀티 인스턴스 환경에서도 직렬화 보장
  4. 갱신 완료 후 큐의 4개 요청도 새 토큰으로 순차 재시도 → 5개 모두 정상

종료 — 토큰 갱신은 1회, 5개 요청 모두 성공.

S4 · 갱신 중 상태 변화

갱신 도중 사용자가 차단(BAN)됨

  1. 401 → 갱신 시도
  2. 서버: 리프레시 토큰은 유효하지만 사용자 상태가 BAN → 거부
  3. 클라이언트: 강제 로그아웃

종료 — 상태 변화가 다음 갱신 시점에 즉시 반영.

S5 · Happy Path

명시적 로그아웃

  1. 프로필 “로그아웃” 탭 → 확인 다이얼로그 → “로그아웃”
  2. 서버: 액세스 토큰 jti를 블랙리스트 → 리프레시 토큰 전체 폐기 → 사용자 상태=LOGOUT, logoutAt 기록
  3. 클라이언트: 로컬 토큰 폐기 → 토스트 플래그 설정 → 로그인 화면 이동
  4. 로그인 화면 진입 시 “로그아웃되었습니다” 토스트 1회 소비

종료 — 같은 사용자의 다른 기기는 다음 호출에서 401 → 자연 로그아웃.

S6 · 견고성

로그아웃 API 실패해도 클라이언트는 정상 처리

  1. 로그아웃 탭 → 네트워크 오류로 서버 호출 실패
  2. 예외를 무시하고 로컬 토큰은 무조건 폐기
  3. 로그인 화면 이동 + 토스트 표시

종료 — 클라이언트는 깔끔히 로그아웃됨. 서버 토큰은 다음 호출에서 자연 만료까지 살아있을 수 있으나 동작에는 영향 없음.

S7 · 갱신 경로 분리

로그인 API의 401은 갱신을 시도하지 않는다

  1. 사용자가 잘못된 비밀번호로 로그인 시도
  2. 서버 401 응답
  3. 인터셉터: 요청 경로가 로그인 API임을 인식 → 갱신 우회
  4. 호출자에게 원래 에러 전달 → 사용자에게 “자격증명 오류” 토스트

종료 — 갱신 루프 없이 즉시 에러 표시.

S8 · 단일 기기

다른 기기 자동 로그아웃

  1. 폰에서 토큰 갱신 → 서버는 사용자의 모든 리프레시 토큰 삭제 후 새 토큰 발급
  2. 태블릿의 리프레시 토큰도 함께 사라짐
  3. 태블릿이 다음 401에서 갱신 시도 → 토큰 없음 → 로그인 화면 이동

종료 — 두 기기를 동시 유지할 수 없는 단일 기기 정책 동작.

S9 · 분산 락

멀티 인스턴스 동시 갱신 충돌

  1. 두 인스턴스가 같은 리프레시 토큰으로 동시에 갱신 요청 수신
  2. 첫 인스턴스가 락 획득 → 갱신 진행
  3. 두 번째 인스턴스는 락 즉시 실패 → INVALID_REFRESH_TOKEN으로 응답
  4. 두 번째 클라이언트는 강제 로그아웃

종료 — 동시 갱신을 직렬화. 정상 상황에서는 클라이언트 큐로 직렬화되므로 발생 빈도는 낮다.

E2E-1 · 스플래시 분기

콜드 스타트 시 비로그인 → 로그인 화면

  1. 토큰 없음 상태로 앱 시작
  2. 스플래시 화면 단일 노출
  3. 토큰 검사 후 비인증 상태 전이 → 최대 10초 내 로그인 화면으로 이동
  4. 로그인 화면 + 슬로건 가시 확인

종료 — 로그인 화면 단독 노출.

E2E-2 · 로그아웃 라운드트립

로그아웃 다이얼로그 → 토스트 → 같은 계정 재로그인

  1. 프로필 → 로그아웃 → 다이얼로그 “로그아웃하시겠습니까?”
  2. 다이얼로그 “로그아웃” 확정 → 로그인 화면 이동
  3. “로그아웃되었습니다” 토스트 가시
  4. 같은 자격증명 재로그인 → 메인 셸 재진입

종료 — 사이클이 깨끗하게 동작, 토큰 재발급 정상.

E2E-3 · 탈퇴 후 차단

계정 탈퇴 → 자동 로그아웃 + 동일 계정 재로그인 차단

  1. 프로필 → 프라이버시 → “계정 탈퇴”
  2. 다이얼로그 안내 “탈퇴 후에는 계정을 복구할 수 없습니다.” 확인
  3. “탈퇴” 확정 → 자동 로그아웃
  4. “계정이 탈퇴되었습니다” 토스트 또는 로그인 화면 가시
  5. 같은 자격증명으로 재로그인 시도 → 메인 셸 진입 차단

종료 — 탈퇴 계정은 메인 셸 진입 불가.

E2E-4 · 라이프사이클 surface

프라이버시 화면이 계정 상태를 정확히 노출

  1. 프라이버시 화면 진입
  2. 내보내기 상태별 surface:
    • 처리 중: “처리 중입니다…”
    • 완료: “완료” + “다운로드”
    • 실패: “실패” + “다시 요청”
    • 없음: “데이터 내보내기 요청”
  3. 삭제 요청 상태별 surface:
    • 없음: “계정 삭제 요청”
    • 대기/승인: “삭제까지 …” 배너 + “삭제 취소”
    • 거부: “계정 삭제 요청”으로 복귀

종료 — 현재 라이프사이클 단계를 한 화면에서 정확히 식별 가능.

E2E-5 · 데이터 내보내기

데이터 내보내기 요청 라운드트립

  1. 프라이버시 → “데이터 내보내기 요청” 단일 CTA 탭
  2. 서버 즉시 응답이 PROCESSING이거나 READY이거나 무관
  3. “처리 중입니다…” 또는 “다운로드 가능” 둘 중 하나가 10초 내 가시

종료 — 요청이 UI에 즉시 반영, 다음 단계가 분명함.

E2E-6 · 삭제 요청·취소

계정 삭제 요청 → 30일 배너 → 취소 사이클

  1. “계정 삭제 요청” → 다이얼로그 → “삭제” 확정
  2. 예정 삭제 배너(“삭제까지 …”) + “삭제 취소” CTA 가시
  3. “삭제 취소” 탭 → 기본 “계정 삭제 요청” CTA로 복귀

종료 — idempotent 사이클이 깨끗하게 동작.

화면 구성 — 사람이 보는 것

본 단위의 “화면”은 사용자가 명시적으로 마주치는 두 곳이 핵심이다 — 스플래시와 로그아웃 다이얼로그.

자주 빠뜨리는 케이스

레드 플래그
  • 로그인 API 401을 갱신 대상으로 잡는 회귀 — 무한 갱신 루프. 경로 매칭으로 반드시 우회.
  • 큐잉 없이 동시 갱신 — 같은 토큰으로 동시에 여러 번 갱신을 시도하면 분산 락 충돌이 사용자 한 명에서 발생.
  • 로그아웃 API 실패 시 로컬 토큰을 안 지우는 회귀 — 사용자가 “로그아웃 눌렀는데 다시 들어가짐” 상태가 된다. 반드시 finally에서 보장.
  • 블랙리스트 누락 — 액세스 토큰이 1시간 동안 살아있을 수 있어 “로그아웃 직후에도 일부 API가 통과”하는 회귀.
  • 새 토큰 발급 시 기존 토큰을 안 지우는 회귀 — 단일 기기 정책이 깨지고 보안 사고로 이어진다.
  • 세션 만료를 토스트로 알리는 시도 — 토스트 대신 자연스러운 로그인 화면 진입으로 처리하는 것이 본 단위의 선택. 사용자는 “이상하다”가 아니라 “세션이 끝났구나”로 인지.
  • 탈퇴 후에도 토큰이 살아있어 일부 API가 통과 — 탈퇴 시 모든 토큰을 즉시 폐기해야 한다.
