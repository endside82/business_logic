F13-06. 계정 삭제 요청 (30일 유예)

즉시 탈퇴(F13-07)와 별도의 “차분한” 종료 경로. 30일 유예 동안 사용자가 마음을 바꾸면 취소할 수 있고, 유예 만료 시 스케줄러가 개인정보를 익명화·삭제한다. 거래 기록은 법적 보존 의무로 남는다.

유예 기간
30일
시나리오
5
스케줄러
매일 04:00
상태
PENDING/APPROVED/COMPLETED/CANCELLED
삭제 정책

개인정보(이름·이메일·전화·생년월일·프로필 이미지·자기소개) + 알림 토큰·결제 자격증명·데이팅 프로필·캘린더 가용시간을 일괄 삭제 또는 익명화. 거래·정산·회계 원장은 법적 보존 요건에 따라 남긴다. 계정 status는 EXIT로 전이.

이 기능 안의 하위 항목

동작 1

30일 유예 삭제 요청

PENDING 요청 INSERT + scheduledAt = now+30d. 사용자에게는 D-day 카드가 노출된다.

동작 2

유예 중 취소

DELETE로 status=CANCELLED. 진행 배너 즉시 사라짐.

동작 3

중복 요청 차단

PENDING이 이미 있으면 INVALID_INPUT.

동작 4

D-day 배너

마이페이지·프라이버시 화면에 D-N 카운트 표시 + “관리” CTA로 취소 진입.

동작 5

스케줄러 익명화·삭제

매일 04:00에 만료된 PENDING을 일괄 처리 → COMPLETED + 사용자 상태 EXIT.

상태 머신

stateDiagram-v2 [*] --> PENDING: 사용자 요청 (scheduledAt=+30d) PENDING --> CANCELLED: 사용자가 취소 PENDING --> COMPLETED: 스케줄러가 만료 처리 CANCELLED --> [*] COMPLETED --> [*]: 사용자 EXIT, 개인정보 제거

요청 → 만료 시퀀스

sequenceDiagram actor U as 사용자 participant API as 사용자 API participant SVC as 탈퇴 서비스 participant DB as DB participant SCH as 스케줄러 U->>API: POST /users/me/data-deletion API->>SVC: 삭제 요청 SVC->>DB: PENDING + scheduledAt=now+30d API-->>U: 토스트 + D-30 카드 Note over U: D-day 진행 동안 마이페이지에 배너 U->>API: (선택) DELETE /users/me/data-deletion API->>DB: status=CANCELLED API-->>U: 진행 배너 사라짐 Note over SCH: 매일 04:00 실행 SCH->>DB: PENDING + scheduledAt < now DB-->>SCH: 만료 행 SCH->>SVC: 익명화·삭제 + status=COMPLETED SCH->>DB: 사용자 status=EXIT

사용자 시나리오 5개

S1 · 요청

30일 유예 삭제 요청

  1. 프라이버시 → “계정 삭제 요청”
  2. 확인 다이얼로그 → POST
  3. PENDING + scheduledAt=+30d INSERT
  4. 토스트 + D-day 카드
S2 · 진행 배너

마이페이지 진입 시 D-N 표시

  1. 삭제 상태 provider 구독
  2. PENDING/APPROVED 상태면 배너 노출
  3. scheduledAt 기준 “D-N” 카운트
  4. “관리” 탭 → 프라이버시 화면
S3 · 취소

유예 중 삭제 취소

  1. “삭제까지 N일 남음” 카드 + “삭제 취소” 버튼
  2. DELETE → status=CANCELLED
  3. 토스트 “계정 삭제가 취소되었습니다” + 배너 사라짐
S4 · 중복

이미 PENDING인 상태 재요청

  1. findByUserIdAndStatus(PENDING) → 발견
  2. INVALID_INPUT → “요청에 실패했습니다”
S5 · 스케줄러

30일 만료 후 자동 익명화

  1. 매일 04:00 실행
  2. PENDING + scheduledAt 과거 행 조회
  3. 사용자/멤버 익명화 + 알림·결제·데이팅·캘린더 개인정보 삭제
  4. 거래 기록은 보존
  5. 요청 status=COMPLETED, 사용자 EXIT

익명화·삭제 대상

도메인처리
users / member이름·생년월일·전화 등 익명화, status=EXIT
알림 디바이스 토큰전부 삭제
결제 수단·자동충전billingKey 등 자격증명 삭제
데이팅 프로필·사진전부 삭제
캘린더 가용시간전부 삭제
거래·정산·회계보존(법적 의무)
리뷰·게시글닉네임만 “탈퇴한 사용자”로 변경, 본문 보존

에러 코드

  • 404 DELETION_REQUEST_NOT_FOUND — 취소 대상 없음

자주 빠뜨리는 케이스

레드 플래그
  • 거래 기록 함께 삭제 — 법적 보존 의무 위반. 영업 기록 유지 필수.
  • 익명화 누락(닉네임만 변경, PK 그대로) — 사용자 식별 가능 정보가 남음.
  • 스케줄러가 트랜잭션 없이 진행 — 중간 실패 시 익명화는 됐는데 EXIT 미적용 등 부분 상태 발생.
  • 유예 기간 중 사용자 로그인 후 즉시 EXIT 표시 — 정책상 PENDING 동안은 정상 로그인 + 배너만.
  • 요청 취소 후 D-day 카드가 즉시 사라지지 않음 — provider invalidate 누락.
