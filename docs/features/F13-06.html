F13-06. 계정 삭제 요청 (30일 유예)

사용자가 "지금 영구 삭제를 예약해 두고 싶다"고 의사를 표시하는 흐름. 신청 즉시 데이터가 사라지지는 않는다 — 30일 동안 유예 상태로 남아 있다가 그 기간이 지나야 매일 새벽에 동작하는 서버 작업이 개인정보를 익명화·삭제한다. 그동안 사용자는 마이페이지 상단 배너로 자기 상태를 인지하고, 마음을 바꿔 취소할 수 있다. 이 흐름이 흔들리면 "지웠는데 안 지워졌다"거나 "안 지웠는데 지워졌다"는 사고가 발생해 서비스 신뢰가 즉시 무너진다.

설계 의도 — "결심"과 "실행"의 시간 사이에 30일을 두어 사용자를 보호한다

계정 삭제는 비가역적이라 사용자가 충동적으로 누른 결과가 실제 데이터 손실로 이어지면 안 된다. 그래서 본 단위는 "신청한 의사"와 "실제로 데이터를 지우는 동작"을 의도적으로 분리한다 — 신청은 사용자 화면에서 즉시 가능하지만, 실제 삭제는 30일 뒤 서버 작업이 수행한다. 그동안 사용자가 다시 로그인해 "역시 안 지워야겠다"고 결정하면 한 번의 취소로 원상 복귀할 수 있다. 동시에 30일은 사용자가 자기 데이터를 다른 곳으로 옮길(F13-05 데이터 내보내기) 충분한 시간이기도 하다. 즉시 비활성화(F13-07)와는 의도가 다르다 — F13-07은 "당장 비공개로 만들고 싶다", 본 단위는 "충분한 회복 가능성을 두고 영구 삭제를 예약한다".

이 단위가 사용자 신뢰의 마지막 동선을 결정한다

계정 삭제 동선이 깔끔하지 않으면 "다시 가입"의 의사 자체가 사라진다
  • "내가 지운 게 정말 지워졌는가"의 확실성 — 사용자가 신청한 뒤 다시 들어와 봐도 자기 상태가 "삭제 진행 중"으로 분명히 보여야 한다. 배너가 사라지거나 정보가 어긋나면 "이거 진짜 신청된 게 맞나" 라는 의심이 남는다.
  • 유예 중 취소의 즉시성 — 사용자가 "취소"를 누른 순간 배너가 사라지고 마이페이지가 원상 복귀해야 한다. 취소했는데 배너가 남아 있으면 사용자가 자기 통제권을 잃은 것처럼 느낀다.
  • 30일 후 실제 삭제의 정확성 — 매일 새벽 4시 작업이 동작하지 않으면 유예 기간이 무한 연장되어 "지웠다고 약속한 데이터가 아직 살아 있다"는 신뢰 무너짐이 생긴다. 반대로 작업이 잘못 동작해 유예 기간 전에 삭제가 시작되면 사용자가 회복 기회를 잃는다.

한 번의 삭제 신청이 통과하는 흐름

신청 즉시 데이터가 사라지지 않는다 — 30일 동안 살아 있다가 새벽 작업이 한 번에 정리한다.

사용자 시나리오

기본 신청

30일 뒤 영구 삭제를 예약

데이터 프라이버시 화면에서 "계정 삭제 요청"을 누른다. 확인 다이얼로그가 떠 한 번 더 의사를 묻는다. 사용자가 확인하면 요청이 저장되고 즉시 화면이 "삭제까지 30일 남음" 카드와 "삭제 취소" 버튼으로 바뀐다. 마이페이지 상단에도 같은 카운트의 배너가 따라붙는다.

이 흐름이 남기는 것 — 사용자가 "지금 결심한 결과"를 화면에서 즉시 확인할 수 있다. 결심과 화면 상태가 어긋나면 신뢰 자체가 무너지는 흐름.

유예 중 인지

며칠 뒤 마이페이지에 다시 들어옴

로그인해 마이페이지를 열면 상단 배너가 여전히 보인다. 남은 일수가 줄어 있다 — "삭제까지 22일 남음". 사용자가 "관리"를 누르면 데이터 프라이버시 화면으로 빠져 거기서 취소할 수도, 그대로 둘 수도 있다.

이 흐름이 남기는 것 — 사용자가 "내가 신청해 둔 일"을 잊지 않게 한다. 배너가 없으면 30일 뒤 갑자기 데이터가 사라진 걸 알게 된다.

유예 중 취소

다시 쓰기로 결심해 취소

"삭제 취소"를 누르면 요청 상태가 "취소됨"으로 바뀌고 마이페이지 배너는 즉시 사라진다. "계정 삭제가 취소되었습니다" 안내가 뜨고 사용자는 평소처럼 서비스를 이어 갈 수 있다.

이 흐름이 남기는 것 — 30일 유예 정책의 본질적 가치 — "결심이 바뀐 사용자를 돌려보낸다". 취소가 매끄럽지 않으면 다음에 사용자가 "신청은 가볍지만 취소는 어렵다" 라고 인지해 신청 자체를 망설이게 된다.

중복 차단

이미 신청한 상태에서 다시 누름

유예 중인 사용자가 어떤 경로로 다시 "계정 삭제 요청"을 누르면 서버가 "이미 진행 중인 요청이 있다"고 거절한다. 화면에는 "요청에 실패했습니다" 안내가 뜨고, 기존 요청은 그대로 살아 있다.

이 흐름이 남기는 것 — 같은 사용자에게 동시에 두 개의 삭제 요청이 살아 있는 상태가 운영·집계 양쪽에서 어떤 의미인지 정의되지 않게 만들지 않는다.

30일 만료

새벽 작업이 실제로 데이터를 정리한다

유예 기간 동안 취소하지 않은 사용자의 요청은 매일 새벽 4시 작업이 일괄 처리한다. 사용자·멤버 정보 익명화, 주소·태그·동의 기록·소셜 로그인·푸시 토큰·결제 자격증명·데이팅 프로필 등이 한 번에 정리되고 계정 상태가 EXIT로 바뀐다.

이 흐름이 남기는 것 — 거래 기록은 법적 보존 기간(현재 정책상 5년)에 따라 보존된다. 사용자에게 "거래 기록도 다 사라진다"고 안내하면 약속을 못 지키는 셈이 되므로 정책 안내 문구에 정확히 반영되어야 한다.

자동 취소

유예 중 로그인하면 안내되는 자동 취소

화면 안내에는 "유예 기간 중 로그인하면 삭제 요청이 취소됩니다"라는 문구가 있다. 현재 구현에서 자동 취소 동작이 어디까지 보장되는지는 운영 관찰 포인트로 남는다 — 사용자가 이 문구를 보고 행동을 정하기 때문에, 약속과 실제 동작이 같지 않으면 정정 우선순위가 높다.

화면 구성 — 사람이 보는 것

영역보이는 것그 자리에 둔 이유
마이페이지 상단 배너 유예 중에만 노출. "계정 삭제 요청 진행 중" 제목, "삭제까지 N일 남음" 카운트, "관리" 텍스트 버튼. 사용자가 어느 화면에 있든 자기 계정 상태를 잊지 않게 한다 — 30일은 충분히 잊어버릴 수 있는 시간.
요청 전 안내 경고 박스 "계정 삭제를 요청하면 30일 유예 기간 후 모든 데이터가 영구 삭제됩니다", 위험 색상의 "계정 삭제 요청" 버튼. 버튼 색상으로 결과의 비가역성을 분명히 한다. 다른 일반 버튼과 같은 색이면 사용자가 결과의 무게를 인지하지 못한다.
유예 중 카드 "삭제까지 N일 남음" 큰 글씨, "유예 기간 중 로그인하면 삭제 요청이 취소됩니다" 안내, 외곽선 형태의 "삭제 취소" 버튼. 취소 버튼은 의도적으로 위험 색이 아니다 — "이 버튼을 누르면 위험한 일이 일어난다"가 아니라 "기존 신청을 되돌린다"는 의미.
요청 확인 다이얼로그 "정말 삭제하시겠습니까?" 본문 + 취소·삭제 두 버튼. 비가역적 결과의 동작에는 반드시 한 단계 의사 확인을 끼운다.

측정해야 하는 것

본 단위가 정상이라고 부르려면 이 다섯 가지가 추적되어야 한다
  • 삭제 요청 → 유예 중 취소율 — 30일 유예의 가치가 실제 회복으로 이어지는 비율. 너무 낮으면 유예 기간이 무의미하고, 너무 높으면 신청 단계가 너무 가벼운 신호.
  • 30일 만료 후 실제 삭제 작업 성공률 — 새벽 작업이 한 건이라도 실패해 만료된 요청이 누적되면 약속 위반.
  • 유예 중 사용자가 다른 기능을 계속 쓰는지 — 신청해 두고도 평소처럼 활동하는 사용자가 많다면 유예 정책의 가치가 잘 작동하는 신호.
  • 삭제 직후 같은 이메일·전화번호로 재가입하는 비율 — 너무 높으면 본 단위가 "계정 정리를 위한 도구"로 잘못 쓰이고 있다는 신호.
  • 마이페이지 배너 노출 후 "관리" 클릭률 — 사용자가 자기 상태를 능동적으로 확인하는지의 지표.

운영자가 알아야 할 것 — 비즈니스 리스크

본 단위에서 무너지면 사용자가 서비스 자체를 신뢰하지 못한다
  • 매일 새벽 4시 작업이 멈추면 정책 자체가 무너진다. 유예 만료된 요청이 누적되어 "지웠다고 한 데이터가 며칠씩 살아 있는" 상태가 된다 — 운영 모니터링의 최우선 알람 대상.
  • 유예 중 취소가 즉시 화면에 반영되지 않으면 사용자가 "취소 안 됐나" 하고 또 누른다. 두 번째 취소는 "이미 취소된 요청"이라 거절되고, 사용자에게는 마치 자기 행동이 의미 없는 것처럼 느껴진다.
  • 마이페이지 배너의 남은 일수 계산이 어긋나면 사용자가 자기 상태를 잘못 인지한다. 특히 음수가 표시되거나 일정 안내 형식이 깨지는 회귀가 자주 나타난다.
  • 거래 기록(포인트·결제·정산)은 법적 보존에 따라 익명화·삭제에서 빠진다. 사용자에게는 "모든 데이터가 사라진다"고 안내하는 카피와 어긋나지 않게 정책 문서를 정렬할 필요가 있다.
  • "유예 기간 중 로그인하면 자동 취소"라는 안내 문구가 화면에 있다. 현재 구현에서 자동 취소가 어디까지 보장되는지가 운영 관찰 항목 — 사용자는 이 문구를 보고 안심하기 때문에 실제 동작과 문구 사이의 차이가 가장 위험한 신호 영역.
  • 같은 사용자에게 두 개의 삭제 요청이 동시에 존재하면 안 된다. 중복 차단 가드가 풀리는 회귀가 들어가면 어떤 요청을 기준으로 30일을 세어야 할지 정의되지 않는다.
