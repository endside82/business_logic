F10-05. 타 사용자 가용성 공개 조회

다른 사용자가 공개해 둔 가용 시간을 보고, 그 시간에 이벤트나 만남을 제안할 수 있도록 한다. 공개 범위는 두 단계 — "전체 공개"와 "친구만 공개". 본 단위는 캘린더 도메인 안에 자체 화면을 두지 않고, 이벤트 제안 폼이나 데이팅 일정 조율 같은 다른 도메인의 화면이 호출해서 쓰는 데이터 소스 역할만 한다.

공개 범위
2종
전체 공개 · 친구 공개
친구 정의
암묵적
같은 이벤트 참석 또는 상호 리뷰 — 사용자에게 직접 노출 안 됨
비공개 슬롯
노출 없음
어떤 외부 조회 경로로도 비공개 슬롯은 응답에 포함되지 않는다
설계 의도 — 가용 시간이 제안을 만들 때 비로소 가치를 가진다

본인이 비워둔 시간을 본인만 보고 있으면 단순한 일정 메모와 다를 게 없다. 다른 사용자가 그 시간을 보고 이벤트나 만남을 제안해 와야 가용 시간이 제 역할을 한다. 그래서 본 단위는 "보는 쪽"이 아니라 "다른 사용자가 보는 면"을 책임진다. 단, 모든 시간이 모두에게 보이는 건 아니다 — 등록자가 정한 공개 범위 안에서만 응답에 포함된다. 비공개 슬롯은 등록자 본인 외에는 그 누구도 보지 못한다.

공개 범위 세 단계

전체 공개

로그인 없이도 조회 가능

등록자가 "공개"로 설정한 슬롯은 누구나 조회할 수 있다. 가입 전 사용자가 "이 사람이 언제 시간이 비어 있는지" 둘러볼 수 있는 진입점.

친구 공개

친구 관계가 확인될 때만

"친구"는 같은 이벤트에 참석한 이력이 있거나, 서로 리뷰를 남긴 사이로 정의된다. 호출자가 친구가 아니면 응답이 거부된다.

비공개

등록자 본인만

어떤 외부 조회 경로에서도 노출되지 않는다. 본인 일정 관리 전용으로 등록한 슬롯이 우연히 다른 사용자에게 노출될 위험이 없다.

사용자 시나리오

기본 흐름

이벤트를 제안하기 전 상대 가용 시간 확인

사용자 A가 사용자 B에게 이벤트를 제안하려 한다. 제안 폼 화면이 B의 공개 가용 시간을 가져와 "이 사람이 비워둔 시간"이 캘린더에 초록 블록으로 표시된다. A가 그 슬롯 중 하나를 선택하면 이벤트 시간 입력에 자동 반영된다.

이 흐름이 남기는 것 — "내가 원하는 시간"이 아니라 "상대가 가능한 시간" 안에서 제안이 시작된다. 매칭 성사율을 끌어올리는 직접적 입력값.

친구 한정

친구 관계가 인정된 상대

A와 B가 같은 이벤트에 함께 참석한 이력이 있다. A가 친구 가용 시간을 호출하면 서버가 친구 관계를 확인한 뒤 B의 친구 공개 슬롯을 응답한다. 두 사람이 만난 적 없는 첫 인사가 아니라, 한 번 같이 시간을 보낸 사이에서만 더 깊은 가용 시간이 공유되는 구조.

친구 거부

처음 보는 사용자의 친구 가용 시간 요청

A가 처음 보는 사용자 C의 친구 가용 시간을 요청하면 서버가 친구 관계 없음을 확인하고 응답을 거부한다. 호출자 화면은 "이 사용자의 가용 시간은 친구만 볼 수 있어요" 등 안내로 처리한다.

자기 자신

본인 가용 시간을 친구 경로로 요청

버그나 디버그로 본인 친구 가용 시간을 요청하면 서버가 명시적으로 거부한다 — 본인 슬롯은 본인이 등록한 화면에서 직접 보는 것이지, 친구 경로로 들어오는 건 잘못된 요청이라는 정책.

빈 결과

상대가 가용 시간을 등록한 적 없음

응답은 정상이지만 슬롯이 0건. 호출자 화면이 "공개된 가용 시간이 없습니다" 같은 안내를 표시한다. "기능 자체가 안 되는 것"이 아니라 "이 사용자가 아직 비워 둔 시간이 없다"는 사실 그대로.

운영자가 알아야 할 것

노출 범위 정책이 사용자 신뢰의 직접적 입력값이다
  • 비공개 슬롯은 어떤 외부 조회 경로로도 노출되지 않는다. 사용자가 "비공개"로 설정한 시간이 다른 사용자에게 한 번이라도 노출되면 가용 시간 기능 자체의 신뢰가 무너진다 — 이 가드는 어떤 변경에서도 풀려서는 안 된다.
  • "친구" 정의가 사용자에게 직접 보이지 않는다. 사용자는 같은 이벤트 참석 이력 또는 상호 리뷰가 친구를 만든다는 것을 모를 수 있다. "왜 이 사람만 친구로 인정되지" 라는 혼란이 발생할 수 있는 지점 — 호출자 화면에 안내 카피를 두는 것을 검토할 가치가 있다.
  • 본 단위는 캘린더 안에 화면이 없다. 이 데이터를 어떤 화면에서 어떻게 보여 줄지는 호출자 도메인(이벤트 제안, 데이팅 일정 조율 등)의 결정. 따라서 사용 빈도와 효과는 호출자 도메인의 동선 설계에 달려 있다.
  • 존재하지 않는 사용자에 대한 조회도 빈 결과로 응답한다. 서버가 사용자 존재 여부를 별도로 검증하지 않으므로, 잘못된 식별자가 들어와도 "공개된 시간이 없습니다"로 보인다 — 호출자 입장에서는 빈 결과와 식별자 오류가 같게 보인다.
  • 전체 공개 조회는 로그인 없이도 가능하도록 작성되어 있다. 실제 운영에서 비로그인 조회를 허용할지는 운영팀의 보안 정책으로 별도 결정한다 — 가입 전 사용자에게 "여기 어떤 사람이 어떤 시간에 비어 있는지" 둘러보게 할 것인가의 정책 판단.
