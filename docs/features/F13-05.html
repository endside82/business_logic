F13-05. 데이터 내보내기

사용자가 자기 데이터의 사본을 JSON 파일로 받는 흐름. 프로필·이벤트 신청·신뢰점수·지갑·알림·데이팅 인증 등 서비스 안에 흩어져 있는 본인 데이터를 한 묶음으로 모아 내려보낸다. 요청 즉시 파일이 만들어지지 않는다 — 서버가 비동기로 데이터를 모아 파일을 만든 뒤, 준비가 끝나면 사용자가 그 파일을 다운로드한다. 사용자에게 "지금 진행 중"임을 분명히 보여 주는 것이 본 화면의 핵심.

처리 방식
비동기
요청 → 처리 → 완료
사용자 안내
최대 48시간
대기 기대 시간
파일 보관
30일
지나면 만료, 다시 요청 필요
설계 의도 — "버튼을 눌렀더니 다운로드가 떨어진다"는 기대를 의도적으로 깬다

본 데이터 묶음에는 여러 도메인의 정보가 들어가므로 즉석에서 모으는 데 시간이 걸린다. 사용자에게 "지금 만들어 두고 있어요" 라는 상태를 분명히 보여 주지 않으면 "이게 다운로드가 되긴 한 건가" 라는 불신이 생긴다. 그래서 본 화면은 요청 직후 진행 표시·경과 시간·"최대 48시간" 안내를 같이 두어 "기다리는 게 정상"이라는 맥락을 만든다. 완료 후 30일이 지나면 만료시켜 파일 보관 책임을 사용자에게 명시적으로 넘긴다.

사용자 시나리오

기본 흐름

처음으로 내 데이터를 요청한다

데이터 프라이버시 화면에서 "데이터 내보내기 요청"을 누른다. 즉시 "요청했습니다" 안내가 뜨고, 화면이 처리 중 상태로 바뀐다 — 진행 인디케이터, "요청 후 N분 전 경과" 라벨, "최대 48시간 소요" 안내가 함께. 요청 버튼은 비활성으로 바뀐다.

이 흐름이 남기는 것 — 한 번 요청을 받은 뒤에는 "다시 누르라"고 안내하지 않는다 — 중복 요청 자체가 사용자를 혼란시킨다.

중간 확인

나중에 다시 들어와서 진행 상태 확인

다음 날 같은 화면을 다시 연다. 진행 인디케이터는 그대로고 새로고침 아이콘이 우측에 보인다. 누르면 상태가 다시 조회된다.

운영 관찰 포인트 — 새로고침을 자주 누르는 사용자가 많다면 자동 갱신 정책을 검토.

완료 후 받음

완료된 파일을 다운로드

화면이 "상태: 완료"와 만료일(D-N), "다운로드" 버튼으로 바뀐다. "다운로드"를 누르면 외부 브라우저나 파일 앱이 열려 사용자가 그곳에서 파일을 받는다.

이 흐름이 남기는 것 — 사용자가 자기 데이터를 다른 곳에 보관해 둘 수 있다는 명시적 통제권. 개인정보 이동권의 사용자 실감.

실패

처리 중 오류로 작업이 실패

서버 처리 중 문제가 생기면 상태가 "실패"로 바뀌고 "다시 요청" 버튼이 뜬다. 사용자가 누르면 새 요청이 만들어진다.

만료

완료 후 오래 두면 파일이 만료된다

완료된 파일을 30일 안에 받지 않으면 상태가 "만료"로 바뀌고 "다시 요청" 버튼이 뜬다. 한 번 만들어진 파일을 영원히 보관하지 않는 것이 보안·저장 비용 양쪽 정책.

화면 구성 — 사람이 보는 것

상태보이는 것그 자리에 둔 이유
요청 전 "데이터 내보내기" 제목, "내 데이터를 JSON 파일로 다운로드할 수 있습니다. 처리에 최대 48시간이 소요됩니다." 설명, "데이터 내보내기 요청" 버튼. 요청 전에 "시간이 걸린다"는 사실을 미리 알려 주어 완료 즉시 다운로드 기대를 깨 둔다.
처리 중 선형 진행 인디케이터, "요청 후 N분/N시간/N일 전 경과", "최대 48시간 소요" 안내, 새로고침 아이콘. 요청 버튼은 비활성. 경과 시간을 보여 주면 사용자가 "내가 너무 오래 기다리고 있는 건가" 자체 판단할 수 있다.
완료 "상태: 완료", 만료일 "yyyy.MM.dd (D-N)", "다운로드" 버튼. 만료일을 카드에 함께 두어 "지금 받지 않으면 다시 요청해야 한다"는 사실을 사용자가 인지하게.
실패·만료 "상태: 실패" 또는 "상태: 만료됨", "다시 요청" 버튼. 실패와 만료 모두 사용자 행동(다시 요청)이 같으므로 같은 UI 패턴.

운영자가 알아야 할 것

데이터 내보내기 정책의 결정점
  • "최대 48시간" 안내는 클라이언트 문구일 뿐 서버가 48시간을 강제하지 않는다. 실제 처리 시간이 그보다 오래 걸리는 케이스가 누적되면 사용자가 "약속이 안 지켜진다"는 인상을 받는다 — 평균 처리 시간 모니터링이 운영 책임.
  • 완료 후 30일이 지나면 파일은 만료된다. 사용자가 같은 파일을 두 번 받으려고 하면 다시 요청이 필요하다는 안내가 보여야 한다.
  • 실패 사유는 사용자에게 노출되지 않는다. 개인 데이터 처리 중 오류의 구체적 원인이 클라이언트로 흘러가면 보안 위험이 있다 — "다시 요청"으로만 안내하고 실패율은 운영 측에서 별도 추적.
  • 거래 기록은 법적 보존 기간에 따라 익명화·삭제 대상에서 빠진다. "내 데이터 사본"이 모든 데이터를 포함한다고 사용자가 오해할 수 있어, 안내 문구에 포함 범위를 명시할 여지가 있다.
