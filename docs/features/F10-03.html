F10-03. 단일 가용 시간 등록·수정·삭제

본인이 비어 있는 시간 슬롯을 캘린더에 명시적으로 표시한다. 공개로 설정하면 다른 사용자가 그 슬롯을 보고 이벤트나 만남을 제안할 수 있고, 비공개로 두면 본인 일정 관리 용도로만 쓰인다. 한 슬롯의 시작·종료 시각·공개 범위를 등록하고, 잘못 만든 슬롯을 수정하거나 삭제한다. 삭제할 때 그 시간대에 잡혀 있는 이벤트가 있으면 한 단계를 더 거쳐 확인을 받는다.

슬롯 최소 길이
30분
30분 단위로 최대 8시간까지
공개 범위
3단계
공개 · 친구 · 비공개 (현재 화면 토글은 2단계)
시간 겹침 정책
금지
같은 사용자의 기존 가용 시간과 겹치면 등록 거부
설계 의도 — 빈 시간을 명시적인 데이터로 만든다

"비어 있다"는 사실은 일반 캘린더에서는 그냥 "일정이 없다"로 표현된다. 본 서비스는 그 빈 시간을 한 단계 더 끌어올려 "이 시간에 무언가 들어와도 좋다"는 의사 표시로 데이터화한다. 그래야 다른 사용자가 이벤트나 만남을 제안할 때 출발 신호가 된다. 그래서 가용 시간은 "비어 있는 시간"이 아니라 "사용자가 의식적으로 비워 둔 시간 슬롯"이다 — 길이·공개 범위가 모두 사용자의 선택이며, 잘못 만든 슬롯을 사용자가 직접 삭제할 수 있어야 그 의사 표시의 통제권이 사용자에게 머무른다.

이 단위가 하는 세 가지 동작

등록

새 슬롯 만들기

날짜·시작·종료·공개 범위를 정해 새 슬롯을 등록한다. 본인의 기존 슬롯과 시간이 겹치면 등록을 막고 "선택한 시간이 기존 가용 시간과 겹칩니다" 경고 배너를 띄운다. 길이가 30분 미만이거나 30분 단위가 아니면 저장 자체가 비활성.

수정

이미 만든 슬롯 바꾸기

같은 화면이 편집 모드로 열리며 기존 값이 채워져 있다. 시간·공개 범위를 바꾸고 저장하면 갱신된다. 자기 자신을 제외한 다른 슬롯과 겹치는지 다시 검사한다.

삭제

슬롯 지우기 (충돌 확인)

편집 화면 우상단의 "삭제"를 누르면 두 단계로 진행. 1단계는 단순 확인, 그 시간대에 본인이 호스팅 중인 이벤트가 있으면 2단계로 "겹치는 이벤트가 N건 있습니다. 그래도 삭제하시겠습니까?"가 떠오른다. "예"를 한 번 더 누르면 가용 시간만 사라지고 이벤트는 그대로 유지된다.

저장 한 번에 일어나는 일

사용자가 "저장"을 누른 한 순간에 시간 형식 검증·과거 시각 차단·시간 겹침 검사·DB 저장이 한 동작으로 진행된다.

삭제할 때 거치는 두 단계

한 번에 지우는 위험을 피하기 위해 1차 확인 + 충돌 시 2차 확인. 충돌이 없으면 1차 확인만으로 종료된다.

사용자 시나리오

기본 흐름

점심시간을 비워 두고 싶다

일간 캘린더 5월 12일에 들어가 FAB을 누른다. 폼 기본값은 09:00-10:00. 시간을 12:00-13:30으로 바꾸고 제목 "점심시간 가용", 공개 범위는 그대로 공개로 두고 저장. 등록 토스트와 함께 캘린더로 돌아오면 해당 일자에 초록 마커가 새로 생긴다.

이 흐름이 남기는 것 — 본인이 의식적으로 비워 둔 시간이 다른 사용자에게 "여기 제안해도 좋다"는 신호로 나타난다.

시간 겹침

같은 시간에 또 비워 두려 할 때

이미 5/12 12:00-13:00 슬롯이 있는 상태에서 12:30-14:00을 새로 등록하려 한다. 저장 순간 노란 배너 "선택한 시간이 기존 가용 시간과 겹칩니다"가 떠오른다. 사용자가 시간을 조정해 다시 저장하면 정상 진행.

이 흐름이 남기는 것 — 같은 사용자의 가용 시간이 한 시각에 두 번 있을 수 없다는 정책. 다른 사용자가 어떤 슬롯에 제안해야 하는지 모호해지는 상황을 막는다.

짧은 길이

15분만 비워 두려 할 때

14:00-14:15로 입력하면 시간 박스 아래에 빨간 안내 "최소 30분 이상의 가용 시간을 설정해주세요"가 즉시 표시되고 저장 버튼이 비활성화된다. 서버에 도달하지도 않는다.

이 흐름이 남기는 것 — "한 슬롯이 너무 짧으면 다른 사용자가 의미 있는 제안을 못 한다"는 정책을 화면에서 미리 차단.

수정

등록한 슬롯의 시간 바꾸기

캘린더에서 그 가용 시간 블록을 탭하면 편집 화면이 열리고 기존 값이 채워져 있다. 시간을 11:30-13:00으로 조정하고 저장하면 "가용 시간이 수정되었습니다" 토스트.

충돌 없는 삭제

그 시간대에 다른 이벤트가 없다

편집 화면 우상단 "삭제" → 1차 확인 → "삭제". 같은 시간대에 본인 이벤트가 없으면 한 번에 사라진다.

충돌 있는 삭제

그 시간에 본인 호스팅 이벤트가 잡혀 있다

5/12 12:00-13:30 슬롯을 지우려는데, 같은 시간에 본인이 만든 모집 중 이벤트가 1건 있다. 1차 확인 후 자동으로 2차 다이얼로그 "이 가용 시간과 겹치는 이벤트가 1건 있습니다. 그래도 삭제하시겠습니까?"가 떠오른다. "예"를 누르면 가용 시간만 사라지고 이벤트는 그대로 유지된다.

이 흐름이 남기는 것 — 가용 시간과 이벤트는 별도 데이터지만, 사용자가 "비워둔 시간"을 지웠을 때 그 시간에 잡혀 있던 이벤트의 존재를 의식하게 한다.

실패

네트워크가 끊긴 상태에서 삭제

1차 확인 후 통신이 실패하면 "삭제에 실패했습니다" 토스트. 슬롯은 그대로 남고 사용자가 다시 시도할 수 있다.

화면에서 사용자가 인지하는 것

영역사용자가 보는 것그 자리에 둔 이유
상단 타이틀·삭제 액션 신규는 "가용 시간 설정", 편집은 "가용 시간 수정". 편집 모드에서 우상단에 빨간 "삭제" 텍스트. 화면 모드를 한눈에 구분하고, 삭제 액션은 편집 모드에서만 노출해 신규 등록 중에 잘못 누르는 사고를 막는다.
날짜·시간 필드 날짜는 오늘 이후로만 선택 가능. 시작·종료 시간은 30분 단위. 가운데에 "~" 표시. 과거 시각에 가용 시간을 등록하는 의미 없는 흐름을 화면에서 미리 막는다.
지속시간 라벨 정상이면 옅은 초록 배경에 "⏱️ X시간 Y분", 30분 미만이면 같은 자리에 빨간 안내 텍스트. 사용자가 시간을 만질 때마다 그 결과가 즉시 보여, 저장 시점에야 거부당하지 않게 한다.
충돌 경고 배너 저장 시도 후 기존 슬롯과 겹쳤을 때만 떠오르는 노란 배너. "선택한 시간이 기존 가용 시간과 겹칩니다. 시간을 조정해주세요." 겹침 검사는 서버에서만 가능한 정책이므로 저장 후에 안내된다. 사용자가 시간을 조정할 때까지 화면에 남는다.
공개 토글 "공개"/"비공개" 두 단계 스위치. 친구 한정 공개는 화면에서 직접 선택할 수 없다. 현재는 두 단계로 단순화. 친구 공개 슬롯이 필요한 사용자는 다른 경로로 설정해야 한다는 한계가 있다.
반복 설정 스위치 신규 모드에서만 노출. 켜면 요일 선택 칩이 인라인으로 노출된다. "매주 반복"이 자연스러운 흐름에서 한 화면에서 끝낼 수 있게. 반복 규칙의 전체 흐름은 F10-04에서 다룬다.

운영자가 알아야 할 것

가용 시간은 다른 사용자가 보는 데이터다 — 정책의 일관성이 중요하다
  • 같은 시간에 두 가용 시간을 만들 수 없다. 다른 사용자가 어떤 슬롯에 제안해야 하는지 모호해지면 가용 시간 기능 자체가 신뢰를 잃는다. 겹침 검사는 등록·수정 둘 다 동일하게 동작한다.
  • 최소 30분, 30분 단위, 최대 8시간. 너무 짧은 슬롯은 의미 있는 제안이 들어오지 않고, 너무 긴 슬롯은 가용 시간이라기보다 그냥 "오늘은 한가하다"에 가까워진다 — 그래서 8시간 상한.
  • 삭제 시 같은 시간대 이벤트가 있으면 한 단계 더 거친다. 가용 시간과 이벤트는 별도 데이터지만, 사용자가 보기에는 한 시간대를 정리하는 동작이라는 점을 인지시키는 장치. "그래도 삭제"를 선택하면 이벤트는 그대로 남는다 — 가용 시간만 정리.
  • 친구 한정 공개는 화면 토글에서 빠져 있다. 데이터 모델은 친구 공개를 지원하지만 화면 노출이 두 단계로 축소되어 있어, 운영 정책상 친구 공개 슬롯이 활용되는 사례가 드물 가능성이 높다 — 필요해지면 토글 확장을 검토할 시점.
  • 다른 사람의 가용 시간을 임의로 편집할 수 없다. 편집 화면을 URL로 강제 진입해도 본인 슬롯이 아니면 서버가 거부한다 — 단, 사용자에게 명확한 안내가 가지 않는 경계가 있어 운영 모니터링 포인트.
