F18-01. 통합 분쟁 케이스 조회

다양한 원천 도메인에서 발생한 분쟁·신고·이의를 단일 목록으로 묶어 보여 주는 화면. source별 조회 기준이 다르다 — 대부분의 source는 신고자·영향받은 당사자 시점만 포함하고, USER_DISPUTE만 reporter/target 양측을 포함한다. 호스트는 정산 이의(개설자 시점)와 환불 분쟁(판매자 시점) 2종의 source에서 별도 케이스를 확인한다. 케이스 상태는 원천 도메인의 고유 상태를 5값으로 정규화한 통합 상태를 사용한다. 7일 이상 미해결 케이스는 운영자 알림이 자동으로 발행된다.

설계 의도 — "내 분쟁이 지금 어떻게 되고 있는지" 한 눈에

이전에는 노쇼 이의는 이벤트 화면, 정산 이의는 정산 화면, 경고 이의는 클럽 화면으로 각자 흩어져 있었다. 이 기능은 그 모든 케이스를 하나의 목록으로 모아 사용자가 현재 진행 중인 분쟁과 종결된 분쟁을 한 화면에서 추적하게 한다. 원천 도메인이 다르더라도 사용자가 보는 상태 레이블은 항상 동일하다.

케이스 상태가 거치는 길

stateDiagram-v2 [*] --> OPEN: 원천 이벤트 발생 또는 사용자 직접 접수 OPEN --> IN_REVIEW: 운영팀 검토 시작 OPEN --> ESCALATED: 바로 에스컬레이션 OPEN --> RESOLVED: 바로 처리 완료 OPEN --> CLOSED: 바로 종결 IN_REVIEW --> ESCALATED: 상위 CS로 IN_REVIEW --> RESOLVED: 처리 완료 IN_REVIEW --> CLOSED: 종결(철회·기각) ESCALATED --> RESOLVED: 처리 완료 ESCALATED --> CLOSED: 종결 RESOLVED --> [*]: 종결 + 1년 후 증빙 정리 CLOSED --> [*]: 종결 + 1년 후 증빙 정리 note right of OPEN evidenceFrozen = true 이의제기 가능 end note note right of RESOLVED evidenceFrozen = false 재접수 CTA 없음 end note
상태 전이는 admin API만 소유

공개 API는 케이스 상태를 직접 변경하는 endpoint를 제공하지 않는다. USER_DISPUTE source는 OPEN으로 생성된 후 admin이 처리할 때까지 상태가 변하지 않는다. 사용자는 현재 상태를 조회하고 이의를 제기하는 것만 가능하다.

사용자 시나리오

기본 흐름 — 참가자

내가 관련된 분쟁 목록 확인

이벤트 노쇼 기록, 클럽 강퇴 이의, 직접 접수한 분쟁이 모두 한 목록에 보인다. "처리 중" 필터를 선택하면 OPEN·IN_REVIEW·ESCALATED 케이스만 추려 볼 수 있다. 카드를 탭하면 상세 화면으로 진입해 타임라인과 증빙을 확인한다.

이 흐름이 남기는 것 — 어느 도메인에서 발생했든 "내 분쟁 이력"이 한 화면으로 집약된다.

기본 흐름 — 호스트

내 이벤트·클럽에 걸린 분쟁 확인

호스트가 확인하는 source는 두 종류다. SETTLEMENT_APPEAL(정산 이의, 개설자 기준)과 REFUND_DISPUTE(환불 분쟁, 판매자 기준). 두 source의 호스트 조회 기준이 다르며, 동일한 목록·필터 구조를 공유한다. 상세 화면에서 호스트에게 허용된 액션 패널이 별도로 표시된다.

이 흐름이 남기는 것 — 호스트 입장에서 처리해야 할 케이스를 놓치지 않게 한다.

상세 조회

타임라인과 증빙으로 맥락 파악

케이스 상세에는 타임라인 항목(누가 언제 어떤 액션을 했는지)과 증빙 파일이 포함된다. 보이는 내용은 역할에 따라 다르다 — 참가자는 양측 공개(PARTIES) 증빙만 볼 수 있고, CS 전용 항목은 표시되지 않는다.

이 흐름이 남기는 것 — "왜 이 결정이 났는지"를 사용자가 직접 추적할 수 있다.

상태 필터

종결된 케이스만 이력으로 보기

"종결" 필터를 선택하면 RESOLVED·CLOSED 케이스만 남는다. sourceType 파라미터로 원천 유형을 좁힐 수 있으나, DisputeSourceType enum과 실제 caseId prefix가 1:1 대응하지 않으므로 필터 결과에 주의가 필요하다. 무한 스크롤로 이력 전체를 탐색한다.

권한 분기

내가 뭘 할 수 있는지 화면이 알려준다

케이스 상세에는 7개의 액션 권한 flag(ActorPermissionFlags)가 포함된다. 이의제기 가능 여부, 노트 전송 가능 여부, 환불 분쟁 처리 가능 여부 등이 이 flag를 기반으로 화면에 버튼으로 표시되거나 숨겨진다. "이의 승인" 버튼은 현재 서버 endpoint가 없어 비활성 상태다.

evidenceFrozen

케이스가 진행 중이면 증빙 보존 안내가 표시된다

증빙 삭제 기능은 현재 미구현(조회 전용)이다. OPEN·IN_REVIEW·ESCALATED 케이스에서는 "증빙 보존 중" 안내 배너가 표시된다. 케이스 종결 후에도 삭제 버튼은 별도 구현 전까지 제공되지 않는다.

운영자가 알아야 할 것

운영 주의사항
  • SLA 7일 알림은 8개 scanOne 호출을 독립 처리한다. 한 source의 스캔이 실패해도 나머지는 계속 처리된다. 알림은 idempotency_key로 중복 발행이 차단된다. PUBLIC_REPORT는 PENDING+IN_REVIEW만 스캔(ESCALATED 제외).
  • 타임라인 CS_ONLY 항목은 사용자에게 보이지 않는다. 서버 filterByActorVisibility가 CS_ONLY를 제거한 결과만 전달한다. PARTIES/HOST_ONLY/PUBLIC_SUMMARY는 유지된다(현재 builder는 HOST_ONLY 미emit).
  • caseId prefix와 DisputeSourceType enum이 1:1 대응하지 않는다. DisputeSourceType enum 21값은 케이스 sourceType 필드 분류용이며, 실제 emit prefix(OPERATIONAL_ISSUE:/WARNING_REPORT:/WARNING_APPEAL:/ SETTLEMENT_APPEAL:/REFUND_DISPUTE:/REPORT:/DATE_BLOCK:/TRANSPORT:/CLUB_MEMBERSHIP_ACTION:/USER_DISPUTE:)와 다르다. EVENT_NO_SHOW는 enum에 없는 appeal dispatcher 전용 prefix다. 새 source 추가 시 클라이언트 enum 동기화가 필요하다.
  • canApproveAppeal flag는 있지만 공개 endpoint가 없다. 호스트/클럽 staff용 이의 승인·거절 공개 API가 미구현 상태다. 현재 admin API로만 처리 가능하다.

레드 플래그

현재 알려진 미구현·주의 사항
  • dispute 딥링크 미배선. 분쟁 관련 FCM 알림(USER_DISPUTE·CLUB_MEMBERSHIP_ACTION 타입)을 클릭하면 홈으로 이동한다. 기존 REFUND_DISPUTE 3개 타입만 처리되어 있다.
  • canApproveAppeal 버튼 동작 불가. 앱 화면에 이의 승인 버튼이 있으나 서버 전이 endpoint가 없어 실제 동작하지 않는다.
  • union 정합성 테스트 없음. 실제 emit source(사용자 8종/호스트 2종) 각각에 대해 caseId 형식·상태 매핑·visibility 필터가 올바른지 검증하는 자동화 테스트가 없다.
