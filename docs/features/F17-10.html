F17-10. 호스트 정산 (flow-through)

코스가 종료되면 코스당 정산 1건이 자동으로 만들어진다. 호스트가 실제로 받는 금액은 유료 포인트로 들어온 매출만이고, 무료 보너스 포인트로 결제된 부분은 플랫폼 보조로 분리되어 호스트 지급 대상이 아니다. 외부 계좌이체 결제는 호스트가 직접 받은 돈이라 정산에 들어가지 않는다. 환불은 정산 이전에 끝나도록 설계되어 정산 후 새 환불이 생기지 않는다.

1건

유료 잔존액
무료는 플랫폼 보조
수수료

플랫폼 수수료·원천징수
flow-through — 유료/무료를 분리해 호스트 수익에 잡는다

플랫폼 포인트는 두 종류 — 유료 포인트(사용자가 현금으로 충전)와 무료 보너스(이벤트로 받은 무료 적립). 결제 시 두 종류가 섞이고 환불도 비례 차감되는데, 정산에서는 유료 잔존액만 호스트 수익으로 잡는다. 무료 잔존액은 플랫폼이 보조한 부분이라 호스트에게 지급되지 않는다 — 분리하지 않으면 무료 포인트로 결제된 코스 매출까지 호스트 지급이 일어나 플랫폼 손실이 된다.

정산 금액 구성

항목의미
매출 (gross)유료 포인트 잔존액 — 결제에서 환불을 뺀 순액
플랫폼 보조 (별도)무료 보너스 잔존액 — 호스트 지급 대상 아님, 정산에 별도 표시
플랫폼 수수료매출의 10%
원천징수매출의 3.3%
실지급액 (net)매출 − 수수료 − 원천징수
계좌이체는 정산 대상 아님

계좌이체로 결제된 부분은 호스트가 외부에서 직접 받은 돈이라 플랫폼 매출이 아니다. 정산에는 한 푼도 포함되지 않으며 "이건 호스트가 직접 받은 거래"라고 분리되어 표기된다.

왜 "정산 후 환불 위험 0"인가

flowchart TD A["호스트: 종료 버튼"] --> B{"모든 회차 확정 완료?"} B -->|"아니오"| BX["거절 — 종료 통과 못 함"] B -->|"예"| C["종료 확정"] C --> D["정산 생성 — 별도 단계"] C --> E{"멤버 자가 취소?"} E -->|"종료 후 시도"| EX["차단 — 시스템이 거절"] D --> F["호스트 정산 보기"]
다층 방어

① 종료는 모든 회차 확정 후에만 통과. ② 정산은 종료 확정이 끝난 다음 별도 단계로 만들어진다. ③ 그 사이 멤버의 자가 취소는 차단된다. ④ 자격박탈 환불은 회차 확정 시점(=종료 이전)에만 발생하므로 정산 매출에 자연 반영된다. 그래서 정산이 만들어진 뒤에 새 환불이 발생할 가능성이 사실상 0이다.

사용자 시나리오

정상 정산

"8명 가입·전부 유료 지갑 결제"

코스 종료 직후 정산 1건 자동 생성. 매출 = 사용자 8명의 유료 결제 합계, 수수료·원천징수 적용 후 실지급액 호스트 지급.

이 흐름이 남기는 것 — 호스트가 종료 버튼만 누르면 정산이 다음 화면에서 보이는 자연스러운 흐름.

중간 환불 있음

"한 명이 3회차 후 취소·환불 받음"

그 사용자의 결제 - 환불 = 유료 잔존액으로 매출에 반영. 환불액은 정산에서 자동 차감된 상태로 계산됨.

이 흐름이 남기는 것 — 환불 발생 후에도 호스트 정산이 정확히 떨어진다.

계좌이체 결제 포함

"5명 지갑·3명 계좌이체"

계좌이체 3명분은 정산에 포함 안 됨. 매출 = 지갑 5명분만. 호스트는 계좌이체 3명에게서 외부로 직접 받은 돈을 따로 관리한다.

이 흐름이 남기는 것 — 호스트가 결제 방식별 자기 수익을 분리해 인지하게.

무료 포인트만

"이벤트 보너스로만 결제된 코스"

유료 잔존액 0원 → 정산 행을 만들지 않음. 호스트 수익 없음, 무료 보너스는 플랫폼 보조로 흡수.

이 흐름이 남기는 것 — 무료 보너스 결제분이 호스트 지급으로 잘못 흘러가지 않게.

정산 준비중

"종료 직후 정산을 보러 갔는데 빈 화면"

자동 생성이 일시 실패하면 안전망이 1시간마다 재시도. 그 사이 호스트 화면은 "정산 준비중" 라벨. 재시도 성공 후 즉시 노출.

이 흐름이 남기는 것 — 자동화 일시 실패가 호스트 신뢰를 깨뜨리지 않게.

중복 정산 차단

"같은 코스 두 번 정산?"

코스당 정산 1건 보장. 어떤 사유로 자동 생성이 두 번 호출되어도 시스템이 두 번째를 막는다.

이 흐름이 남기는 것 — 호스트 이중 지급 사고 차단.

화면에서 사용자가 인지하는 것

영역사용자가 보는 것그 자리에 둔 이유
금액 분해매출·플랫폼 보조·수수료·원천징수·실지급액이 표 형태로 노출.호스트가 자기 수익이 어떻게 산정됐는지 의문 없이 인지하게.
유료/무료 분리 라벨"플랫폼 보조" 행을 별도 색상으로 구분 표기.호스트가 "왜 매출이 결제 총액보다 적지?"를 인지하게.
준비중 라벨정산 자동 생성 실패 시 "정산 준비중 — 잠시 후 확인" 안내.한 시간 안에 자동 재시도가 돌고 있음을 사용자에게 알린다.
실지급일 표기"실지급액은 X일에 지급 예정" 안내.호스트가 자기 자금 흐름을 예측할 수 있게.
잔여 환불 책임 표기미정리 환불(자격박탈 환불 실패 등)이 있으면 정산 지급이 보류된다는 안내.호스트가 모든 환불 책임이 끝나야 지급이 진행됨을 인지하게.

운영자가 알아야 할 것

멱등·재시도·잔액 가드
  • 코스당 정산 1건 보장 — 같은 코스에서 두 번 정산이 만들어지지 않는다.
  • 참조 거래가 없으면 정산 생성 중단 — 데이터 무결성이 깨진 상태에서는 정산을 만들지 않고 다음 안전망 주기에서 재시도.
  • 유료 잔존액이 0 이하면 정산 행을 만들지 않는다 — 무료 포인트만으로 채워진 코스는 호스트 지급 대상이 아님.
  • 승인·지급은 일반 모임 정산 파이프라인 재사용 — 미정리 환불(잔여 환불 책임)이 남으면 지급 보류.
  • 종료 직후 정산이 "준비중"으로 잠시 보일 수 있다 — 자동 생성 성공 후 즉시 가시화, 실패 시 최대 1시간 대기.

레드 플래그

분쟁이 일어날 수 있는 케이스
  • 안전망 재시도까지 호스트가 정산을 못 보는 시간 — 자동 생성 실패 시 최대 1시간 대기. UI 안내 필수.
  • 무료 포인트만으로 결제된 코스 — 호스트가 "왜 정산이 0원이지?"라고 묻는 케이스. 무료 보너스는 호스트 수익 아님을 사전 안내 필요.
  • 정산 이의제기 흐름 부재 — 1차 출시는 일반 모임 정산의 이의제기 흐름을 재사용. 정기모임 전용 분쟁 화면은 후속.
  • 유료/무료 섞인 결제 라벨링 — 정산 분해 라벨이 명확해야 호스트가 의문을 갖지 않는다.
