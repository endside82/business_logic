F17-10. 호스트 정산 (flow-through)

코스가 종료되면 코스당 정산 1건이 만들어진다. 호스트가 실제로 받는 금액은 유료 포인트로 들어온 매출만이고, 무료 보너스 포인트로 결제된 부분은 플랫폼 보조로 분리되어 호스트 지급 대상이 아니다. 외부 계좌이체 결제는 호스트가 직접 받은 돈이라 정산에 들어가지 않는다. 환불은 정산 이전에 끝나도록 설계되어 정산 후 새 환불이 생기지 않는다.

flow-through — 유료/무료를 분리해 호스트 수익에 잡는다

플랫폼 포인트는 두 종류 — 유료 포인트(사용자가 현금으로 충전)와 무료 보너스(이벤트로 받은 무료 적립). 결제 시 두 종류가 섞이고 환불도 비례 차감되는데, 정산에서는 유료 잔존액만 호스트 수익으로 잡는다. 무료 잔존액은 플랫폼이 보조한 부분이라 호스트에게 지급되지 않는다 — 분리하지 않으면 무료 포인트로 결제된 코스 매출까지 호스트 지급이 일어나 플랫폼 손실이 된다.

정산 금액 구성

항목의미
매출 (gross)유료 포인트 잔존액 — 결제에서 환불을 뺀 순액
플랫폼 보조 (별도)무료 보너스 잔존액 — 호스트 지급 대상 아님, 정산에 별도 표시
플랫폼 수수료매출의 10%
원천징수매출의 3.3%
실지급액 (net)매출 − 수수료 − 원천징수
계좌이체는 정산 대상 아님

계좌이체로 결제된 부분은 호스트가 외부에서 직접 받은 돈이라 플랫폼 매출이 아니다. 정산에는 한 푼도 포함되지 않으며 "이건 호스트가 직접 받은 거래"라고 분리되어 표기된다.

왜 "정산 후 환불 위험 0"인가

flowchart TD A["호스트: 종료 버튼"] --> B{"모든 회차 확정 완료?"} B -->|"아니오"| BX["거절 — 종료 통과 못 함"] B -->|"예"| C["종료 확정"] C --> D["정산 생성 — 별도 단계"] C --> E{"멤버 자가 취소?"} E -->|"종료 후 시도"| EX["차단 — 시스템이 거절"] D --> F["호스트 정산 보기"]
다층 방어

① 종료는 모든 회차 확정 후에만 통과. ② 정산은 종료 확정이 끝난 다음 별도 단계로 만들어진다. ③ 그 사이 멤버의 자가 취소는 차단된다. ④ 자격박탈 환불은 회차 확정 시점(=종료 이전)에만 발생하므로 정산 매출에 자연 반영된다. 그래서 정산이 만들어진 뒤에 새 환불이 발생할 가능성이 사실상 0이다.

정산 생성은 종료 확정 직후 자동으로 일어난다(호스트 수동 호출 없음). 만약 자동 생성 단계가 실패하면 안전망 스케줄러가 1시간마다 재시도한다 — 호스트는 정산이 만들어질 때까지 "준비중" 화면을 본다.

운영자가 알아야 할 것

멱등·재시도·잔액 게이트
  • 코스당 정산 1건 보장 — 같은 코스에서 두 번 정산이 만들어지지 않는다(시스템이 막음).
  • 참조 거래가 없으면 정산 생성을 중단하고 재시도 — 데이터 무결성이 깨진 상태에서는 정산을 만들지 않고 다음 안전망 주기에서 재시도.
  • 유료 잔존액이 0 이하면 정산 행을 만들지 않는다 — 무료 포인트만으로 채워진 코스는 호스트 지급 대상이 아님.
  • 승인·지급은 일반 모임 정산 파이프라인을 그대로 재사용 — 미정리 환불(잔여 환불 책임)이 남으면 지급이 보류된다.
  • 호스트가 종료 직후 정산을 보면 "준비중" 메시지를 볼 수 있다 — 자동 생성 성공 후에는 즉시 보임. 실패 시 최대 1시간 대기.

레드 플래그

분쟁이 일어날 수 있는 케이스
  • 안전망 재시도까지 호스트가 정산을 못 보는 시간 — 자동 생성이 실패하면 최대 1시간 대기. UI 안내 필수.
  • 무료 포인트만으로 결제된 코스 — 호스트가 "왜 정산이 0원이지?" 라고 묻는 케이스. 무료 보너스는 호스트 수익 아님을 사전 안내 필요.
  • 호스트 정산 이의제기 흐름 부재 — 1차 출시는 일반 모임 정산의 이의제기 흐름을 재사용. 정기모임 전용 분쟁 화면은 후속.
  • 코스가 무료/유료 섞여 끝났을 때 보고 라벨 — 정산에 "유료 매출 / 플랫폼 보조 / 수수료 / 원천징수 / 실지급액"이 모두 나오므로 분해 라벨이 명확해야 함.
