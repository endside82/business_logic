F01-07. 관심사 태그 관리

추천 알고리즘에 들어가는 사용자 선호 신호를 만들고 유지한다. 온보딩에서 처음 입력되지만 이후 프로필·설정 화면에서 추가·이름 변경·가중치 조정·삭제가 모두 가능하다. 본 단위는 등록 진입점에 화면이 있고, 조회·수정·삭제는 외부 단위에서 호출하는 리소스 단위의 기능이다.

이 기능이 다루는 범위

한 사용자의 관심사 태그 목록(태그명·가중치)을 CRUD 단위로 관리한다. 본 단위에서 화면이 있는 부분은 온보딩 단계 2(F01-06 안의 일괄 등록)만이고, 그 외 조회/수정/삭제는 프로필·설정 영역의 외부 화면에서 호출한다. 추천 알고리즘 자체는 별도 단위로, 본 단위는 입력값만 책임진다.

이 기능 안의 하위 항목

하위 동작 1

태그 등록

한 태그씩 등록. 중복(같은 이름) 검사 + 사용자당 한도(20개) 검사. 온보딩에서 N개를 묶어 등록할 때도 서버 호출은 N회 분할 호출이라 부분 저장 위험이 있다. 멱등 처리는 클라이언트 책임.

409 중복400 한도 초과
하위 동작 2

본인 태그 목록 조회

가중치 내림차순으로 정렬되어 반환. 가장 관심도 높은 태그가 위쪽에 노출되도록 외부 단위 화면이 그대로 사용한다.

가중치 정렬
하위 동작 3

태그 수정 (이름 또는 가중치)

이름 변경 시 중복 검사. 이름이 같고 가중치만 바뀌면 중복 검사를 생략. 가중치 범위는 0.0~5.0이며 서버에서 강제.

422 범위
하위 동작 4

태그 삭제

소유권 검증(본인 태그만 삭제 가능). 다른 사용자의 태그 ID로 시도하면 404로 응답해 태그 존재 여부 자체를 노출하지 않는다.

사용자 시나리오 10개

S1 · Happy Path

온보딩에서 3개 태그 일괄 등록

  1. 관심사 단계에서 3개 선택 → “다음 (3개 선택)” 탭
  2. 3회 분할 호출 모두 성공 → 가중치 기본 1.0으로 저장
  3. 온보딩 완료 플래그 갱신 후 다음 단계로 이동

종료 — 3개 행 INSERT.

S2 · 부분 저장 회귀

중간 실패 후 재시도 시 중복 에러

  1. S1의 두 번째 호출에서 네트워크 끊김
  2. 첫 번째 태그만 저장된 상태
  3. 사용자가 다시 “다음” → 첫 태그 재등록 시도 → 409 DUPLICATE
  4. 토스트로 사용자가 진행 불가

종료 — 부분 저장 상태에서 막힘. 알려진 한계: 클라이언트에서 409를 “이미 등록됨”으로 무시하고 다음 태그로 진행하는 멱등 처리를 추가할 후보.

S3 · 한도 초과

21번째 태그 등록 시도

  1. 이미 20개 보유 사용자
  2. 외부 단위 화면에서 추가 시도
  3. 서버: 400 MAX_PREFERENCE_TAGS_EXCEEDED

종료 — 추가 차단.

S4 · 중복

이미 가진 태그를 다시 등록

  1. 같은 이름의 태그를 다시 등록 요청
  2. 서버: 409 DUPLICATE_PREFERENCE_TAG

종료 — 변경 없음.

S5 · 조회

본인 태그 목록 조회

  1. 프로필/설정 화면 진입 → 목록 GET
  2. 가중치 내림차순으로 정렬되어 응답
  3. 관심도 높은 태그가 위쪽에 노출

종료 — 조회 성공.

S6 · 이름 변경

‘러닝’을 ‘마라톤’으로 변경

  1. 외부 단위에서 PUT 호출
  2. 서버: 소유권 확인 → 이름 다름 → 새 이름 중복 검사 → 통과 → UPDATE

종료 — 이름 변경.

S7 · 보안

다른 사용자의 태그 수정 시도

  1. A 사용자가 B 사용자의 태그 ID로 PUT 시도
  2. 서버: 본인 소유 태그 조회에서 미발견 → 404 PREFERENCE_TAG_NOT_FOUND

종료 — 차단. 의도: 403이 아닌 404로 응답해 태그 존재 여부 자체를 숨긴다.

S8 · 가중치

이름은 그대로, 가중치만 변경

  1. 같은 이름 + 가중치 3.5로 PUT
  2. 서버: 이름 동일 인식 → 중복 검사 생략 → 가중치만 UPDATE

종료 — 다음 조회 시 정렬 위치가 변경됨.

S9 · 범위 위반

가중치 6.0으로 시도

  1. PUT 요청에 weight=6.0
  2. 서버: 범위 검증 실패 → 422 VALIDATION_ERROR

종료 — 변경 거부.

S10 · 삭제

태그 삭제

  1. DELETE 요청
  2. 서버: 소유권 확인 → DELETE → 204

종료 — 태그 제거, 다음 추천부터 신호 변동.

서버 계약 한눈에

MethodPath인증핵심 동작
POST/api/v1/users/me/preference-tags로그인태그 1개 등록. 가중치 기본 1.0.
GET/api/v1/users/me/preference-tags로그인본인 태그 목록. 가중치 내림차순.
PUT/api/v1/users/me/preference-tags/{tagId}로그인이름/가중치 수정. 소유권 + 중복 + 범위 검증.
DELETE/api/v1/users/me/preference-tags/{tagId}로그인삭제. 본인 태그만.
에러 코드 매트릭스
  • 409 DUPLICATE_PREFERENCE_TAG — 같은 이름 태그 보유 중
  • 400 MAX_PREFERENCE_TAGS_EXCEEDED — 한도 20개 초과
  • 404 PREFERENCE_TAG_NOT_FOUND — 미보유 또는 다른 사용자 태그
  • 422 VALIDATION_ERROR — 가중치 범위(0.0~5.0) 위반

자주 빠뜨리는 케이스

레드 플래그
  • 온보딩 일괄 등록의 비멱등 회귀 — 부분 저장 후 재시도 시 사용자가 갇히는 알려진 한계.
  • 이름 동일 가중치만 변경할 때도 중복 검사 호출 — 본인이 본인 태그를 “중복”으로 자기검출하는 버그.
  • 다른 사용자 태그 시도 시 403 응답 — 존재 여부 노출. 404로 통일.
  • 가중치 클라이언트 측 범위 미검증 — 422가 사용자 화면 끝에서 노출되어 UX 저하. UI에서 슬라이더 범위로 사전 차단.
  • 한도 20개 클라이언트 미강제 — 21번째에 막혀 “선택했는데 안 됨” 경험. UI 칩 그리드에서 사전 비활성화.
