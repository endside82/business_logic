F18-05. 호스트 운영 인박스

호스트가 처리해야 할 이벤트 문의, 참가 신청, 정산 이의, 공개 신고, 결제·환불 이슈, 운영 이슈, 분쟁 케이스를 한 화면에서 확인하고 처리하는 운영 대시보드. 8가지 source를 NEEDS_RESPONSE·IN_PROGRESS·DONE 3탭으로 분류하고, 미응답 이벤트 문의에는 건수 배지가 표시된다. 분쟁 케이스 카드는 케이스 상세 화면으로 바로 연결된다.

설계 의도 — 호스트가 "오늘 내가 뭘 처리해야 하는지" 한 눈에

이전에는 이벤트 문의는 이벤트 화면, 참가 신청은 신청 관리, 정산 이의는 정산 화면으로 흩어져 있었다. 운영 인박스는 이 모든 항목을 하나의 처리 큐로 모아 호스트가 우선순위를 파악하고 놓치는 것 없이 대응하게 한다. 특히 이벤트 문의 미응답 건수 배지가 "답장 안 한 게 있다"는 것을 즉각 알려준다.

8가지 source와 처리 흐름

사용자 시나리오

기본 흐름

인박스 열면 오늘 할 일이 한눈에

호스트가 인박스에 진입하면 "응답 필요" 탭에 미처리 항목이 모인다. 상단 헤더에는 총 응답 필요 건수와 응답률, source별 카운트가 보인다. EVENT_MESSAGE 카드에 "미응답 3"이 표시되어 있으면 답장이 필요한 스레드가 3개다.

이 흐름이 남기는 것 — 호스트가 여러 화면을 돌아다니지 않아도 처리해야 할 것이 얼마나 남았는지 즉시 인지한다.

분쟁 케이스 처리

인박스에서 분쟁 케이스 상세로 바로 이동

DISPUTE_CASE 카드에는 케이스 id가 연결되어 있다. 카드를 탭하면 호스트 분쟁 케이스 상세 화면(/host/disputes/{caseId})으로 바로 이동한다. 상세 화면에서 타임라인·증빙을 확인하고 호스트 액션 패널로 처리한다.

이벤트 문의

미응답 문의가 있다는 알림

EVENT_MESSAGE 카드의 "미응답 N" 배지는 스레드 단위로 집계된다. 배지를 보고 해당 스레드로 이동해 답장을 보내면 응답 필요 탭에서 빠진다. 실시간 업데이트가 없으므로 최신 상태는 pull-to-refresh로 갱신한다. 화면 재진입 시 자동 갱신되지 않는다.

통계 헤더

응답률과 평균 응답 시간 확인

인박스 상단에는 전체 응답률(%)과 평균 응답 시간(분), source별 항목 수가 표시된다. "이번 주 응답률 82%" 같은 형태로 호스트 자신의 운영 품질을 인지하게 한다.

탭 분류

처리 단계별로 나누어 보기

"응답 필요" 탭은 지금 당장 처리해야 할 항목, "처리 중" 탭은 진행 중, "완료" 탭은 마무리된 항목이다. 탭은 NEEDS_RESPONSE/IN_PROGRESS/DONE 3개뿐이며 source type 필터는 없다. pull-to-refresh로 최신 데이터를 재조회한다.

운영자가 알아야 할 것

운영 주의사항
  • DISPUTE_CASE 카드의 disputeCaseId가 null이면 fallback 동작이다. 정상 케이스라면 disputeCaseId가 있어야 한다. null이면 item.id로 대신 이동하는데, 이 경우 상세 화면으로 정확히 이동하지 않을 수 있다.
  • 실시간 업데이트가 없다. 새 문의가 도착해도 화면이 자동으로 갱신되지 않는다. 화면을 벗어났다 돌아와야 최신 목록이 나타난다. 긴급 대응이 필요한 상황에서 호스트가 놓칠 수 있다.
  • 대량 항목 시 성능 저하 가능성. stats 호출 시 size=500 파라미터로 수집을 시작하지만, 개별 finder는 자체 제한(호스트 이벤트 200건, 분쟁 케이스 100건, APPLICATION·REPORT·정산 이의·FailedRefund는 전량 로드)을 따른다. 수집 후 메모리 정렬이 적용되며 최종 페이지 크기는 최대 100건으로 캡된다. 활성 이벤트가 많은 호스트는 인박스 로딩이 느려질 수 있다.
  • dispute 딥링크 미배선. FCM 알림에서 분쟁 관련 신규 타입을 클릭하면 인박스 화면이 아닌 홈으로 이동한다. REFUND_DISPUTE 기존 3개 타입만 배선되어 있다.

레드 플래그

현재 알려진 미구현·주의 사항
  • dispute 딥링크 미배선. 분쟁 관련 FCM 알림 클릭 시 인박스 또는 케이스 상세로 이동하지 않는다. 알림을 받아도 어디서 확인해야 할지 찾아야 한다.
  • 실시간 업데이트 없음. WebSocket이나 polling이 없어 새 항목을 즉각 인지할 방법이 없다. 빠른 응답이 필요한 문의 처리에 지장을 줄 수 있다.
  • 성능 위험. HostInboxService stats가 size=500으로 수집을 시작하지만, source별 finder 자체 제한(호스트 이벤트 200건, 분쟁 100건 등)을 거친 후 메모리 정렬·페이징 적용, 최종 출력은 최대 100건이다. 그럼에도 이벤트가 많은 호스트는 응답 지연과 메모리 부하가 발생할 수 있다.
  • 노쇼 환불 화면 직접 진입 경로 없음. 노쇼 환불 화면(/host/disputes/:caseId/no-show-refund/:applicationId)은 호스트 분쟁 상세에서만 들어갈 수 있다. 인박스에서 직접 진입하는 경로가 없다.
