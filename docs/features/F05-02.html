F05-02. 자동완성 서제스트

사용자가 검색창에 두 글자 이상 입력하면 비슷한 카테고리 라벨과 곧 시작하는 이벤트 제목이 짧은 드롭다운으로 떠오른다. 사용자가 단어를 끝까지 치지 않아도, 한 단어가 무엇과 닿아 있는지 보면서 선택할 수 있다. 입력 한 글자마다 호출하지는 않고 짧은 대기 후 한 번만 보낸다.

설계 의도 — 검색을 "단어 끝까지 쳐야 하는 일"로 만들지 않는다

사용자가 떠올리는 단어는 모호한 경우가 많다. "등" 한 글자만 입력했을 때 서비스가 "등산"을 카테고리로, "북한산 등산 모임" 같은 이벤트 제목을 함께 보여 주면 사용자는 자기가 찾는 것이 카테고리 둘러보기인지 특정 이벤트인지 자연스럽게 분기한다. 입력 한 글자마다 매번 호출하면 사용자도 서버도 지치므로, 두 글자 이상 + 짧은 대기 후 한 번만 보낸다 — "빨리 치는 사용자"의 모든 키 입력이 한 번의 호출로 묶이는 것이 이 단위의 핵심이다.

사용자 시나리오

기본 흐름

"등"만 입력했더니 후보가 떠오른다

사용자가 "등산"이라고 치려다 "등"을 입력한 순간엔 아무것도 뜨지 않는다. 두 번째 글자를 입력해 "등산"이 되는 순간, 잠깐 멈추면 "등산"(카테고리) 그리고 "북한산 등산 모임" 같은 실제 이벤트 제목이 한 줄씩 떠오른다. 사용자가 "북한산 등산 모임"을 누르면 그 단어로 검색이 실행된다.

이 흐름이 남기는 것 — 한 단어가 카테고리·실제 이벤트 어느 쪽과 닿아 있는지 보여 주어 사용자가 검색 의도를 자연스럽게 좁힌다.

빠른 타이핑

빠르게 친 네 글자가 한 번에 묶인다

"독서모임"을 한 번에 빠르게 친다. 사용자가 손을 멈춘 짧은 순간이 지나야 서버 호출이 한 번 떠난다. 중간 글자마다 호출이 가지 않으므로 사용자도 서버도 부담이 없다.

결과 없음

어떤 후보도 떠오르지 않을 때

"asdfqwer" 같은 무의미한 단어를 입력하면 드롭다운 자체가 나타나지 않는다. 사용자에게는 평소의 입력 전 화면이 그대로 남아, "추천이 없다"는 별도 안내로 시야가 흐트러지지 않는다.

이 흐름이 남기는 것 — 후보가 없는 상태가 "오류"로 보이지 않게 한다. 사용자는 그대로 엔터를 눌러 일반 검색으로 갈 수 있다.

입력 취소

입력을 지우면 후보도 같이 사라진다

드롭다운이 떠 있는 상태에서 X 버튼을 누르면 입력값과 함께 후보 목록도 즉시 사라진다. 화면은 입력 전 상태(최근 검색어 + 트렌딩 키워드)로 자연스럽게 돌아간다.

키워드 빠르게 바꿈

"등산"에서 "요리"로 갈아탔다

"등산"을 친 직후 결과를 보기 전에 마음을 바꿔 "요리"로 다시 입력한다. 늦게 도착하는 "등산"의 응답이 화면을 잠시라도 덮어쓰지 않게, 가장 최근 입력의 결과만 사용자에게 보인다.

이 흐름이 남기는 것 — 빠르게 마음을 바꾸는 사용자에게 "방금 친 단어"의 결과만 보여 주는 것이 검색 신뢰의 기본. 이 보호 장치가 풀리면 "내가 친 게 아닌 결과"가 잠깐 떠올라 사용자가 혼란을 겪는다.

운영자가 알아야 할 것

자동완성의 정책 결정점
  • 최소 두 글자. 한 글자에서는 후보의 의미가 너무 흐려져, 사용자에게도 서버에도 부담만 늘어난다. 두 글자 이상에서만 호출이 떠난다.
  • 이벤트 제목 후보는 미래 이벤트만 대상으로 한다 — 이미 끝난 이벤트가 자동완성에 떠오르면 사용자가 진입한 뒤에야 종료 사실을 알게 되어 검색의 신뢰가 떨어진다.
  • 자동완성은 검색 기록에 남기지 않는다. 후보를 둘러보는 행위 자체는 사용자가 의도한 검색이 아니다. 사용자가 후보를 눌러 검색을 실행하는 순간(F05-01)에 비로소 기록된다.
  • UI 스펙은 8건, 서버 기본값은 10건으로 노출 한도가 어긋나 있다. 현재 클라이언트가 8건으로 강제하므로 사용자에게는 항상 8건까지만 보이지만, 한쪽의 기준을 맞추는 정리 작업이 필요하다.
