F17-08. 환불 (비례 산식 · 세 가지 트리거)

고정형 환불은 이미 진행/제공된 회차만큼만 빼고 나머지를 돌려준다(비례 환불). 세 가지 트리거 — 멤버 본인 취소, 호스트 모임 취소, 노쇼 자격박탈 — 가 모두 같은 골격을 쓰되 "차감 기준"이 조금씩 다르다. 100원 단위로 절삭되며, 음수가 되면 0원으로 클램프된다.

설계 의도 — "내가 못 받은 만큼만 돌려받는다"

코스 단위로 결제했더라도, 환불은 회차 단위 공정성이 필요하다. 시작된 회차는 회사가 이미 자원을 썼다고 보고 환불 대상에서 빼고, 미래 미제공 회차만 비례로 돌려준다. 호스트가 모임을 닫는 경우만 예외 — 호스트가 미제공한 회차는 100% 돌려준다(노쇼한 멤버 책임 아님).

세 가지 트리거 — 같은 골격, 다른 차감 기준

트리거발동차감 기준 (회차)
멤버 본인 취소멤버가 모집 중 모임 취소 버튼이미 시작된 회차 (노쇼 포함)
호스트 모임 취소호스트가 모임 자체 취소이미 종료(제공)된 회차만
자격박탈노쇼 한도 도달로 자동 박탈이미 시작된 회차 (= 멤버 취소와 동일)
예시 — 8회 코스, 80,000원 결제

3회까지 진행된 시점에 환불이 발생한다면:
멤버 취소 → 5회 미제공 → 50,000원 환불.
호스트 취소 → 호스트가 5회 미제공 → 50,000원 환불 (동일).
자격박탈 → 멤버가 5회를 더 못 받게 됨 → 50,000원 환불.

결제 방식에 따른 환불 경로

지갑 결제 환불

즉시 자동 환불

플랫폼 포인트로 자동 복원. 거래 이력에 환불이 기록되고 회계도 자동 분개된다.

계좌이체 환불

호스트 직접 송금 → 마감

플랫폼 외부 거래라 호스트가 직접 사용자 계좌로 송금한 뒤 "환불 완료" 버튼을 눌러야 마감된다. 그 사이 멤버 입장에서는 "환불 진행중" 라벨.

환불 실패

자동 재시도 큐

지갑 처리 실패 등 일시적 문제는 실패 큐로 들어가 자동 재시도된다. 호스트 취소처럼 다수 환불이 한꺼번에 발사될 때 일부 실패해도 모임 취소 자체는 완료된다.

자격박탈 환불

코스 종료 이전에 발생

노쇼로 박탈되는 시점은 회차 확정 단계 — 즉 코스 종료 이전이다. 그래서 환불 금액이 정산 금액에 자연 반영되어, 정산 후 환불 같은 위험이 없다.

환불 트리거 자체는 멤버 취소(F17-06) / 호스트 취소(F17-04) / 자격박탈(F17-09) 흐름 안에 묶여 있어 별도 API가 없다. 위 endpoint는 계좌이체 환불 마감 전용.

운영자가 알아야 할 것

금액 산정 규칙
  • 100원 단위 절삭 — 원 단위 금액은 자동으로 100원 미만이 깎인다(일반 결제와 동일).
  • 0 클램프 — 차감이 결제 금액을 넘어가면 0원으로 클램프(음수 환불 금지).
  • 회차 시작 시각 기준 — 멤버 취소/박탈 산식에서 "시작된 회차"는 시작 시각이 지난 회차(노쇼 포함).
  • 호스트 미제공 회차는 무조건 환불 — 호스트가 닫은 회차는 멤버 귀책과 무관하게 100% 환불.
  • 자동 재시도 큐로 환불 실패 추적 — 모든 실패는 별도 큐로 들어가 운영팀이 모니터링.

레드 플래그

분쟁이 일어날 수 있는 케이스
  • 계좌이체 환불 마감 누락 — 호스트가 송금만 하고 마감 버튼 안 누르면 멤버 입장에서 영구 "환불 진행중". 호스트 알림 미배선이라 잊기 쉬움.
  • 지갑 환불로 음수 잔액 발생 — 사용자가 환불액보다 적은 잔액을 가진 경우 마이너스 잔액이 일시 발생할 수 있음(다음 적립 시 우선 차감).
  • "24시간 전 100% 환불" 같은 시간 비례 정책 미적용 — 1차 출시는 회차 단위 차감만. 시간 비례 환불은 후속.
  • 출석 인정 결과(공결·세션 취소)의 환불 정책 — 1차는 멤버 취소 산식과 동일하게 처리. 약관 검토 후 별도 정책 가능.
