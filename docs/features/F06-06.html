F06-06. 포인트 결제 · 환불

이벤트 참여비·티켓·구독 등 모든 포인트 결제의 공통 경로. 유료/무료 잔액 우선순위 차감, 충전 lot의 FIFO 소비, 회계 분개, 잔액 부족 시 자동충전 분기, 정책별 환불 비율, 이중 결제 가드까지를 한 트랜잭션 안에서 묶는다. 이 단위가 깨지면 잔액·정산·회계가 동시에 어긋난다.

핵심 규칙 세 가지
  1. 가격은 항상 서버 사실 — 클라이언트가 보낸 금액은 검증용. 이벤트의 실제 가격과 다르면 즉시 거절.
  2. 중복 결제 가드를 가장 먼저 — 같은 사용자×같은 참조×같은 referenceId면 wallet lock 이전에 차단.
  3. 잔액 → 자동충전 → 재시도 순서 고정 — 잔액이 부족하면 자동충전을 시도한 후에만 재시도. 자동충전이 꺼져 있으면 즉시 부족액 정보 반환.

이 기능 안의 하위 항목

동작 1

결제 (Wallet.pay)

중복 가드 → 가격 검증 → wallet row lock → 유료/무료 우선순위 차감 → 거래·결제 기록 → FIFO lot 소비 → 회계 분개 → 알림. 한 트랜잭션 안에서 모든 부수효과를 묶는다.

한 트랜잭션
동작 2

잔액 부족 분기

자동충전 ON이면 부족분을 충전(최소 충전 금액과 부족액 중 큰 값)한 뒤 결제 재시도. 자동충전 OFF면 부족 페이로드(필요금액·현재잔액·부족액)를 그대로 반환해 클라이언트가 충전 화면을 prefill로 띄울 수 있게 한다.

PaymentShortfall
동작 3

FIFO lot 소비

충전 시점의 lot(만료일·결제수단·PG 키 포함)을 FIFO로 차감. 회계 분개에 lot 기준 정보가 동행해 나중에 환불할 때 정확한 lot로 되돌리거나 PG 측 환불을 매칭할 수 있게 한다.

동작 4

환불 (정책 + 비율)

이벤트 시작 시각과 환불 정책(STANDARD/STRICT)에 따라 100%/50%/30%/0% 비율 결정. 사용자 잔액은 즉시 복구, PG 카드 환불은 비동기 워커가 별도 처리.

정책 머신
동작 5

실패 처리 + failed_refund

PG 환불 실패 시 failed_refund 큐로 적재. 운영자 수동 대사 + 재시도 스케줄러. 사용자 잔액은 영향 없이 보장.

사용자 시나리오 10개

S1 · Happy Path

이벤트 참여비 결제

잔액 50,000P, 8,000P 결제.

  1. “참여하기” 탭 → 확인 모달(“잔액 50,000 → 42,000”)
  2. 중복 가드 통과 → 가격 검증 통과 → wallet 락
  3. 유료/무료 우선순위로 8,000 차감 → 잔액 42,000
  4. 결제 거래(EVENT_PAYMENT) + 결제 기록(COMPLETED) INSERT
  5. FIFO lot 소비 → consumption 매핑 → 회계 분개(사용자 지갑 → 크리에이터/플랫폼)
  6. 호스트에게 PAYMENT_COMPLETED 푸시 + 클라이언트 지갑 invalidate

종료: 호스트는 정산 대상의 grossAmount에 누적될 예정.

S2 · 자동충전

잔액 부족 + 자동충전 ON

잔액 4,500P, 8,000원 결제, 자동충전(임계 5,000/충전 30,000) 활성.

  1. POST /pay → 잔액 부족 → tryAutoCharge 호출
  2. 부족 3,500. chargeAmount = max(30,000, 3,500) = 30,000
  3. dev 환경은 시뮬레이션 PG 키(SIM_AUTO_…) 즉시 부여, prod는 실제 PG billingPayment
  4. +30,000 충전 + 충전 거래·결제 기록·회계 분개 INSERT
  5. tryAutoCharge 성공 → 본 결제 재시도 → 8,000 차감 성공
  6. 거래 내역에 충전 +30,000P + 결제 −8,000P 두 행, 최종 잔액 26,500P
S3 · prefill 분기

잔액 부족 + 자동충전 OFF

  1. POST /pay → 자동충전 설정 없음 → tryAutoCharge가 즉시 false
  2. 서버: PaymentFailureVo(필요 8000, 현재 4500, 부족 3500, 메시지)와 함께 400 INSUFFICIENT_BALANCE
  3. 클라이언트: ErrorInterceptor가 PaymentShortfall로 변환 → 충전 화면을 부족액 prefill로 push
  4. 사용자가 5,000원 충전 후 다시 결제 → 성공

설계 의도: 자동충전을 끈 사용자는 “부족액만큼만” 충전하도록 안내. 풀 가격 prefill은 의도 위배.

S4 · 환불

24시간 전 100% 환불 (STANDARD)

  1. “참여 취소” → 환불 정책 모달(100%)
  2. POST /wallet/refund → RefundPolicyService가 정책·잔여시간으로 비율 결정
  3. RefundLotAllocator가 lot 단위 환불 분배(무료/유료 비율 보존)
  4. 무료 부분은 즉시 wallet 복구, 유료 부분도 즉시 wallet 복구 + RefundRequest 큐(현재 dormant)
  5. 환불 거래(EVENT_REFUND) INSERT + 회계 역분개 + REFUND_COMPLETED 푸시

주의: RefundRequest 워커는 현재 비활성(PG_SKIPPED). v3에서 활성 예정.

S5 · 환불

12-24h 전 50% 환불

  1. 정책 모달: “50% 환불”
  2. POST /refund → 50% 비율 적용 → wallet 50%만 복구
  3. 나머지 50%는 호스트 정산에 영향 없이 보존
S6 · 환불 거절

12h 미만 환불 시도

  1. POST /refund → RefundPolicyService.allowed=false → 400 REFUND_NOT_ALLOWED
  2. 토스트로 정책 안내, 사용자 흐름 유지
S7 · 중복 차단

이중 결제 시도

  1. 결제 직후 푸시 지연으로 사용자가 다시 탭
  2. 중복 가드(사용자×참조×referenceId)가 wallet lock 이전에 차단
  3. 409 DUPLICATE_PAYMENT → 클라는 ApiError.conflict로 매핑 → 화면 재조회로 정확한 상태 정렬

핵심: 가드가 lock 이전에 실행되어 잔액·거래 행 모두 무변동을 보장.

S8 · 위변조 차단

가격 조작 시도(amount=1)

  1. 비정상 클라이언트가 8,000원 이벤트를 amount=1로 결제 시도
  2. 서버: event.price와 amount 불일치 → 400 INVALID_PAYMENT_AMOUNT
S9 · PG 실패

PG 측 카드 환불 실패

  1. wallet은 환불 시점 즉시 복구
  2. 비동기 워커가 PG cancelPayment 호출 → 실패 → failed_refund 큐 적재
  3. 거래 상세에 “관리자 확인 중” 표기
  4. 운영자 수동 대사 + RefundLedgerFailureRetryScheduler 재시도

설계 의도: 사용자 경험과 PG 후처리를 분리해, 사용자에게는 실패가 전파되지 않게.

S10 · 유료 승인제

승인 후 결제 확정 (정책 목표)

  1. 호스트 승인으로 APPROVED_PENDING_PAYMENT 상태
  2. 사용자가 “결제하고 참석 확정” → POST /pay
  3. 서버: 신청 상태·결제 기한·중복 결제 검증 → 결제 성공 후에만 ATTENDING + currentCapacity++
  4. 실패: PAYMENT_WINDOW_EXPIRED / PAYMENT_NOT_ALLOWED / PaymentShortfall / DUPLICATE_PAYMENT 분기

현재 구현 갭: 결제 서비스가 Application 상태를 조회하지 않음. 결제 후처리 경로 미구현. 별도 라운드로 마이그레이션 예정.

자주 빠뜨리는 케이스

레드 플래그
  • 중복 가드를 wallet lock 이후로 옮기는 회귀 — 잔액이 차감되었다 롤백되면서 잔액 잠금이 길어진다. 가드는 트랜잭션의 첫 단계.
  • 가격 검증을 클라이언트만 — 위변조 즉시 통과. 서버에서 event.price 기준으로 강제.
  • PaymentShortfall 페이로드 누락 — 클라이언트가 부족액을 모르고 풀 가격 prefill 하면 사용자가 과충전.
  • FIFO lot 소비를 안 함 — 환불 시점에 어떤 lot을 되돌릴지 결정 불가, PG 환불 매칭 깨짐.
  • 회계 분개 빠뜨림 — 호스트 정산·플랫폼 수수료 집계가 어긋난다.
  • 호스트가 자기 이벤트에 결제 가능 — 현재는 신청 단계에서 막히지만 결제 자체에는 가드 없음. co-host도 마찬가지. 정책 강화 시 추가.
  • PG 환불 워커 활성화 전에 PG_SKIPPED 처리 누락 — 모든 RefundRequestItem이 PG_SKIPPED로 생성되어야 이중 환불 회피.
