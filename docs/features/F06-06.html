F06-06. 포인트 결제·환불

이벤트 참가비, 호스팅 티켓, 멤버십 구독 — 서비스 안의 모든 유료 행위가 보유 포인트에서 차감되는 공통 동작. 본 단위는 어느 화면에도 자체로 떠올리지 않는다. 결제는 이벤트·티켓·구독 화면이 띄우는 결제 확인 안에서 일어나고, 환불은 모임 취소·신청 취소 같은 동작에서 시작된다. 결과는 거래 내역(F06-03)과 지갑 메인(F06-01)에 즉시 보인다. 환불은 시작된 시점이 모임 시작 몇 시간 전인지에 따라 100%, 50%, 0%로 정해지는 명확한 정책을 따른다. 이 정책이 어긋나면 사용자에게는 "환불이 안 된다", "왜 일부만 돌려받지" 같은 신뢰 손상으로 즉시 나타난다.

설계 의도 — 결제는 잔액만 다루고, 환불은 명문화된 비율만 따른다

결제는 외부 카드 결제와 분리되어 잔액 차감만 다룬다 — 카드사 위험이 본 단위에 들어오지 않는다. 환불은 사용자가 모임 시작 몇 시간 전에 취소했는지에 따라 100·50·0% 또는 100·30·0% 비율이 자동으로 정해진다 — 운영자가 매 환불마다 개별 판단하지 않게 둔 명문화된 정책. 포인트는 사용자 잔액에 즉시 돌아오지만, 그 결제가 외부 카드로 일어났던 부분은 영업일 기준 며칠이 더 걸린다 — 이 두 시점을 거래 상세에서 명확히 분리해 사용자에게 보여 준다. 결제·환불의 일관성이 흔들리면 데이팅의 매칭이나 클럽 가입 같은 다른 핵심 가치보다 빠르게 사용자가 떠난다 — 돈 문제는 한 번의 사고가 회복하기 가장 어려운 영역.

이 단위가 신뢰의 핵심인 이유

결제와 환불에 한 번이라도 사고가 나면 사용자는 거의 돌아오지 않는다
  • 한 모임에 두 번 결제되는 사고가 없어야 한다. 결제 직후 푸시 지연이나 네트워크 재시도로 사용자가 같은 결제를 두 번 시도할 수 있다. 두 번째 시도는 그 자리에서 거절되어야 하고, 사용자 잔액은 한 번만 차감되어야 한다.
  • 가격 조작 시도가 통과되어서는 안 된다. 사용자가 입력한 금액과 시스템이 알고 있는 모임 가격이 다르면 결제 자체가 거절된다. 사용자 화면에서 가격을 임의로 바꿔 결제가 통과되는 일이 일어나면 정산·회계가 동시에 어긋난다.
  • 환불 비율은 정책으로만 결정된다. "이번만 풀어 달라"는 사용자 요청을 매번 개별 처리하면 환불 정책이 흐려지고, 호스트가 받는 정산도 불안정해진다 — 명문화된 비율을 그대로 적용하는 것이 사용자 신뢰와 호스트 신뢰 양쪽을 동시에 보호한다.

환불 비율이 정해지는 흐름

사용자가 모임 취소를 누르면 시스템이 현재 시각과 모임 시작 시각을 비교해 자동으로 비율을 정한다. 본 흐름은 운영자의 개별 판단이 들어가지 않는다 — 명문화된 정책의 자동 적용.

사용자 시나리오

기본 흐름

이벤트 참가비를 잔액으로 결제

잔액 50,000원을 가진 사용자가 8,000원 모임에 "참여하기"를 누른다. 결제 확인 안내가 떠오르고 "8,000원 결제 / 잔액 50,000 → 42,000"이 미리 보인다. "결제 확인"을 누르면 잔액이 차감되고 신청이 확정된다.

이 흐름이 남기는 것 — 결제 한 번에 카드 입력이 한 번도 안 들어간다 — 충전(F06-02)이 외부 카드 결제를 격리해 둔 결과.

잔액 부족 + 자동 충전

자동 충전이 켜져 있어 막힘 없이 결제 완료

잔액 4,500원인 사용자가 8,000원 결제를 시도한다. 자동 충전이 켜져 있어 시스템이 30,000원을 카드에서 자동으로 충전하고, 그 자리에서 결제 8,000원을 차감한다. 사용자에게는 "결제 완료"가 한 번 보이고 거래 내역에는 자동 충전 한 줄과 결제 한 줄이 함께 남는다.

이 흐름이 남기는 것 — 결제 의도가 끊기지 않는다 — 자동 충전(F06-05) 가입의 직접 가치가 본 시점에 발현.

잔액 부족 + 자동 충전 없음

충전 화면이 자동으로 떠오름

자동 충전이 꺼진 사용자가 잔액 4,500원으로 8,000원 결제를 시도한다. 결제가 잔액 부족으로 거절되고 충전 화면이 자동으로 열린다 — 부족한 3,500원이 입력칸에 미리 채워진 상태로. 사용자가 충전을 마치고 다시 결제 흐름으로 돌아가 결제를 완료한다.

이 흐름이 남기는 것 — 잔액 부족 안내만 띄우고 끝내지 않는다 — 그 자리에 사고 싶은 사용자를 충전으로 연결.

환불 — 100%

모임 시작 36시간 전에 취소

표준 정책의 모임을 시작 36시간 전에 취소한다. 안내 모달이 "24시간 전 · 100% 환불" 정책을 보여 주고 사용자가 동의한다. 잔액이 즉시 복구되고 거래 내역에 환불 한 줄이 추가된다. 카드로 결제됐던 부분은 영업일 기준 별도로 진행되어 며칠 뒤 카드 명세서에 반영.

환불 — 50%

모임 시작 18시간 전에 취소

표준 정책의 모임을 시작 18시간 전에 취소한다. "12~24시간 · 50% 환불"이 안내되고 사용자가 동의한다. 결제한 금액의 절반이 잔액으로 돌아오고, 나머지 절반은 호스트 정산에 반영된다 — 호스트는 일부라도 보상을 받는다.

이 흐름이 남기는 것 — 시간 임박 취소가 호스트에게 끼치는 손실을 정책으로 분담. 호스트가 자기 모임을 안전하게 운영할 수 있다는 신뢰의 근거.

환불 — 거절

모임 1시간 전에 취소 시도

표준 정책의 모임이 시작 1시간 앞으로 다가왔다. 취소를 시도하면 "환불 가능 시간이 지났습니다" 안내가 떠오르고 환불은 진행되지 않는다. 사용자는 모임에 그대로 참여하거나, 노쇼로 둘 수 있다 — 노쇼라도 환불은 없다.

이 흐름이 남기는 것 — 환불 정책의 마지막 경계. 사용자가 "왜 정확히 12시간이냐"고 묻지 않게 사전에 명확한 정책으로 둔다.

이중 결제 차단

푸시 지연으로 두 번 결제를 시도

결제 직후 푸시가 늦게 도착해 사용자가 "결제됐나" 싶어 다시 한 번 결제 버튼을 누른다. 두 번째 시도는 "이미 결제하셨습니다"로 거절된다 — 잔액은 한 번만 차감되고, 화면이 자동으로 신청 완료 상태로 갱신된다.

이 흐름이 남기는 것 — 통신 지연·재시도가 사용자 잔액에 영향을 주지 않게 막는 가드. 한 번의 회귀가 매출·회계 양쪽을 동시에 흐트러뜨릴 수 있는 자리.

가격 조작 차단

가격을 바꿔 결제를 시도

비정상 동작으로 8,000원 모임을 1원으로 결제하려는 시도가 들어온다. 시스템이 자기 쪽에 등록된 모임 가격과 사용자가 보낸 금액을 비교해 자동으로 거절한다 — 모임 가격이 다르면 결제 자체가 통과되지 않는다.

이 흐름이 남기는 것 — 매출·정산·회계가 한 번에 어긋날 수 있는 사고를 사전에 차단.

유료 승인제

호스트 승인을 받은 뒤 결제로 참석 확정

호스트 승인이 필요한 유료 모임의 흐름: 신청 → 호스트 승인 → 결제 → 참석 확정. 결제 화면에는 "결제하고 참석 확정" 버튼이 있고, 결제 성공 시점에 비로소 자리가 확보된다 — 결제 전까지는 정원에 자리가 잡혀 있지 않다. 결제 기한이 지나면 결제 자체가 거절되고 신청은 자동 만료.

이 흐름이 남기는 것 — 호스트가 "결제 안 한 사람이 자리를 점유하는" 사고를 겪지 않게 한다. 단, 본 시점에 결제 기한·승인 상태가 시스템에 완전히 갖춰지지 않아 향후 보강이 필요한 영역.

화면에서 사용자가 인지하는 것

본 단위는 자체 화면이 없다 — 결제는 이벤트·티켓·구독 화면 안의 확인 모달에서, 환불은 모임 취소 동작 안의 정책 안내에서 일어난다.

영역사용자가 보는 것그 자리에 둔 이유
결제 확인 안내 이벤트/티켓/구독 이름, 결제 금액, 현재 잔액, 결제 후 잔액, "환불 규정 보기" 링크, 결제 확인/취소 두 버튼. 사용자가 결제 직전에 한 번 더 결과(결제 후 잔액)와 환불 규정을 확인하게 — 결제 후 후회가 가장 흔히 일어나는 시점에 둔 마지막 확인.
환불 규정 안내 "24시간 전 100% · 12시간 전 50% · 그 이후 환불 불가" 같은 표 형식. 모달 또는 시트. 환불을 결정하기 전에 사용자가 정책을 명확히 인지하게 — "왜 환불이 50%냐"고 묻기 전에 확인하게 두는 자리.
환불 완료 안내 "환불이 완료되었습니다 / 카드 환불은 영업일 기준 N일 소요됩니다" 같은 두 시점 안내. 포인트 즉시 복구와 외부 카드 환불의 시점 차이를 사용자에게 명확히 분리해 알려, "환불됐다는데 카드사에 안 보인다"는 신고를 줄인다.
잔액 부족 분기 자동 충전 ON이면 자동 충전 → 결제 한 번에 진행. 자동 충전 OFF면 충전 화면이 부족한 금액을 미리 채워 둔 채로 자동으로 떠오름. "사고 싶다"는 의도가 가장 강한 시점에 사용자가 변수 입력에 시간을 쓰지 않게 한다.
거래 상세의 환불 진행 포인트 환불(즉시 완료) + 외부 카드 환불 진행 상태 두 줄. "환불이 어디까지 갔는지" 사용자가 거래 내역만으로 추적할 수 있게 — 운영팀 문의를 줄이는 자족적 동선.

측정해야 하는 것

본 단위가 정상이라고 부르려면 이 다섯 가지 지표가 추적되어야 한다
  • 결제 시도 → 결제 성공률 — 잔액 부족 거절은 별도로 분리해 측정. 너무 낮으면 가격 조작 가드나 중복 결제 가드의 거짓 차단을 의심.
  • 잔액 부족 → 충전 후 결제 복귀율 — 자동 진입의 효과. 너무 낮으면 충전 화면 자동 진입이 끊기거나 안내가 사용자에게 명확하지 않다는 신호.
  • 환불 비율 분포 — 100% / 50% / 30% / 0% 각각의 발생 빈도. 거절 비율이 비정상적으로 높으면 사용자가 시간 임박에 취소하는 경향 — 모임 정보의 명확성을 의심.
  • 외부 카드 환불 처리 시간 — 사용자에게 "영업일 기준 N일"이라고 안내한 시간이 실제로 지켜지는지. 늘어지면 사용자 신고가 늘어난다.
  • 이중 결제 가드 발동 빈도 — 사용자가 한 번의 결제를 두 번 시도한 비율. 너무 높으면 결제 화면의 응답성·확인 안내가 사용자에게 명확하지 않다는 신호.

운영자가 알아야 할 것 — 비즈니스 리스크

결제·환불에서의 한 번의 사고가 사용자 신뢰의 회복을 가장 어렵게 만든다
  • 한 모임에 두 번 결제되는 사고는 절대 일어나면 안 된다. 중복 결제 가드가 풀리면 사용자 잔액과 플랫폼 매출·정산이 동시에 어긋난다 — 사후 보정은 가능하지만, 사용자 신뢰는 그 시점에 무너진다.
  • 환불 비율을 운영자가 개별로 조정하지 않는다. "이번만 풀어 달라"는 요청에 응하는 환불 처리가 늘어나면 환불 정책 자체가 약해지고, 호스트가 받는 정산 신뢰도가 떨어진다 — 명문화된 비율을 그대로 적용하는 것이 사용자·호스트 양쪽 신뢰의 근거.
  • 가격 조작 차단이 풀리면 매출·정산·회계가 한 번에 어긋난다. 시스템이 알고 있는 모임 가격과 사용자가 보낸 금액이 다르면 결제가 거절되어야 한다 — 이 가드는 어떤 경우에도 풀려서는 안 된다.
  • 외부 카드 환불은 시스템 측에서 별도 절차로 진행된다. 포인트 잔액은 즉시 복구되지만 카드 환불은 영업일 기준 진행이라, 그 사이에 카드사 측 환불이 실패하면 시스템 내부 잔액과 외부 결제 사이에 불일치가 잠시 생긴다 — 이 불일치는 운영팀이 수동으로 정합을 맞추는 흐름이 별도로 존재한다.
  • 호스트와 공동 호스트가 자기 모임에 결제하는 흐름은 현재 차단되지 않는다. 본인이 호스트인 모임에 본인이 결제를 시도하면 신청 단계에서 차단되지만, 공동 호스트는 신청·결제 모두 허용된다 — 이 정책은 알면서 둔 현재 상태이며, 향후 결정에 따라 차단으로 바뀔 수 있다.
  • 유료 승인제 모임의 결제 흐름은 현재 일부 보강이 필요하다. "호스트 승인 → 결제 → 참석 확정" 순서가 시스템에서 자동으로 보장되는 결제 후처리 경로가 아직 완성되지 않았다 — 호스트 승인 없이 결제가 통과되는 경로가 막혀 있지 않으므로 정책 보강이 필요한 영역.
