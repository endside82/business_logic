F03-04. 이벤트 생명주기 관리 (호스트)

발행된 이벤트를 호스트가 운영하는 동안 일어나는 모든 동작 — 발행, 일정 변경, 종료, 취소, 삭제, 공지 발송, 위치 공유 정책 — 의 자리. 사용자 데이터를 크게 흔드는 동작(취소·환불·일정 변경)이 본 단위에서 시작되므로, 무엇이 같은 동작 안에서 함께 일어나는지(취소와 환불, 일정 변경과 알림)가 본 단위의 핵심 결정.

설계 의도 — 데이터를 크게 흔드는 동작은 분리될 수 없는 한 동작으로

유료 이벤트를 취소하는 한 번의 동작 안에 (1) 이벤트 상태를 취소로 바꾸기, (2) 참가자별로 환불 처리, (3) 환불 결과에 따라 다른 알림을 발송하기, (4) 환불 실패가 있다면 호스트에게 별도 알림을 보내기까지가 모두 일어난다. 이 다섯 가지가 따로 처리되거나 누락되면 "이벤트는 취소됐는데 환불이 안 됐다", "환불은 됐는데 알림이 안 왔다" 같은 분쟁이 즉시 발생한다. 데이터를 크게 흔드는 동작은 분리될 수 없는 한 동작으로 처리하는 것이 본 단위의 안전 기준선.

이 단위가 책임지는 동작

호스트가 자기 이벤트에 가하는 7가지 운영 행동.

동작 1

발행 (임시저장 → 모집)

임시저장 상태의 이벤트를 외부 사용자에게 공개. 한 번 발행한 뒤에는 다시 임시저장으로 되돌릴 수 없다 — "공개됐다"는 사실은 사용자 인지에 영향을 주기 때문.

동작 2

메타 수정 (임시저장에 한해)

제목·설명·카테고리·썸네일·정원·가격 같은 정보를 자유롭게 수정. 발행 이후에는 일정 변경을 제외한 메타 수정은 막힌다 — 신청자가 본 정보가 사후에 바뀌면 분쟁의 시작.

동작 3

일정 변경 (모집 중)

변경 규모에 따라 두 경로로 처리된다. 경미한 변경(60분 미만·장소 그대로·가격 동결)은 즉시 반영되고 "경미한 변경으로 바로 반영됐습니다" 토스트가 뜬다. 참가자에게 변경 알림만 간다.

큰 변경(60분 이상·장소 변경·가격 인상)은 참가자 합의 절차로 전환된다. 참가자 한 명 한 명에게 "일정 변경 제안" 알림이 발송되고, 48시간 이내 동의 또는 거절을 선택한다. 거절하면 패널티 없이 100% 환불된다. 전원 응답 후 호스트가 "확정"을 눌러야 이벤트가 변경된다. 자세한 내용은 F03-19 일정 변경 제안·참가자 합의.

동작 4

종료

모집 중 → 종료. 더 이상 신청을 받지 않지만 사진첩·리뷰는 살아 있다. 한 번 종료한 뒤에는 모집으로 되돌릴 수 없다.

동작 5

취소 + 환불

모집 중 → 취소. 유료 이벤트라면 참가자별 환불이 같은 동작 안에서 처리된다. 환불 성공자에게는 "취소됐고 환불 완료" 알림, 실패자에게는 "고객센터 처리 예정" 알림, 호스트에게는 환불 실패 건수 별도 알림.

동작 6

삭제 (임시저장에 한해)

임시저장 상태의 이벤트만 삭제 가능. 발행된 이벤트는 삭제할 수 없고 "취소" 또는 "종료"를 사용해야 한다 — 신청자 기록을 보존하기 위해.

동작 7

공지 발송

참석 확정자 전원에게 푸시 알림을 보낸다. 호스트 본인은 발송 대상에서 제외. 분당 3건 한도로 스팸을 방지하고, 한 공지에 외부 링크는 최대 2개까지 허용.

이벤트 상태가 거치는 길

한 이벤트가 거치는 다섯 상태. 임시저장과 모집으로의 되돌림은 허용하지 않는다.

사용자 시나리오

기본 흐름

임시저장 이벤트를 수정 후 발행

호스트가 임시저장 이벤트를 열어 수정 화면 진입 → 시작 시각·정원·가격 수정 → 저장 → 상세 복귀 → "발행" 버튼 → 확인 다이얼로그 → 발행 완료. 발행과 동시에 일반 사용자 노출 시작.

이 흐름이 남기는 것 — 발행 전 최종 검토 동선. 발행 즉시 외부 노출이 시작되므로 한 번 더 확인할 단계가 본 동선의 핵심.

참석자 영향

일정을 1시간 연기 + 참석자 알림

장소 사정으로 시작을 1시간 미룬다. 새 시각과 변경 사유("장소 사정으로 연기")를 입력하면 참석 확정자와 대기자 전원에게 푸시 알림이 발송된다. 알림 본문에 변경 전/후 시각과 사유가 함께 포함된다.

이 흐름이 남기는 것 — 변경 사유는 필수. 사용자가 변경의 의도를 모르면 "왜 갑자기"라는 의심이 누적된다.

취소·환불

유료 이벤트를 취소

호스트가 모집 중 이벤트를 취소. 확인 다이얼로그에 "참가자 N명에게 X원이 자동 환불됩니다" 안내가 먼저 보인다. 확인하면 한 동작 안에서 상태 변경·환불·알림이 모두 일어나고, 환불 성공자에게는 "취소됐고 환불 완료", 환불 실패자에게는 "고객센터 처리 예정", 호스트에게는 환불 실패 건수 요약 알림이 도달한다.

이 흐름이 남기는 것 — 비즈니스 분쟁이 가장 자주 발생하는 동작. 환불·알림이 같은 동작 안에서 처리되지 않으면 "환불은 됐는데 알림이 안 왔다", "취소됐는데 환불이 없다"는 분쟁이 즉시 일어난다.

공지

참석자 전원에게 준비물 안내

호스트가 더보기 → "공지 보내기" → 제목·본문 입력 → 발송. 호스트 본인을 제외한 참석 확정자 전원에게 푸시 알림. 한 공지에 외부 링크 최대 2개까지.

이 흐름이 남기는 것 — 호스트의 운영 진입로. 별도 채팅 없이도 참석자에게 닿을 수 있어 호스트의 운영 비용이 낮다.

스팸 방지

1분 안에 4번째 공지 발송 시도

분당 한도 초과 → "공지 발송 한도를 초과했습니다. 잠시 후 다시 시도해주세요" 안내. 한도가 풀린 뒤 재시도 가능.

이 흐름이 남기는 것 — 호스트가 흥분 상태에서 같은 정보를 여러 번 보내는 사고를 방지. 사용자 인지에 부담을 주지 않는 운영 한도.

삭제 제한

발행된 이벤트를 삭제 시도

임시저장이 아닌 이벤트는 "삭제" 버튼이 노출되지 않는다. 우회해도 "발행된 이벤트는 삭제할 수 없습니다. 취소 또는 종료를 사용해주세요" 안내. 신청자 기록을 보존하기 위해 발행 이후에는 삭제하지 않는다.

권한

일반 사용자가 수정 화면 진입 시도

일반 사용자가 수정 화면 URL로 진입해도 "접근 권한이 없습니다" 안내만 뜨고 폼은 열리지 않는다. 우회 호출은 서버에서 차단된다.

반복 이벤트

반복 그룹의 미래 회차 일괄 취소

호스트가 반복 그룹 화면에서 "미래 이벤트 취소"를 누르면 오늘 이후 자식 이벤트들이 한 번에 취소된다. 각 자식 이벤트의 참석자·환불·알림도 자식별로 같이 처리된다.

종료된 이벤트

종료·취소된 이벤트의 수정 시도

이미 종료되거나 취소된 이벤트는 수정 화면 자체가 열리지 않는다. "종료되었거나 취소된 이벤트는 수정할 수 없습니다" 안내만 보이고, 호스트는 회고·리뷰 같은 다른 동선으로 이동.

운영자가 알아야 할 것

생명주기 관리의 결정점
  • 유료 이벤트 취소 시 상태 변경·환불·알림은 같은 동작 안에서 처리된다. 세 가지가 분리되거나 일부가 누락되면 즉시 분쟁이 발생한다. 환불 실패는 별도 큐에 기록되어 고객센터가 수동 처리하고, 호스트에게도 별도 알림이 간다.
  • 발행 이후 메타 수정은 일정 변경을 제외하고 차단된다. 신청자가 본 정보가 사후에 바뀌면 분쟁의 시작. 정원·가격·환불 정책 변경이 필요하면 호스트는 취소 후 새 이벤트 생성으로 안내해야 한다.
  • 공지 발송은 분당 3건 한도. 호스트가 흥분 상태에서 같은 내용을 여러 번 보내는 사고 방지. 한도를 풀려면 정책 변경이 필요하다.
  • 한 공지에 외부 링크 최대 2개. 스팸·피싱 방지 정책. 더 많은 링크가 필요하면 호스트는 별도 채널을 사용해야 한다.
  • 발행된 이벤트는 삭제하지 않는다. 신청 기록·결제 기록을 보존해야 정산·환불 후속 처리가 가능. 호스트의 "지운 모임이 사라진 모임"으로 인지되지 않도록 "취소" 동작과 명확히 구분한다.
  • 반복 그룹의 일괄 동작은 자식 단위의 환불·알림을 모두 발생시킨다. 호스트가 "1년치 일괄 취소"를 누르면 그만큼의 환불·알림이 한 번에 발생할 수 있어, 운영 측에서 부하 모니터링 필요.
  • 일정 변경 시 변경 사유는 필수. 알림 본문에 함께 포함되어 사용자가 변경 의도를 납득하게 한다. 사유 없는 일정 변경이 가능해지면 "왜 갑자기"라는 의심이 즉시 누적.

측정해야 하는 것

생명주기 관리가 정상이라고 부르려면
  • 유료 이벤트 취소 시 환불 성공률 — 100%가 정상. 1건이라도 실패하면 환불 실패 큐에 들어가고, 고객센터의 수동 처리가 필요한 시점.
  • 일정 변경 → 참석자 알림 도달률 — 알림 발송 실패가 누적되면 사용자가 "변경됐는데 안 알려줬다"는 분쟁을 제기한다. 푸시 알림 도달 실패는 운영 모니터링 필수 항목.
  • 공지 발송 한도 초과 발생률 — 한도 초과가 자주 일어나는 호스트가 있다면 운영 측에서 사용 패턴을 확인할 신호.
  • 발행 후 24시간 내 취소 비율 — 호스트가 "발행 직후 후회"하는 비율. 너무 높으면 발행 전 미리보기 단계가 불충분하다는 신호.
  • 반복 그룹의 일괄 취소 빈도 — 빈도가 높으면 반복 설정 UI가 호스트 의도와 어긋난다는 신호. 반복 그룹 운영 도구의 직접 측정.
