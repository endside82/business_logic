F08-15. 크리에이터 매출 귀속 보정

플랜 마켓의 세 가지 구매(개별 아이템·번들·플랜 직접)가 모두 크리에이터 매출 원장에 잡힌다. 유료 결제분에만 플랫폼 수수료·원천징수가 붙고, 무료 보너스 결제분은 수수료 없이 별도로 적립된다. 번들 매출은 큐레이터가 아니라 포함 아이템 원작자에게 정가 비례로 배분되어 원작자 권리를 보호한다. 이 보정이 끝나야 F08-14 환불이 정확히 동작한다 — 환불은 매출 원장을 잠그거나 역전시키기 때문이다.

왜 이 보정이 환불보다 먼저인가

환불은 "이 구매가 만든 크리에이터 매출"을 정확히 찾아내 잠그거나(아직 정산 전), 회수하거나(이미 정산 완료) 해야 한다. 매출 원장이 구매 단위로 식별되지 않으면 환불 회계가 망가진다. 그래서 F08-14 환불 PRD가 F08-15에 하드 의존한다 — 매출 귀속이 정확해야 환불도 정확하다.

사용자 시나리오

개별 아이템 구매

"마켓에서 단품 데이트 플랜 구매"

구매자가 10,000원 결제. 그 아이템을 만든 크리에이터에게 매출 1건이 잡힌다. 플랫폼 수수료·원천징수 적용 후 다음 정산 주기에 크리에이터에게 지급.

이 흐름이 남기는 것 — 단순 구매에서 가장 직관적인 매출 흐름. 환불 발생 시 이 매출 행을 찾아 처리.

번들 구매 — 원작자 배분

"3개 아이템 묶음 30,000원 구매"

아이템 A(정가 10,000) + B(15,000) + C(8,000) 묶음. 결제 30,000원 → 각 아이템 정가 비율로 분배 → A 작가 9,090원, B 작가 13,640원, C 작가 7,270원에 매출 행이 따로 만들어진다.

이 흐름이 남기는 것 — 큐레이터 한 명이 매출을 독식하지 않고 원작자 권리를 보호.

플랜 직접 구매

"플랜 자체를 구매(이벤트 가입 아님)"

플랜 작성자에게 매출 1건. 1차 출시에 마켓 아이템과 동일한 회계 경로로 정산 대상에 포함.

이 흐름이 남기는 것 — 플랜 작성자도 마켓 판매자와 동등하게 수익을 받는 구조.

유료/무료 혼합 결제

"결제 10,000원 중 7,000은 유료·3,000은 무료 보너스"

매출 행이 두 부분으로 분리되어 기록 — 유료 7,000원에는 수수료 적용, 무료 3,000원은 수수료 없는 "무료 보조"로 적립. 호스트 정산 시 유료 부분만 지급 대상.

이 흐름이 남기는 것 — 사용자가 무료 보너스 포인트로 결제한 부분이 호스트 지급으로 흘러가지 않게.

환불 보류

"환불 신청 들어옴"

그 구매가 만든 매출 행 상태가 "환불 보류"로 들어간다. 월간 정산에서 제외되어 정산 후 환불로 인한 역회수 위험을 미연에 차단.

이 흐름이 남기는 것 — 환불 진행 중인 매출이 호스트에게 잘못 지급되지 않게.

정산 후 환불

"이미 호스트에게 지급된 매출에 환불 들어옴"

매출 행이 "역전" 상태로 들어가고 회수 절차가 별도로 돈다. 호스트에게 회수 안내.

이 흐름이 남기는 것 — 정산 후 환불도 회계가 깨끗하게 떨어지게.

화면에서 사용자(호스트·운영자)가 인지하는 것

영역사용자가 보는 것그 자리에 둔 이유
매출 원장 라벨"개별 아이템·번들·플랜 직접" 출처별로 매출 카드 라벨이 구분.호스트가 자기 매출의 출처를 한눈에 인지하게.
유료/무료 분리"유료 매출 + 무료 보조" 두 항목이 별도 행으로 노출.호스트가 자기 실제 지급 대상과 플랫폼 보조를 구분하게.
번들 배분 안내번들 매출 카드에 "정가 비례로 배분됨" 작은 글자 안내.원작자가 "왜 매출이 더 적지?" 의문 갖지 않게.
매출 상태"대기·정산 완료·환불 보류·환불 완료·역전" 다섯 상태가 카드에 색으로 노출.호스트가 자기 매출이 어디 단계에 있는지 인지하게.
역전 안내정산 후 환불이 들어온 매출은 별도 알림 + 회수 안내.호스트가 회수가 일어남을 명확히 알게 해 혼란 방지.

운영자가 알아야 할 것

정산·환불 영향
  • 매출 행은 구매 ID 기준으로 식별 — 환불 시 "이 구매가 만든 매출"을 정확히 찾을 수 있다.
  • 월간 정산은 "대기" 상태 매출만 — 환불 보류·환불 완료·역전 상태는 정산에서 제외.
  • 이벤트 정산의 유료/무료 분리는 후속 — 마켓·플랜은 1차에서 분리 완료. 이벤트는 다음 슬라이스에서.
  • 번들의 정가 합계가 0이면 균등 배분 — 안전망(예외 케이스 대비).
  • 번들 보너스 포인트 비용 — 플랫폼이 부담하는 마케팅 비용으로 처리(원작자 매출에서 빼지 않음).

레드 플래그

자주 발생하는 케이스
  • 지갑 환불의 유료/무료 분리 보존(2026-06-06 해소) — 환불이 결제했던 유료/무료 구성 그대로 복원되며, 장부와 실제 지갑이 같은 기준으로 기록된다.
  • 번들 큐레이터와 원작자 혼동 — 환불 권한자는 큐레이터, 매출 귀속은 원작자. 분쟁 발생 시 구분 안내 필요.
  • 이벤트 정산 분리 미이관 — 이벤트 결제의 유료/무료 분리는 아직 legacy 경로. 환불 시 마켓과 동작이 다를 수 있어 주의.
  • 정산 후 환불은 역전 처리 — 매출 행이 "역전" 상태로 들어가고 회수 절차가 별도로 돈다. 호스트에게 안내 필요.
