F01-08. 소셜 계정 연결 해제

소셜 가입자가 본인 계정에서 특정 제공자(카카오/네이버/구글/애플)와의 연결을 끊을 수 있게 한다. 본 단위의 모든 분기는 단 하나의 보호 규칙으로 수렴한다 — “마지막 로그인 수단은 절대로 해제할 수 없다.” 이 규칙이 깨지면 사용자가 자기 계정에 다시 들어올 길을 잃는다(계정 잠김).

핵심 안전 규칙

해제 가능 여부는 다음 식으로 결정한다 — (연결된 소셜 수) + (비밀번호 보유 시 1, 아니면 0) > 1. 이 식이 만족되지 않으면 무조건 차단(400 LAST_LOGIN_METHOD). 비밀번호도 없고 소셜도 하나뿐인 사용자가 그 하나를 해제하는 즉시 계정에 다시 들어올 수단이 없어진다.

이 기능 안의 하위 항목

하위 동작

제공자별 연결 해제

URL의 경로 변수로 어느 제공자를 해제할지 지정한다. 서버는 사용자의 비밀번호 보유 여부와 연결된 소셜 수를 합쳐 로그인 수단이 2개 이상 남는 경우에만 해제를 진행한다. 해제 후에는 같은 제공자로 다시 로그인하면 새로운 사용자 레코드가 만들어진다(데이터 분리 가능성).

의사결정 매트릭스

사용자가 보유 중인 로그인 수단 조합에 따라 결과가 달라진다.

비밀번호 연결된 소셜 해제 시도 결과
있음카카오 1개카카오허용 (이메일 로그인 남음)
없음카카오 1개카카오차단 (마지막 수단)
없음카카오 + 구글카카오허용 (구글 남음)
없음카카오 + 구글두 번째 해제차단 (이때부터 마지막 수단)
있음/없음—연결 안 된 제공자400 INVALID_INPUT

사용자 시나리오 8개

S1 · Happy Path

이메일 가입자가 카카오 연동을 해제

  1. 이메일/비밀번호로 가입 후 편의를 위해 카카오 연동했던 사용자
  2. 설정에서 “카카오 연결 해제” 탭 → 확인 다이얼로그 → “확인”
  3. 서버: 비밀번호 있음 + 카카오 1개 = 2 → 통과 → 카카오 소셜 로그인 수단 DELETE

종료 — 카카오 연결 사라짐. 이메일/비밀번호로만 로그인 가능.

S2 · 안전 규칙

카카오만 쓰는 사용자가 카카오 해제 시도

  1. 소셜 가입자, 비밀번호 없음, 카카오 1개
  2. 해제 시도 → 비밀번호 없음 + 카카오 1개 = 1 → 400 LAST_LOGIN_METHOD
  3. 외부 단위에서 “마지막 로그인 수단은 해제할 수 없습니다” 안내

종료 — 해제 차단. 사용자는 비밀번호 설정 또는 다른 소셜 추가 후 재시도 가능.

S3 · 다중 소셜

카카오 + 구글 사용자가 카카오 해제

  1. 소셜 2개 + 비밀번호 없음
  2. 해제 시도 → 2 → 통과 → 카카오 DELETE

종료 — 구글 1개 남음. 다음 해제 시도(구글)는 차단된다.

S4 · 잘못된 입력

지원하지 않는 제공자 문자열

  1. URL 경로에 “FACEBOOK” 같은 미지원 값
  2. 제공자 파싱 실패 → 400

종료 — 변화 없음. 참고: 클라이언트는 4종으로 고정되어 있어 정상 흐름에서는 발생 불가.

S5 · 미연동 제공자

연동되지 않은 제공자 해제 시도

  1. 카카오만 보유 + 비밀번호 있음 사용자가 실수로 “구글 해제” 호출
  2. 1차 검증(수단 수)은 통과 → 2차 검증(해당 소셜 보유)에서 발견 안 됨 → 400 INVALID_INPUT

종료 — 변화 없음. 잘못된 요청을 정확한 에러로 응답.

S6 · 운영 노트

해제 후 같은 제공자로 다시 로그인 시 새 계정 생성

  1. S1 후 사용자가 다시 카카오 로그인
  2. 서버: 카카오 식별자 검증 → 매칭되는 소셜 로그인 수단 없음 → 신규 가입 흐름 진입
  3. 새 사용자 레코드 생성(별도 userId), 기존 사용자는 그대로 남음

종료 — 데이터 분리 발생. 판단 필요: 동일 카카오 식별자를 동일 사용자에 다시 매핑하려면 별도 식별 정책이 필요(향후 운영 결정).

S7 · 인증

인증 없이 호출

  1. 토큰 없이 DELETE 요청
  2. 인증 필터에서 401

종료 — 변화 없음.

S8 · 동시성

두 기기 동시 해제 시도

  1. 같은 사용자가 카카오+구글 보유 상태에서 두 기기에서 동시에 “카카오 해제” 호출
  2. 첫 호출: 수단 2개 → 통과 → 카카오 DELETE
  3. 두 번째 호출: 카카오는 이미 삭제 → 수단 1개 → 400 LAST_LOGIN_METHOD

종료 — 첫 호출만 성공. 안전: 트랜잭션 내에서 수단 수가 다시 계산되므로 동시 호출이 같은 보호 규칙으로 수렴.

에러 코드 매트릭스
  • 400 LAST_LOGIN_METHOD — 해제 시 로그인 수단이 0개가 됨
  • 400 INVALID_INPUT — 해당 제공자가 사용자에게 연결되어 있지 않음
  • 400 — 경로 변수가 잘못된 제공자 문자열
  • 401 — 인증 토큰 없음

이 단위가 만들거나 바꾸는 데이터

대상변경
소셜 로그인 수단해당 제공자의 행 DELETE (다른 제공자는 영향 없음)
사용자 본체변경 없음 (이메일·비밀번호는 그대로)
리프레시 토큰변경 없음 (다른 기기 세션 유지)

자주 빠뜨리는 케이스

레드 플래그
  • “비밀번호 보유”를 카운트에서 빠뜨리는 회귀 — 이메일 가입자가 마지막 소셜을 해제하지 못하는 모순. 비밀번호도 “로그인 수단”으로 같이 센다.
  • 해제 후 다시 로그인 시 동일 사용자 매핑 가정 — 현재 구현은 새 계정을 만든다. UI에서 사용자에게 “해제 후 같은 소셜로 다시 가입하면 새로운 계정이 됩니다” 안내가 필요.
  • 해제 후 다른 기기 세션을 종료하지 않음 — 본 단위는 토큰을 폐기하지 않는다. 보안 강화 시 추가 정책 후보.
  • 마지막 수단 차단을 클라이언트에서만 검사 — 서버에서 반드시 강제. 클라이언트 우회 시도에도 동일하게 막아야 한다.
  • 403이 아닌 200으로 응답하는 회귀 — 마지막 수단 차단을 사용자에게 명확히 알리는 400 응답을 유지.
