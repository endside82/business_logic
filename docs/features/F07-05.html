F07-05. 분담금 납부 (Transfer Pay)

정산 도메인의 심장. 참가자가 본인 transfer를 포인트·외부 송금 앱·혼합 중 어느 방법으로든 결제할 수 있고, 마지막 결제 시점에 정산 자체가 자동 COMPLETED로 전이된다. 이의가 진행 중인 transfer는 결제 자체가 차단되어 호스트가 먼저 해결해야 한다.

결제 시점 가드 4단계
  1. PENDING 상태 — 이미 완료/취소된 transfer는 차단
  2. 본인 transfer — fromUserId 일치
  3. settlement ACTIVE — 취소·완료된 정산에서 결제 차단
  4. PENDING appeal 없음 — 이의 진행 중인 subject 결제 차단

이 기능 안의 하위 항목

동작 1

포인트 결제

유료 포인트로 즉시 차감 → 수취자 지갑 +. 회계 분개 동시. 한 호출로 양 쪽 잔액과 회계가 정렬된다.

즉시 정산
동작 2

외부 송금 앱(토스 등)

딥 링크로 외부 앱을 열어 사용자가 직접 송금. 서버는 자동으로 알 수 없으며 호스트가 입금 확인 후 F07-06으로 처리.

호스트 확인 필요
동작 3

혼합 결제 (MIXED)

POINT 일부 + BANK 일부. 합계가 정확히 일치해야 통과. POINT 부분은 즉시 정산, BANK 부분은 외부 송금 선언.

동작 4

self-refund (수취자 환불)

정산이 취소된 BANK 입금분을 수취자가 직접 송금자에게 포인트로 환불. 회계 역분개로 정산.

동작 5

자동 완료 트리거

마지막 transfer가 COMPLETED 되는 순간 settlement도 자동 COMPLETED. 호스트에게 알림.

사용자 시나리오 9개

S1 · Happy Path

포인트로 즉시 결제

분담금 16,000, 잔액 50,000.

  1. “정산 시작” 푸시 → my-shares 화면 → “포인트로 결제”
  2. 가드 4단계 통과
  3. 송금자 −16,000 + 수취자 +16,000 + 회계 분개
  4. transfer COMPLETED + audit TRANSFER_PAID_POINT
  5. 수취자에게 MEETING_SETTLEMENT_PAID 푸시 “정산금 16,000원이 입금되었습니다.”
S2 · 외부 송금

토스로 송금

  1. “송금 앱 열기” → 시트 → “토스로 송금”
  2. 토스 딥 링크 생성 → 외부 앱 오픈
  3. 사용자가 토스에서 송금 완료 후 앱 복귀
  4. 서버는 자동 인지 불가 — 호스트가 입금 확인 후 F07-06에서 “이체 확인” 처리

Fallback: 토스 미설치 시 웹 페이지로 이동, 그것도 실패하면 토스트.

S3 · 혼합 결제

POINT 10k + BANK 6k

  1. 혼합 결제 다이얼로그에 분배 입력
  2. 합계가 amount와 일치 검증 → 불일치 시 MIXED_AMOUNT_MISMATCH
  3. POINT 부분 즉시 차감 + 회계 분개
  4. BANK 부분은 외부 송금 선언으로 기록만
  5. transfer COMPLETED (paymentMethod=MIXED) + audit TRANSFER_PAID_MIXED
S4 · 항목 단위

share 직접 결제(저빈도)

  1. HOST_COLLECT 모드가 아닌 흐름에서 항목 holder에게 직접 결제
  2. UI는 기본적으로 transfer 흐름이 우세 → share 결제 버튼은 일부 화면에서만 노출
S5 · self-refund

정산 취소 후 수취자가 직접 환불

  1. 호스트가 정산 취소 → 본인이 받은 BANK 입금이 PENDING_MANUAL_REFUND로 전이
  2. MEETING_SETTLEMENT_REFUND_REQUIRED 푸시
  3. “포인트 환불” → 본인 지갑 −16,000 / 송금자 지갑 +16,000 + 역분개
  4. transfer CANCELLED + audit TRANSFER_SELF_REFUNDED
  5. 송금자에게 REFUND_COMPLETED 푸시
S6 · 잔액 부족

포인트 결제 잔액 부족

  1. deductPaidOnly에서 WALLET_INSUFFICIENT_BALANCE
  2. 트랜잭션 롤백 → transfer 상태 변동 없음
  3. “포인트가 부족합니다” + 충전 안내(F06)
S7 · 중복 차단

이미 완료된 transfer 재결제

  1. 더블 탭/재시도 → 두 번째 호출이 transfer.status≠PENDING 감지
  2. 409 MEETING_TRANSFER_ALREADY_COMPLETED
S8 · 이의 차단

이의 진행 중 결제 시도

  1. 다른 사용자가 자기 share/item에 이의 제기
  2. 본인 transfer 결제 시도 → validateNoPendingAppeal 실패
  3. 400 MEETING_SETTLEMENT_SUBJECT_UNDER_APPEAL
  4. “이의제기 진행 중” 토스트, 호스트 처리 후 재시도 안내

설계 의도: 이의가 진행 중인 동안 잔액이 옮겨지면 환원이 어려워진다. 결제 자체를 사전 차단.

S9 · 자동 완료

마지막 transfer가 settlement를 종료시킨다

  1. 5명 중 4명 완료, 5번째가 결제
  2. checkAndCompleteSettlement → 모든 transfer + share COMPLETED 확인
  3. settlement.status = COMPLETED + audit SETTLEMENT_COMPLETED(SYSTEM)
  4. 호스트(생성자)에게 MEETING_SETTLEMENT_COMPLETED 푸시
  5. 액션바가 “정산이 완료되었습니다” 라벨로 전환

자주 빠뜨리는 케이스

레드 플래그
  • 송금자·수취자 한 쪽만 잔액 변경 — 회계 분개와 wallet이 어긋남.
  • 이의 가드를 빼고 결제 허용 — 이의 해결 후 환원이 매우 복잡해진다.
  • 혼합 결제 합계 검증 누락 — POINT 10k + BANK 5k로도 통과되어 부분 결제 후 완료 처리.
  • 마지막 결제 시 자동 완료 누락 — settlement가 ACTIVE로 영원히 남는다.
  • self-refund 역분개 누락 — 정산 취소 후 회계 잔재.
  • 외부 송금을 자동 COMPLETED 처리 — 서버가 인지할 수 없는 상태를 임의로 변경하면 호스트 확인 절차가 무력화.
