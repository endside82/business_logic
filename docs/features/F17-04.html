F17-04. 정기모임 생명주기 (발행·종료·취소·재오픈)

모임 본체는 초안 → 모집 중 → 종료(또는 어디서든 취소)로 흘러간다. 종료는 모든 회차의 출석이 확정된 다음에만 통과한다 — 미확정 회차가 남아 있으면 버튼이 막힌다. 취소는 멤버 전원에게 자동 환불을 발사한다. 정산은 종료가 확정된 뒤 별도 단계로 만들어져, "정산 후 새 환불"이 생기는 사고를 막는다.

취소는 종착 상태
머니 안전 — 종료와 정산 사이에 빈틈 없음

종료는 모든 회차의 출석이 확정된 뒤에만 통과한다. 종료가 확정된 직후 정산이 만들어지고, 그 사이 멤버 자가 취소는 차단된다. 자격박탈 환불은 회차 확정 시점(=종료 이전)에만 발생하므로 정산 금액에 자연 반영된다 — 정산이 만들어진 뒤에 환불이 새로 발생할 가능성이 0이다.

네 가지 상태 전이

사용자 시나리오

발행

회차 8개 다 채우고 모집 연다

호스트가 초안 모임에서 "발행하기" 누름. 시스템이 회차가 약속한 8개에 도달했는지 확인하고 통과시킨다. 발행 직후부터 일반 검색에 노출되고 신청이 들어오기 시작한다.

이 흐름이 남기는 것 — 회차 부족 모임이 외부에 노출되어 사용자가 가입 후 "회차가 없네?" 하는 경험을 차단.

발행 차단

고정형 8회 약속인데 5회만 등록함

"발행하기" 누르면 "회차를 3개 더 추가해야 발행할 수 있어요" 안내. 시스템이 거절.

이 흐름이 남기는 것 — 호스트가 회차 셋업을 다 마치고 발행하게 강제, 사용자 신뢰 보호.

종료

모든 회차 출석 확정 후 종료

호스트가 마지막 회차 출석을 확정한 뒤 "종료하기" 누름. 즉시 정산이 자동 생성되어 호스트가 정산 화면으로 진입할 수 있다.

이 흐름이 남기는 것 — 정산이 종료 직후 가시화되어 호스트가 별도 알림 없이도 다음 단계로 흐른다.

종료 차단

마지막 회차 출석 미확정

"종료하기" 누르면 "미확정 회차가 있어요" 안내. 미확정 회차 목록을 보여주고 출석 확정 화면으로 진입할 수 있게 한다.

이 흐름이 남기는 것 — 회차 노쇼 판정 없이 정산이 만들어지는 사고를 차단.

취소

호스트가 모임 자체를 접는다

"취소하기" 누르고 확인 dialog("멤버에게 미진행 회차분 환불됩니다"). 활성 멤버 전원에게 비례 환불이 발사되고 미래 회차도 일괄 취소된다. 한 번 취소된 모임은 재개 불가.

이 흐름이 남기는 것 — 호스트가 모임을 접을 때 모든 멤버 환불 책임이 자동 처리되어 분쟁 여지를 줄인다.

재오픈

변동형 자동 종료를 풀어 다시 모집

주로 변동형 모임이 한동안 회차 없이 흘러 자동 종료된 경우, 호스트가 "재오픈"으로 다시 모집을 연다. 기존 취소 회차는 되살아나지 않는다.

이 흐름이 남기는 것 — 호스트가 한 번 닫은 모임을 새 모임으로 다시 만들지 않게, 같은 모임 ID를 유지하며 재시작 가능.

초안 수정

초안 본체 정보 손보기

초안 단계에서만 본체(제목·설명·카테고리·가격 등)를 수정 가능. 발행 후에는 수정 버튼 비활성.

이 흐름이 남기는 것 — 발행 후 본체 정보가 흔들리지 않게 해, 가입 시점 약속을 끝까지 유지.

초안 삭제

"이 초안은 안 쓰겠다"

초안 단계에서만 영구 삭제 가능. 발행된 모임은 멤버·결제·회차 이력 보존 의무로 삭제 버튼 비활성.

이 흐름이 남기는 것 — 호스트가 시도 단계의 초안을 깔끔히 정리할 수 있게.

운영자가 알아야 할 것

현금 가드
  • 종료가 막혀 있으면 출석 미확정 회차가 남은 것 — 호스트에게 어떤 회차가 남았는지 안내해 다음 액션이 명확하게.
  • 취소 환불은 비동기로 멤버마다 따로 처리 — 일부 실패해도 모임 취소 자체는 완료되고, 실패한 환불은 자동 재시도 큐로 들어간다.
  • 재오픈은 변동형 위주 — 고정형도 가능하지만 회차 확정을 다시 갈아엎고 닫아야 하므로 실제 운영 케이스 드묾.
  • 본체 수정·삭제는 초안만 — 발행 이후엔 두 버튼 모두 비활성. 가입자 노출 정보가 흔들리지 않도록.
  • 취소는 종착 — 되돌릴 수 없다 — 호스트에게 확인 dialog 필수.

레드 플래그

분쟁이 일어날 수 있는 케이스
  • 호스트가 출석 확정을 미루면 종료 버튼이 막혀 있음 — 다음 회차로 못 넘어가는 멤버 항의 가능. 자동 마감 스케줄러는 기본 비활성(opt-in).
  • 취소 환불 일부 실패 — 자동 재시도가 돌지만 사용자 입장에서는 환불이 늦어 보일 수 있음. 안내 필요.
  • 재오픈 후 회차 추가 → 다시 종료 — 정산이 멱등이라 한 번만 생성되지만, 호스트 입장에서 "정산이 안 늘었네?" 혼동 가능.
  • 발행 차단을 무한히 시도 — 회차 부족 같은 사유가 명확히 표시되어야 호스트가 다음 액션을 안다.
