F11-03. 리뷰 수정 · 삭제

본인이 작성한 후기를 정정하거나 지우는 면이다. 작성 직후의 단순 오타부터, 시간이 지나 마음이 바뀐 경우까지를 모두 받는다. 평판 데이터가 영구히 박히지 않도록 본인 통제권을 열어 두되, 무제한 수정으로 후기 신뢰가 흔들리지 않도록 횟수 한도(5회)와 수정 흔적("수정됨") 표시가 같이 따른다.

수정 한도
5회
초과 시 더는 수정 불가
수정 흔적
"수정됨" 노출
한 번이라도 수정되면 카드에 표시
삭제 정책
행 보존
표시는 사라지지만 데이터는 남는다
설계 의도 — 통제권은 본인에게, 흔적은 다음 사용자에게

작성자가 자기 후기에 대한 통제권을 가지지 못하면 사용자는 작성 자체를 망설인다. 그래서 수정과 삭제는 본인 한정으로 열어 둔다. 동시에 후기는 다음 참가자의 결정 단서이므로 "원래 어떻게 적혔는지"가 사라지면 신뢰가 흔들린다 — 수정될 때마다 직전 내용을 따로 보관하고, 다음 사용자에게는 "수정된 후기"라는 사실만 알린다. 삭제는 행을 통째로 지우지 않고 "지워진 표시"만 한다 — 1인 1리뷰 정책(삭제 후 재작성 불가)이 유지되어야 하기 때문.

사용자 시나리오

기본 흐름 — 수정

다음날 보니 별점이 너무 박했다

본인 후기 카드의 더보기 메뉴에서 "수정"으로 들어온다. 작성 폼에 직전 별점과 본문이 미리 채워진 상태로 열리고, 별점을 4에서 5로 올린 뒤 "수정"을 누른다. 토스트가 뜨고 목록으로 돌아오면 카드 옆에 "(수정됨)" 라벨이 붙어 있다.

이 흐름이 남기는 것 — 작성자가 자기 평가를 차분히 다시 정리할 수 있는 여유. 동시에 다음 참가자는 "이 후기는 수정된 적이 있다"는 사실을 알 수 있다.

한도 도달

같은 후기를 다섯 번째 수정한 뒤 또 손대려 함

이미 다섯 번 수정한 후기에 또 들어와 수정을 시도한다. "리뷰 수정 한도(5회)를 초과했습니다" 토스트가 뜨고 더 이상 변경되지 않는다.

이 흐름이 남기는 것 — 후기를 무한히 바꿔 다음 사용자의 판단 기준이 사라지는 일을 막는다. 다섯 번이면 의도 표현은 충분하다.

기본 흐름 — 삭제

그냥 지우고 싶다

본인 카드 더보기에서 "삭제"를 누르면 "리뷰를 삭제할까요? 삭제된 리뷰는 복구할 수 없습니다" 확인 다이얼로그가 떠오른다. 확정하면 토스트와 함께 목록에서 사라지고, 호스트의 신뢰점수도 다시 계산된다.

이 흐름이 남기는 것 — "내 마음에 안 드는 후기"를 본인이 통제할 수 있다는 안전감. 호스트 점수는 즉시 재계산되므로 평판 합계가 어긋나지 않는다.

삭제 회피

삭제를 누르고 나서 마음이 바뀜

다이얼로그가 떠 있는 상태에서 "취소"를 누르면 아무 변동 없이 닫힌다. 실제 삭제 호출은 발생하지 않는다.

이 흐름이 남기는 것 — 위험한 동작(삭제) 앞에 한 번 더 묻는 가드가 일관되게 작동한다는 신뢰.

권한 차단

남의 후기를 수정·삭제하려는 시도

본인이 작성하지 않은 후기에 대해 직접 호출이 들어오면 "본인 리뷰만 수정·삭제할 수 있습니다" 안내가 떠오른다. 화면에서는 더보기 메뉴 자체가 보이지 않는 자리이지만, 어떤 경로로든 우회 시도가 막힌다.

이 흐름이 남기는 것 — 평판 데이터의 무결성을 가장 기본적인 자리에서 지키는 가드.

화면 동기화

이미 삭제된 후기를 캐시된 화면에서 또 손대려 함

다른 기기에서 이미 삭제한 후기를, 캐시된 화면에서 다시 수정·삭제하려 한다. "리뷰를 찾을 수 없습니다" 안내가 뜨고 목록은 새로 받아 와 그 카드가 사라진다.

이 흐름이 남기는 것 — 사용자가 여러 기기로 같은 후기를 다룰 때 화면이 자동으로 정렬되도록 한다.

본인 카드에서만 보이는 더보기 메뉴

수정·삭제 진입은 리뷰 목록(F11-02)의 카드 더보기에서 시작된다. 본인이 작성한 카드와 남이 작성한 카드는 그 자리에 다른 아이콘이 보인다.

카드 종류카드 우상단에 보이는 것그렇게 둔 이유
본인이 쓴 후기 카드 더보기(케밥) 메뉴. 탭하면 "수정" / "삭제" 두 항목이 시트로 열린다. 본인 후기는 정정·삭제의 대상이고, 신고의 대상이 아니다 — 같은 자리에 두 아이콘을 띄우면 사용자가 헷갈린다.
다른 사람이 쓴 후기 카드 신고 아이콘(깃발). 탭하면 신고 화면(F11-04)으로 진입. 남의 후기에 직접 손댈 수 없다는 사실을 아이콘 자체로 명확히 한다.
한 번이라도 수정된 카드 닉네임 옆에 "(수정됨)" 라벨이 작게 붙는다. 다음 참가자가 후기를 읽을 때 "이 평가는 한 번 변동된 적이 있다"는 사실을 알 수 있게 한다. 횟수까지는 노출하지 않는다.

운영자가 알아야 할 것

수정·삭제가 일으키는 후속 동작
  • 수정과 삭제 모두 호스트 신뢰점수를 다시 계산시킨다. 평판 합계가 어긋난 채로 남지 않게 하는 안전장치 — 호스트 점수의 즉시성은 이 호출에 달려 있다.
  • 삭제는 행을 지우는 것이 아니라 "삭제됨" 표시만 한다. 조회·통계에서는 빠지지만, 1인 1리뷰 검사에는 여전히 잡힌다 — 본인이 한 모임에서 후기를 지우고 다시 작성하는 우회 경로를 막는다.
  • 수정 직전 본문은 따로 보관된다. "원래 어떻게 적혔는지"가 운영 시점에서 필요해질 수 있다(악의적 수정 패턴, 운영자 검수 등) — 이력은 5건까지 누적된다.
  • 5회 한도는 호출 시점에 검사된다. 화면에서 사전에 "수정 가능 횟수"를 알려 주지는 않는다 — 사용자가 다섯 번째에서 잘리는 경험이 갑작스럽지 않게 하려면, 응답에 잔여 횟수를 함께 내려 화면에 보여 주는 보강이 필요하다(미구현 영역).
  • "수정됨" 라벨은 카드 옆에 한 번만 표기되고, 횟수까지는 보여 주지 않는다. 다섯 번 수정한 후기와 한 번 수정한 후기를 같은 시각적 인상으로 다루는 게 현재 정책이다 — 다음 사용자에게 "신뢰의 정도 차이"를 노출할지 여부는 별도 논의가 필요하다.
