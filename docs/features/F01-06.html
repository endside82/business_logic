F01-06. 온보딩 (서비스 소개 → 프로필 → 관심사 → 위치)

가입을 끝낸 사용자가 처음 들어가는 네 단계. 서비스가 무엇을 하는 곳인지 보여주고, 본인 정보를 채우고, 관심사를 고르고, 동네를 설정해서 추천이 가능한 상태로 만든다. 모든 단계는 건너뛸 수 있지만 건너뛴 만큼 첫 화면이 비어 보이게 된다 — 강제하지 않으면서도 채우는 게 본인에게 이득임을 사용자가 자연스럽게 느끼게 만드는 흐름이다.

온보딩은 신규 사용자가 가입 직후 "이 앱이 나에게 줄 게 있는지" 확인하는 5분이다. 이 5분에 사용자가 본인 관심사·동네를 입력하느냐 아니냐가 첫 홈 화면의 풍부함을 결정하고, 첫 홈 화면의 풍부함이 재방문 여부를 결정한다. 그래서 본 단위는 각 단계를 모두 건너뛸 수 있게 두되, 건너뛴 결과(빈 추천 화면, 위치 기반 기능 비활성)가 자연스럽게 사용자에게 도달하도록 둔다 — 다음에 본인이 채울 동기가 생기게.

이 단위가 도메인 잔류·이탈을 결정한다

가입을 끝낸 사용자가 다시 돌아올지 떠날지는 이 5분에 결정된다
  • 관심사가 비어 있으면 추천이 비어 보인다. 온보딩에서 관심사를 한 개도 안 고른 사용자는 첫 홈 화면에서 맞춤 추천 영역이 비거나 일반 콘텐츠로 채워진다 — "이 앱은 나를 모른다"는 인상이 첫 화면에서 형성된다. 그래서 본 단위는 "관심사 최소 3개"를 강하게 권장한다.
  • 위치를 설정해야 거리 기반 기능이 동작한다. 내 주변의 이벤트·모임을 보여주는 기능 전체가 위치를 입력값으로 쓴다 — 위치를 건너뛴 사용자는 "근처의 ~" 종류의 추천을 받을 수 없다.
  • 프로필 사진과 닉네임은 사용자 본인이 다른 사용자에게 어떻게 보일지를 결정한다. 본 단위에서 사진을 안 올린 사용자는 기본 아바타로 모임·매칭에 나타나서 다른 사용자의 의사 결정에 불리해진다 — 본인 노출 효과를 사용자가 본 단위에서 처음 인지하게 한다.

네 단계의 흐름

앞 단계에서 입력한 결과가 다음 단계에서 쓰이지는 않는다 — 단계들은 서로 독립적이고 각자 건너뛸 수 있다.

사용자 시나리오

기본 흐름

네 단계를 모두 채워서 풍부한 홈으로 진입

가입 직후 4페이지 소개를 본 뒤 프로필 사진을 올리고 닉네임을 손본다. 관심사 그리드에서 등산·카페탐방·영화 세 개를 고른다. "현재 위치 자동 감지"로 동네를 설정하고 홈으로 들어간다 — 첫 홈 화면의 추천 영역이 본인 관심사로 채워져 있다.

이 흐름이 남기는 것 — 첫 홈이 풍부하게 보일수록 다음날 재방문 비율이 높아진다. 본 시나리오가 본 단위의 핵심 성공 경로.

빠른 진입

모든 단계를 건너뛰고 곧장 홈으로

"빨리 안을 보고 싶은" 사용자가 각 단계에서 "건너뛰기"만 누른다. 30초 안에 홈에 도달하지만 추천 영역이 비어 있고 위치 기반 기능이 동작하지 않는다 — 사용자에게는 "이 앱이 비어 있네"라는 인상이 첫 화면에서 형성된다.

이 흐름이 남기는 것 — 강제하지 않은 결과를 그대로 보여줌으로써 사용자가 본인 선택의 영향을 즉시 인지하게 한다. 다음에 본인이 프로필로 가서 관심사를 추가할 동기가 만들어진다.

중단·재개

프로필까지 하고 앱 종료 → 며칠 뒤 다시 진입

프로필을 채운 뒤 일정이 생겨 앱을 닫는다. 며칠 뒤 앱을 다시 켜면 로그인 화면이 아니라 관심사 화면이 곧장 뜬다 — 본인이 어디까지 했는지를 앱이 기억한다. 사용자는 처음부터 다시 입력할 필요가 없다.

권한 분기

위치 권한을 거부한 사용자

"현재 위치 자동 감지" 버튼을 누르고 OS 권한 다이얼로그에서 거부한다. "위치 권한이 필요합니다" 안내가 한 번 뜨고, 사용자는 "건너뛰기"로 위치 없이 홈에 진입한다. 거리 기반 기능은 비활성 상태이지만 다른 기능은 정상.

이 흐름이 남기는 것 — 권한 거부가 온보딩 전체 진행을 막지 않는다. 권한 정책과 진행 정책을 분리.

실패

프로필 사진 업로드가 실패

사진을 골라 업로드를 시도하지만 네트워크 끊김으로 실패한다. "업로드에 실패했습니다" 안내가 뜨고 미리보기는 그대로 보이지만 실제 저장은 안 된 상태. 사용자가 "다음"을 눌러도 프로필 사진은 비어 있는 채로 다음 단계로 넘어간다.

중복

닉네임이 이미 사용 중

소셜 가입자가 자동 생성된 닉네임을 더 좋은 것으로 바꿔보려고 한다. "이미 사용 중인 닉네임입니다" 안내가 뜬다 — 다른 닉네임으로 다시 시도하거나 그냥 자동 닉네임으로 다음 단계로 넘어갈 수 있다.

권한 부재 우회

위치 권한 없이 주소 직접 검색

위치 권한을 주기 싫은 사용자가 "다른 주소 검색" 동선으로 본인 동네를 직접 입력한다. 우편번호 검색 모달이 뜨고 선택한 주소를 위치로 설정하면 거리 기반 기능이 동작한다 — 권한 없이도 위치 입력은 가능.

이 흐름이 남기는 것 — 권한과 입력 경로가 분리되어 있어서 한쪽이 막혀도 다른 경로로 같은 결과에 도달 가능.

화면에서 사용자가 인지하는 것

단계마다 다른 화면이지만 공통적으로 진행 인디케이터와 건너뛰기 동선이 같은 자리에 있다.

단계사용자가 보는 것그 자리에 둔 이유
1. 서비스 소개 4페이지 캐러셀 — 만남·이벤트·클럽·내 주변. 페이지 인디케이터(점)와 "건너뛰기" 링크 가입 직후 사용자가 본 단위 진입 이유를 자기 언어로 정리하게 한다. 4페이지는 한 페이지당 한 가지 가치만 보여서 인지 부담을 낮춤.
2. 프로필 3단계 진행 표시 + "1/3 프로필 사진과 닉네임을 설정해주세요" 안내. 큰 원형 아바타·닉네임·자기소개 칸. 뒤로가기 차단 가입에서 자동 생성된 닉네임을 사용자가 한 번 더 직접 확인하게. 사진은 본인이 어떻게 보일지 첫 결정의 자리.
3. 관심사 카테고리별 태그 그리드 (스포츠/음식/문화). "3개 이상 선택" 안내와 현재 선택 카운트 배지. "다음 (N개 선택)" 버튼 "3개 이상"이라는 권장이 현재 선택 개수와 함께 보여서 사용자가 본인의 진행 상태를 매 순간 인지한다.
4. 위치 "주변 이벤트와 클럽을 추천받으려면 위치를 설정해주세요" 안내. "현재 위치 자동 감지" 큰 버튼. 감지 후에는 지도 미리보기 + "이 위치로 설정" 위치 입력이 추천 품질의 직접 입력값임을 안내 문구에서 명시 — 권한 요청 직전에 사용자가 "왜 이걸 묻는지" 알고 결정하게.
공통 — 건너뛰기 링크 모든 단계 하단에 동일하게 위치한 작은 텍스트 링크 "건너뛰기"가 부차적 동선임을 시각적 위계에서 분명히 — 채우는 게 기본 행동.

측정해야 하는 것

본 단위가 정상이라고 부르려면 이 다섯 지표가 추적되어야 한다
  • 온보딩 시작 → 완료 전환율 — 본 단위 진입 후 마지막 단계를 끝낸 사용자 비율. 본 단위 효과의 가장 큰 신호.
  • 단계별 건너뛰기 비율 — 어느 단계가 가장 많이 건너뛰어지는지. 건너뛰기율이 높은 단계는 화면 설계 개선 후보.
  • 관심사 3개 이상 선택 비율 — 권장 기준을 채운 사용자 비율. 추천 품질의 직접 입력 신호.
  • 위치 권한 허용 비율 — 권한 거부 비율이 너무 높으면 권한 요청 시점 또는 안내 문구를 손봐야 한다.
  • 온보딩 완료 후 다음날 재방문 비율 — 본 단위의 진짜 성과 지표. 풍부한 첫 홈을 만든 사용자와 빈 첫 홈을 본 사용자 사이의 재방문 격차를 추적.

운영자가 알아야 할 것 — 비즈니스 리스크

본 단위에서 어긋나면 신규 사용자 잔류가 떨어진다
  • 관심사 태그 카테고리는 클라이언트에 하드코딩되어 있다. 운영팀이 새로운 카테고리를 추가하려면 앱 업데이트가 필요하다 — 시즌성 콘텐츠(예: 여름 한정 야외 활동)를 빠르게 반영하기 어려움. 카테고리 동적 관리 기능이 필요해질 시점을 모니터링.
  • 관심사 등록은 한 개씩 순차로 일어난다. 중간에 네트워크가 끊기면 일부만 저장되고 나머지가 다음 진입에서 중복으로 거절될 수 있다 — 사용자가 답답한 상태로 막힐 위험. 이 동작의 사용자 경험은 운영 측에서 사용자 응답으로 모니터링해야 한다.
  • 관심사 0개로 메인 진입한 사용자가 재방문하지 않으면 본 단위의 의미가 사라진다. "건너뛰기" 허용 자체는 정책이지만 그 결과 빈 추천을 본 사용자에게 다시 관심사 추가를 권하는 후속 알림이 필요한지 검토 필요.
  • 위치 권한 거부 사용자는 거리 기반 기능 전반에서 가치를 받지 못한다. 본 단위 외부(예: 홈 추천)에서 권한 거부 안내 띠를 노출하는 정책과 함께 움직여야 한다.
  • 중간 단계에서 종료한 사용자는 재진입 시 같은 자리부터 보여지지만, 너무 오래 안 돌아오면 본인이 어디까지 했는지 잊는다. 온보딩 중단 후 며칠이 지나도 재진입이 없으면 별도 알림으로 안내하는 정책을 검토할 수 있다.
  • 온보딩 단계의 상태는 사용자 기기에 저장되어 있다. 사용자가 앱을 삭제 후 재설치하면 처음부터 다시 보게 된다 — 본 단위의 정상 동작이지만 CS 문의로 들어올 수 있는 행동이라 매뉴얼에 포함.
  • 4페이지 캐러셀의 콘텐츠는 서비스 정체성을 결정한다. 마케팅·기획에서 가치 메시지가 바뀔 때 본 화면의 4페이지 카피와 일치하는지 점검 필수 — 어긋나면 사용자가 받은 첫 인상과 실제 사용 경험이 다르게 느껴진다.
