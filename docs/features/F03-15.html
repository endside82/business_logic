F03-15. 이벤트 카풀

이동수단이 CARPOOL인 이벤트에서 운전자가 좌석을 제공하고, 호스트가 참가자를 차량에 배정하는 운영 도구. 참가자는 개별 이동을 선택하거나 카풀 배정을 받을 수 있고, 호스트는 정원·픽업 가능 인원·권한을 기준으로 배정을 조정한다.

카풀은 "누가 탈 수 있나"보다 "누가 누구 차에 배정됐나"가 중요하다. 운전자 지원, 호스트 확정, 참가자 배정, 배정 해제, 개별 이동 선택을 모두 분리해야 당일 혼선이 줄어든다.

운영 흐름

flowchart TD A["mode=CARPOOL 이벤트"] --> B["운전자 offer 등록"] B --> C{"호스트 심사"} C -->|"확정"| D["offer=CONFIRMED"] C -->|"거절"| E["offer=REJECTED"] D --> F["참가자 passenger row"] F --> G{"이동 선택"} G -->|"카풀 배정"| H["assigned_offer_id 기록"] G -->|"개별 이동"| I["transport_choice=SELF"] H --> J["배정 변경 로그"] H -->|"해제"| K["assigned_offer_id=NULL"]

사용자 시나리오

운전자

참가자가 운전자로 지원

좌석 수와 픽업 가능 인원을 입력해 offer를 만든다. 호스트가 확정하기 전까지는 배정 대상 차량이 아니다.

호스트

운전자 확정 후 참가자 배정

호스트가 offer를 CONFIRMED로 바꾼 뒤 참가자를 해당 차량에 배정한다. 픽업 가능 인원 초과 배정은 차단된다.

참가자

개별 이동 선택

`allowsSelfTransport=true`이면 참가자가 SELF를 선택할 수 있다. 이 참가자는 호스트의 카풀 배정 대상에서 빠진다.

권한

참가자는 자기 row만 본다

일반 참가자가 passenger 목록을 조회하면 본인 row만 보인다. 호스트는 전체 passenger를 보며 운영한다.

mode guard

CARPOOL이 아닌 이벤트에서 호출

mode가 NONE 또는 BUS이면 모든 carpool endpoint가 실패한다. 하위 기능은 F03-14의 mode를 신뢰한다.

DRAFT 전환

카풀 설정 후 버스로 바꿈

DRAFT에서 mode를 BUS로 바꾸면 stale carpool 데이터가 삭제된다. 공개 전 변경이므로 사용자에게 알림은 필요 없다.

서버 계약

영역대표 동작사용자 의미
Offer운전자 지원·수정·확정·거절차량 제공 후보를 호스트가 운영 가능한 상태로 전환
Passenger탑승자 목록 조회·이동 선택참가자의 카풀/개별 이동 의사와 배정 상태 표시

운영자가 알아야 할 것

당일 혼선을 줄이는 제약
  • CONFIRMED offer만 배정 대상이다. 지원만 한 운전자의 차량에는 참가자를 배정하지 않는다.
  • 픽업 가능 인원을 넘길 수 없다. 좌석 수와 픽업 가능 인원은 다를 수 있다.
  • 배정 해제도 알림 대상이다. 참가자의 당일 이동 계획이 바뀌므로 배정/해제 모두 알림이 필요하다.
  • 1차 문서는 backend 중심이다. Flutter 화면은 후속 운영 화면에서 더 구체화될 수 있다.
