F11-02. 리뷰 목록 조회 (이벤트별 · 사용자별)

이미 작성된 후기들을 모아 보여 주는 면이다. 한 화면이 두 가지 진입을 함께 처리한다 — 모임을 신청할지 고민하는 사람은 이벤트별 리뷰를 보고, 호스트의 평판을 가늠하려는 사람은 사용자별 리뷰를 본다. 평균 별점과 분포는 다음 결정의 단서가 되고, 부적절한 후기는 카드에서 바로 신고로 빠질 수 있다.

진입 모드
2가지
이벤트별 · 사용자별
평균·분포
화면 산정
전체 리뷰를 받은 뒤 즉시 계산
신고 진입
이벤트 모드만
사용자 프로필에서는 카드 신고 미노출
설계 의도 — 같은 데이터를 두 의도로 보여 준다

"이 모임이 어땠는지"와 "이 사람이 받아온 평가가 어땠는지"는 의사결정의 맥락이 다르다. 이벤트별 모드에서는 평균과 분포가 가장 먼저 보여 신청 판단을 돕고, 사용자별 모드에서는 그 사람이 받은 후기 흐름이 중심이다 — 분포 막대는 노이즈가 되어 의도적으로 빠진다. 카드에서 신고 진입이 보이는 자리도 다르다. 사용자 프로필 내부에서 다른 사람을 즉시 신고하는 경로는 오용 여지가 있어 화면 자체에서 빼고, 이벤트 모임 안에서 발견된 부적절한 후기만 그 자리에서 받는다.

두 진입 모드의 차이

같은 화면이지만 보여 주는 것과 받을 수 있는 행동이 다르다.

모드사용자가 보는 것그 자리에 둔 이유
이벤트별 상단에 큰 평균 별점 + "리뷰 N개" + 5점부터 1점까지 분포 막대. 그 아래로 작성자 아바타·닉네임·별점·상대시간·본문이 담긴 카드 리스트. 카드 우상단에 신고 아이콘. 모임을 신청할지 결정할 때 평균보다 "분포가 어떻게 갈리는가"가 더 큰 단서가 된다 — 평균 4점이라도 5점이 많은 경우와 3·5점이 양극화된 경우는 의미가 다르다.
사용자별 평균 별점 카드 + 받은 리뷰 카드 리스트. 분포 막대는 빠지고, 카드의 신고 아이콘도 노출되지 않는다. 한 사람의 평판을 가늠할 때는 누적 평균이 의사결정의 핵심. 분포 막대는 후기 한 건 한 건이 어느 모임에서 나왔는지 흩어져 있어 의미가 흐려진다.

사용자 시나리오

신청 직전

모임 상세에서 "리뷰" 탭

신청을 고민하던 사용자가 이벤트 상세에서 리뷰 탭으로 진입한다. 평균 별점 4.3, 리뷰 12개, "5점이 6명·4점이 4명·3점이 1명" 같은 분포가 한 화면에 보이고 그 아래로 후기 카드들이 깔린다.

이 흐름이 남기는 것 — 평균만으로 결정하기 망설이는 사용자가 "어느 정도로 갈렸는지"까지 확인하고 신청을 결정한다.

방금 작성

내가 막 후기를 남기고 돌아옴

리뷰 작성을 마친 사용자가 목록으로 돌아온다. 화면을 아래로 당기면 방금 작성한 후기가 카드 리스트에 즉시 등장한다.

이 흐름이 남기는 것 — "내 후기가 등록됐다"는 즉시 확인은 사용자가 후기 작성에 다시 동참할 동기를 만든다.

빈 상태

아직 후기가 한 건도 없는 모임

새로 열린 모임이라 후기가 0건이다. 평균과 분포 자리는 비고 "아직 리뷰가 없습니다" 안내만 표시된다.

이 흐름이 남기는 것 — 데이터가 없다는 사실 자체를 사용자가 명확히 인지하게 한다 — 평균 0.0 같은 오해 소지의 숫자는 보여 주지 않는다.

즉시 안전

이상한 후기를 발견하고 카드에서 신고

카드 우상단 신고 아이콘을 누르면 신고 화면으로 즉시 빠진다(F11-04). 신고 대상은 자동으로 그 후기로 잡혀 있다.

이 흐름이 남기는 것 — 부적절한 후기를 발견한 사용자가 별도 경로를 찾을 필요 없이 한 번에 운영자에게 전달할 수 있다. 안전 동선은 짧을수록 작동한다.

호스트 평판

이 호스트가 어떤 사람인지 보려고 함

사용자 프로필에서 "리뷰" 탭으로 들어온 사용자가 호스트가 그동안 받아온 후기들을 본다. 분포 막대는 없고 평균 + 카드 리스트만 깔린다. 카드에는 신고 아이콘이 보이지 않는다.

이 흐름이 남기는 것 — 평판 가늠은 누적 평균이 핵심이라는 결정. 동시에 프로필 안에서 즉흥적인 신고 진입을 막아 오용을 줄인다.

일시 장애

잠시 응답이 안 옴

네트워크가 흔들려 응답이 오지 않으면 카드 자리에 "다시 시도" 버튼이 떠오른다. 사용자가 누르면 같은 요청을 다시 보내 채워진다.

이 흐름이 남기는 것 — 자동 재시도로 배터리·트래픽을 낭비하지 않고 복구 시점을 사용자가 통제한다.

운영자가 알아야 할 것

조회 단위에서 운영이 알아야 할 것
  • 평균과 분포는 화면에서 즉시 계산된다. 서버는 단순 리스트로 보내고, 평균·분포·정렬은 클라이언트가 직접 만든다 — 후기 데이터가 한 화면에 모두 들어와야 정확한 평균이 나온다. 모임당 후기가 수백 건 이상으로 늘어나는 시점이 오면 페이지 단위 조회 + 서버 측 집계 전환이 필요하다.
  • "최신순" 라벨은 정적 표기다. 현재는 정렬 옵션 변경이 동작하지 않는다 — 서버가 보내는 순서를 그대로 보여 준다. 사용자가 정렬을 바꿔도 결과가 같다는 보고가 들어오면 이 라벨의 노출 자체를 조정해야 한다.
  • 이벤트별 모드에서만 카드 신고 아이콘이 보인다. 사용자 프로필 안에서 다른 사용자의 후기를 즉시 신고하는 동선은 오용 가능성이 있어 의도적으로 닫혀 있다 — 사용자 신고가 필요하면 프로필 상단의 별도 "사용자 신고" 경로(F11-04)로 진입한다.
  • 본인이 작성한 카드에는 신고 대신 더보기 메뉴가 보인다. 여기서 수정·삭제(F11-03)가 트리거된다 — "내 후기를 내가 신고하는" 혼란이 없도록 분리되어 있다.
  • 빈 상태와 응답 실패는 다른 화면으로 보인다. "아직 리뷰가 없습니다"는 정상이고, "다시 시도" 카드가 뜨면 일시 장애다 — 두 상태를 같이 다루지 않는다.
