F10-04. 반복 가용 시간 규칙 관리

"매주 평일 점심", "격주 토요일 오전"처럼 주기적으로 반복되는 가용 시간을 한 번의 규칙 설정으로 묶어 관리한다. 규칙 한 건을 만들면 서버가 그 자리에서 다음 3개월치 슬롯을 펼쳐 캘린더에 등록한다. 무기한 반복(종료일 없음)은 매일 새벽 자동으로 3개월씩 미래로 연장된다. 캘린더 위에서는 단일 가용 시간과 동일하게 보이지만, 수정과 삭제는 규칙 단위로 일괄 적용된다.

"매주 월·수·금 점심을 비워 두고 싶다"는 의사 표시는 단일 슬롯 등록을 50번 하는 것과 본질이 다르다. 사용자는 한 번 결정을 내렸고, 같은 결정을 매번 다시 내리고 싶지 않다. 그래서 본 단위는 결정을 "규칙"으로 저장하고, 실제 시간 슬롯은 서버가 자동으로 펼친다. 사용자는 규칙 한 건을 만들고 한 건을 삭제할 뿐, 그 안의 50개 슬롯을 직접 다루지 않는다. 이 분리가 깨지면 사용자가 "지웠다"고 생각한 것과 실제 캘린더의 상태가 어긋난다.

한 번의 규칙으로 결정하는 것

주기

매주 · 격주 · 매월

매주는 선택한 요일마다, 격주는 선택한 요일이지만 한 주 건너서, 매월은 시작일의 일자에 매월 한 번.

요일

매주/격주에서만 사용

월~일 중 다중 선택. 매월 주기는 요일 대신 시작일의 일자를 사용한다(28일 이후는 그 달의 마지막 날로 보정).

시간대

시작·종료 시각

30분 단위, 최소 30분~최대 8시간. 단일 슬롯과 같은 시간 규칙.

기간

시작일·종료일(또는 무기한)

종료일을 지정하면 그날까지만 펼친다. "종료일 없음"으로 두면 무기한 — 스케줄러가 미래로 자동 연장.

규칙 등록 한 번에 일어나는 일

사용자가 "저장"을 누른 순간 서버는 (1) 규칙 한 건을 저장하고 (2) 다음 3개월치 슬롯을 펼쳐 등록한다. 이 펼침 도중 단일 가용 시간과 겹치는 인스턴스는 조용히 건너뛴다.

sequenceDiagram actor U as 사용자 participant 화면 as 반복 규칙 화면 participant 서버 as 가용 시간 서비스 participant DB as 데이터 저장소 U->>화면: 주기·요일·시간·기간 입력 후 저장 화면->>서버: 규칙 생성 요청 서버->>서버: 시간 형식 / 길이 / 시작일 검증 서버->>DB: 규칙 한 건 저장 서버->>서버: 시작일~종료일(또는 +3개월) 범위로 인스턴스 펼침 loop 각 인스턴스 서버->>DB: 같은 사용자의 기존 슬롯과 겹침 검사 alt 겹침 없음 서버->>DB: 슬롯 저장 else 겹침 있음 서버->>서버: 조용히 건너뜀(로그만 남김) end end 서버-->>화면: 규칙 + 실제 만들어진 슬롯 수 화면-->>U: 등록 토스트 + 캘린더 복귀

무기한 반복은 매일 자동으로 연장된다

규칙에 종료일을 두지 않으면 서버는 처음 3개월치만 펼친다. 그 다음부터는 매일 새벽 스케줄러가 마지막 인스턴스 날짜를 확인해 앞으로 3개월씩 미래로 밀어준다.

사용자 시나리오

기본 흐름

매주 평일 점심을 정기로 비워두기

반복 규칙 화면에서 주기 "매주", 요일 월·화·수·목·금, 시간 12:00-13:00, 시작일 내일, "종료일 없음" 유지. 저장하면 다음 3개월치(약 65건)가 한 번에 캘린더에 펼쳐지고, 그 이후는 매일 새벽 자동으로 연장된다.

이 흐름이 남기는 것 — "매주 같은 시간"이라는 약속이 한 번의 등록으로 모든 미래에 적용된다. 사용자가 매번 등록을 반복할 필요가 없다.

격주

격주 토요일 오전만

주기 "격주", 요일 토, 시간 10:00-12:00, 시작일 토요일. 저장하면 그 토요일부터 두 주 간격으로 슬롯이 펼쳐진다. 매 주가 아니라는 점이 단일 가용 시간이나 매주 반복으로 표현할 수 없는 가치다.

매월

매월 25일 점심

주기 "매월", 시작일 5월 25일. 매월 25일에 슬롯이 한 건씩 생긴다. 2월처럼 25일 이후로 일수가 부족한 달은 그 달의 마지막 날로 자동 보정된다.

미리보기

저장 전에 몇 개가 생기는지 확인

편집 모드에서 "미리보기"를 누르면 서버가 펼친 결과를 미리 받아 "총 N개의 가용 시간이 생성됩니다" 안내한다. 신규 모드에서는 화면이 추정만 보여 주고, 실제 결과는 저장 후에 확정된다 — 추정과 실제가 다를 수 있다.

수정

규칙의 시간을 바꾸면 기존 슬롯이 모두 다시 만들어진다

매주 평일 12:00-13:00을 11:30-13:00으로 바꾸려 한다. 저장하면 서버는 기존 펼침 슬롯 전체를 지우고 새 시간으로 다시 펼친다. 이 과정에서 단일 가용 시간과 겹치는 인스턴스는 건너뛰어 빈 칸이 생긴다.

이 흐름이 남기는 것 — 규칙은 진실 공급원이고, 슬롯은 그 결과물이라는 분리. 사용자가 그 사실을 의식하지 않아도 결과는 항상 규칙과 일치한다.

삭제

이 반복을 더 이상 안 한다

편집 화면 "삭제" → 확인 → 규칙과 펼쳐진 모든 슬롯이 한 번에 사라진다. "이번만", "이후만" 같은 부분 선택지는 현재 제공하지 않는다 — 항상 전체 삭제.

충돌

일부 날짜에 이미 단일 슬롯이 있다

매주 평일 12-13시 규칙을 등록할 때 5/12에 이미 단일 가용 시간 12:00-13:30이 있다. 5/12 인스턴스는 서버가 조용히 건너뛰고, 다른 날짜는 정상 등록. 사용자에게는 "성공" 토스트만 보이고 "1건이 건너뛰어졌습니다" 같은 부분 충돌 안내는 별도로 가지 않는다.

이 흐름이 남기는 것 — 부분 충돌이 등록 자체를 막지 않는다는 정책. 다만 사용자가 인지하지 못한 채 인스턴스 한두 건이 빠지는 가능성을 안고 간다.

운영자가 알아야 할 것

규칙·인스턴스 분리가 흐트러지면 사용자 인지와 실제 캘린더가 어긋난다
  • 삭제는 항상 전체 삭제다. 사용자가 "이 일정만 삭제", "이후 모든 일정 삭제"를 기대할 수 있지만 현재 서버 정책은 그 분기를 지원하지 않는다. 화면이 그 선택지를 제시하면 사용자에게 거짓 약속을 하게 되므로, 화면도 단일 "삭제"만 두고 있다. "이번만 빼고 싶다"는 사용자에게는 단일 슬롯 삭제(F10-03)를 안내하는 우회로가 유일하다.
  • 수정은 "삭제 + 재펼침"으로 동작한다. 시간 변경 한 번이 기존 인스턴스 전체 삭제 + 재생성을 일으킨다 — 단일 가용 시간과 충돌하는 일자에서는 펼침이 건너뛰어진다. 사용자가 수정 후 "왜 어떤 날만 빈 칸이지" 라고 인지할 가능성이 있다.
  • 부분 충돌은 사용자에게 가시화되지 않는다. "성공" 토스트만 보이고 N건이 건너뛰어졌다는 안내는 가지 않는다. 사용자가 한참 뒤에 캘린더에서 빈 자리를 발견하는 회귀 가능성이 있다 — 사용자 피드백을 보고 부분 충돌 안내 추가를 검토할 시점.
  • 무기한 반복은 새벽 스케줄러에 의해 미래가 자동으로 채워진다. 운영 측이 이 스케줄러를 멈추면 한 달 안에 사용자 캘린더의 미래 슬롯이 비기 시작한다. 분산 락이 풀리지 않거나 무한 대기에 빠지면 무기한 규칙 사용자 전체에 영향이 가는 단일 실패 지점이다.
  • 요일 표기 일관성. 본 반복 일정 전용 화면은 약어(MON, TUE 등)로 서버와 일치하지만, 단일 가용 시간 화면에서 켜는 인라인 반복 토글이 풀네임(MONDAY 등)을 사용하는 비대칭이 있어 등록 자체가 실패할 가능성이 있다 — 화면 수정이 후속 과제로 남아 있다.
  • 매월 주기는 요일을 사용하지 않는다. 화면에 요일 칩이 노출되더라도 매월 주기에서는 무시되고 시작일의 일자가 기준이 된다. 사용자 입장에서 요일 칩이 보이는데 무시된다는 사실이 혼란을 줄 수 있다.
