F06-02. 포인트 충전 (외부 카드 결제)

서비스의 모든 유료 행위(이벤트 참가비, 호스팅 티켓, 멤버십)는 지갑 포인트로 처리된다. 외부 카드 결제가 직접 이루어지는 자리는 오직 이 화면 하나다. 사용자가 금액을 정하고 카드 결제를 마치면 결제 결과가 서버로 두 경로(앱과 카드사 알림)로 들어와 어느 쪽이 먼저 도착하든 잔액이 정확히 한 번만 늘어난다. 이 흐름이 어긋나면 사용자에게는 즉시 "결제했는데 포인트가 없네", "두 번 결제됐네" 같은 신뢰 손상으로 나타난다.

이벤트 결제, 티켓 구매, 구독 모두 포인트에서 차감된다. 외부 카드 결제는 오직 이 충전 화면 하나에서만 일어난다. 그 결과 카드사 연동의 위험 — 한 결제가 두 번 반영되거나, 결제했는데 포인트가 안 들어오거나, 환불이 어긋나거나 — 이 한 자리에 집중되고, 나머지 결제 흐름은 잔액만 다루는 단순한 동작으로 분리된다. 이 분리가 무너지면(예: 이벤트 결제 화면에서 카드 결제를 직접 받게 되면) 카드사 위험이 도메인 전체로 흩어지면서 한 번의 회귀가 다섯 화면에서 동시에 터진다.

이 화면이 매출에 미치는 영향

충전이 막히면 매출이 멈춘다 — 잔액 부족이 어떻게 다시 충전 의도로 돌아오는가
  • 이벤트 결제·티켓 구매·구독에서 잔액이 부족하면 사용자는 본 화면으로 자동으로 옮겨진다. 그 시점에 이미 사용자는 "사고 싶다"는 의도가 분명한 상태고, 본 화면은 부족한 금액을 미리 채워 놓고 안내 띠를 띄워 — 사용자가 입력에 시간을 거의 쓰지 않게 한다.
  • 충전 화면이 막히거나 느려지면 그 직전 단계의 결제가 함께 멈춘다. "충전이 안 돼서 모임도 못 신청한다"는 신호가 곧장 매출 손실로 이어진다.
  • 자동 충전(F06-05)이 활성 상태인 사용자는 본 화면에 거의 들어오지 않는다. 본 화면이 매출에 영향을 주는 비중은 자동 충전 비활성 사용자 비중에 비례한다 — 자동 충전 가입을 권하는 동선의 효과가 본 화면의 진입 빈도로 측정된다.

충전이 잔액에 안착하는 흐름

사용자가 금액을 정하고 카드 결제를 마치면, 결과가 두 경로로 서버에 도착한다. 앱이 받은 결제 성공 신호와 카드사가 직접 보내는 알림. 둘 중 어느 쪽이 먼저 도착해도 잔액은 정확히 한 번만 늘어난다.

사용자 시나리오

기본 흐름

지갑에서 들어와 50,000원 충전

잔액이 모자라다고 느낀 사용자가 지갑의 "충전하기"를 누른다. 프리셋 칩(예: 5,000 · 10,000 · 30,000 · 50,000) 중 50,000을 누르면 입력칸이 채워지고 "충전 후 잔액 75,000원"이 미리 보인다. 결제수단은 기본으로 설정된 카드가 이미 골라져 있어 별도 선택 없이 "충전하기"로 이어진다. 카드 결제 창이 떠오르고 결제 완료 → 성공 안내 → 지갑으로 돌아오면 새 잔액이 보인다.

이 흐름이 남기는 것 — 충전에서 사용자가 결정해야 할 변수(금액·카드)를 한 화면 안에 모아, 카드 결제 창에 갈 때까지 두 번 이상의 탭이 거의 들어가지 않게 한다.

자동 진입

이벤트 결제 직전에 잔액이 모자라 본 화면으로 옮겨짐

8,000원 모임에 신청하려는데 잔액이 5,000원밖에 없는 사용자. 결제 시도가 잔액 부족으로 막히면 본 화면이 자동으로 열리고, 부족한 3,000원이 입력칸에 미리 채워져 있다. 화면 상단에 "잔액 부족" 안내 띠가 한 줄 떠 있어 사용자가 "왜 지금 충전 화면이 떠 있는지"를 즉시 인지. 사용자가 그대로 충전하거나, 더 큰 금액을 직접 입력해 한 번에 채워 놓는다.

이 흐름이 남기는 것 — "사고 싶다"는 의도가 가장 강한 시점에 변수를 최소로 줄여 충전 전환을 높인다. 이 자동 진입이 끊기면 그 직전 단계의 매출이 즉시 빠진다.

실패

카드 한도 초과로 결제 실패

카드 결제 창에서 한도 초과로 실패한다. 본 화면이 카드 결제 창을 닫고 실패 안내 다이얼로그를 띄운다. 사용자가 다른 결제수단으로 다시 시도하면 새 충전 시도가 만들어진다 — 앞서 실패한 충전은 잔액에 영향을 주지 않고 서버에서 일정 시간 뒤 자동 정리.

이 흐름이 남기는 것 — 카드사 측 실패가 사용자 잔액·과금에 영향을 주지 않게 격리.

중복 방지

앱과 카드사 알림이 거의 동시에 도착

결제가 끝나면 두 경로로 서버에 결과가 도착한다 — 앱이 보낸 성공 신호와 카드사가 직접 보내는 결제 알림. 서버는 같은 주문이면 결과를 한 번만 잔액에 반영하고, 두 번째 도착은 첫 결과를 그대로 돌려준다. 사용자에게는 어떤 경우든 잔액이 한 번만 늘어난 모습으로 보인다.

이 흐름이 남기는 것 — "결제는 한 번인데 잔액이 두 번 늘었다"는 사고가 발생하지 않게 막는 가드. 이 가드가 풀리면 매출·회계 모두 어긋난다.

취소

실수로 충전한 금액을 한 푼도 안 쓰고 취소

50,000원을 충전한 직후 "실수였다"고 인지한 사용자. 거래 내역에서 해당 충전을 열면 "충전 취소" 버튼이 보이고, 누르면 카드사에 결제 취소를 요청하고 잔액에서도 다시 빼낸다. 사용자에게는 환불 거래가 거래 내역에 한 줄 추가되고, 카드 환불은 영업일 기준 진행된다.

이 흐름이 남기는 것 — 사용자에게 "실수했을 때 되돌릴 수 있다"는 통제감.

취소 거절

일부 사용한 충전을 취소하려 함

50,000원 충전 후 10,000원을 결제에 썼다. 이 상태에서 "충전 취소"를 시도하면 거절된다. 본 화면이 사용자에게 "이미 일부 사용한 충전은 취소할 수 없다"는 안내를 보여 준다 — 회계 추적의 일관성을 위한 정책.

이 흐름이 남기는 것 — 충전이 결제에 섞여 들어간 뒤에는 단순 취소가 아닌 환불(F06-06)을 거치게 한다.

중단

카드 결제 창에서 앱을 강제 종료

카드 결제 창이 뜬 상태에서 앱이 꺼졌다. 결제 결과가 서버에 도달하지 못한 채로 충전이 대기 상태로 남는다. 서버가 일정 시간 뒤 이 대기 충전을 자동으로 실패 처리한다 — 잔액에 들어오지 않고 카드사에도 청구되지 않는다.

화면에서 사용자가 인지하는 것

영역사용자가 보는 것그 자리에 둔 이유
잔액 부족 안내 띠 자동 진입일 때만 화면 상단에 한 줄. "잔액이 부족합니다" 같은 사유와 미리 채워진 금액. 사용자가 "내가 왜 지금 충전 화면에 있는지"를 즉시 인지하게 — 외부 카드 결제 직전에 맥락이 깨지는 것이 가장 빠른 이탈 요인.
금액 프리셋 4개 자주 쓰는 금액(예: 5,000 · 10,000 · 30,000 · 50,000) 칩. 사용자별로 자주 쓴 금액이 별도로 더 노출되기도 함. 충전은 의사결정 변수를 최소화하는 것이 핵심. 칩 하나로 입력 단계가 끝나면 카드 결제 창까지의 거리가 짧아진다.
직접 입력칸 천 단위 콤마 자동, 최소 1,000원 · 최대 500,000원 한도. 프리셋이 안 맞는 사용자에게 자유로운 금액을 받는다. 한도는 사용자가 실수로 큰 금액을 입력하는 사고를 막는 안전장치.
충전 후 잔액 미리보기 "충전 금액 N원 / 충전 후 잔액 M원" 박스. 사용자가 카드 결제 창에 가기 전에 결과를 한 번 더 확인. 잘못 입력한 금액으로 결제를 진행하는 사고를 줄인다.
결제수단 선택 기본 결제수단이 미리 선택되어 있고, "변경" 링크가 작게. 기본값으로 동작을 끝낼 수 있어야 한다 — 매 충전마다 카드를 다시 고르게 하면 충전 빈도가 떨어진다.
자동충전 설정 링크 화면 하단에 "자동충전 설정 →" 작은 링크. 본 화면에 자주 들어오는 사용자가 자동충전으로 옮겨 가게 안내. 본 화면 진입 빈도가 자동충전 가입의 신호다.
카드 결제 창 화면 위에 외부 카드사 결제 화면이 떠오름. 백 버튼은 결제 종료를 묻는 동작으로만 사용. 카드사 결제 화면 안에서 사용자가 실수로 빠져나오면 결제가 어중간하게 남는 사고가 일어난다 — 명시적 종료만 허용.

측정해야 하는 것

본 화면이 정상이라고 부르려면 이 다섯 가지 지표가 추적되어야 한다
  • 충전 화면 진입 → 충전 완료 전환율 — 입력 단계에서 사용자가 떨어지는 비율. 외부 카드 결제까지 갔다가 돌아오는 비율과 분리해 측정.
  • 자동 진입(잔액 부족) → 충전 완료 전환율 — 결제 직전의 자동 진입은 일반 진입보다 전환이 훨씬 높아야 한다. 같다면 안내 띠·미리 채워진 금액이 사용자 의도를 받아 내지 못한다는 신호.
  • 카드 결제 창 실패율 — 카드사 측 실패가 사용자 결제수단의 문제인지 카드사 연동의 문제인지 분리해야 한다.
  • 대기 상태로 남았다가 자동 정리된 충전 건수 — 사용자가 카드 결제 창에서 중단한 비율. 이 수치가 높으면 결제 창의 진행이 사용자에게 어렵다는 신호.
  • 충전 직후 사용까지의 시간 — 충전한 포인트가 얼마나 빨리 결제로 이어지는지. 짧을수록 충전이 매출의 직접 신호로 작동하고 있다는 의미.

운영자가 알아야 할 것 — 비즈니스 리스크

본 화면에서 무너지면 결제 도메인 전체가 무너진다
  • 잔액이 한 번의 결제로 두 번 늘어나서는 안 된다. 앱과 카드사 알림이 동시에 도착하는 경우에도 결과는 한 번만 반영되어야 한다. 이 가드가 풀리면 사용자 회계와 플랫폼 회계가 동시에 어긋난다.
  • 외부 카드 결제는 본 화면에서만 일어나야 한다. 이벤트·티켓·구독 결제 화면에서 카드사 결제 창이 떠오르는 회귀가 들어가면, 본 화면이 격리하던 카드사 위험이 도메인 전체로 흩어진다.
  • 잔액 부족 자동 진입이 깨지면 그 직전 단계의 매출이 함께 멈춘다. 이벤트 결제·티켓 구매·구독 시도가 잔액 부족으로 거절되는 모든 흐름은 본 화면으로 정확히 옮겨져야 한다. 사용자가 충전 화면을 직접 찾아가게 만들면 전환이 크게 떨어진다.
  • 일부 사용한 충전의 취소는 절대 허용하지 않는다. 이 가드가 풀리면 회계 추적이 어긋난다 — 충전·결제·환불의 회계 분개가 서로 맞지 않게 된다.
  • 대기 상태 충전은 일정 시간 안에 정리되어야 한다. 자동 정리가 멈추면 사용자 거래 내역에 "결제 중" 상태로 쌓여 사용자 신뢰가 떨어진다.
  • 충전 금액 한도(1,000~500,000원)는 사용자 사고 방지용이지 매출 한도가 아니다. 운영 결정으로 한도를 바꿀 때, 그 변경이 자동충전(F06-05)의 한 회 충전 금액 한도와 정합하게 움직여야 한다.
