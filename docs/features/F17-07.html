F17-07. 고정형 결제 (지갑·계좌이체)

결제 대기 상태가 된 멤버가 코스 전체 금액을 한 번에 납부한다. 두 가지 방식 — 지갑(포인트)은 즉시 차감되고 회계에 잡혀 자동 참가 확정으로 이어진다. 계좌이체는 호스트가 직접 받는 외부 송금이라 회계에 잡히지 않고, 호스트가 입금을 확인해야 참가 확정으로 마감된다. 멤버 한 명에게는 동시에 살아 있는 결제가 최대 1건뿐이라, 같은 화면이 두 번 열려도 두 번 결제되지 않는다.

두 방식의 회계상 차이

지갑 결제는 플랫폼이 중개하므로 매출·수수료·정산이 자동으로 잡힌다. 계좌이체는 호스트가 직접 받는 외부 송금이라 회계에 안 잡히고 정산 대상도 아니다(호스트 직거래). 그래서 환불도 지갑 결제는 자동, 계좌이체는 호스트가 직접 송금하고 "환불 완료"를 눌러야 마감된다.

사용자 시나리오

지갑 결제 happy

잔액 충분 → 즉시 확정

사용자가 결제 화면에서 "지갑" 선택. 잔액이 코스 가격보다 많아 즉시 차감되고 "참가 확정"으로 들어간다. 모든 회차 자동 참석 명단에 등록.

이 흐름이 남기는 것 — 가입 결정 후 가장 짧은 동선으로 자리 확보. 사용자가 결제 화면을 두 번 보지 않는다.

잔액 부족

지갑 결제하려는데 모자라다

지갑 선택 후 "결제하기" 누르면 "잔액이 부족해요 — 충전하기" 안내. 충전 화면에서 부족분 자동 입력 + 충전 완료 후 결제 화면으로 복귀.

이 흐름이 남기는 것 — 잔액 부족 사용자가 다른 모임으로 이탈하지 않게, 충전 동선을 한 흐름으로 연결.

계좌이체 신청

계좌이체로 코스 가입

"계좌이체" 선택. 호스트가 별도로 알려준 계좌로 직접 송금한 뒤 "송금 완료" 누름. 결제는 "확인 대기" 상태로 들어가 72시간 카운트다운 시작. 자리는 예약된 채로.

이 흐름이 남기는 것 — 카드 결제가 어려운 사용자도 가입할 수 있는 경로 유지(특히 50~60대).

호스트 입금 확인

호스트가 송금을 확인했다

호스트가 멤버 화면에서 "입금 확인" 누름. 결제 상태가 "완료"로 바뀌고 멤버는 "참가 확정"으로. 모든 회차 자동 참석 명단 등록.

이 흐름이 남기는 것 — 호스트의 명시적 확인이 멤버 가입을 마무리, 무책임한 호스트 운영을 막는다.

호스트 거절

호스트가 입금을 못 확인했다

호스트가 "입금 확인 안 됨" 사유로 거절. 결제는 "취소"로 들어가지만 멤버는 "결제 대기"로 유지되어 다시 결제할 수 있다.

이 흐름이 남기는 것 — 송금 누락·잘못된 송금자명 등 실수에 한 번 더 기회.

72시간 만료

송금 안 함·확인 안 됨

72시간이 지나도 결제 미완료면 자동 만료. 멤버 상태가 "결제 만료"로 가고 자리는 다음 대기자에게 자동 승격된다.

이 흐름이 남기는 것 — 자리가 무기한 점유되지 않게, 정원을 다른 사용자에게 흐르게 한다.

중복 결제 차단

실수로 결제 화면을 두 번 열었다

같은 멤버에게 살아 있는 결제가 이미 있으면 새 결제 시도가 거절된다. "이미 진행 중인 결제가 있어요" 안내.

이 흐름이 남기는 것 — 사용자가 우발적으로 두 번 결제되어 환불 분쟁이 일어나는 일을 차단.

화면에서 사용자가 인지하는 것

영역사용자가 보는 것그 자리에 둔 이유
결제 방식 선택"지갑·계좌이체" 라디오. 지갑 선택 시 현재 잔액이 같이 노출.잔액이 부족할 때 사용자가 한 번에 인지하게.
코스 총액"총 80,000원" 큰 글씨. 회차당 가격이 아니라 코스 전체.코스 단위 결제임을 강조해, 사용자가 "회차당 가격"으로 오인하지 않게 한다.
계좌 정보계좌이체 선택 시 호스트가 입력한 계좌·예금주·입금자명 가이드.외부 송금이므로 사용자가 자기 송금 책임을 인지하게.
72시간 카운트다운결제 대기 상태에서 실시간 카운트다운.기한 만료 = 자리 빼앗김. 시각적 긴급성으로 사용자 행동 유도.
잔액 부족 안내"X원 부족 — 충전하기" 버튼.충전 동선을 한 흐름으로 묶어 이탈 방지.
중복 결제 차단이미 살아 있는 결제가 있으면 새 결제 버튼 비활성.버튼 단계 차단으로 실수 결제 방지.

운영자가 알아야 할 것

결제·환불 책임 분리
  • 지갑 결제는 정산 대상 — 호스트 수익으로 잡혀 종료 시 정산에 포함.
  • 계좌이체는 정산 대상 아님 — 호스트가 직접 받고 직접 환불하는 외부 거래. 플랫폼은 상태 추적만 한다.
  • 잔액 부족 → 충전 화면 → 재시도 — 한 흐름으로 묶여 있어 사용자 이탈을 줄인다.
  • 결제 만료 시 대기자 자동 승격 — 자리는 항상 다음 대기자에게 흘러간다.
  • 변동형은 본 화면 사용 안 함 — 변동형 회차는 일반 이벤트 결제 흐름을 그대로 쓴다.

레드 플래그

분쟁이 일어날 수 있는 케이스
  • 계좌이체 입금 확인 무대응 — 호스트가 확인 안 하면 자리가 예약된 채 72시간 흘러간다. 다른 대기자가 못 들어옴.
  • 계좌이체 환불 마감 무대응 — 호스트가 송금만 하고 "환불 완료"를 안 누르면 멤버 입장에서 영구 "환불 진행중".
  • 지갑 잔액 부족 → 충전 동선 이탈 — 충전 화면에서 멤버가 다른 화면으로 새면 가입 미완료.
  • 잘못된 송금자명 — 호스트가 누가 누구 송금인지 매칭이 안 되면 입금 확인이 늦어지거나 거절될 수 있음.
