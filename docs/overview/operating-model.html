PRD 운영 모델 — Source-first

2026-05-18부터 기능 PRD의 작성 기준이 바뀌었다. “시나리오 수 나열”에서 “실제 서버·Flutter 소스를 직접 대조한 판단 문서”로 옮겨갔다. 이 페이지는 그 새 운영 모델의 골자를 한 자리에 정리한다.

왜 모델을 바꿨나

기존 자동 생성 PRD는 “…를 안전하게 처리해야 한다”, “이해 가능한 결과를 볼 수 있어야 한다” 같은 범용 문장으로 가득 차 있어, 읽고 나도 “이 기능이 지금 어디까지 동작하는가”에 답할 수 없었다. 새 모델은 그 답이 PRD 한 페이지 안에서 나오도록 강제한다.

새 PRD의 6가지 질문 — 모두 답할 수 있어야 PRD다
  1. 이 기능은 현재 실제로 어디까지 동작하는가
  2. 서버 계약과 Flutter가 기대하는 계약이 맞는가
  3. 상태·권한별로 버튼·화면이 어떻게 달라지는가
  4. 사용자가 피해를 볼 수 있는 Gap/Risk는 무엇인가
  5. 어떤 테스트가 이 판단을 뒷받침하는가
  6. 제품·운영·법무 결정 없이는 못 닫는 항목은 무엇인가

5개 라벨로 사실과 판단을 분리

새 PRD는 한 문장이 어느 라벨에 속하는지 항상 분명해야 한다.

라벨정의
Fact실제 소스에서 직접 확인한 사실. Controller 경로, VO 필드, enum 값, Repository 호출 등.
Derivedbackend/frontend 대조로 도출한 판단. 사실은 아니지만 사실에 기반한 추론.
Gap서버·프론트·테스트 사이의 불일치, 미구현, 미검증.
Risk지금 당장 실패하지 않아도 사용자 피해·회귀 가능성이 큰 지점.
Decision Needed사업·운영·법무·외부 credential 없이는 닫을 수 없는 항목.

금지 패턴 — 이 문장이 보이면 PRD가 아니다

기능 PRD에 나타나면 자동 생성 초안으로 간주
  • “해당 상황에서 사용자가 이해 가능한 성공/실패 결과와 다음 행동을 볼 수 있어야 한다.”
  • “사용자가 이 조건을 인지하거나 시스템이 자동으로 안전하게 처리해야 한다.”
  • “조건부/없음”만 있고 실제 영향 판단이 없는 영향 범위 표.
  • source path 없이 “서버에서 처리한다”, “앱에서 처리한다”라고 쓰는 문장.
  • 서버 enum에 없는 상태를 제품 상태처럼 확정해서 쓰는 문장.

새 PRD 표준 템플릿

기능 PRD는 다음 10개 섹션으로 구성한다. 비어 있는 섹션은 “없음”이라고 명시한다.

#섹션내용
1결론3–5문장. 어디까지 닫혀 있고 어디가 미구현인지.
2실사 근거Backend·Frontend·Verification 소스 경로 표.
3전체 동작 흐름사용자 진입부터 화면 결과까지 단계별.
4서버 계약endpoint·요청·응답·enum·에러·side effect.
5프론트 계약route·provider·repository·screen·widget·CTA.
6상태·권한 매트릭스사용자/상태 × 서버 근거 × 프론트 분기 × 사용자 결과.
7정합성 판단항목 × Backend × Frontend × 판단.
8Gap / Risk등급·항목·근거·영향·다음 조치.
9수용 기준Given/When/Then 또는 관찰 가능한 조건.
10미결정 / 후속사업·운영·법무·credential·테스트 보강 분리.

품질 체크리스트 — PRD 한 건이 완료되려면

  • Backend source — Controller·Service·VO/DTO·enum 중 해당 기능 파일을 직접 확인했다
  • Frontend source — API·Repository·Provider·Screen/Widget 중 해당 기능 파일을 직접 확인했다
  • Contract comparison — 서버 응답 필드와 클라이언트 소비 필드를 비교했다
  • State matrix — 상태·권한별 CTA 또는 차단 동작이 표로 정리되어 있다
  • Gap/Risk — 발견된 불일치·미구현·미검증이 숨겨지지 않고 등급이 붙어 있다
  • Acceptance criteria — 범용 문장이 아니라 관찰 가능한 조건으로 작성됐다
  • Verification — 현재 검증 자산과 검증 공백이 분리되어 있다

이 시각화 문서와의 관계

이 시각화 묶음(prd_visualization/)은 제품 구조와 인벤토리를 빠르게 훑는 입구이지, 기능별 source-of-truth가 아니다. 특정 기능의 현재 계약·trace·Gap/Risk가 필요하면 반드시 02_feature_prds/의 해당 기능 PRD와 units/<domain>/<feature>/ 실사 자료를 본다.
