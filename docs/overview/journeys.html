7대 사용자 여정

기능 목록만 보면 “화면이 많다”라는 인상만 남는다. 이 페이지는 사용자가 실제로 무엇을 하는지의 흐름으로 제품을 다시 본다. 각 여정은 시작 상태 → 사용자 행동 → 시스템 판단 → 종료 상태 순으로 읽는다.

여정 1. 모임 발견에서 리뷰까지

참가자의 가장 흔한 골든 패스. 발견·신청·결제·체크인·리뷰의 다섯 단계를 거친다.

flowchart LR A["홈·검색에서 이벤트 발견"] --> B["이벤트 상세 확인"] B --> C{"신청 조건 확인"} C -->|"무료/즉시 참석"| D["참석 확정"] C -->|"승인 필요"| E["신청서 제출"] C -->|"유료"| F["지갑 결제"] E --> G["호스트 승인/거절"] F --> D G -->|"승인"| D G -->|"거절"| H["참석 불가"] D --> I["캘린더 반영"] I --> J["현장 QR 체크인"] J --> K["리뷰 작성"] K --> L["신뢰·취향 데이터 반영"]

단계별 체크포인트

단계확인할 것
발견이 이벤트가 추천·검색 결과에 왜 노출됐는지 사용자가 짐작할 수 있는가
상세비로그인·호스트·참가자에게 CTA가 어떻게 달라지는가
신청승인 필요·정원 초과·대기열·유료 조건이 어떻게 갈라지는가
참석캘린더·알림·위치·길찾기 연결이 자동으로 일어나는가
체크인체크인 가능한 시간 게이트와 권한은 무엇인가
리뷰실제 참석자만 작성 가능한가, 중복 작성은 막히는가

여정 2. 호스트가 이벤트를 만들고 운영한다

호스트의 풀 사이클. 초안 작성·발행·신청 처리·운영·종료까지.

flowchart TD A["초안 작성"] --> B["발행"] B --> C["모집중"] C --> D{"신청 처리 방식"} D -->|"자동 참석"| E["정원·대기열 자동 관리"] D -->|"호스트 승인"| F["신청서 검토"] E --> G["참가자 확정"] F --> G G --> H["공지·일정 변경"] G --> I["QR 체크인"] I --> J["종료"] J --> K["리뷰 유도"] J --> L["정산 생성"]
주제호스트가 가장 자주 묻는 질문
생성필수 입력을 다 채우기 전 발행을 막는가
모집승인제·선착순·대기열 정책을 사용자에게 어떻게 보여주는가
운영일정 변경·취소 시 누구에게 어떤 알림이 가는가
종료정산·리뷰·사진첩으로 어떻게 이어지는가

여정 3. 클럽 가입에서 공동 정산까지

클럽은 단발 이벤트가 아니라 “관계가 이어지는” 단위다. 멤버 활동이 누적되고, 공동 비용이 생기면 정산으로 이어진다.

flowchart LR A["검색·추천에서 클럽 발견"] --> B["클럽 상세"] B --> C{"가입 방식"} C -->|"즉시 가입"| D["멤버"] C -->|"승인 필요"| E["가입 신청"] E --> F["운영자 승인"] F --> D D --> G["게시판·댓글·사진첩 활동"] D --> H["정기모임 참여"] H --> I["이벤트로 연결"] I --> J["공동 비용 발생"] J --> K["모임 정산"] K --> L["클럽 기금 반영 가능"]
단계확인할 것
가입공개 클럽 vs 승인제 클럽의 CTA 차이
권한소유자·관리자·일반 멤버·차단 사용자의 액션 차이
활동게시글·댓글·사진첩 신고와 삭제 권한
재무기금·후원·출금·공동 정산이 어디서 갈라지는가

여정 4. 정산 생성과 납부

모임이 끝난 뒤 돈을 나누는 흐름. 핵심은 “돈을 나누는 방식”이 아니라 “상태가 어떻게 흘러가는가”다.

sequenceDiagram actor H as 호스트 actor P as 참가자 participant S as 정산 화면 participant W as 지갑·계좌 participant N as 알림 H->>S: 종료된 모임에서 정산 생성 H->>S: 항목·참여자 선택 H->>S: 정산 활성화 S->>N: 참가자에게 납부 알림 P->>S: 내 분담금 확인 P->>W: 포인트 또는 계좌이체로 납부 W-->>S: 납부 상태 반영 H->>S: 계좌이체 건 확인 S->>N: 완료 또는 미납 알림
상태 우선 원칙

DRAFT는 호스트가 만들고 편집하는 단계, ACTIVE는 참가자에게 납부 요청이 열린 단계, COMPLETED는 모든 납부·확인이 끝난 단계, CANCELLED는 취소된 단계다. DRAFT 상태에서는 참가자에게 어떤 알림도 가서는 안 된다.

여정 5. 플랜 작성·판매·구매·활용

크리에이터와 구매자가 같은 데이터를 보지만, 다른 역할로 다르게 본다.

flowchart LR A["크리에이터가 플랜 초안 작성"] --> B["블록으로 코스 구성"] B --> C["발행 조건 검토"] C --> D["마켓 등록"] D --> E["구매자가 마켓에서 탐색"] E --> F["상세 확인"] F --> G["포인트 구매"] G --> H["내 컬렉션 저장"] H --> I["이 플랜으로 이벤트 만들기"] I --> J["사용 후 리뷰"]
주제질문
작성초안 상태에서만 편집 가능한 항목은 무엇인가
발행가격·커버·설명·블록 등 발행 요건은 무엇인가
구매이미 구매한 상품·잔액 부족·번들 일부 중복은 어떻게 처리하는가
활용구매한 플랜을 이벤트 생성으로 넘길 때 어떤 정보가 복사되는가

여정 6. 데이팅 매칭과 안전 흐름

다른 도메인과 가장 다른 영역. 안전·차단·인증이 모든 단계에 끼어든다.

flowchart TD A["본인 인증"] --> B["데이팅 프로필 작성"] B --> C["프로필 노출 ON"] C --> D["후보자 카드 확인"] D --> E{"상대에게 액션"} E -->|"LIKE"| F{"상호 LIKE?"} E -->|"PASS"| D F -->|"아니오"| D F -->|"예"| G["매칭 성사"] G --> H["채팅"] H --> I["만남 제안"] I --> J{"상대 응답"} J -->|"수락"| K["캘린더·장소 연결"] J -->|"거절"| H K --> L["만남 후 평가"] H --> M["차단 가능"] D --> M I --> M
단계확인할 것
인증미인증 사용자는 어디까지 볼 수 있는가
프로필노출 ON/OFF, 사진 개수, 소개 입력 조건
스와이프일일 LIKE 한도, 매칭 성사 모달, 후보자 없음 상태
채팅차단 시 채팅방·매칭 상태가 어떻게 바뀌는가
만남제안·수락/거절·장소·일정·안전 기능 노출 기준

여정 7. 알림과 설정의 공통 흐름

알림은 “기능”이 아니라 다른 기능의 “결과를 전달하는 보조 흐름”이다.

flowchart LR A["도메인 이벤트 발생"] --> B{"사용자 수신 설정"} B -->|"허용"| C{"방해금지 시간?"} B -->|"차단"| D["알림 미발송 또는 인박스만"] C -->|"아님"| E["푸시 발송"] C -->|"해당"| F["보류 또는 무음 처리"] E --> G["알림함 적재"] F --> G D --> G
기획 순서 주의

알림을 기획할 때는 “알림 화면에 무엇을 보여줄까”가 아니라 “어떤 사건이 발생했을 때 누구에게 어떤 문구로 알려야 하는가”를 먼저 정한다. 알림 화면 디자인은 그 다음 단계다.

여정 검토 기준

  • 여정을 그릴 때는 첫 화면에 들어오는 사람이 누구인지부터 적는다. 게스트인지, 로그인한 일반 사용자인지, 호스트·운영자처럼 책임이 있는 사람인지에 따라 흐름이 달라진다.
  • 여정 도중에 돈·위치·알림·리뷰 중 하나라도 움직이면 부수효과 매트릭스를 같이 펴 둔다.
  • 흐름이 중간에 막혔을 때 사용자가 그대로 멈추지 않도록 다시 시도, 이전 화면으로 복귀, 다른 길로 우회 중 적어도 하나는 열어 둔다.
