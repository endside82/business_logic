원문 PRD가 source-first 모델로 전환됐다. 기능 PRD는 이제 “시나리오 나열”이 아니라 실제 서버·Flutter 소스를 직접 대조한 판단 문서로 운영된다. 자세한 골자는 PRD 운영 모델에서, 168개 기능의 PRD 문서 상태와 Trace/Risk 후보 수는 PRD 문서 상태표에서 확인한다. 시각화 변경 내역은 변경 기록 페이지에 있다.

이 허브를 보는 법

왼쪽 사이드바는 “이해 → 도메인(7개 줄기) → 기능 → 정책 → QA” 순서로 정렬돼 있다. 도메인은 줄기별로 묶여 있다 — 들여쓰인 항목(정기모임·모임 정산·마일리지·경고)은 큰 줄기(이벤트·클럽)에 속한 위성 도메인이다. 처음이라면 서비스 정의 → 정보구조 → 7대 여정을 차례로 읽고, 담당 영역 도메인 페이지로 내려가는 흐름을 권장한다. 담당 기능을 직접 다루려면 운영 모델 → PRD 문서 상태표 → 해당 기능 PRD 순서로 본다.

community는 어떤 제품인가

한 줄 정의와 큰 흐름만 빠르게 확인한다. 자세한 정의는 서비스 정의 페이지에 있다.

community는 관심사 기반의 오프라인 활동을 찾고, 참여하고, 운영하고, 정산하고, 다시 이어가는 전 과정을 하나의 앱에서 처리하는 커뮤니티 플랫폼이다. 발견·참여만 다루는 SNS와 달리, 결제·정산·환불 같은 돈의 흐름과 리뷰·신고·차단 같은 신뢰 관리가 같은 비중으로 들어간다.

왜 이 문서가 필요한가

168개 기능이 한 데 모이면, 한 기능을 바꿀 때 어디가 같이 바뀌는지 잊어버리기 쉽다. 예를 들어 “이벤트 신청” 한 줄을 고치면 알림, 결제 흐름, 캘린더 동기화, 호스트 정산, 리뷰 자격이 동시에 영향받는다. 이 문서들은 그 연결을 머릿속이 아니라 종이로 남겨두기 위해 있다.

반대로, 이 문서로 결정하지 않는 영역이 있다. 사업 KPI, 출시 우선순위, 법무 최종 문구, 운영 SLA, UX 카피는 별도 의사결정이 필요한 사안이다. 여기서는 “기능이 동작하기 위한 정의”와 “QA가 검증할 수용 기준”까지만 다룬다.

모든 문서는 사이드바에서 바로 열 수 있다. 정적 HTML이라서 파일을 직접 더블클릭해도 동작한다.
