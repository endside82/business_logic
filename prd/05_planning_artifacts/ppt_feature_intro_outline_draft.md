# community 기능 소개 PPT 항목 초안

<!-- supporting-doc-status: 2026-05-18 -->

> 문서 상태: **보조 문서**. 기능별 현재 계약, source trace, Gap/Risk 판단은 [PRD_MIGRATION_STATUS.md](../PRD_MIGRATION_STATUS.md)와 각 기능 PRD를 우선한다. 이 문서는 인벤토리, 정책, QA, 기획 운영 기준을 보조하며, 기능 세부 판단은 [FEATURE_PRD_STANDARD.md](../FEATURE_PRD_STANDARD.md) 기준으로 재확인한다.

## 1. 문서 목적

이 문서는 `business_logic/prd`의 제품 PRD, 도메인 PRD, 사용자 여정, 영향 매트릭스, MVP 범위 문서를 바탕으로 기능 소개용 PPT에 들어갈 항목을 먼저 정리한 초안이다.

목표는 기능을 단순 나열하는 것이 아니라, 이 프로젝트가 왜 일반 모임 앱보다 넓은 사업 기회를 갖는지 보여주는 발표 흐름을 만드는 것이다.

## 2. 발표 핵심 메시지

community는 관심사 기반 오프라인 활동을 발견하는 앱에 머물지 않는다. 이벤트 참여, 클럽 운영, 결제, 정산, 플랜 거래, 캘린더, 위치, 리뷰, 신고, 신뢰 관리까지 연결해 오프라인 커뮤니티의 전체 운영 흐름을 하나의 플랫폼 안에 묶는다.

핵심 주장은 다음 세 가지다.

| 주장 | 설명 |
|---|---|
| 발견에서 재참여까지 닫힌 루프 | 홈/검색에서 시작한 활동이 이벤트, 클럽, 캘린더, 알림, 리뷰를 거쳐 다시 추천과 재참여로 돌아온다. |
| 운영과 돈 흐름까지 포함한 모임 인프라 | 단순 참가 모집이 아니라 정원, 대기열, 승인, 체크인, 지갑, 환불, 모임 정산, 클럽 기금까지 다룬다. |
| 신뢰와 안전을 제품 중심축으로 내장 | 리뷰, 신고, 신뢰점수, 데이팅 인증, 차단, 위치 공유 opt-in/out, 데이터 내보내기/삭제가 기능 PRD 안에 포함되어 있다. |

## 3. 권장 덱 구성

권장 분량은 15~17장이다. 외부 투자/사업 소개용이면 핵심 기능과 리스크 대응을 압축해 15장 내외로 구성하고, 내부 제품 리뷰용이면 상세 기능 부록을 붙여 17장 이상으로 확장한다.

| 순서 | 슬라이드 역할 | 제목 초안 | 핵심 내용 | 추천 시각화 |
|---:|---|---|---|---|
| 1 | Cover | 오프라인 커뮤니티의 발견, 운영, 결제, 신뢰를 하나로 묶는 플랫폼 | 제품 한 줄 정의와 14개 도메인, 117개 기능, 987개 시나리오의 범위를 짧게 제시한다. | 제품 루프를 감싸는 큰 원형 구조 |
| 2 | Problem | 기존 모임 경험은 발견 이후 운영과 신뢰가 끊긴다 | 모임 발견, 참가 관리, 돈 정산, 후기/신뢰, 재참여가 서로 다른 도구로 분산되는 문제를 정리한다. | 분절된 도구 체인 vs 통합 플랫폼 비교 |
| 3 | Product Thesis | community는 모임 앱이 아니라 오프라인 활동 운영체제다 | 가입/온보딩, 탐색, 참여/운영, 돈, 관계/신뢰, 개인정보/설정으로 이어지는 전체 정보구조를 소개한다. | 6개 플랫폼 축 맵 |
| 4 | Core Loop | 발견에서 리뷰까지 이어지는 재참여 루프가 제품의 중심이다 | 홈/검색 → 이벤트 상세 → 신청/결제 → 캘린더 → QR 체크인 → 리뷰/신뢰 → 추천으로 돌아오는 흐름을 설명한다. | 사용자 여정 루프 다이어그램 |
| 5 | Feature Cluster 1 | 이벤트는 단순 모집이 아니라 운영 상태를 관리한다 | 이벤트 생성, 승인제, 유료/무료, 정원/대기열, 일정 변경, 취소/환불, QR 체크인, 사진첩, 내 이벤트 로그를 묶는다. | 이벤트 라이프사이클 타임라인 |
| 6 | Feature Cluster 2 | 클럽은 일회성 모임을 장기 커뮤니티 자산으로 전환한다 | 클럽 탐색/가입, 멤버/권한, 게시판, 댓글, 공지, 사진첩, 클럽 이벤트, 기금, 후원, 구독, 출금을 소개한다. | 클럽 운영 스택 |
| 7 | Feature Cluster 3 | 결제와 정산은 플랫폼 신뢰의 핵심 인프라다 | 지갑, 포인트 충전, 결제수단, 자동충전, 포인트 결제/환불, 모임 정산 생성/활성화/납부/이체확인/이의제기를 정리한다. | 돈 흐름 상태도 |
| 8 | Feature Cluster 4 | 플랜 마켓은 오프라인 경험을 상품화하는 크리에이터 레이어다 | 플랜 작성, 블록 에디터, 발행, 마켓 등록, 구매, 컬렉션, 구매 플랜 기반 이벤트 생성, 리뷰를 설명한다. | 크리에이터 → 구매자 → 이벤트 전환 퍼널 |
| 9 | Feature Cluster 5 | 데이팅은 인증, 매칭, 만남, 안전을 하나의 폐쇄 루프로 다룬다 | 본인 인증, 데이팅 프로필, 후보자 스와이프, 매칭, 채팅, 만남 제안, 차단, 프로필 조회 이력을 소개한다. | 인증 기반 매칭/안전 플로우 |
| 10 | Trust & Safety | 리뷰, 신고, 차단, 신뢰점수가 커뮤니티 품질을 지킨다 | 이벤트 리뷰, 리뷰 수정/삭제, 신고, 신뢰점수, 취향 프로필, 차단 정책을 소개한다. | 신뢰 데이터가 도메인에 환류되는 구조 |

| 13 | Business Strength | 수익화 지점이 한 번의 참가비에 갇히지 않는다 | 참가비/포인트, 호스팅 티켓, 개인 구독, 클럽 기금/후원/구독, 플랜 마켓 구매, 크리에이터 경제를 사업 레이어로 정리한다. | 수익원 포트폴리오 맵 |

## 4. 핵심 기능 묶음

### 4.1 발견과 탐색

PPT에서는 홈 피드와 검색을 별도의 큰 기능으로 길게 설명하기보다, 전체 루프의 시작점으로 다루는 것이 좋다.

| 포함 기능 | 발표 포인트 |
|---|---|
| 홈 피드 | 추천 콘텐츠와 주요 진입점을 제공하는 첫 화면 |
| 검색/필터 | 이벤트, 클럽, 플랜 등 여러 콘텐츠를 한 번에 찾는 탐색 축 |
| 저장 검색 | 검색 결과가 없거나 새 결과가 필요할 때 재방문을 만드는 알림 루프 |
| 위시리스트 | 즉시 참가하지 않는 관심을 보존하는 경량 전환 장치 |

핵심 메시지: 탐색 기능은 단순 목록이 아니라 이벤트, 클럽, 플랜 마켓으로 사용자를 분기시키는 상단 퍼널이다.

### 4.2 이벤트 운영

이벤트는 가장 중요한 기능 군이다. 발표에서는 사용자의 참여 흐름과 호스트의 운영 흐름을 동시에 보여줘야 한다.

| 포함 기능 | 발표 포인트 |
|---|---|
| 이벤트 생성 | 초안/발행, 유료/무료, 사전결제, 반복, 프라이빗 조건 |
| 이벤트 상세 | 게스트, 참가자, 호스트에 따라 CTA가 달라지는 상태 기반 화면 |
| 신청/참석 | 승인제, 유료 승인제, 정원, 대기열, 자동 승격 |
| 신청서 승인/거절 | 호스트가 정원과 결제 상태를 함께 고려하는 운영 기능 |
| 정원/대기열 | 수동 승격, 자동 승격, 모집 마감/재개 |
| QR 체크인 | 실제 참석 증빙과 리뷰 자격의 근거 |
| 사진첩/내 이벤트 | 참여 후 기록과 회고, 재참여 동선 |

핵심 메시지: 이벤트 기능은 “모집 공고”가 아니라 신청, 결제, 참석, 현장 확인, 리뷰까지 이어지는 상태 머신이다.

### 4.3 클럽과 장기 커뮤니티

클럽은 일회성 이벤트와 구분되는 장기 리텐션 축이다.

| 포함 기능 | 발표 포인트 |
|---|---|
| 클럽 발견/상세/가입 | 자유가입, 승인가입, 대기열, 초대 |
| 멤버/권한 | 소유자, 관리자, 멤버, 비회원, 차단 사용자별 액션 차이 |
| 게시판/댓글/공지/사진첩 | 활동과 관계를 유지하는 커뮤니티 표면 |
| 클럽 이벤트/캘린더 | 클럽 내부 활동을 다시 이벤트로 연결 |
| 기금/후원/출금 | 커뮤니티 재무 흐름을 플랫폼 안에 편입 |
| 클럽 구독 | 장기 멤버십과 반복 매출 가능성 |

핵심 메시지: 클럽은 이벤트 이후 관계가 끊기는 문제를 해결하고, 운영자와 멤버가 반복적으로 돌아올 이유를 만든다.

### 4.4 결제, 지갑, 정산

결제와 정산은 이 프로젝트의 사업적 차별점이다. 일반 모임 앱에서 외부 계좌이체나 카카오톡 공지로 처리되는 영역을 제품화한다.

| 포함 기능 | 발표 포인트 |
|---|---|
| 지갑/포인트 | 충전, 잔액, 거래내역, 포인트 결제 |
| 결제수단/자동충전 | 결제 실패와 잔액 부족을 줄이는 인프라 |
| 환불 | 이벤트 취소, 선입금, 포인트 환불 조건 |
| 모임 정산 | DRAFT → ACTIVE → COMPLETED/CANCELLED 상태 |
| 분담금 납부 | 포인트/계좌이체/혼합 결제 가능성 |
| 이체 확인/이의제기/감사로그 | 돈 관련 분쟁을 줄이는 운영 장치 |

핵심 메시지: 돈 흐름을 상태 기반으로 다루기 때문에 신뢰, 운영 효율, 수익화가 함께 강화된다.

### 4.5 플랜 마켓

플랜 마켓은 다른 모임 앱과 가장 다르게 보일 수 있는 확장 기능이다.

| 포함 기능 | 발표 포인트 |
|---|---|
| 플랜 작성/블록 에디터 | 오프라인 코스와 모임 계획을 구조화 |
| 발행/마켓 등록 | 크리에이터가 콘텐츠를 상품화 |
| 구매/컬렉션 | 구매자가 검증된 플랜을 보관하고 재사용 |
| 플랜 기반 이벤트 생성 | 콘텐츠 구매가 실제 오프라인 모임 생성으로 이어짐 |
| 리뷰/통계 | 크리에이터 품질과 판매 가능성을 강화 |

핵심 메시지: community는 모임을 소비하는 앱이 아니라, 모임 기획 자체를 거래 가능한 자산으로 만든다.

### 4.6 데이팅과 안전

프라이빗 데이팅은 별도 앱처럼 보일 수 있으므로, 발표에서는 “오프라인 만남의 신뢰/안전 기능을 확장한 영역”으로 설명하는 것이 좋다.

| 포함 기능 | 발표 포인트 |
|---|---|
| 본인 인증 | 데이팅 진입의 신뢰 기반 |
| 프로필/스와이프/매칭 | 인증 사용자끼리 후보를 탐색하고 매칭 |
| 채팅/만남 제안 | 매칭 후 실제 만남으로 전환 |
| 캘린더/장소/안전 | 만남 일정과 장소가 플랫폼 기능과 연결 |
| 차단/조회 이력 | 안전과 통제권 강화 |

핵심 메시지: 데이팅 기능은 단순 매칭이 아니라 인증, 안전, 위치, 캘린더, 차단 정책까지 포함한 오프라인 만남 관리다.

### 4.7 신뢰, 알림, 캘린더, 위치

이 네 영역은 독립 기능이라기보다 전체 플랫폼의 연결 조직이다.

| 영역 | 발표 포인트 |
|---|---|
| 리뷰 & 신고 | 실제 참석자 리뷰, 신고, 신뢰점수, 취향 데이터 |
| 알림 | 상태 변화, 납부 요청, 승인/거절, 저장 검색, 방해금지, 권한 안내 |
| 캘린더 | 이벤트, 가용시간, 데이팅 만남을 시간축으로 통합 |
| 위치 & 길찾기 | 장소 안내, 참석자 위치 공유, opt-out, 프라이버시 제어 |

핵심 메시지: 사용자가 다음 행동을 놓치지 않게 만들고, 민감한 정보는 권한과 상태로 통제한다.

## 5. 다른 모임 앱 대비 차별점

특정 경쟁사를 직접 지목하기보다, 일반적인 기능 유형과 비교하는 방식이 안전하다.

| 비교 축 | 일반 모임/커뮤니티 앱 | community 차별점 |
|---|---|---|
| 발견 | 모임 목록과 카테고리 중심 | 홈, 검색, 저장 검색, 플랜 마켓, 클럽을 연결한 탐색 구조 |
| 참여 | 신청 또는 RSVP 중심 | 승인제, 유료 승인제, 정원, 대기열, 자동 승격, 취소/환불까지 관리 |
| 운영 | 운영자가 수동으로 공지/관리 | 신청서 검토, 정원 관리, QR 체크인, 알림, 캘린더, 참석 로그 제공 |
| 커뮤니티 | 이벤트 이후 관계가 약함 | 클럽, 게시판, 공지, 사진첩, 클럽 이벤트, 기금으로 장기 관계 유지 |
| 결제 | 외부 결제/계좌이체 의존 | 지갑, 포인트, 결제수단, 자동충전, 거래내역, 환불, 정산 상태 관리 |
| 정산 | 모임 후 별도 도구 필요 | 정산 생성, 납부, 이체 확인, 리마인드, 이의제기, 감사로그 내장 |
| 콘텐츠 자산 | 모임 자체만 소비 | 플랜을 작성, 판매, 구매, 컬렉션화하고 이벤트 생성에 재사용 |
| 신뢰/안전 | 후기나 신고가 부가 기능 | 리뷰 자격, 신고, 차단, 신뢰점수, 데이팅 인증, 위치 opt-in/out 설계 |
| 재방문 | 새 모임 알림 정도 | 캘린더, 알림, 저장 검색, 클럽 활동, 리뷰, 플랜 활용으로 반복 접점 생성 |

## 6. 비즈니스 강점

### 6.1 여러 사용자 집단이 같은 플랫폼 안에서 다른 가치를 얻는다

| 사용자 | 얻는 가치 | 사업적 의미 |
|---|---|---|
| 참가자 | 관심사 기반 모임 발견, 안전한 참여, 간편 결제/정산 | 사용자 획득과 반복 참여 |
| 호스트 | 이벤트 생성, 신청 관리, 체크인, 정산, 리뷰 확보 | 공급자 생산성 향상 |
| 클럽 운영자 | 멤버십, 게시판, 이벤트, 기금, 구독 운영 | 장기 커뮤니티 락인 |
| 플랜 크리에이터 | 플랜 상품화, 판매, 리뷰, 통계 | 크리에이터 공급 확장 |
| 데이팅 사용자 | 인증 기반 매칭과 안전한 만남 조율 | 고신뢰 오프라인 관계 확장 |

### 6.2 수익원이 단일 참가비에 의존하지 않는다

| 수익화 후보 | 연결 기능 |
|---|---|
| 포인트 충전/사용 | 지갑, 포인트 결제, 환불 |
| 유료 이벤트/사전결제 | 이벤트 생성, 신청/참석, 환불 정책 |
| 호스팅 티켓 | 호스트의 이벤트 생성/운영 |
| 개인 구독 | 결제 & 지갑의 구독 관리 |
| 클럽 후원/구독/기금 | 클럽 재무와 장기 멤버십 |
| 플랜 마켓 거래 | 크리에이터 플랜 판매, 구매자 컬렉션 |
| 정산/운영 프리미엄 가능성 | 모임 정산, 이체 확인, 감사로그, 리마인드 |

### 6.3 운영 리스크를 제품 정책으로 흡수한다

PRD는 정상 흐름만 정의하지 않는다. 권한, 상태 변경, 입력 검증, 외부 실패, 중복/동시성, 알림/결제/위치/신뢰 영향까지 검토한다. 이는 기능이 많다는 뜻보다, 실제 운영 중 생길 수 있는 분쟁과 장애를 제품 안에서 흡수하려는 설계라는 점이 중요하다.

### 6.4 데이터와 신뢰가 누적될수록 추천과 운영 효율이 좋아진다

온보딩 관심사, 이벤트 참석, 클럽 활동, 리뷰, 취향 평가, 저장 검색, 캘린더, 위치 동의 같은 데이터가 쌓이면 추천, 신뢰 판단, 운영 자동화의 기반이 된다. 이 누적 데이터는 단순 모임 목록 앱보다 방어력이 높은 자산이 될 수 있다.

## 7. 리스크와 극복 지점

이 프로젝트의 리스크는 기능이 많다는 사실 자체보다, 오프라인 활동에서 발생하는 상태, 돈, 권한, 안전, 개인정보를 한 플랫폼 안에서 일관되게 처리해야 한다는 점에 있다. 발표에서는 리스크를 숨기기보다 “어떤 난이도가 있고, PRD가 이미 어떤 방식으로 흡수하고 있는가”를 보여주는 편이 설득력이 높다.

| 리스크 | 왜 중요한가 | 극복 지점 | PPT 표현 방식 |
|---|---|---|---|
| 초기 범위가 넓어 MVP가 흐려질 수 있음 | 14개 도메인, 117개 기능, 987개 시나리오를 모두 첫 출시로 보이면 실행력이 약해 보인다. | 핵심 루프를 P0로 닫고, 체크인/사진/구독/위치/고급 추천/운영 SLA는 P1 이후로 단계화한다. | “비전은 넓고, 출시는 좁게”라는 로드맵 슬라이드로 처리한다. |
| 상태와 권한이 복잡함 | 이벤트 신청, 유료 승인제, 대기열, 클럽 권한, 정산, 데이팅 차단은 상태가 조금만 어긋나도 사용자 피해가 생긴다. | 상태 전이 문서, 역할/액션 매트릭스, QA 수용 기준을 기준으로 CTA와 허용 액션을 검산한다. | 복잡성을 약점이 아니라 운영 방어력의 근거로 제시한다. |
| 결제, 환불, 정산 분쟁 리스크 | 돈이 움직이는 기능은 PG 실패, 콜백 지연, 중복 결제, 환불 기준, 계좌이체 확인, 이의제기가 모두 사용자 신뢰와 직결된다. | 지갑/거래내역, 정산 상태, 이체 확인, 이의제기, 감사로그, 환불 정책을 제품 흐름에 포함한다. | “돈 흐름을 외부 대화가 아니라 제품 상태로 관리한다”는 메시지로 압축한다. |
| 운영/CS 부담이 커질 수 있음 | 신고, 환불, 정산 이의, 차단, 클럽 기금 출금, 플랜 심사는 사람이 판단해야 하는 구간이 남는다. | 결정 등록부에 운영 SLA, 환불 기준, 기금 운영, 플랜 심사를 별도 결정 항목으로 분리하고 출시 전 확정한다. | 미결정 리스크와 결정 책임자를 분리한 운영 준비표로 보여준다. |
| 안전과 개인정보 리스크 | 데이팅, 위치 공유, 신고/차단, 데이터 내보내기, 계정 삭제는 법무/정책 판단이 없으면 출시 리스크가 크다. | 본인 인증, 명시적 opt-in/out, 위치 공유 중지, 차단, 신고, 데이터 보관/삭제 문구를 핵심 정책으로 둔다. | “신뢰 기능이 부가 기능이 아니라 진입 조건”임을 강조한다. |
| 외부 시스템 의존성 | 소셜 로그인, 이메일, PG, 푸시, 지도/지오코딩, 파일 업로드 실패는 주요 흐름을 중단시킬 수 있다. | pending 상태, 재조회, 재시도, 실패 원인 분리, 인앱 알림함 fallback, OS 설정 안내를 수용 기준에 포함한다. | 외부 실패에도 사용자가 다음 행동을 알 수 있는 복구 설계를 보여준다. |
| 중복 실행과 동시성 | 정원 마지막 자리, 대기열 자동 승격, 결제 버튼 연타, PG 중복 콜백, 멀티 디바이스 토큰 갱신은 실제 운영에서 자주 터진다. | 버튼 잠금, 멱등 처리, 최신 상태 재조회, 상태 불일치 안내, 서버 기준 최종 판정을 원칙으로 둔다. | “오프라인 모임은 동시에 여러 사람이 움직이는 시스템”이라는 설명과 함께 제시한다. |
| 초기 공급 품질과 콜드스타트 | 이벤트, 클럽, 플랜 마켓은 양질의 호스트/운영자/크리에이터가 없으면 탐색 경험이 약해진다. | 초기에는 검증된 호스트/클럽/플랜을 선별하고, 리뷰/신뢰점수/플랜 리뷰를 통해 품질 데이터를 누적한다. | 기능 소개보다 공급자 생산성, 반복 운영, 품질 신호를 강조한다. |
| 수익화 가정 검증 필요 | 포인트, 유료 이벤트, 개인 구독, 클럽 후원/구독, 플랜 마켓 등 수익원이 많지만 실제 전환율은 아직 PRD 근거가 없다. | 첫 단계에서는 결제/정산 인프라와 핵심 GMV 지표를 먼저 검증하고, 구독/프리미엄은 사용 패턴 확인 후 확장한다. | “여러 수익원 후보”와 “검증 순서”를 분리해 과장으로 보이지 않게 한다. |
| 포지셔닝이 넓어 보일 수 있음 | 모임, 클럽, 정산, 플랜 마켓, 데이팅이 모두 들어가면 제품 정체성이 흐려질 수 있다. | 모든 기능을 “오프라인 활동의 발견 → 운영 → 돈 → 신뢰 → 재참여” 루프에 배치한다. | 기능 나열 대신 하나의 운영체제 프레임으로 묶는다. |

장표화할 때는 리스크를 별도 방어 슬라이드로만 두지 말고, 사업 강점과 연결하는 편이 좋다. 예를 들어 결제/정산 리스크는 동시에 수익화와 신뢰 인프라의 근거가 되고, 상태/권한 복잡도는 동시에 운영 방어력과 모방 난도의 근거가 된다.

## 8. 덱에서 강조할 차별화 문장 후보

- 모임을 찾는 앱이 아니라, 오프라인 활동을 끝까지 운영하는 플랫폼.
- 이벤트, 클럽, 플랜, 데이팅을 하나의 신뢰/결제/알림 인프라 위에 얹는다.
- 참여 전에는 발견과 신뢰, 참여 중에는 운영과 결제, 참여 후에는 리뷰와 재참여를 연결한다.
- 지갑과 정산을 내장해 오프라인 모임의 가장 번거로운 돈 흐름을 제품화한다.
- 플랜 마켓을 통해 좋은 모임 기획을 일회성 경험이 아니라 거래 가능한 콘텐츠로 전환한다.
- 클럽은 재방문을 만들고, 정산과 기금은 운영자의 지속 가능성을 만든다.
- 데이팅과 위치 공유는 인증, 차단, opt-in/out을 전제로 설계되어 안전을 기능 중심에 둔다.

- 게시판/댓글/공지/사진첩
- 클럽 이벤트
- 기금/후원/구독/출금

시각화:

- 클럽 운영 스택: Membership, Content, Events, Finance.

### Slide 7. Payment & Settlement

제목 후보: `돈 흐름을 내장할수록 플랫폼 신뢰와 수익화가 커진다`

넣을 항목:

- 지갑과 포인트
- 결제수단과 자동충전
- 포인트 결제/환불
- 정산 DRAFT/ACTIVE/COMPLETED/CANCELLED
- 이체 확인, 이의제기, 감사로그

시각화:

- 참가자와 호스트 사이의 돈 흐름 + 상태 레이어.

### Slide 8. Plan Market

제목 후보: `좋은 모임 기획을 거래 가능한 콘텐츠로 전환한다`

넣을 항목:

- 블록 기반 플랜 작성
- 발행/마켓 등록
- 구매/컬렉션
- 구매 플랜으로 이벤트 생성
- 리뷰와 크리에이터 통계

시각화:

- Creator → Market → Buyer → Event Creation → Review flywheel.

### Slide 9. Dating & Safety

제목 후보: `데이팅은 인증과 안전을 전제로 한 오프라인 만남 확장이다`

넣을 항목:

- 본인 인증
- 후보자 스와이프
- 매칭/채팅
- 만남 제안
- 캘린더/장소 연결
- 차단/조회 이력

시각화:

- 인증 게이트를 통과한 매칭 플로우.

### Slide 10. Trust Layer

제목 후보: `신뢰 기능은 부가 기능이 아니라 모든 활동의 품질 관리 장치다`

넣을 항목:

- 참석자 리뷰 자격
- 신고와 자기 신고/중복 신고 방지
- 차단
- 신뢰점수
- 취향 프로필
- 데이팅 인증과 안전 흐름

시각화:

- 리뷰/신고/차단/인증 데이터가 이벤트, 클럽, 데이팅에 환류되는 구조.

### Slide 11. Context Layer

제목 후보: `캘린더, 알림, 위치가 온라인 의도를 실제 행동으로 바꾼다`

넣을 항목:

- 통합 캘린더
- 가용 시간과 반복 규칙
- 알림 목록, 카테고리, 방해금지, 푸시 기기
- 위치 공유 opt-in/out
- 길찾기와 주소 관리

시각화:

- Time, Notification, Location 3축으로 이벤트/클럽/데이팅을 연결.

### Slide 12. Differentiation

제목 후보: `차별점은 기능 수가 아니라 연결 밀도다`

넣을 항목:

- 일반 모임 앱 대비: 운영/결제/정산/신뢰까지 포함.
- 커뮤니티 앱 대비: 실제 오프라인 이벤트와 돈 흐름까지 포함.
- 결제/정산 도구 대비: 모임 상태와 참가자 맥락을 알고 있음.
- 데이팅 앱 대비: 캘린더, 장소, 안전, 차단 정책과 연결.

시각화:

- 비교 매트릭스. 행은 핵심 역량, 열은 일반 모임 앱/커뮤니티 앱/정산 도구/community.

### Slide 13. Business Strength

제목 후보: `수익화 지점이 여러 사용자 집단과 여러 순간에 분산되어 있다`

넣을 항목:

- 참가자: 포인트/유료 이벤트/플랜 구매
- 호스트: 호스팅 티켓/정산/수익 대시보드
- 클럽: 후원/구독/기금
- 크리에이터: 플랜 판매
- 데이팅: 인증 기반 고신뢰 만남 기능 확장

시각화:

- 사용자 집단별 수익화 맵.

### Slide 14. Defensibility

제목 후보: `상태와 정책 커버리지가 쌓일수록 운영 방어력이 커진다`

넣을 항목:

- 987개 시나리오
- 권한/역할 매트릭스
- 알림/결제/캘린더/위치/리뷰 영향 매트릭스
- 예외 범주: 권한, 상태 변경, 입력 검증, 외부 실패, 중복/동시성

시각화:

- 기능 수보다 시나리오/정책 레이어가 두껍게 쌓이는 피라미드.

### Slide 15. Risk & Mitigation

제목 후보: `큰 리스크는 기능 수가 아니라 상태, 돈, 안전을 다루는 방식에 있다`

넣을 항목:

- MVP 범위 리스크: P0/P1/보류로 단계화.
- 상태/권한 리스크: 상태 전이와 역할 매트릭스로 검산.
- 결제/정산 리스크: 거래내역, 환불 정책, 이의제기, 감사로그.
- 안전/개인정보 리스크: 인증, opt-in/out, 차단, 신고, 보관/삭제 정책.
- 외부 의존성 리스크: pending, 재시도, fallback, 재조회.
- 콜드스타트 리스크: 검증된 공급자와 품질 신호부터 누적.

시각화:

- `리스크 → 대응 설계 → 사업적 의미` 3열 매트릭스.

### Slide 16. MVP Path

제목 후보: `첫 출시는 핵심 루프를 닫고, 고급 운영 기능은 단계적으로 확장한다`

넣을 항목:

- P0: 인증, 홈, 검색, 이벤트, 클럽, 지갑/결제, 정산, 플랜 마켓, 리뷰/신고, 알림.
- P1: 체크인, 사진, 위시리스트, 저장 검색, 카테고리 알림, 가용시간, 컬렉션/리뷰.
- 보류/후속: 복잡한 정산 감사, 고급 위치 프라이버시, 고도화 추천, 운영 SLA/법무 문구.

시각화:

- P0/P1/Future 3단계 로드맵.

### Slide 17. Closing

제목 후보: `community는 오프라인 활동의 전 생애주기를 제품화한다`

넣을 항목:

- 사용자는 안전하게 발견하고 참여한다.
- 호스트는 운영 부담을 줄인다.
- 클럽은 장기 관계와 기금을 운영한다.
- 크리에이터는 플랜을 상품화한다.
- 플랫폼은 결제, 정산, 신뢰, 알림 데이터를 누적한다.

시각화:

- 4개 사용자 가치와 플랫폼 자산을 연결한 마지막 요약.

## 10. PPT 제작 전 확인할 미결정 항목

| 항목 | 필요한 이유 |
|---|---|
| 발표 대상 | 투자자, 내부 임원, 제품팀, 파트너에 따라 강조점이 달라진다. |
| 실제 서비스명/브랜드 톤 | community가 임시명인지 최종명인지 확인 필요. |
| 시장/경쟁사 범위 | 특정 앱과 직접 비교할지, 일반 카테고리 비교로 갈지 결정 필요. |
| 사업 KPI | 가입자, 활성 모임, GMV, 반복 참여율 등 수치가 있으면 비즈니스 슬라이드가 강해진다. |
| 출시 범위 | MVP와 장기 비전의 경계를 명확히 해야 과장으로 보이지 않는다. |
| 실사용 화면/디자인 자산 | PPT에 앱 화면을 넣을지, PRD 기반 구조도 중심으로 갈지 결정 필요. |

## 11. 현재 초안에서 의도적으로 제외한 것

- 실제 시장 규모, TAM/SAM/SOM: PRD 안에 근거가 없으므로 별도 리서치 필요.
- 구체 매출 전망: 가격, 수수료, 전환율, 거래액 가정이 필요.
- 경쟁사 실명 비교: 최신 경쟁사 기능 확인 없이는 위험하다.
- 법무/운영 SLA: PRD에서 별도 결정 필요로 분리되어 있다.
- 최종 UX 카피: 제품/디자인 리뷰 후 확정해야 한다.
